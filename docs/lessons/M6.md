## 2026-05-08 — M6.1.a: manifest-seed migration + privilege-boundary phrase contract + authority enum reuse

**PR**: pending
**Merged**: pending

### Context

M6.1.a seeded the Watchmaster manifest (personality, language, orchestrator system
prompt) and authority_matrix entries declaring which privileged actions Watchmaster
may request, request-with-approval, or is forbidden from invoking. The manifest
needed to be canonical, idempotent, and referenceable by future code without
re-parsing migration files. The privilege boundary rule "Watchmaster MUST NEVER
execute Slack App creation directly" needed to survive LLM rewrites of the system
prompt. Authority actions reused the enum from M5.5.b.c.c.b.

### Pattern

**Manifest-seed-as-migration**: When a system role (Watchmaster, Coordinator, etc.)
needs a canonical pre-built manifest, the cleanest mechanism is a SQL seed migration
with stable UUID literals and `ON CONFLICT (id) DO NOTHING`. Idempotent, reversible
on rollback, testable. Lives at `deploy/migrations/<next-num>_<role>_manifest_seed.sql`
(note: `deploy/`, not `keep/` — the latter is a binary, not a directory). The seed
manifest's id is exported as a Go package-level constant in `core/pkg/manifest/<role>.go`
(e.g., `WatchmasterManifestID`) so future callers reference it without reparsing the
migration. Reusable for M6.2/M6.3, M7 Coordinator, M8 Reviewer, etc.

**Privilege-boundary phrase contract**: When a role's system prompt needs to encode
an architectural rule (e.g., "Watchmaster MUST NEVER execute Slack App creation
directly"), pin the exact phrase in the manifest_version row's system_prompt field
AND assert its presence in a Go test. The test acts as a regression guard against an
LLM-friendly rewrite that softens the rule.

**Authority matrix value enum reuse**: M5.5.b.c.c.b established `"allowed" |
"lead_approval" | "forbidden"` as the authority_matrix value enum. M6.1.a's
Watchmaster manifest reuses this verbatim. When a future role needs to declare
authority over an action, the values must come from this enum — no per-role bespoke
values.

### References

- Files: `deploy/migrations/*_watchmaster_manifest_seed.sql`, `core/pkg/manifest/watchmaster.go`, `core/pkg/manifest/watchmaster_test.go`
- Docs: `docs/ROADMAP-phase1.md` §M6.1.a

## 2026-05-08 — M6.1.b: privilege boundary as HTTP-RPC seam + wire-level auth rejection via .strict()

**PR**: pending
**Merged**: pending

### Context

M6.1.b implemented the privileged RPC that owns Slack App creation credentials, exposed
to Watchmaster as a tool. The core challenge: how to ensure Watchmaster (an LLM-driven
agent) executes privileged actions without ever holding credentials, and how to prevent
credential or audit-trail spoofing via untrusted wire payloads.

### Pattern

**Privilege boundary as HTTP-RPC seam**: When an untrusted agent (Watchmaster) needs
the EFFECT of a privileged action without DIRECT credential access, the cleanest
pattern is a core-owned RPC that owns credentials, validates the agent's authority
claim against the manifest's `authority_matrix`, audits pre-call and post-call to
Keeper's Log, and returns only the result. The agent calls a tool by NAME; the tool
routes through harnessrpc (bidirectional JSON-RPC seam from M5.5.d) to the core RPC;
the core executes. Agent never holds credentials; audit trail is provable. Reusable
for M6.2's `propose_spawn` (lead-approval-gated), M6.3's `adjust_personality`
(same), M7.1's spawn saga (same pattern). This pattern creates a hard privilege
boundary that survives agent rewrites.

**Wire-level agent_id rejection via `.strict()` zod**: When a tool's payload SHOULD
have `agent_id` injected from manifest (server-side trust), NOT from wire (client
untrusted), the zod schema must use `.strict()`. A wire-level `agent_id` is then
rejected as "unknown property". This blocks the audit-spoofing-via-wire-agent_id
attack. Reusable pattern for any future tool mixing server-trusted + client fields.

**Approval-token opacity as forward-compat**: M6.1.b's `approval_token` field
validates only as non-empty. M6.3's lead-approval flow will own cryptography
(token generation + signature verification). The non-empty validation is a
future-friendly seam — M6.3 can swap in real signature checks without breaking
M6.1.b's caller contract.

### References

- Files: `core/pkg/rpc/slack_app.go`, `core/rpc.go`, `cmd/harness/builtin.go`, harness RPC tests
- Docs: `docs/ROADMAP-phase1.md` §M6.1.b

## 2026-05-08 — M6.2.a: Read-only tools skip authority_matrix consultation by design

**PR**: #101
**Merged**: yes

### Context

M6.2.a implemented three read-only Watchmaster tools: `list_watchkeepers` (queries
watchkeeper table), `report_cost` (aggregates keepers_log cost rollups), and
`report_health` (returns runtime health metrics). Unlike M6.1.b's privilege boundary
(core-owned RPC enforcing authority_matrix consultation), read-only tools don't
require approval gates or manifest authority checks. The design question: should
read-only tools be gated by authority_matrix, or does membership in the toolset
suffice?

### Pattern

**Read-only tools skip authority_matrix consultation by design**: Watchmaster
toolset membership (M5.5.b.a harness ACL: manifest-id, toolset-membership gate)
is sufficient privilege control for read-only operations. Authority_matrix
consultation is reserved for write-side actions (`propose_spawn`, `retire_watchkeeper`,
`promote_to_keep`, etc.) where approval gates protect state mutations. Read-only
tools have no state-change consequences, so the lighter-weight gate (toolset
membership) is proportionate. This reflects the least-privilege principle: simpler
tools get fewer guards. Reusable for M6.2.b/c/d and future read-only agents.

**Real-fakes-end-to-end test pattern for read tools**: M6.2.a's
`TestReportCost_RealKeepersLogWriter_E2E` reuses the M6.1.b lesson (M5.5.b pattern):
real `keeperslog.Writer` instance writing cost entries into a fake `LocalKeepClient`
mock, then query-back via `LogTail` to verify aggregation. Demonstrates that
read-side E2E tests can use real writers + fake storage backends without breaking
test isolation. Scaling this pattern to M6.2.b (manifest-bump tools) with real
manifest operations against a seeded Watchmaster manifest is next.

- Files: `core/pkg/watchkeeper/tools.go`, `cmd/harness/builtin.go`, `core/pkg/keeperslog/keeperslog_test.go`
- Docs: `docs/ROADMAP-phase1.md` §M6.2.a
