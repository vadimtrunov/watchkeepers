## 2026-05-08 — M6.1.a: manifest-seed migration + privilege-boundary phrase contract + authority enum reuse

**PR**: pending
**Merged**: pending

### Context

M6.1.a seeded the Watchmaster manifest (personality, language, orchestrator system
prompt) and authority_matrix entries declaring which privileged actions Watchmaster
may request, request-with-approval, or is forbidden from invoking. The manifest
needed to be canonical, idempotent, and referenceable by future code without
re-parsing migration files. The privilege boundary rule "Watchmaster MUST NEVER
execute Slack App creation directly" needed to survive LLM rewrites of the system
prompt. Authority actions reused the enum from M5.5.b.c.c.b.

### Pattern

**Manifest-seed-as-migration**: When a system role (Watchmaster, Coordinator, etc.)
needs a canonical pre-built manifest, the cleanest mechanism is a SQL seed migration
with stable UUID literals and `ON CONFLICT (id) DO NOTHING`. Idempotent, reversible
on rollback, testable. Lives at `deploy/migrations/<next-num>_<role>_manifest_seed.sql`
(note: `deploy/`, not `keep/` — the latter is a binary, not a directory). The seed
manifest's id is exported as a Go package-level constant in `core/pkg/manifest/<role>.go`
(e.g., `WatchmasterManifestID`) so future callers reference it without reparsing the
migration. Reusable for M6.2/M6.3, M7 Coordinator, M8 Reviewer, etc.

**Privilege-boundary phrase contract**: When a role's system prompt needs to encode
an architectural rule (e.g., "Watchmaster MUST NEVER execute Slack App creation
directly"), pin the exact phrase in the manifest_version row's system_prompt field
AND assert its presence in a Go test. The test acts as a regression guard against an
LLM-friendly rewrite that softens the rule.

**Authority matrix value enum reuse**: M5.5.b.c.c.b established `"allowed" |
"lead_approval" | "forbidden"` as the authority_matrix value enum. M6.1.a's
Watchmaster manifest reuses this verbatim. When a future role needs to declare
authority over an action, the values must come from this enum — no per-role bespoke
values.

### References

- Files: `deploy/migrations/*_watchmaster_manifest_seed.sql`, `core/pkg/manifest/watchmaster.go`, `core/pkg/manifest/watchmaster_test.go`
- Docs: `docs/ROADMAP-phase1.md` §M6.1.a

## 2026-05-08 — M6.1.b: privilege boundary as HTTP-RPC seam + wire-level auth rejection via .strict()

**PR**: pending
**Merged**: pending

### Context

M6.1.b implemented the privileged RPC that owns Slack App creation credentials, exposed
to Watchmaster as a tool. The core challenge: how to ensure Watchmaster (an LLM-driven
agent) executes privileged actions without ever holding credentials, and how to prevent
credential or audit-trail spoofing via untrusted wire payloads.

### Pattern

**Privilege boundary as HTTP-RPC seam**: When an untrusted agent (Watchmaster) needs
the EFFECT of a privileged action without DIRECT credential access, the cleanest
pattern is a core-owned RPC that owns credentials, validates the agent's authority
claim against the manifest's `authority_matrix`, audits pre-call and post-call to
Keeper's Log, and returns only the result. The agent calls a tool by NAME; the tool
routes through harnessrpc (bidirectional JSON-RPC seam from M5.5.d) to the core RPC;
the core executes. Agent never holds credentials; audit trail is provable. Reusable
for M6.2's `propose_spawn` (lead-approval-gated), M6.3's `adjust_personality`
(same), M7.1's spawn saga (same pattern). This pattern creates a hard privilege
boundary that survives agent rewrites.

**Wire-level agent_id rejection via `.strict()` zod**: When a tool's payload SHOULD
have `agent_id` injected from manifest (server-side trust), NOT from wire (client
untrusted), the zod schema must use `.strict()`. A wire-level `agent_id` is then
rejected as "unknown property". This blocks the audit-spoofing-via-wire-agent_id
attack. Reusable pattern for any future tool mixing server-trusted + client fields.

**Approval-token opacity as forward-compat**: M6.1.b's `approval_token` field
validates only as non-empty. M6.3's lead-approval flow will own cryptography
(token generation + signature verification). The non-empty validation is a
future-friendly seam — M6.3 can swap in real signature checks without breaking
M6.1.b's caller contract.

### References

- Files: `core/pkg/rpc/slack_app.go`, `core/rpc.go`, `cmd/harness/builtin.go`, harness RPC tests
- Docs: `docs/ROADMAP-phase1.md` §M6.1.b

## 2026-05-08 — M6.2.a: Read-only tools skip authority_matrix consultation by design

**PR**: #101
**Merged**: yes

### Context

M6.2.a implemented three read-only Watchmaster tools: `list_watchkeepers` (queries
watchkeeper table), `report_cost` (aggregates keepers_log cost rollups), and
`report_health` (returns runtime health metrics). Unlike M6.1.b's privilege boundary
(core-owned RPC enforcing authority_matrix consultation), read-only tools don't
require approval gates or manifest authority checks. The design question: should
read-only tools be gated by authority_matrix, or does membership in the toolset
suffice?

### Pattern

**Read-only tools skip authority_matrix consultation by design**: Watchmaster
toolset membership (M5.5.b.a harness ACL: manifest-id, toolset-membership gate)
is sufficient privilege control for read-only operations. Authority_matrix
consultation is reserved for write-side actions (`propose_spawn`, `retire_watchkeeper`,
`promote_to_keep`, etc.) where approval gates protect state mutations. Read-only
tools have no state-change consequences, so the lighter-weight gate (toolset
membership) is proportionate. This reflects the least-privilege principle: simpler
tools get fewer guards. Reusable for M6.2.b/c/d and future read-only agents.

**Real-fakes-end-to-end test pattern for read tools**: M6.2.a's
`TestReportCost_RealKeepersLogWriter_E2E` reuses the M6.1.b lesson (M5.5.b pattern):
real `keeperslog.Writer` instance writing cost entries into a fake `LocalKeepClient`
mock, then query-back via `LogTail` to verify aggregation. Demonstrates that
read-side E2E tests can use real writers + fake storage backends without breaking
test isolation. Scaling this pattern to M6.2.b (manifest-bump tools) with real
manifest operations against a seeded Watchmaster manifest is next.

### References

- Files: `core/pkg/watchkeeper/tools.go`, `cmd/harness/builtin.go`, `core/pkg/keeperslog/keeperslog_test.go`
- Docs: `docs/ROADMAP-phase1.md` §M6.2.a

## 2026-05-08 — M6.2.b: Manifest-bump pattern symmetric to slack_app_create + GetManifest pre-emit short-circuit + interface seam

**PR**: pending
**Merged**: pending

### Context

M6.2.b implemented three manifest-bump write tools (`propose_spawn`, `adjust_personality`,
`adjust_language`) that share M6.1.b's privilege-RPC seam pattern but diverge in a key
detail: all three tools read a prerequisite document (the current Manifest version) BEFORE
emitting an audit row. This read-before-write order creates a subtle correctness issue:
if the read fails, the tool should abort before any audit side-effect, ensuring no orphaned
audit rows for partial-state events.

### Pattern

**Manifest-bump pattern symmetric to slack_app_create**: M6.1.b's `slack_app_create` RPC
follows the pattern: auth check → token lookup → audit-requested (logged) → privileged
action → audit-succeeded/failed. M6.2.b's manifest-bump tools mirror this gate stack
exactly: auth check (authority_matrix) → read current manifest → audit-requested → write
new manifest_version → audit-succeeded. The symmetry is intentional — any future
Watchmaster write tool follows the same three-RPC pattern (request gate, execute gate,
result gate). Reusable for M6.2.c/d and all future Watchmaster write tools.

**GetManifest pre-emit short-circuit**: When a write tool reads a prerequisite document
before the audit row, a read failure must abort BEFORE emit. M6.2.b implements this as:
`GetManifestVersion()` is called first; if it returns an error, the RPC returns early
with that error, and no audit-requested row is written. This protects against partial-state
events where the audit trail claims an action was attempted but no manifest change occurred.
Reusable for any future read-then-write tool (e.g., promote_to_keep reads notebook before
writing Keep).

**Interface seam parallel between read and write tools**: M6.2.a introduced
`WatchmasterReadClient` (interface for list_watchkeepers, report_cost, report_health).
M6.2.b introduces `WatchmasterWriteClient` in parallel — explicit interface for
propose_spawn, adjust_personality, adjust_language, with compile-time assertion
`*keepclient.Client implements WatchmasterWriteClient`. Mirrors the M6.1.b pattern
(HTTP RPC owns credentials, agent calls tool, harnessrpc routes). The parallel interfaces
make toolset membership testable + enforce contract at compile time. Reusable template
for future read-only and write-only agent roles.

### References

- Files: `core/pkg/rpc/manifest.go`, `cmd/harness/builtin.go`, `core/pkg/watchkeeper/tools.go`, harness RPC tests
- Docs: `docs/ROADMAP-phase1.md` §M6.2.b

## 2026-05-08 — M6.2.c: retire_watchkeeper (status-flip write tool)

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

M6.2.c added a lifecycle-status tool (`retire_watchkeeper`) that flips a Watchkeeper's
active state to retired via `UpdateWatchkeeperStatus(ctx, id, "retired")`. Deliberately
separated from M6.2.b's manifest-bump pattern to keep PR scope tight. No pre-read required;
the Keep server enforces state-machine rules (active→retired only), rejecting invalid
transitions with typed errors that the audit trail captures.

### Pattern

**No pre-read for status-flip writes**: Unlike M6.2.b's read-before-write manifest bumps,
status-flip tools follow ProposeSpawn's write-only shape. `UpdateWatchkeeperStatus` is
called directly; if the server rejects the transition (e.g., already retired), the typed
error is wrapped and captured in an audit `failed` row, avoiding an orphaned request-only
entry. Reusable for any future status-mutation RPC where the server enforces invariants.

**Audit-event helpers per tool family**: M6.2.c defines local helpers (`retireRequestedEvent`,
`retireSucceededEvent`, `retireFailedEvent`) rather than reusing M6.2.b's manifest-bump
family. Pattern: each tool family with semantically distinct payload shapes (manifest-bump
carries `manifest_version_id`; retire carries nothing) gets its own trio. ~12 LOC duplication
is acceptable to keep audit payloads cleanly separable and prevent future divergence.

**Empty success envelope**: For tools with no return value, prefer a typed empty struct
over `null`. TS callers don't need null guards; Go callers get real types. Cheaper than
inheritance and more explicit.

- Files: `core/pkg/spawn/retire_watchkeeper.go`, `core/pkg/harnessrpc/retire_watchkeeper.go`, `harness/src/builtinTools.ts`

## 2026-05-08 — M6.2.d: promote_to_keep (read-then-write across stores)

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

M6.2.d added a lead-approval-gated tool (`promote_to_keep`) that reads a notebook entry
and writes it as a Keep chunk. Unlike M6.2.b's manifest-bump tools that read the same
store they write, promote_to_keep bridges two packages (notebook → keep) with a
read-then-write semantic: read must succeed before the audit row is written; read failure
aborts immediately (pre-emit short-circuit). The domain event `notebook_promoted_to_keep`
appears in the audit chain between the security-audit triplet (`requested` → `succeeded`).

### Pattern

**Cross-store read-then-write generalizes M6.2.b pattern**: The M6.2.b lesson taught
GetManifest pre-emit short-circuit (read failure aborts before audit-requested row). That
pattern is NOT manifest-specific: promote_to_keep reuses it with a different read source
(notebook.DB) and write sink (Keep). The invariant holds: read fails → return early,
zero audit rows; read succeeds → audit-requested → write → audit-succeeded/failed.
Reusable for any future Watchmaster tool crossing package boundaries.

**Two interface seams when crossing package boundaries**: Cross-package tools need
TWO separate interfaces, NOT one combined. promote_to_keep uses `WatchmasterWriteClient`
(for Keep's Store call) + new `WatchmasterNotebookClient` (for notebook's PromoteToKeep
call). Avoids tangling unrelated package contracts. Reusable template for future tools
touching multiple stores: each store gets its own seam with compile-time assertions.

**Three-event happy path for tools with named domain events**: Most Watchmaster write
tools emit 2 events (requested + succeeded). When the ROADMAP names a specific domain
event (notebook_promoted_to_keep), the audit chain becomes 3: `tool_requested` (security)
→ `notebook_promoted_to_keep` (domain, downstream subscribers) → `tool_succeeded` (security).
Gives downstream consumers a clean filter while keeping security audit chain intact.
Reusable when future ROADMAP items name domain events.

**Application-range RPC sentinel for new domain errors**: Adding `-32011 ToolNotFound`
to harnessrpc when notebook.ErrNotFound needs distinct classification (not the same as
ToolUnauthorized or ApprovalRequired). Allocate from application range (-32000 to -32099)
rather than reusing existing sentinel. TS callers branch on raw wire `code` via
RpcRequestError; TS enum entry can be added later (M6.3 wire vocab work).

## 2026-05-08 — M6.3.a: Slack inbound webhook scaffolding pattern (signature verify + dispatcher skeleton)

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

M6.3.a added inbound Slack webhook scaffolding: an HTTP handler factory that accepts
`POST /v1/slack/events` (Events API, DM messages) and `POST /v1/slack/interactions`
(Interactivity API, button clicks). The handler verifies request signatures (HMAC-SHA256
over raw body with 5-minute timestamp window), dispatches to skeleton interfaces, and
audits. No business logic — M6.3.b–d own event interpretation.

### Pattern

**Inbound-webhook-scaffold-pattern reusable template**: The `core/pkg/messenger/slack/inbound/`
package exposes `http.Handler` factory taking signing-secret + dispatcher hooks (interfaces).
Pattern: verify signature → re-supply raw body via bytes.NewReader → dispatch by type → ACK 200.
Dispatcher interfaces are skeleton no-ops; concrete handlers live in M6.3.b–d. Template
is reusable for multi-workspace OAuth callbacks, Slash Commands, or future Slack integrations
without duplicating signature/body-read logic.

**Read raw body once, re-supply via bytes.NewReader (timing attack guard)**: Go's
`http.Request.Body` is read-once. For HMAC-over-raw-body, read into `[]byte`, compute
HMAC, then unmarshal via `json.Unmarshal(rawBytes, &envelope)`. Silent trap: calling
`json.NewDecoder(r.Body)` AFTER `io.ReadAll` reads from an empty body (decoder sees EOF,
returns zero-value). Pattern: store raw bytes, validate HMAC with constant-time `hmac.Equal`
(NOT `==`/`bytes.Equal`), then `json.Unmarshal`.

**Body-size cap before HMAC compute (DoS guard)**: Wrap body with `http.MaxBytesReader`
BEFORE `io.ReadAll` to bound HMAC compute time. Otherwise attacker submits 1 GB body,
forcing HMAC over entire payload before rejection. Cap is cheap (413 response), prevents
CPU exhaustion.

**Constant-time signature compare via hmac.Equal**: Mandatory regression guard. AC13
pins `hmac.Equal` usage in tests to block timing-attack regressions via `==` or
`bytes.Equal`. Pattern: `hmac.Equal(computed, headerValue)` always, no shortcuts.

### References

- Files: `core/pkg/messenger/slack/inbound/` (doc.go, verify.go, handler.go, dispatcher.go, \*\_test.go)
- Docs: `docs/ROADMAP-phase1.md` §M6.3.a

## 2026-05-08 — M6.3.b: Pending-approval DAO pattern + cross-package replayer seam + token-prefix truncation

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

M6.3.b implemented the manifest-draft approval-card renderer for four tools (propose_spawn,
adjust_personality, adjust_language, retire_watchkeeper) and the approval-decision handler that
decodes Slack button payloads, resolves the DAO, and optionally re-invokes the tool. The
dispatcher must NOT render raw approval_token into operator-facing Slack cards; must NOT
conflate malformed action_id with malformed button-value errors; must NOT leak param-struct
field names into audit payloads.

### Pattern

**Pending-approval DAO pattern (in-memory + Postgres-deferred)**: M6.3.b introduces a typed
DAO interface with in-memory impl in spawn package; Postgres-backed adapter deferred to M6.3.c.
Pattern: ship in-memory DAO + tests in the same PR as the consumer (here the approval
dispatcher), defer Postgres adapter + migrations to a follow-up PR if the adapter pulls in new
packages. Keeps PR scope tight; DAO contract is locked by consumer tests.

**Cross-package replayer interface seam**: M6.3.b's approval dispatcher calls a `Replayer`
interface for M6.2.x tool re-invocation, NOT direct spawn-package calls. Pattern: when a handler
needs to invoke a peer Watchmaster tool, define a narrow per-handler interface (1-method seam)
so the handler can be tested with a fake without pulling spawn-package state into test fixtures.
Production wiring composes the seam later in the runtime entry-point (M6.3.c).

**action_id format for Slack button payloads**: `approval:<tool>:<token>` with strict 3-part
SplitN. Reasons: (a) tool name in the action_id pins the dispatcher's tool-specific routing
without parsing the whole payload; (b) the closed-set of known tool names lets DecodeActionID
reject malformed strings cheaply; (c) button-value branch is a SEPARATE failure mode from
action_id-shape — needs separate sentinel `ErrInvalidButtonValue`.

**Token-prefix display for Slack cards**: Operator-facing approval cards must NOT render the
raw approval_token. Render `tok-<first-6-chars>…` for operator correlation; the full token
round-trips inside action_id (opaque to the human reader). Pattern: any Slack card body
containing a security-bearing token should truncate; full token belongs only in action payloads
the dispatcher decodes server-side.

## 2026-05-08 — M6.3.c: DM intent router pattern (closed-set classifier + EventDispatcher seam) + render-before-persist invariant + loud token-mint failure

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

M6.3.c wired admin Slack DM messages end-to-end through Watchmaster, parsing inbound intents (read-only queries, manifest-bump proposals) via a deterministic keyword/regex classifier, dispatching to M6.2 tools, rendering approval cards or read results, and posting outbound DMs. The dispatcher had to enforce three correctness invariants: (1) DM-event PII discipline (no message text in audit), (2) pending-approval DAO rows created only AFTER successful card render (orphan-row prevention), (3) silent token-mint fallback must never occur (noisy error propagation required).

### Pattern

**DM intent router pattern (closed-set classifier + EventDispatcher seam)**: M6.3.c established the inbound DM flow: `inbound.EventDispatcher` impl filters non-DM/bot/empty events early (zero audits), parses intent via a closed-set keyword/regex classifier (NOT NL), branches on intent class to call typed seams (`ReadToolDispatcher`, `Proposer`, `Outbound`). Pattern: keep the parser deterministic and limited to a closed set of intents; off-topic phrasings fall to `IntentUnknown` with a help-text response. Reusable for any future Slack DM-driven agent surface.

**Render-before-persist invariant for approval flows**: M6.3.c established that DAO inserts of pending-approval rows must happen AFTER successful card render, NOT before. Inverting the order means a render failure orphans the row (state=pending forever, no card to act on). Pattern: `mint token → render card → on success persist DAO → post outbound; on render failure no DAO row, apology DM`. The invariant `pending row exists ⇒ user has an approval card` is preserved.

**Loud failure for token-mint regressions**: `uuid.NewV7()` has no realistic failure mode in practice, but a silent literal-string fallback (`"tok-error"`) on rare failure converts a regression into a unique-constraint collision masking the root cause. Pattern: token-mint helpers return `(string, error)` — propagate the error to a dedicated audit reason (`token_mint_error`) + apology DM. Never substitute a deterministic literal that two failures could collide on.

**Outbound seam adapter pattern**: M6.3.c introduced a typed `Outbound` interface + thin `SlackOutbound` adapter wrapping `*slack.Client.SendMessage`. The adapter currently posts plain-text fallback only because `messenger.Message` lacks a portable `Blocks` field; renderers also produce text fallback. Pattern: when a Slack send surface needs to evolve (Blocks, attachments, etc.), introduce an interface seam BEFORE the messenger struct grows, so test fakes never touch HTTP. The seam stays stable across the messenger evolution.

- Files: `core/pkg/messenger/slack/inbound/intent/`, `core/pkg/messenger/slack/inbound/dm/`, `core/pkg/messenger/slack/inbound/outbound/`
- Docs: `docs/ROADMAP-phase1.md` §M6.3.c
