## 2026-05-11 ‚Äî M8.2.d: GitHub PR tool (`find_stale_prs`) + new `core/pkg/github` adapter + V4 manifest seed

**PR**: #131
**Merged**: 92e43ed on 2026-05-11

### Context

M8.2.d ships the sixth and final Coordinator tool the M8.2 roadmap
calls out ‚Äî `find_stale_prs` ‚Äî together with the **new** stdlib-only
GitHub REST API adapter the handler consumes. M8.2.d's magnitude is
comparable to M8.1 because of the adapter; the prior three M8.2
sub-items (a/b/c) all rode on existing M8.1 / M4.2 adapters.

`core/pkg/github` is a sibling to `core/pkg/jira` and
`core/pkg/messenger/slack`:

- **`Client.ListPullRequests(ctx, owner, repo, opts)`** ‚Äî `GET
/repos/{owner}/{repo}/pulls` with GitHub's documented
  page-number pagination via the `Link` response header.
- **`TokenSource`** per-call bearer resolver (mirrors
  `slack.TokenSource`; `jira.BasicAuthSource` shape). `WithTokenSource`
  is REQUIRED at [NewClient] time ‚Äî fail-closed default.
- **Default base URL** is `https://api.github.com`; [WithBaseURL]
  overrides for GitHub Enterprise Server. Unlike jira (which rejects
  path prefixes), GHES REQUIRES `/api/v3` ‚Äî path prefixes are
  accepted.
- **Rate-limit handling** ‚Äî both HTTP 429 AND the documented `403 +
X-RateLimit-Remaining: 0` shape map to `ErrRateLimited`; `RetryAfter`
  is computed from `X-RateLimit-Reset` (Unix-epoch seconds, NOT
  Retry-After).

The `find_stale_prs` handler (`core/pkg/coordinator/find_stale_prs.go`)
drives page-1 ... page-N pagination via the adapter, filters to PRs
where (a) `reviewer_login` is in `RequestedReviewers` (case-
insensitive) AND (b) `UpdatedAt` is non-zero AND strictly before
`now - age_threshold_days`. Output drops author logins and reviewer
echoes per the M8.2.c lesson #10 PII discipline.

The V4 seed lands as a NEW migration `027_coordinator_manifest_v4_seed.sql`
‚Äî same append-only `manifest_version` evolution pattern as M8.2.b/c
(NEW stable id `24000000-‚Ä¶`, `version_no=4`, V1/V2/V3 rows preserved
verbatim). `CoordinatorManifestVersionV4ID` joins the Go constants in
`core/pkg/manifest/coordinator.go`; the migration-shape test family
extends from three tests (V1+V2+V3) to four (V1+V2+V3+V4).

### Pattern

1. **New `core/pkg/github` adapter mirrors `core/pkg/jira` shape
   verbatim ‚Äî five files (`doc.go`, `client.go`, `options.go`,
   `errors.go`, `types.go`) + one per-operation file
   (`list_pull_requests.go`)** ‚Äî
   `core/pkg/github/{doc.go, client.go, options.go, errors.go,
types.go, list_pull_requests.go}` mirror the jira adapter layout
   1:1: package doc-block enumerating sentinels + design choices;
   `Client` + `clientConfig` + functional `ClientOption`s
   (`WithBaseURL` / `WithTokenSource` / `WithHTTPClient` /
   `WithLogger` / `WithClock`); `doParams` + `do` transport carrying
   auth + JSON + redaction discipline; sentinel errors backed by an
   `APIError` with `endpointKind` Unwrap; wire structs decoded by
   per-shape `toX` projectors. Mirroring saves the reader's
   onboarding cost ‚Äî fluency in jira transfers to github verbatim.
   For future M9.x adapters (Linear, Notion), open the jira / github
   pair side-by-side and copy the structure.

2. **Default base URL + accept path prefixes ‚Äî divergence from jira
   driven by GHES** ‚Äî `core/pkg/github/client.go::defaultBaseURL =
"https://api.github.com"`; [WithBaseURL] overrides without
   path-prefix rejection. Atlassian Cloud has one tenant per
   subdomain so requiring [WithBaseURL] is the right discipline;
   GitHub has a single canonical host so requiring it would be hostile
   to the common case. GHES uses `/api/v3` path prefix, which jira
   would reject as a Confluence sub-product mishandling ‚Äî for github
   the prefix is REQUIRED by the deployment, so the adapter accepts
   any non-empty path. The fail-closed concern is `TokenSource`, not
   base URL ‚Äî auth is the actual security boundary. The two
   divergences are documented in `client.go::WithBaseURL`'s doc-block
   so a future reader sees the rationale without spelunking.

3. **`TokenSource` bearer-only ‚Äî GitHub auth surface is narrower than
   jira's** ‚Äî
   `core/pkg/github/client.go::TokenSource` returns a single token
   (no email companion). GitHub's documented auth scheme is
   `Authorization: Bearer <token>` for both PATs and GitHub App
   installation tokens; basic-auth is legacy / unsupported. Sibling
   `jira.BasicAuthSource` returns `(email, token)` because Atlassian
   uses HTTP Basic with the email as the username. The adapter
   surface tracks the actual scheme rather than forcing a unified
   shape ‚Äî readers see "bearer vs basic" and immediately know which
   is which. M9 adapters with token+secret or signing-key shapes get
   their own resolver type; do not retrofit a single godlike
   `AuthSource` interface across adapters.

4. **Rate-limit handling: BOTH 429 AND 403-with-remaining=0, with
   `X-RateLimit-Reset` as Unix-epoch seconds** ‚Äî
   `core/pkg/github/errors.go::APIError.Unwrap` maps both 429 AND a
   403 carrying `X-RateLimit-Remaining: 0` to [ErrRateLimited]. A
   plain 403 (token lacks the scope) does NOT alias rate-limit ‚Äî
   callers may need to surface "scope missing" distinctly. The
   `RetryAfter` field is computed from `X-RateLimit-Reset` (Unix-epoch
   seconds, NOT Retry-After / HTTP-date) via
   `parseResetHeader(raw, clock)`. Jira surfaces 429 + Retry-After
   (HTTP-date OR delta-seconds, RFC 9110 ¬ß10.2.3); github surfaces
   429/403 + X-RateLimit-Reset (Unix epoch). The two adapters
   correctly model the platform-specific shapes ‚Äî a uniform "wait
   time" API would lie about which time source is authoritative.
   Pinned via `TestDo_429ReturnsErrRateLimited` +
   `TestDo_403WithZeroRemaining_AliasesRateLimited` +
   `TestDo_403WithNonZeroRemaining_DoesNotAliasRateLimited`.

5. **Link-header pagination via anchored regex; rel="next" must
   appear explicitly** ‚Äî
   `core/pkg/github/client.go::linkNextPagePattern` =
   `<[^>]*[?&]page=(\d+)[^>]*>;\s*rel="next"`. The regex is anchored
   on the `rel="next"` marker so a header advertising only
   rel="prev" / rel="first" / rel="last" returns 0
   (terminal-page marker). Tests cover (a) empty header, (b) only
   first/last (no next ‚Üí 0), (c) next first in the list, (d) next
   later in the list, (e) page= mixed with other query params, (f)
   rel-next URL with no page= param. Mirrors the M8.2.c lesson #4
   "discriminant patterns" generalisation: an identifier regex MUST
   require the load-bearing marker, not just permit the character
   class. For GitHub Link, the `rel="next"` marker IS the
   discriminant.

6. **Consumer-owned single-method interface `GitHubPRLister` +
   compile-time satisfaction assertion** ‚Äî
   `core/pkg/coordinator/find_stale_prs.go` declares
   `GitHubPRLister` (single method mirroring `github.Client.ListPullRequests`
   exactly) at the consumer, not at the producer, per the M8.2.a/b/c
   "interfaces belong to the consumer" precedent. `var _
GitHubPRLister = (*github.Client)(nil)` locks the contract at
   build time ‚Äî an M8.2.d signature drift fails build, not
   production. Tests inject hand-rolled `fakeGitHubPRLister` without
   touching the HTTP client.

7. **`find_stale_prs` handler shape mirrors M8.2.b `find_overdue_tickets`
   verbatim (clock injection, per-arg readers, pagination cap matrix,
   PII discipline, source-grep AC)** ‚Äî
   `core/pkg/coordinator/find_stale_prs.go` carries the exact same
   handler skeleton as `find_overdue_tickets.go`:
   `newFindStalePRsHandlerWithClock(lister, clock)` test factory ‚Üí
   public `NewFindStalePRsHandler(lister)` wrapping with
   `defaultStalePRsNow = time.Now`; per-arg `readRepoOwnerArg` /
   `readRepoNameArg` / `readReviewerLoginArg` /
   `readStalePRsAgeDaysArg` readers each refusing without echo;
   `paginateStalePRs` loop with `maxStalePRs=200` /
   `maxStalePages=10` caps and the M8.2.b/c symmetric defence on
   every pagination signal (per-PR cap inside a page surfaces
   `truncated=true` whenever ANY of: more PRs in page OR more pages
   exist; page cap fall-through unconditionally `truncated=true`);
   `projectStalePRs` mapping `github.PullRequest` to the agent-facing
   wire shape dropping author + reviewer logins; source-grep AC tests
   `TestFindStalePRsHandler_ValidationPrecedesListCall_SourceGrep` +
   `TestFindStalePRsHandler_NoAuditOrLogAppend_SourceGrep` matching
   the M8.1 lesson #7 + M8.2.b lesson #6 family. Reusable across all
   future M9+ "fetch from external read-only API and project" tools.

8. **Case-insensitive reviewer match (`strings.EqualFold`) ‚Äî GitHub
   logins are displayed case-insensitively but the API surfaces the
   stored case** ‚Äî
   `core/pkg/coordinator/find_stale_prs.go::matchesStaleFilter`
   compares the supplied `reviewer_login` against every entry in
   `pr.RequestedReviewers` via `strings.EqualFold`. GitHub's display
   convention is case-insensitive (the login `Alice` and `alice`
   route to the same account), but the REST API surfaces whatever
   case is stored. An agent passing `"alice"` finds a reviewer
   recorded as `"Alice"` and vice versa. Pinned via
   `TestFindStalePRsHandler_ReviewerMatchIsCaseInsensitive`. Mirrors
   M4.2 slack lookup discipline (case-folding email-to-user matching
   on the slack lookup_user surface).

9. **Zero `UpdatedAt` is EXCLUDED from stale, not treated as
   infinitely old** ‚Äî
   `core/pkg/coordinator/find_stale_prs.go::matchesStaleFilter`
   short-circuits `pr.UpdatedAt.IsZero()` ‚Üí `return false`. A parse
   failure on the GitHub timestamp produces a zero `time.Time`; if
   the filter treated zero as "infinitely old", a parse-failure PR
   would silently surface as stale and the agent would dispatch a
   false-positive nudge. Pinned via
   `TestFindStalePRsHandler_ZeroUpdatedAtDropped`. Mirrors the M8.2.b
   "guard against zero updated" discipline.

10. **PII reach in success output applies to GitHub logins too ‚Äî drop
    author + reviewer echoes; only PR number + title + html_url +
    updated_at + age_days survive** ‚Äî
    `core/pkg/coordinator/find_stale_prs.go::projectStalePRs` emits
    `{number, title, html_url, updated_at, age_days}` per PR. Author
    login (`pr.AuthorLogin`) is NOT echoed; reviewer logins
    (`pr.RequestedReviewers`) are NOT echoed; scope drops
    `reviewer_login` for the same reason. Mirrors the M8.2.c iter-1
    codex Major #3 PII generalisation: walking EVERY nested level of
    the output for identifier echoes, not just the root. The agent
    has every input in its call args; the success path is a PII reach
    surface and the M5.6.b reflector ingests result Output verbatim.
    Pinned via `TestFindStalePRsHandler_HappyPath` (asserts
    `prs[0]["author_login"]` / `["user"]` / `["requested_reviewers"]`
    - scope `["reviewer_login"]` are all absent).

### Iter-1 review patterns (codex + critic)

The parallel iter-1 review (codex via `omc ask` + `oh-my-claudecode:critic`
agent) surfaced **4 Major + 3 Minor** findings that landed on the same
branch before commit. Each pattern is reusable for M9+ work.

a. **`repoPattern` admits `.`/`..`/leading-`.`/leading-`-` ‚Äî
path-traversal gate must restrict the FIRST char, not just the
character class (iter-1 critic Major #1)** ‚Äî
`core/pkg/github/client.go::repoPattern` originally was
`^[A-Za-z0-9_.\-]{1,100}$`. The character class admits `.`, `..`,
`.foo`, `-foo`, `..foo` as legal repo names. Composing
`ListPullRequests(ctx, "octo", "..", ‚Ä¶)` produces a request to
`/repos/octo/../pulls`; `net/http` does NOT normalise on the client,
but server fronts (GHES nginx, fronting proxies) MAY normalise
before authentication and dispatch the call elsewhere. The fix:
require the leading character be alphanumeric or `_`:
`^[A-Za-z0-9_][A-Za-z0-9_.\-]{0,99}$`. The handler-mirror regex
(`core/pkg/coordinator/find_stale_prs.go::stalePRsRepoPattern`)
needed the same tightening. Lesson generalisation (extends M8.2.b
lesson #9 "character-class is necessary but not sufficient"): for
identifiers that interpolate into a URL path, the regex MUST also
restrict the LEADING-CHAR class, not just the body class. Path-
traversal canaries (`.`, `..`, `...`, `.foo`, `-foo`) belong in the
shape-AC table for every URL-path-interpolating handler arg.

b. **Pagination cursor fails open on Link-header drift (iter-1 codex
Major #1)** ‚Äî `core/pkg/github/client.go::parseLinkHeaderNextPage`
originally returned `0` for any unparseable `rel="next"` entry,
and `ListPullRequests` treated `0` as a clean terminal page. Any
GHES / proxy / format drift (a `rel="next"` URL without a parseable
`page=`, a malformed `Link` header, a non-anchored relation type)
silently truncated the scan after the current page. M8.1's
analogous guard (`isLast=false && cursor==""` returns
`ErrInvalidArgs`) explicitly rejects the symmetric server-contract
violation; M8.2.d's first iteration regressed against it. Fix:
disambiguate "no `rel="next"` marker" (legitimate terminal page ‚Üí
`(0, nil)`) from "marker present but URL unparseable" (drift ‚Üí
`(0, ErrInvalidArgs)`). `parseLinkHeaderNextPage` returns
`(int, error)`; `ListPullRequests` surfaces the error from the
`headerFn`. Pinned via
`TestListPullRequests_LinkHeaderRelNextWithoutPage_SurfacesError`.
Lesson: any pagination cursor whose terminus is implicit (absence
of a "next" marker) MUST disambiguate "absent" from "present but
unparseable" ‚Äî the latter is fail-loud, not fail-open.

c. **Case-insensitive Link relation type per RFC 8288 (iter-1 critic
Major #2)** ‚Äî `core/pkg/github/client.go::linkNextPagePattern`
originally used a case-sensitive `rel="next"` match. RFC 8288 ¬ß3
defines relation type values as case-insensitive; GitHub.com
always emits lowercase but GHES builds / load-balancer rewrites /
future RFC-strict changes may capitalise. `rel="Next"` silently
returned 0 ‚Üí pagination terminated after page 1 ‚Üí data-loss
regression. Fix: prepend `(?i)` to the regex. Pinned via
`TestParseLinkHeaderNextPage_CaseInsensitive*` test cases. Lesson
generalisation (extends M8.2.c lesson #4 "discriminant patterns"):
when a regex anchors on an external-spec marker, treat every
spec-permitted casing as load-bearing ‚Äî case sensitivity is a
silent data-loss axis unless the spec explicitly forbids
case-variance.

d. **Unstable cursor: `updated asc` page-number pagination admits
duplicates/skips under concurrent activity (iter-1 codex Major #2)** ‚Äî
`core/pkg/coordinator/find_stale_prs.go::paginateStalePRs` uses
GitHub's page-number pagination with sort key `updated` (which IS
mutable). A PR's `updated` shifting between page fetches can cause
duplicates (PR appears on two adjacent pages) or skips (PR falls
off the prior page before the scan reaches the next). The handler
accepts the tradeoff because the scan runs at human-scale
frequency and the agent re-plans on `truncated=true`; the doc-block
on `paginateStalePRs` now spells this out explicitly (a future M9
stable-cursor migration would require GitHub GraphQL). Lesson:
when a paginated REST API offers no stable cursor, the consumer
MUST document the duplicate/skip semantics in the function
doc-block ‚Äî silent acceptance breeds debugging confusion later
when the M5.6.b reflector ingests an obvious-to-the-reader-but-
nonobvious-to-the-debugger duplicate row.

e. **Adapter Options must reject negative values, not silently coerce
(iter-1 codex Minor #1)** ‚Äî `ListPullRequestsOptions.Page` and
`.PerPage` originally accepted any int; values `<= 0` were just
omitted from the query (treated as "use the GitHub default").
This is a sharp edge on the new adapter API surface ‚Äî a
downstream consumer passing `-1` to "skip pagination" gets a
silent first-page response instead of an error. Fix:
`ListPullRequests` rejects `Page < 0` and `PerPage < 0` with
wrapped `ErrInvalidArgs`. Lesson: adapter Option struct fields
that interpolate into a request body or query string MUST reject
typed zero / negative values explicitly when the wire contract
expects positive integers. Silent coercion is a polite-default
that downstream consumers will reach for as a defensive default
and then debug for half an hour.

f. **Dead-code helper removed for jira parity (iter-1 critic Minor 3)** ‚Äî
`core/pkg/github/client.go::errMustNotBeEmpty` was defined
but never called (`validateOwner` / `validateRepo` inlined their
own emptiness checks). The jira sibling uses the equivalent
helper consistently. Either delete OR wire it through; M8.2.d
deletes since the inline checks are equally clear. Lesson: when
mirroring a sibling adapter's layout, audit unused helpers
specifically ‚Äî they signal a refactor that didn't complete, and
the M9 reviewer mirroring this package will replicate the dead
code if it survives.

g. **Test gaps surfaced by iter-1 critic** ‚Äî
`TestListPullRequests_GHESPathPrefixComposedCorrectly` (confirms
`/api/v3` base-URL path prefix composes correctly per the GHES
contract, not stripped or doubled),
`TestDo_403WithMissingRateLimitHeader_DoesNotAliasRateLimited`
(pins the absent-header default = -1 behaviour so a future
regression that changes the default to 0 fails CI), and
`TestFindStalePRsHandler_NoNestedPIIEcho` (recursive walk of
Output asserting no nested field carries the author canary ‚Äî
generalises the M8.2.c iter-1 codex Major #3 "walk EVERY nested
level" lesson with a per-package test harness).

## 2026-05-11 ‚Äî M8.2.c: Slack tools bundle (`fetch_watch_orders` + `nudge_reviewer` + `post_daily_briefing`) + V3 manifest seed + M4.2 adapter extension

**PR**: #130
**Merged**: 364093e on 2026-05-11

### Context

M8.2.c ships the three Coordinator Slack tools the M8.2 roadmap calls
out ‚Äî one read (`fetch_watch_orders`) and two writes (`nudge_reviewer`,
`post_daily_briefing`). All three consume the M4.2 slack adapter, but
the adapter as shipped in M4.2 only covered `chat.postMessage` /
`users.info` / Socket-Mode / CreateApp+InstallApp ‚Äî it had NO
read-history surface. So M8.2.c also extends the slack adapter with
two new methods sibling to `Client.SendMessage` / `Client.LookupUser`:

- `Client.ConversationsHistory(ctx, channelID, opts) (HistoryResult, error)`
  ‚Üí wraps Slack `conversations.history`; cursor pagination + Slack-`ts`
  bounds; projects the wire shape onto a portable `HistoryMessage`
  (cross-platform spine `{TS, UserID, Text, ThreadTS, Subtype}` plus
  Slack-specific `Metadata` bag per the M4.1 LESSON).
- `Client.OpenIMChannel(ctx, userID) (channelID, error)` ‚Üí wraps
  Slack `conversations.open` for the 1:1 DM case so the Coordinator
  can address the lead by user id without callers pre-resolving the
  opaque IM channel id.

Both extensions add the matching rate-limit tier entries
(`conversations.history ‚Üí Tier3`, `conversations.open ‚Üí Tier3`),
two new sentinels (`ErrMissingScope`, `ErrCannotDMBot`), and lift
the documented Slack codes onto their portable counterparts via
the existing `lift*Error` discipline.

The V3 seed lands as a NEW migration `026_coordinator_manifest_v3_seed.sql`
‚Äî same append-only `manifest_version` evolution pattern as M8.2.b
(NEW stable id `23000000-‚Ä¶`, `version_no=3`, V1/V2 rows preserved
verbatim). `CoordinatorManifestVersionV3ID` joins the Go constants
in `core/pkg/manifest/coordinator.go`; the migration-shape test pair
(V1+V2 from earlier sub-items) extends to a triple covering V3.

Shared `briefing_formatter.go` renders structured `{title + sections}`
into Slack-mrkdwn text with idempotent zero-width-space escaping of
control characters (asterisk, underscore, tilde, backtick, gt). Both
`post_daily_briefing` (multi-section briefing) and `nudge_reviewer`
(single-bullet DM) call through it so the escape discipline lives in
one place.

### Pattern

1. **M4.2 adapter extension for a sub-item's read surface ‚Äî methods
   land in the adapter package, NOT in the consumer (Coordinator)** ‚Äî
   `core/pkg/messenger/slack/{conversations_history.go, open_im.go}`
   ship `Client.ConversationsHistory` + `Client.OpenIMChannel` as
   siblings to `Client.SendMessage` / `Client.LookupUser`. The
   alternative shape (a coordinator-internal helper calling
   `Client.Do(ctx, "conversations.history", ...)` directly) was
   rejected because (a) `Client.Do` is the low-level HTTP exchange
   plumbing and exposing it cross-package would leak the retry /
   rate-limit / logger discipline; (b) the per-method JSON-envelope
   types are adapter-owned and bleed into the consumer if defined
   there. The consumer pattern is "single-method interface at the
   consumer" (`SlackHistoryReader`, `SlackIMOpener`, `SlackMessenger`)
   ‚Äî the M8.2.a [JiraFieldUpdater] / M8.2.b [JiraSearcher] precedent ‚Äî
   and the adapter is the only place new Slack methods live. Mirror
   for any future M9+ adapter extension that surfaces a new tool
   surface to a downstream meta-agent.

2. **Three consumer-side single-method interfaces ‚Äî one per Slack
   verb** ‚Äî
   `core/pkg/coordinator/fetch_watch_orders.go:99-118` declares
   `SlackIMOpener` (`OpenIMChannel`) + `SlackHistoryReader`
   (`ConversationsHistory`), and `core/pkg/coordinator/nudge_reviewer.go:88-95`
   declares `SlackMessenger` (`SendMessage`). Each carries the
   compile-time satisfaction assertion `var _ ‚Ä¶ = (*slack.Client)(nil)`
   so a future signature drift on the M4.2 adapter fails build, not
   production. The three interfaces stay separate (not one fat
   `SlackTools` interface) so each tool's test fake only stubs what
   it needs ‚Äî the M8.2.a/M8.2.b "consumer-owned single-method
   interface" lesson scales linearly to multi-verb adapters.

3. **PII-classification of slack-open errors: refuse via
   ToolResult.Error vs Go-error wrap** ‚Äî
   `core/pkg/coordinator/fetch_watch_orders.go::openIMOrLiftRefusal`
   (line ~230) inspects the error from `OpenIMChannel` and routes
   PII-class failures (`user_not_found` / `cannot_dm_bot` /
   `missing_scope`) into a refusal text on `ToolResult.Error`, while
   transport / rate-limit / unclassified errors wrap with
   `coordinator: fetch_watch_orders: %w`. Rationale: the Slack
   `*APIError` for the PII-class codes carries enough context (status
   - method + code) for the agent to re-plan, but the runtime's
     M5.6.b reflector ingests Go-error chains via `%w` and writes a
     `lesson` row carrying the error string. A future regression that
     added `%q` of the failing user id to the adapter's error wrap
     would silently leak through. The refusal-text channel sidesteps
     that surface entirely. Mirrors the M8.2.a iter-1 PII finding
     (issue-key pre-validation precedes adapter call) extended from
     "validate inputs" to "classify adapter errors by PII reach".

5. **Shared formatter for two write tools, idempotent ZWS escape for
   Slack-mrkdwn** ‚Äî

   ZWS sandwich is idempotent: a second pass adds more ZWS but

### Iter-1 review patterns (codex + critic)

The parallel iter-1 review (codex via `omc ask` + `oh-my-claudecode:critic`
agent) surfaced **3 Major + 4 Minor** findings that landed on the same
branch before commit. Each pattern is reusable for M8.2.d / M9.x work.

a. **Adapter doc must match WIRE behaviour, not intuition (iter-1
codex Major #1)** ‚Äî
`core/pkg/messenger/slack/conversations_history.go::HistoryOptions.Oldest`
originally documented the filter as `ts >= Oldest`. The Slack API
default is EXCLUSIVE (`ts > Oldest`); the inclusive form requires
`inclusive=true` in the request. The handler
(`fetch_watch_orders.go::paginateHistory`) was sending `Oldest`
without `Inclusive: true`, dropping any message whose timestamp
matched `oldest_ts` exactly ‚Äî silent boundary hole.
Fix: `Inclusive: true` is now always sent; the adapter doc states
the EXCLUSIVE default explicitly. Lesson: when documenting an
option that maps onto an external API, cite the API's documented
behaviour and the call-site default ‚Äî do not assume the natural
semantic.

b. **Pagination signals are independent ‚Äî guard EVERY combinatoric
server-contract violation (iter-1 codex Major #2)** ‚Äî
`core/pkg/coordinator/fetch_watch_orders.go::paginateHistory`
originally treated `has_more=true` as implying a non-empty
`next_cursor`, with a comment in `conversations_history.go`
claiming the same equivalence. Slack documents the two fields as
INDEPENDENT signals: `has_more=true` with an empty `next_cursor`
is a server-side "more data exists but no continuation token" case
that, untreated, would re-request page 1 indefinitely (until the
page-cap or message-cap fired) and surface duplicates to the
agent. M8.2.b lesson #8 covered three of the four signal-source
combinations; this iter-1 finding completes the matrix.
Fix: `if HasMore && NextCursor == ""` returns `truncated=true`
immediately. Pinned via
`TestFetchWatchOrdersHandler_TruncatesOnHasMoreTrueWithEmptyCursor`.
Lesson generalisation: for paginated APIs with multiple signal
sources, enumerate the 2√ó2 matrix of `(HasMore, CursorEmpty)` and
ensure every cell has an unambiguous handler. The M8.1 jira guard
covered `IsLast=false && cursor==""`; M8.2.b lesson #8 added
`IsLast=true && cursor!=""`; M8.2.c iter-1 completes the matrix.

c. **Exported-but-typo'd struct field invites future wire-shape drift
(iter-1 critic Major #1)** ‚Äî
`core/pkg/messenger/slack/open_im.go` originally declared
`PreventCreat bool` (missing the trailing `ion`). The wire tag was
correct (`json:"prevent_creation,omitempty"`) so HTTP behaviour was
fine, but the symbol itself reads as a slip. A future caller (e.g.
the M8.2.d audit tool wanting `prevent_creation: true`) referencing
the field by name would either propagate the typo or rename it,
and a rename touches the wire-bound tag ‚Äî risking a silent
wire-shape drift the test harness (which only asserted `users` +
`return_im`) would not catch.
Fix: `PreventCreat` ‚Üí `PreventCreation`; the test wire struct now
includes the field. Lesson: even unused-today fields are API
surface inside the package; review for typos at write time, and
add the field to the wire-AC test struct so a future caller cannot
rename without breaking the assertion.

d. **Slack-mrkdwn escape table must cover the mention/link envelope
`<‚Ä¶>` (iter-1 critic Minor #1)** ‚Äî
`core/pkg/coordinator/briefing_formatter.go::escapeMrkdwn`
originally wrapped only asterisk / underscore / tilde / backtick /
greater-than runes. Slack parses angle-bracket envelopes as
link/mention syntax (an `<@U‚Ä¶>` token is a real ping; an
`<https://‚Ä¶>` envelope is a link) ‚Äî a bullet text containing such
an envelope would
render as a real notification surface even though the agent's
system prompt forbids it.
Fix:`<`added to the escape switch. Pinned via
`TestEscapeMrkdwn\*NeutralisesSlackMentionSyntax`(asserts
`<@U12345>`does not survive verbatim). Lesson: when implementing
an escape table for an external parser, enumerate every
syntactically-special character the parser documents ‚Äî not just
the ones the test happens to think of. For Slack-mrkdwn the
minimum table is`\*\*~\`<>`(and`&`for HTML-entity precedence in
   text contexts, intentionally omitted here because the formatter
   output is fed to`chat.postMessage.text`, which does not
HTML-entity-decode).

e. **Retract the idempotence claim when behaviour is not actually
idempotent (iter-1 codex Minor #1)** ‚Äî
`briefing_formatter.go::escapeMrkdwn`'s docblock originally claimed
"idempotent on input that already contains zero-width spaces". The
actual behaviour grows the output on every pass:
`escapeMrkdwn("*")` ‚Üí `<ZWS>*<ZWS>`; second pass ‚Üí `<ZWS><ZWS>*<ZWS><ZWS>`.
Slack renders both visually identically (extra ZWS collapse to
no-ops) but the structural string and rune-count are NOT stable
across passes.
Fix: docblock now states "NOT IDEMPOTENT" with the precise
behaviour and a note that the production call path invokes the
function exactly once per token. Pinned via
`TestEscapeMrkdwn_NotIdempotent_DocBlockMatchesBehavior` which
asserts the documented behaviour (pass2 strictly longer than
pass1). Lesson: when a docblock claims an invariant, either add a
test pinning it OR retract the claim. "Idempotent" is a load-
bearing word ‚Äî a reader who relies on it is entitled to a test
that proves it.

f. **Source-grep AC on shared helpers, not just on handler files
(iter-1 critic Missing #5)** ‚Äî
The handler files all carry `*_NoAuditOrLogAppend_SourceGrep`
tests asserting they do not import `keeperslog` or call `.Append(`.
The shared `briefing_formatter.go` had no such AC, so a future
inline-it-everywhere refactor could re-introduce audit calls in
the formatter and the test surface would not catch it.
Fix: `TestBriefingFormatter_NoAuditOrLogAppend_SourceGrep` added.
Lesson: when a helper is shared across multiple handlers that each
carry an AC, the helper itself needs its own AC of the same shape
‚Äî otherwise the boundary is unguarded at the lowest-shared layer.

g. **Test gaps surfaced by iter-1 critic (Missing #1, #2)** ‚Äî added
in iter-1: 429 path test for OpenIMChannel (so the lift discipline
is pinned independent of the shared `Client.Do` path); explicit

## 2026-05-11 ‚Äî M8.2.b: Jira read tool (`find_overdue_tickets`) + V2 manifest seed via append-only `manifest_version` row

**PR**: #129
**Merged**: 0f035fc on 2026-05-11

### Context

M8.2.b ships the first Jira READ tool the Coordinator can dispatch:
`find_overdue_tickets`. It composes a JQL string from four agent-
supplied args (`project_key`, `assignee_account_id`, `status` array,
`age_threshold_days`), drives the M8.1 cursor pagination via
`jira.Client.Search` up to a hard cap, and projects results through the
shared `issueWire.toIssue()` decoder so the agent never sees Atlassian
wire shapes. The tool's authority entry is `self` (read-only) per the
roadmap.

The seed extension is delivered via a NEW migration ‚Äî
`025_coordinator_manifest_v2_seed.sql` ‚Äî that INSERTs a SECOND
`manifest_version` row (`version_no=2`, fresh stable id
`22000000-‚Ä¶`) under the same Coordinator manifest. Migration 024 stays
untouched so the M8.2.a baseline is recoverable from the migration
sequence alone, and `keepclient.GetManifest` returns the highest-
`version_no` row (`core/pkg/keepclient/read_manifest.go:63-67`) so the
runtime auto-loads V2 once 025 has run. M8.2.c / M8.2.d follow the same

### Pattern

1. **Append-only `manifest_version` evolution: NEW migration with NEW
   stable id + incremented `version_no`, V1 row preserved verbatim** ‚Äî
   `deploy/migrations/025_coordinator_manifest_v2_seed.sql:99-126`
   INSERTs a fresh row (id `22000000-‚Ä¶`, `version_no=2`) carrying the
   extended toolset + authority matrix; migration 024 (id
   `21000000-‚Ä¶`, `version_no=1`) is NEVER edited. The
   `UNIQUE (manifest_id, version_no)` constraint
   (`002_core_business_tables.sql:48`) guarantees a re-run with a
   different id would fail loudly rather than spawn a duplicate
   `version_no=2`. The `keepclient.GetManifest` "highest version_no
   wins" semantic means the runtime auto-promotes to V2 with no
   loader change. Companion Go const
   `CoordinatorManifestVersionV2ID` lands in
   `core/pkg/manifest/coordinator.go:42-58` so test fixtures and
   M8.2.c/d migrations reference one symbol. Reusable for every
   future Coordinator toolset extension; the cost is one new
   migration file per increment, the win is the M8.2.a baseline
   stays recoverable from `git log -- deploy/migrations/`.

2. **Per-version migration-shape test pinning load-bearing literals
   in BOTH 024 and 025 (M8.2.a baseline + M8.2.b extension)** ‚Äî
   `core/pkg/manifest/coordinator_seed_test.go:147-185`
   (`TestCoordinatorSeed_MigrationContainsExpectedShape`, unchanged
   from M8.2.a) reads migration 024 off disk and asserts the V1
   shape; the new
   `TestCoordinatorSeedV2_MigrationContainsExpectedShape` (same file,
   ~line 240) reads migration 025 and asserts the V1 entries are
   preserved AND the V2 additions are present (the
   `find_overdue_tickets` toolset entry, the `"find_overdue_tickets":
"self"` authority entry, the `version_no=2` literal `"  2,"`,
   etc.). The M8.2.a fixture (`coordinatorSeedFixture`) is reused
   verbatim by V2 via a single-call wrapper
   `coordinatorSeedV2Fixture()` that mutates `ID`, `VersionNo`,

   system prompt. The runtime test
   `TestCoordinatorSeed_LoadsViaLoadManifest` migrates to the V2
   fixture (because keepclient picks it in production); the V1

   reading 024 off disk. Drift on either side fails CI, not
   production.

3. **Functional-clock injection via unexported test factory beats
   package-level `var nowFunc = time.Now`** ‚Äî
   `core/pkg/coordinator/find_overdue_tickets.go:204-222` exports
   `NewFindOverdueTicketsHandler(searcher)` that wraps the
   unexported `newFindOverdueTicketsHandlerWithClock(searcher, clock)`
   with `defaultNow = time.Now`. Tests reach for the unexported
   factory with a fixed clock so per-test overrides stay scoped to
   one handler instance ‚Äî no package-level mutable shared state, no
   race under `-parallel`. The first iteration of M8.2.b shipped a
   package-level `var nowFunc = time.Now` with a `withFixedNow(t,
...)` swap helper; `go test -race` flagged it instantly because
   parallel tests race on the read in `projectIssues` against the
   `t.Cleanup` restore in another test. The age_days assertion
   surfaced 125 days instead of 10 because by the time
   `projectIssues` ran, another test's `t.Cleanup` had already
   restored `nowFunc` to `time.Now`, so the production clock leaked
   into the assertion. Lesson: any "test override" via package-level
   variable is a race waiting to happen under `t.Parallel()`. Either
   thread the dependency through the factory (this fix), serialize
   the affected tests, or use `sync/atomic` ‚Äî but the per-instance
   factory injection is the cleanest because it keeps the production
   surface unchanged and the test surface honest about what it's
   substituting. Mirrors the M8.1 jira-adapter clock-injection
   precedent (`core/pkg/jira/client.go::clientConfig.clock`).

4. **JQL composition with safe quote-escaping AND pre-network shape
   pre-validation: defence-in-depth against agent-supplied injection
   shapes** ‚Äî
   `core/pkg/coordinator/find_overdue_tickets.go::composeOverdueJQL`
   (line ~340) builds the JQL string with `jqlQuote` (line ~360)
   wrapping every user-supplied value; `jqlQuote` escapes `\` BEFORE
   `"` so a `\"` input becomes `\\\"` (literal backslash + escaped
   quote), not `\\"` (escaped backslash + raw quote that closes the
   literal). The composer is defence-in-depth: every user input has
   ALREADY passed shape pre-validation (`projectKeyPattern`,
   `accountIDPattern`, `statusNamePattern`,
   `readAgeThresholdDaysArg` range check) by the time
   `composeOverdueJQL` runs, so the quoting is redundant on the
   happy path. But a future regression that admits a raw quote into
   a status entry would still emit syntactically well-formed JQL,
   not a parse error masquerading as an injection. Pinned via
   `TestComposeOverdueJQL_QuoteEscaping` and
   `TestJqlQuote_BackslashEscapedBeforeDoubleQuote`. The shape
   pre-validation IS the authoritative boundary; the quoting is the
   safety net. Reusable for any future tool composing a downstream
   string from agent-supplied parts (e.g. M8.2.c slack-search query,
   M8.2.d GitHub search query).

5. **Auto-paginate with both per-issue cap AND per-page cap, agent
   sees only `truncated=true`** ‚Äî
   `core/pkg/coordinator/find_overdue_tickets.go::paginateOverdue`

0; page < maxOverduePages; page++` loop, breaks early when the

   `searchPageSize=50`); the issue cap fires first under the

   iteration; the agent wants the full bounded set in one call.
   `ORDER BY updated ASC` in the JQL surfaces the oldest issues
   first so a truncation drops the freshest tail, not the staler
   head ‚Äî the operationally useful set survives the cap. Pinned

   /M8.2.d github-PR-search where the agent similarly wants a
   bounded scan, not a cursor loop.

7. **PII-canary harness: shape-INVALID canaries per arg surface so
   the gate actually rejects them, with `//nolint:gosec` on the

   `core/pkg/coordinator/find_overdue_tickets_test.go::TestFindOverdueTicketsHandler_RefusesMalformedProjectKey_NoEcho`

   `//nolint:gosec // G101: synthetic redaction-harness canary`
   annotation per the M*.c.* convention.

8. **Truncation flag must reflect ALL three signal sources, not just
   `!IsLast` ‚Äî page-cap, server-contract violation, AND unread tail
   (iter-1 codex+critic Major #1)** ‚Äî
   `core/pkg/coordinator/find_overdue_tickets.go::paginateOverdue`
   (line ~470) computes
   `more := !res.IsLast || res.NextPageToken != "" || i+1 < len(res.Issues)`
   when the issue cap fires INSIDE a page, AND treats
   `res.IsLast=true && res.NextPageToken!=""` as truncated rather
   than a clean terminus. The first iteration shipped only the
   `!res.IsLast` term; both reviewers flagged it independently from
   different angles. The bug bites when Atlassian (a) overshoots
   `MaxResults` on a final page (the M8.1 doc only one-sidedly
   guards "may return fewer", not "may return more") so an
   exact-cap last page silently drops the unread tail and reports
   `truncated=false`, OR (b) returns the symmetric server-contract
   violation `IsLast=true + non-empty cursor` (M8.1 only guards
   the inverse `IsLast=false + empty cursor`). Pinned via
   `TestFindOverdueTicketsHandler_TruncatesAtMaxIssues_OnLastPageWithUnread`
   and `TestFindOverdueTicketsHandler_TruncatesOnIsLastWithNonEmptyCursor`.
   Lesson: pagination truncation flags need defence-in-depth on
   every signal the underlying adapter exposes; "trust IsLast" is
   the wrong default when the upstream API only one-sidedly
   guarantees its contract. Reusable for M8.2.c slack-history
   pagination and M8.2.d github-PR pagination.

9. **Character-class regex is necessary but NOT sufficient for
   identifier whitelists ‚Äî pair with a discriminant pattern
   (iter-1 codex Major #2)** ‚Äî
   `core/pkg/coordinator/find_overdue_tickets.go::accountIDPattern`
   (line ~140) restricts the character set
   (`^[A-Za-z0-9][A-Za-z0-9:_\-]*$`); the new
   [accountIDDiscriminantPattern] (`[0-9:]`) requires at least one
   digit OR colon ‚Äî every Atlassian accountId carries one. The
   first iteration shipped only the character class; an
   all-uppercase env-var leak like `THE_API_KEY_VALUE` satisfied it
   and would have reached the M8.1 layer. The discriminant adds
   zero false-positives against real Atlassian accountIds (legacy
   `557058:abc-uuid` has both digits and colon; modern 24-char hex
   contains 0-9) while rejecting the token-shape leak path. Pinned
   via `TestFindOverdueTicketsHandler_RefusesTokenShapedAssigneeID_NoEcho`
   with the canary `THE_API_KEY_VALUE_DO_NOT_LOG` (passes
   character class, fails discriminant). Reusable for any future
   tool taking opaque-but-shape-constrained identifiers from agent
   args (Slack user ids, GitHub login names, Linear assignee
   handles). Lesson: when a regex permits a character class the
   target identifier ALSO permits, the regex is a "is this
   plausibly the right shape?" check, not a security boundary.
   Add a discriminant for the cheapest invariant the real shape
   carries (digit, colon, length, prefix) and require BOTH.

10. **Output success-path is a PII reach surface ‚Äî never echo
    GDPR-regulated identifiers back into [agentruntime.ToolResult.Output]
    even after shape validation (iter-1 critic Major M2)** ‚Äî
    `core/pkg/coordinator/find_overdue_tickets.go` (line ~290)
    surfaces a structured `scope` map keyed by `project_key`,
    `status`, `age_threshold_days` ‚Äî the `assignee_account_id`
    is INTENTIONALLY OMITTED, and the composed JQL string is NOT
    echoed (it would carry the accountId verbatim). The first
    iteration shipped `Output["jql"] = composedJQL` for "agent
    self-debug"; critic flagged that Atlassian classifies
    `accountId` as PII subject to GDPR (the 2018 migration to
    opaque ids was driven exactly by this), and the success-path
    Output is the surface that future audit-pipe integrations
    (M5.6.b reflector, downstream M5.6.c keepers_log writes) would
    silently serialise. The agent already has its call args; it
    does not need a copy in the result. Lesson: the M8.2.a iter-1
    "pre-validate to keep raw values out of adapter ERROR path"
    discipline (lesson #8) extends to the adapter SUCCESS path ‚Äî
    every echo of a user-supplied identifier into Output must be
    audited as a PII reach. Same discipline applies to M8.2.c
    (Slack user ids), M8.2.d (GitHub login names), and any future
    tool whose args include identifiers third parties classify as
    personal data.

1. **Pre-ship decomposition by adapter affinity (with a separate

2. **Stable seed migration mirrors `017_watchmaster_manifest_seed.sql`
   column-for-column** ‚Äî

   system tenant `00000000-0000-4000-8000-000000000000` from

   intentional ordering** ‚Äî

## 2026-05-10 ‚Äî M8.1: Jira adapter ‚Äî REST API v3, stdlib HTTP, fail-closed field whitelist

**PR**: #127
**Merged**: 06d7c45 on 2026-05-10

### Context

M8.1 introduced `core/pkg/jira` ‚Äî an Atlassian Cloud REST API v3
adapter shipping the four operations the M8.2 Coordinator manifest
will consume:

- `Search(ctx, jql, opts) (SearchResult, error)` ‚Äî
  `POST /rest/api/3/search/jql` with cursor pagination.
- `GetIssue(ctx, key, fields) (Issue, error)` ‚Äî
  `GET /rest/api/3/issue/{key}`.
- `AddComment(ctx, key, body) (Comment, error)` ‚Äî
  `POST /rest/api/3/issue/{key}/comment`, plain text in / out, ADF
  wrap/unwrap internal to the adapter.
- `UpdateFields(ctx, key, fields) error` ‚Äî
  `PUT /rest/api/3/issue/{key}` with synchronous whitelist
  enforcement.

This is the first M8 entry; the package layout, naming, and option
patterns mirror the M4.2 slack adapter (the closest sibling) so a
reader fluent in one is fluent in the other. The M8.2 tool layer
(authority matrix, role-scoped action gating) sits ON TOP of, not
INSTEAD of, the transport-level guards this PR introduces.

### Pattern

1. **Stdlib-only HTTP, no third-party Atlassian SDK** ‚Äî `core/pkg/jira`
   depends only on the Go standard library, mirroring the slack
   adapter's M4.2.a "Stdlib-only" decision (`core/pkg/messenger/slack/doc.go`
   ¬ß"Design choices") and the older keepclient discipline (M2.8.a).
   `go.mod` carries no `go-jira`/`andygrunwald` line. Rationale: each
   external SDK ships its own concurrency, retry, JSON-shape, and
   error-wrapping opinions that conflict with the project's own
   conventions; a thin self-rolled client keeps the dependency surface
   small, the tests fast, and the code shape uniform across every
   external integration. Reusable across any future `core/pkg/<svc>`
   adapter (Linear, GitHub, Notion).

2. **API v3 + ADF wrap on the wire, plain text on the seam** ‚Äî
   The adapter targets Atlassian Cloud REST API v3 (the
   forward-compatible path; v2 is documented-deprecated for
   sub-APIs). Comments require ADF (Atlassian Document Format) on
   the wire; `Client.AddComment` accepts plain text and wraps it in
   a minimal `doc ‚Üí paragraph[] ‚Üí text` envelope, splitting on `\n`
   to one paragraph per line. The reverse projection
   (`adfExtractText` in `core/pkg/jira/add_comment.go`) walks the
   ADF tree and joins text leaves with `\n`; complex blocks
   (mentions, codeBlock, tables) are documented best-effort and
   lossy. Pattern: when an external API forces a heavy on-wire
   shape, hide it inside the adapter and expose the simplest seam
   types the consumer needs. M8.2 tool authors never see ADF.

3. **Field whitelist as constructor option, fail-closed default,
   rejection BEFORE the network** ‚Äî `WithFieldWhitelist(fields ...string)`
   in `core/pkg/jira/options.go` configures the closed set of
   Atlassian field IDs `Client.UpdateFields` may write. A nil/empty
   whitelist refuses ALL writes
   (`core/pkg/jira/client.go::Client.fieldWhitelistContains`).
   Rejection is synchronous and precedes the HTTP exchange ‚Äî the
   source-grep AC `TestUpdateFields_WhitelistEnforcement_SourceGrep`
   in `core/pkg/jira/update_fields_test.go` pins the order
   (`fieldWhitelistContains` index < `c.do(` index in the source
   file). Rationale: the adapter is the platform-facing security
   boundary; the M8.2 role authority matrix layers on top of, not
   instead of, this transport guard. Reusable for any future
   capability-restricted adapter call (e.g. a future Linear adapter
   with whitelisted state transitions).

4. **Single shared wire decoder for issue payloads** ‚Äî
   `issueWire.toIssue()` in `core/pkg/jira/get_issue.go` is the
   canonical projection from Atlassian's nested
   `{id, key, fields:{summary, status, assignee, reporter, ‚Ä¶}}`
   shape onto the typed `Issue` surface. Both `Client.GetIssue` and
   `Client.Search` (via `searchWire.Issues []issueWire`) share the
   one decoder; no per-call duplication. The raw
   `Fields map[string]json.RawMessage` is preserved verbatim so
   callers driving custom-field extraction never lose fidelity.
   Pattern: when one wire shape feeds multiple operations, hoist the
   decoder onto the wire type itself; the operation files stay
   one-screen tall.

5. **APIError with `kind endpointKind` tag for endpoint-disambiguated
   Unwrap** ‚Äî `core/pkg/jira/errors.go` ships ONE `APIError` type
   carrying an unexported `kind endpointKind` (generic / issue /
   search). `APIError.Unwrap` consults `kind` so 404 maps to
   `ErrIssueNotFound` only on issue-keyed endpoints and 400 maps to
   `ErrInvalidJQL` only on `/search/jql`. Avoids the per-endpoint
   error-type sprawl an Atlassian SDK would impose ‚Äî five sentinels
   ([`ErrInvalidAuth`], [`ErrIssueNotFound`], [`ErrFieldNotWhitelisted`],
   [`ErrInvalidJQL`], [`ErrRateLimited`]) cover every M8.2-callable
   discriminant. Reusable when an HTTP API multiplexes the same
   status code onto different semantics across endpoints.

6. **`BasicAuthSource` interface mirrors slack `TokenSource`** ‚Äî
   `core/pkg/jira/client.go::BasicAuthSource` is a per-call resolver
   shaped exactly like `core/pkg/messenger/slack/client.go::TokenSource`.
   `StaticBasicAuth{Email, Token}` is the trivial static impl for
   tests + bootstrap; production wires the resolver to the M3.4.b
   secrets interface so token rotation is observable. Rationale: the
   M7.1.c.c lesson "per-call resolver beats process-global static"
   for tenant-scoped values applies here too ‚Äî Atlassian API tokens
   rotate per-account, and a resolver-shaped interface makes
   rotation transparent to call sites.

7. **Layered source-grep ACs over runtime-mock observation** ‚Äî
   Two layers of negative-AC tests use source-grep + `os.ReadFile`
   mutation discipline (per M7.1.c.a's "hollow runtime-mock AC pin"
   anti-pattern). Iter-1 review surfaced that a single negative
   substring list is brittle (a kv-key rename like `, "auth_kv",
token` would slip past `"token"`-only forbidden lists), so the
   pin layers two scans:
   - File-level forbidden list (narrow, structural): patterns that
     should never appear ANYWHERE in `client.go` outside comments
     (`"api_token"`, `"request_body"`, `"response_body"`,
     `req.Body`).
   - Logger.Log call-site scan (rigorous): `extractLoggerCalls`
     paren-balances every `c.cfg.logger.Log(...)` block in the
     source and asserts NONE contains forbidden value-leak shapes
     (`, token)`, `, email)`, `, body)`, `, req.Body`, `+token`,
     `+email`, etc.). Catches the "rename the kv key, leak the
     value" regression class the file-level scan would miss.

   Companion runtime test
   `TestDo_LoggerRedactsAtTransportErrorPath` uses unique probe
   sentinels (`REDACTION_PROBE_TOKEN_M81_DO_NOT_LOG` /
   `REDACTION_PROBE_EMAIL_M81_DO_NOT_LOG` / a tenant-text probe in
   the API-error envelope) and asserts the runtime logger never
   surfaced any of them. Generic test credentials would only catch
   verbatim-equality regressions; probe-shape sentinels catch
   subtler kv leaks.

   `TestUpdateFields_WhitelistEnforcement_SourceGrep` scans
   `update_fields.go` and asserts the source-order
   `fieldWhitelistContains` index < `c.do(` index ‚Äî a refactor
   that moves the whitelist check after the network call fails the
   test even though the runtime "happy paths still pass". A small
   `stripGoComments` helper in `client_test.go` removes line +
   block comments so allowed mentions inside docstrings (e.g. the
   "redaction discipline" doc paragraph naming `"token"` and
   `"body"`) do not false-positive.

   Lesson on iteration: the original M8.1 PR shipped only the
   file-level scan; iter-1 review caught that rename-the-key
   defeats it. The fix is layered + probe-sentinels + adversarial
   test scope, not a wider-but-still-substring forbidden list.
