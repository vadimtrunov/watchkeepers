## 2026-05-11 — M8.2.b: Jira read tool (`find_overdue_tickets`) + V2 manifest seed via append-only `manifest_version` row

**PR**: #129
**Merged**: 0f035fc on 2026-05-11

### Context

M8.2.b ships the first Jira READ tool the Coordinator can dispatch:
`find_overdue_tickets`. It composes a JQL string from four agent-
supplied args (`project_key`, `assignee_account_id`, `status` array,
`age_threshold_days`), drives the M8.1 cursor pagination via
`jira.Client.Search` up to a hard cap, and projects results through the
shared `issueWire.toIssue()` decoder so the agent never sees Atlassian
wire shapes. The tool's authority entry is `self` (read-only) per the
roadmap.

The seed extension is delivered via a NEW migration —
`025_coordinator_manifest_v2_seed.sql` — that INSERTs a SECOND
`manifest_version` row (`version_no=2`, fresh stable id
`22000000-…`) under the same Coordinator manifest. Migration 024 stays
untouched so the M8.2.a baseline is recoverable from the migration
sequence alone, and `keepclient.GetManifest` returns the highest-
`version_no` row (`core/pkg/keepclient/read_manifest.go:63-67`) so the
runtime auto-loads V2 once 025 has run. M8.2.c / M8.2.d follow the same
pattern with V3 / V4 rows.

### Pattern

1. **Append-only `manifest_version` evolution: NEW migration with NEW
   stable id + incremented `version_no`, V1 row preserved verbatim** —
   `deploy/migrations/025_coordinator_manifest_v2_seed.sql:99-126`
   INSERTs a fresh row (id `22000000-…`, `version_no=2`) carrying the
   extended toolset + authority matrix; migration 024 (id
   `21000000-…`, `version_no=1`) is NEVER edited. The
   `UNIQUE (manifest_id, version_no)` constraint
   (`002_core_business_tables.sql:48`) guarantees a re-run with a
   different id would fail loudly rather than spawn a duplicate
   `version_no=2`. The `keepclient.GetManifest` "highest version_no
   wins" semantic means the runtime auto-promotes to V2 with no
   loader change. Companion Go const
   `CoordinatorManifestVersionV2ID` lands in
   `core/pkg/manifest/coordinator.go:42-58` so test fixtures and
   M8.2.c/d migrations reference one symbol. Reusable for every
   future Coordinator toolset extension; the cost is one new
   migration file per increment, the win is the M8.2.a baseline
   stays recoverable from `git log -- deploy/migrations/`.

2. **Per-version migration-shape test pinning load-bearing literals
   in BOTH 024 and 025 (M8.2.a baseline + M8.2.b extension)** —
   `core/pkg/manifest/coordinator_seed_test.go:147-185`
   (`TestCoordinatorSeed_MigrationContainsExpectedShape`, unchanged
   from M8.2.a) reads migration 024 off disk and asserts the V1
   shape; the new
   `TestCoordinatorSeedV2_MigrationContainsExpectedShape` (same file,
   ~line 240) reads migration 025 and asserts the V1 entries are
   preserved AND the V2 additions are present (the
   `find_overdue_tickets` toolset entry, the `"find_overdue_tickets":
"self"` authority entry, the `version_no=2` literal `"  2,"`,
   etc.). The M8.2.a fixture (`coordinatorSeedFixture`) is reused
   verbatim by V2 via a single-call wrapper
   `coordinatorSeedV2Fixture()` that mutates `ID`, `VersionNo`,
   `Tools`, `AuthorityMatrix` — no copy-paste of the unchanged
   system prompt. The runtime test
   `TestCoordinatorSeed_LoadsViaLoadManifest` migrates to the V2
   fixture (because keepclient picks it in production); the V1
   shape is independently pinned by the migration-shape test
   reading 024 off disk. Drift on either side fails CI, not
   production.

3. **Functional-clock injection via unexported test factory beats
   package-level `var nowFunc = time.Now`** —
   `core/pkg/coordinator/find_overdue_tickets.go:204-222` exports
   `NewFindOverdueTicketsHandler(searcher)` that wraps the
   unexported `newFindOverdueTicketsHandlerWithClock(searcher, clock)`
   with `defaultNow = time.Now`. Tests reach for the unexported
   factory with a fixed clock so per-test overrides stay scoped to
   one handler instance — no package-level mutable shared state, no
   race under `-parallel`. The first iteration of M8.2.b shipped a
   package-level `var nowFunc = time.Now` with a `withFixedNow(t,
...)` swap helper; `go test -race` flagged it instantly because
   parallel tests race on the read in `projectIssues` against the
   `t.Cleanup` restore in another test. The age_days assertion
   surfaced 125 days instead of 10 because by the time
   `projectIssues` ran, another test's `t.Cleanup` had already
   restored `nowFunc` to `time.Now`, so the production clock leaked
   into the assertion. Lesson: any "test override" via package-level
   variable is a race waiting to happen under `t.Parallel()`. Either
   thread the dependency through the factory (this fix), serialize
   the affected tests, or use `sync/atomic` — but the per-instance
   factory injection is the cleanest because it keeps the production
   surface unchanged and the test surface honest about what it's
   substituting. Mirrors the M8.1 jira-adapter clock-injection
   precedent (`core/pkg/jira/client.go::clientConfig.clock`).

4. **JQL composition with safe quote-escaping AND pre-network shape
   pre-validation: defence-in-depth against agent-supplied injection
   shapes** —
   `core/pkg/coordinator/find_overdue_tickets.go::composeOverdueJQL`
   (line ~340) builds the JQL string with `jqlQuote` (line ~360)
   wrapping every user-supplied value; `jqlQuote` escapes `\` BEFORE
   `"` so a `\"` input becomes `\\\"` (literal backslash + escaped
   quote), not `\\"` (escaped backslash + raw quote that closes the
   literal). The composer is defence-in-depth: every user input has
   ALREADY passed shape pre-validation (`projectKeyPattern`,
   `accountIDPattern`, `statusNamePattern`,
   `readAgeThresholdDaysArg` range check) by the time
   `composeOverdueJQL` runs, so the quoting is redundant on the
   happy path. But a future regression that admits a raw quote into
   a status entry would still emit syntactically well-formed JQL,
   not a parse error masquerading as an injection. Pinned via
   `TestComposeOverdueJQL_QuoteEscaping` and
   `TestJqlQuote_BackslashEscapedBeforeDoubleQuote`. The shape
   pre-validation IS the authoritative boundary; the quoting is the
   safety net. Reusable for any future tool composing a downstream
   string from agent-supplied parts (e.g. M8.2.c slack-search query,
   M8.2.d GitHub search query).

5. **Auto-paginate with both per-issue cap AND per-page cap, agent
   sees only `truncated=true`** —
   `core/pkg/coordinator/find_overdue_tickets.go::paginateOverdue`
   (line ~378) drives `jira.Client.Search` in a `for page :=
0; page < maxOverduePages; page++` loop, breaks early when the
   cumulative `len(collected) >= maxOverdueIssues`, and surfaces
   `truncated=true` whenever either cap fires before the M8.1 layer
   reports `IsLast=true`. Caps are both at the package level
   (`maxOverdueIssues=200`, `maxOverduePages=10`,
   `searchPageSize=50`); the issue cap fires first under the
   documented 50/page default. The agent does NOT drive cursor
   loops — `find_overdue_tickets` is a SCAN operation, not an
   iteration; the agent wants the full bounded set in one call.
   `ORDER BY updated ASC` in the JQL surfaces the oldest issues
   first so a truncation drops the freshest tail, not the staler
   head — the operationally useful set survives the cap. Pinned
   via `TestFindOverdueTicketsHandler_TruncatesAtMaxIssues` (single
   page > 200 issues) and `TestFindOverdueTicketsHandler_TruncatesAtMaxPages`
   (10+ pages of 1 issue each). Reusable for M8.2.c slack-search
   /M8.2.d github-PR-search where the agent similarly wants a
   bounded scan, not a cursor loop.

6. **Source-grep AC for "every per-arg validator precedes the network
   call" pinned by helper CALL pattern, not bare symbol** —
   `core/pkg/coordinator/find_overdue_tickets_test.go::TestFindOverdueTicketsHandler_ValidationPrecedesSearchCall_SourceGrep`
   reads `find_overdue_tickets.go` off disk, strips Go comments via
   the M8.1 lesson #7 `stripGoComments` helper, and asserts every
   per-arg reader CALL (`readProjectKeyArg(`,
   `readAssigneeAccountIDArg(`, `readStatusArg(`,
   `readAgeThresholdDaysArg(`) appears in the source BEFORE the
   `searcher.Search(` site. Mirrors the M8.2.a iter-1 codex Major
   #1 lesson: anchor source-grep ACs to the runtime-guard CALL
   pattern, NOT the bare symbol — the const declarations / shape
   regex `var` declarations would otherwise satisfy a symbol-only
   search even if the runtime guards moved below the network call.
   Companion `TestFindOverdueTicketsHandler_NoAuditOrLogAppend_SourceGrep`
   asserts the handler contains no `keeperslog.` and no `.Append(`
   outside comments — the runtime's tool-result reflection layer
   (M5.6.b) is the audit boundary, not the handler.

7. **PII-canary harness: shape-INVALID canaries per arg surface so
   the gate actually rejects them, with `//nolint:gosec` on the
   const decl** —
   `core/pkg/coordinator/find_overdue_tickets_test.go::TestFindOverdueTicketsHandler_RefusesMalformedProjectKey_NoEcho`
   uses `redaction-probe-token-m82b-project-do-not-log` (lowercase +
   dashes) — NOT
   `REDACTION_PROBE_TOKEN_M82B_PROJECT_DO_NOT_LOG` (the original
   draft) which silently SATISFIES `projectKeyPattern`
   (`^[A-Z][A-Z0-9_]+$`) and would never trip the gate. The first
   draft of this test failed exactly that way — the canary slipped
   through the regex and the test asserted "no echo" against an
   empty refusal text, false-positive-passing. Lesson: every
   PII-canary test must EXPLICITLY assert the canary is shape-
   invalid against the regex it is testing — otherwise the canary
   becomes a no-op. Same discipline for `assignee_account_id` (the
   `!@#$` suffix on the canary kills the regex) and `status`
   (the leading `")` opens a JQL injection that the
   `statusNamePattern` rejects). Each canary const carries a
   `//nolint:gosec // G101: synthetic redaction-harness canary`
   annotation per the M*.c.* convention.

8. **Truncation flag must reflect ALL three signal sources, not just
   `!IsLast` — page-cap, server-contract violation, AND unread tail
   (iter-1 codex+critic Major #1)** —
   `core/pkg/coordinator/find_overdue_tickets.go::paginateOverdue`
   (line ~470) computes
   `more := !res.IsLast || res.NextPageToken != "" || i+1 < len(res.Issues)`
   when the issue cap fires INSIDE a page, AND treats
   `res.IsLast=true && res.NextPageToken!=""` as truncated rather
   than a clean terminus. The first iteration shipped only the
   `!res.IsLast` term; both reviewers flagged it independently from
   different angles. The bug bites when Atlassian (a) overshoots
   `MaxResults` on a final page (the M8.1 doc only one-sidedly
   guards "may return fewer", not "may return more") so an
   exact-cap last page silently drops the unread tail and reports
   `truncated=false`, OR (b) returns the symmetric server-contract
   violation `IsLast=true + non-empty cursor` (M8.1 only guards
   the inverse `IsLast=false + empty cursor`). Pinned via
   `TestFindOverdueTicketsHandler_TruncatesAtMaxIssues_OnLastPageWithUnread`
   and `TestFindOverdueTicketsHandler_TruncatesOnIsLastWithNonEmptyCursor`.
   Lesson: pagination truncation flags need defence-in-depth on
   every signal the underlying adapter exposes; "trust IsLast" is
   the wrong default when the upstream API only one-sidedly
   guarantees its contract. Reusable for M8.2.c slack-history
   pagination and M8.2.d github-PR pagination.

9. **Character-class regex is necessary but NOT sufficient for
   identifier whitelists — pair with a discriminant pattern
   (iter-1 codex Major #2)** —
   `core/pkg/coordinator/find_overdue_tickets.go::accountIDPattern`
   (line ~140) restricts the character set
   (`^[A-Za-z0-9][A-Za-z0-9:_\-]*$`); the new
   [accountIDDiscriminantPattern] (`[0-9:]`) requires at least one
   digit OR colon — every Atlassian accountId carries one. The
   first iteration shipped only the character class; an
   all-uppercase env-var leak like `THE_API_KEY_VALUE` satisfied it
   and would have reached the M8.1 layer. The discriminant adds
   zero false-positives against real Atlassian accountIds (legacy
   `557058:abc-uuid` has both digits and colon; modern 24-char hex
   contains 0-9) while rejecting the token-shape leak path. Pinned
   via `TestFindOverdueTicketsHandler_RefusesTokenShapedAssigneeID_NoEcho`
   with the canary `THE_API_KEY_VALUE_DO_NOT_LOG` (passes
   character class, fails discriminant). Reusable for any future
   tool taking opaque-but-shape-constrained identifiers from agent
   args (Slack user ids, GitHub login names, Linear assignee
   handles). Lesson: when a regex permits a character class the
   target identifier ALSO permits, the regex is a "is this
   plausibly the right shape?" check, not a security boundary.
   Add a discriminant for the cheapest invariant the real shape
   carries (digit, colon, length, prefix) and require BOTH.

10. **Output success-path is a PII reach surface — never echo
    GDPR-regulated identifiers back into [agentruntime.ToolResult.Output]
    even after shape validation (iter-1 critic Major M2)** —
    `core/pkg/coordinator/find_overdue_tickets.go` (line ~290)
    surfaces a structured `scope` map keyed by `project_key`,
    `status`, `age_threshold_days` — the `assignee_account_id`
    is INTENTIONALLY OMITTED, and the composed JQL string is NOT
    echoed (it would carry the accountId verbatim). The first
    iteration shipped `Output["jql"] = composedJQL` for "agent
    self-debug"; critic flagged that Atlassian classifies
    `accountId` as PII subject to GDPR (the 2018 migration to
    opaque ids was driven exactly by this), and the success-path
    Output is the surface that future audit-pipe integrations
    (M5.6.b reflector, downstream M5.6.c keepers_log writes) would
    silently serialise. The agent already has its call args; it
    does not need a copy in the result. Lesson: the M8.2.a iter-1
    "pre-validate to keep raw values out of adapter ERROR path"
    discipline (lesson #8) extends to the adapter SUCCESS path —
    every echo of a user-supplied identifier into Output must be
    audited as a PII reach. Same discipline applies to M8.2.c
    (Slack user ids), M8.2.d (GitHub login names), and any future
    tool whose args include identifiers third parties classify as
    personal data.

### Addendum to M8.2.a lesson #10 (refusal-prefix const naming)

The M8.2.a lesson #10 advised "Encoded as a `refusalPrefix` const
concatenated onto every refusal text in the handler so a future
`<tool>` rename is a one-line change." The advice is correct WITHIN a
single handler file but breaks across multiple coordinator handler
files in the same Go package: at package scope, `refusalPrefix`
collides between `update_ticket_field.go` and the new
`find_overdue_tickets.go`. M8.2.b resolves this by prefixing each
const with the tool name (`findOverdueRefusalPrefix`); M8.2.c/d
authors should follow the same per-tool prefix convention rather than
re-introducing the bare-name collision. The trade-off is a
slightly-noisier const name in exchange for package-level uniqueness;
the original lesson #10 framing assumed one handler per file but did
not state the assumption.

## 2026-05-10 — M8.2.a: Coordinator manifest seed + tool dispatch primitive + first Jira-write tool (`update_ticket_field`)

**PR**: #128
**Merged**: a7ecd6c on 2026-05-10

### Context

M8.2 in `docs/ROADMAP-phase1.md` originally bundled a Coordinator
manifest seed, a tool-dispatch primitive, and 6 tools spanning three
adapters (Jira / Slack / GitHub). One of those tools, `find_stale_prs`,
required a `core/pkg/github` adapter that did not exist in the
codebase. The pre-ship decomposition (commit `bf2a79f`) split M8.2
along adapter affinity into 4 sub-items, mirroring the M7.3
decomposition pattern (commit `2370593`):

- **M8.2.a** — Coordinator manifest seed + tool dispatch primitive +
  first Jira-write tool (`update_ticket_field`). This entry.
- **M8.2.b** — Jira read tool (`find_overdue_tickets`).
- **M8.2.c** — Slack tools bundle (`fetch_watch_orders` +
  `nudge_reviewer` + `post_daily_briefing`).
- **M8.2.d** — GitHub PR tool (`find_stale_prs`) — REQUIRES a new
  `core/pkg/github` adapter; magnitude comparable to M8.1.

M8.2.a delivers the foundation every later sub-item layers onto: the
seeded manifest the runtime loads at Coordinator boot, the dispatch
primitive the tool registry calls into, and one concrete handler that
exercises both end-to-end against the M8.1 jira write path. The
authority matrix shape, refusal-text channel, and per-arg discipline
shipped here are the templates M8.2.b/c/d will follow.

### Pattern

1. **Pre-ship decomposition by adapter affinity (with a separate
   `docs(roadmap): decompose` commit on main, no PR)** —
   `docs/ROADMAP-phase1.md` M8.2 was decomposed BEFORE M8.2.a opened
   a branch, exactly as M7.3 was (commit `2370593`). The decompose
   commit lives directly on `main` (no PR — the project's prior art
   for `docs(roadmap): cascade` and `docs(roadmap): decompose` is
   direct push). The decomposition carried the missing-adapter
   discovery into the roadmap text itself: M8.2.d's description
   states "REQUIRES new `core/pkg/github` adapter" so future
   ship-roadmap-item runs are not blocked at PR-open time. Reusable
   for any large milestone where shipping the whole item in one PR
   would force review-iteration to drown in surface area.

2. **Stable seed migration mirrors `017_watchmaster_manifest_seed.sql`
   column-for-column** —
   `deploy/migrations/024_coordinator_manifest_seed.sql` reuses the
   system tenant `00000000-0000-4000-8000-000000000000` from
   migration 017 (idempotent `ON CONFLICT (id) DO NOTHING` insert
   makes the duplicate a no-op), declares fresh stable UUIDs for the
   Coordinator manifest (`20000000-…`) and initial manifest_version
   (`21000000-…`), and pins the `\n||` escape-string concatenation
   shape so the system-prompt text stays under sqlfluff's 120-char
   line cap while the runtime stores the joined string. Companion
   Go const file `core/pkg/manifest/coordinator.go` exports
   `CoordinatorManifestID` + `CoordinatorManifestVersionID`; the
   migration-shape test
   `TestCoordinatorSeed_MigrationContainsExpectedShape` reads the
   SQL off disk and asserts every load-bearing literal (UUIDs,
   role-boundary phrase `"NEVER reassign tickets"`, lead-deferral
   phrase `"ALWAYS surface a"`, runtime-vocab matrix entries,
   model + autonomy, idempotency clause). Drift fails CI, not
   production. Same M5/M6 seed pattern; reusable for any future
   per-role meta-agent seed.

3. **Authority-matrix vocabulary divergence: `runtime/authority.go`
   (`self`/`lead`/`operator`/`watchmaster`) versus M5.5.b.c.c.b
   (`allowed`/`lead_approval`/`forbidden`)** — the Watchmaster seed
   (migration 017) uses the M5.5 enum because Watchmaster's
   `autonomy = supervised` short-circuits `RequiresApproval` to
   "every action requires approval" regardless of matrix value (the
   `default` arm of the autonomy switch in
   `core/pkg/runtime/authority.go::RequiresApproval`). The
   Coordinator runs `autonomy = autonomous` so the matrix vocabulary
   IS consulted on every dispatch — only the runtime vocabulary
   produces the intended gate semantics. A drift to `lead_approval`
   here would silently fail closed via the unrecognised-value
   `default` arm of the matrix switch in the same function,
   blocking every Coordinator tool call. Pin via
   `TestCoordinatorSeed_AuthorityMatrixUsesRuntimeVocabulary`
   (asserts every value ∈ {self, lead, operator, watchmaster}). The
   tension is documented inline in both `coordinator.go` (top doc
   block) and migration 024's authority-matrix comment so a future
   reader sees the divergence without spelunking. Anchor by symbol
   name, not line number — `authority.go` line numbers will rot.

4. **Tool-dispatch primitive — six-step gate sequence with
   intentional ordering** —
   `core/pkg/runtime/tooldispatch.go::ToolDispatcher.Dispatch` runs:
   (1) `ctx.Err()` pre-check (no map work on cancel),
   (2) empty `call.Name` → `ErrInvalidToolCall`,
   (3) `call.Name` ∉ `Manifest.Toolset.Names()` → `ErrToolUnauthorized`,
   (4) `RequiresApproval` returns `(true, reason)` →
   `*ApprovalRequiredError`,
   (5) handler missing → `ErrToolHandlerMissing`,
   (6) dispatch to handler.
   Order matters: membership (3) precedes approval (4) so a manifest
   typo surfaces as "unauthorised" not "needs approval"; approval
   (4) precedes handler-presence (5) so a wiring bug never leaks
   through an approval bypass. Pinned via paired-gate tests
   `TestDispatch_ToolsetGateRunsBeforeApprovalGate` and
   `TestDispatch_HandlerMissing_RunsAfterApprovalGate`. Handler
   dispatch happens AFTER the read-lock is released so a re-entrant
   `Dispatch`/`Register` call from inside a handler does not
   self-deadlock — covered by `TestDispatch_ConcurrentRegisterAndDispatch`
   (32 goroutines mixed register + dispatch under `-race`).

5. **`ApprovalRequiredError`: typed error + sentinel duality
   (`errors.Is(err, ErrApprovalRequired)` AND
   `errors.As(err, &*ApprovalRequiredError)`)** —
   `core/pkg/runtime/errors.go` defines both
   `ErrApprovalRequired` (sentinel, matchable via `errors.Is`) and
   `*ApprovalRequiredError{Action, Reason}` (typed, unwrappable via
   `errors.As`). The typed value implements `Is(target error) bool`
   so the sentinel comparison hits via the `errors.Is` chain even
   though they are different concrete types. Wrapping with
   `fmt.Errorf("…: %w", err)` preserves BOTH access patterns —
   pinned via `TestApprovalRequiredError_FmtErrorfWraps`. Reusable
   pattern for any future error that needs both "did this happen?"
   short-circuits and "what specifically?" detail extraction.

6. **Authoritative `assignee` refusal in the handler, with a
   deployment-side jira whitelist as opt-in transport-layer
   reinforcement, plus iter-1-strengthened source-grep AC** —
   `core/pkg/coordinator/update_ticket_field.go` refuses any args
   containing the `assignee` key BEFORE calling
   `updater.UpdateFields` — this is the AUTHORITATIVE boundary the
   M8.2.a system prompt advertises. The M8.1 jira whitelist
   ADDITIONALLY excludes `assignee` when the deployment configures
   `WithFieldWhitelist(...)` accordingly (M8.1 lesson #3) — a
   transport-layer reinforcement that is opt-in at deploy time, not
   enforced by M8.2.a code. Iter-1 review (critic M1) flagged an
   earlier framing that read this as automatic dual-defense; the
   shipped framing scopes the transport claim to "when the
   deployment configures the whitelist" so an operator who configures
   a permissive whitelist does not silently break the prompt's
   promise. The handler's source-grep AC
   `TestUpdateTicketFieldHandler_AssigneeRefusalPrecedesUpdaterCall_SourceGrep`
   reads `update_ticket_field.go` off disk, strips Go comments via
   `stripGoComments` (the M8.1 lesson #7 helper), and asserts the
   runtime-guard pattern `fields[AssigneeFieldKey]` precedes
   `updater.UpdateFields(`. Iter-1 codex Major #1 caught that an
   earlier version matched the bare symbol `AssigneeFieldKey` —
   which landed on the const declaration and silently passed even
   if the runtime guard moved below the network call. Anchor
   source-grep ACs to the GUARD pattern, not the symbol, when the
   symbol also has a const/declaration use elsewhere in the file.
   The roadmap M8.2 phrase "no reassignment without lead approval"
   is realised as a hard handler refusal in M8.2.a; a future
   sub-item lifts the refusal into a real lead-approval card.

7. **Consumer-owned single-method interface + compile-time
   satisfaction assertion for handler dependencies** —
   `core/pkg/coordinator/update_ticket_field.go` declares
   `JiraFieldUpdater` (a single-method interface mirroring
   `jira.Client.UpdateFields`'s signature exactly) at the consumer,
   not at the producer, per the project's `ManifestFetcher`
   precedent. A `var _ JiraFieldUpdater = (*jira.Client)(nil)` line
   locks the contract at build time: an M8.1 signature drift fails
   build, not production. Tests inject hand-rolled fake
   `fakeJiraUpdater` without touching the HTTP client — same
   M*.c.* "no mocking lib" discipline.

8. **Pre-network `issue_key` shape validation in the handler so the
   M8.1 layer never echoes the raw key (iter-1 codex Major #2 /
   critic M3)** —
   `core/pkg/coordinator/update_ticket_field.go::readIssueKeyArg`
   runs `issueKeyPattern.MatchString` on the user-supplied
   `issue_key` BEFORE forwarding to `jira.Client.UpdateFields`. The
   M8.1 layer's `validateIssueKey` (`core/pkg/jira/client.go`)
   wraps a malformed key with `%q` in its `ErrInvalidArgs`, so a
   token-shaped `issue_key` would otherwise leak verbatim through
   the wrapped Go-error chain into the M5.6.b auto-reflection
   layer (which writes a `lesson` row with the error string). The
   handler's refusal text NEVER includes the raw value; the agent
   already has the value in its call args and can re-plan without
   it being re-prompted. Pinned via
   `TestUpdateTicketFieldHandler_RefusesMalformedIssueKey_NoEcho`
   (asserts a probe-sentinel `REDACTION_PROBE_TOKEN_M82A_DO_NOT_LOG`
   passed as `issue_key` does NOT appear in the refusal text) plus
   `TestUpdateTicketFieldHandler_RefusesMalformedIssueKey_TableDriven`
   covering nine boundary shapes (lowercase project, missing dash,
   leading-zero numeric tail, embedded slash, etc.). Lesson:
   adapter-layer error formatting that includes user-supplied
   values is a PII leak vector when the adapter is called from a
   higher-level handler that accepts agent-provided arguments;
   the handler MUST pre-validate to keep the value out of the
   adapter's error path.

9. **Defensive shallow-copy of reference-typed handler args before
   adapter dispatch (iter-1 critic M2)** —
   `core/pkg/coordinator/update_ticket_field.go` calls
   `maps.Clone(fields)` before passing to `updater.UpdateFields`.
   Today `*jira.Client.UpdateFields` JSON-marshals the map
   immediately and does not retain it, so the clone is invariance
   not correctness — but a future updater impl (a retry decorator,
   an audit-logging wrapper, an async batcher) that retained the
   map would observe post-call mutations from the LLM-driven
   runtime that re-uses [agentruntime.ToolCall.Arguments]. Mirrors
   the M7.1.c.c "defensive deep-copy of reference-typed config"
   lesson, scoped here to handler-args. The clone is shallow:
   nested mutable values still alias; M8.2.b/c handlers with
   nested-map args need to upgrade. Pinned via
   `TestUpdateTicketFieldHandler_DefensiveCopiesFieldsMap` (asserts
   post-call mutation of the original does not bleed into the
   updater's recorded `gotFields`, AND a sentinel write into the
   updater's map does not bleed back).

10. **Refusal-string namespace: `coordinator: <tool>: <reason>`
    matching the adapter-layer convention** —
    Iter-1 critic m6 flagged that handler refusal strings without a
    package prefix make multi-tool refusal text ad-hoc as M8.2.b/c
    land. Adopting `coordinator: <tool>:` (mirroring `jira: <op>:`
    and `runtime: <signal>`) gives every Coordinator tool a uniform
    parsing convention. Encoded as a `refusalPrefix` const concatenated
    onto every refusal text in the handler so a future `<tool>`
    rename is a one-line change.

## 2026-05-10 — M8.1: Jira adapter — REST API v3, stdlib HTTP, fail-closed field whitelist

**PR**: #127
**Merged**: 06d7c45 on 2026-05-10

### Context

M8.1 introduced `core/pkg/jira` — an Atlassian Cloud REST API v3
adapter shipping the four operations the M8.2 Coordinator manifest
will consume:

- `Search(ctx, jql, opts) (SearchResult, error)` —
  `POST /rest/api/3/search/jql` with cursor pagination.
- `GetIssue(ctx, key, fields) (Issue, error)` —
  `GET /rest/api/3/issue/{key}`.
- `AddComment(ctx, key, body) (Comment, error)` —
  `POST /rest/api/3/issue/{key}/comment`, plain text in / out, ADF
  wrap/unwrap internal to the adapter.
- `UpdateFields(ctx, key, fields) error` —
  `PUT /rest/api/3/issue/{key}` with synchronous whitelist
  enforcement.

This is the first M8 entry; the package layout, naming, and option
patterns mirror the M4.2 slack adapter (the closest sibling) so a
reader fluent in one is fluent in the other. The M8.2 tool layer
(authority matrix, role-scoped action gating) sits ON TOP of, not
INSTEAD of, the transport-level guards this PR introduces.

### Pattern

1. **Stdlib-only HTTP, no third-party Atlassian SDK** — `core/pkg/jira`
   depends only on the Go standard library, mirroring the slack
   adapter's M4.2.a "Stdlib-only" decision (`core/pkg/messenger/slack/doc.go`
   §"Design choices") and the older keepclient discipline (M2.8.a).
   `go.mod` carries no `go-jira`/`andygrunwald` line. Rationale: each
   external SDK ships its own concurrency, retry, JSON-shape, and
   error-wrapping opinions that conflict with the project's own
   conventions; a thin self-rolled client keeps the dependency surface
   small, the tests fast, and the code shape uniform across every
   external integration. Reusable across any future `core/pkg/<svc>`
   adapter (Linear, GitHub, Notion).

2. **API v3 + ADF wrap on the wire, plain text on the seam** —
   The adapter targets Atlassian Cloud REST API v3 (the
   forward-compatible path; v2 is documented-deprecated for
   sub-APIs). Comments require ADF (Atlassian Document Format) on
   the wire; `Client.AddComment` accepts plain text and wraps it in
   a minimal `doc → paragraph[] → text` envelope, splitting on `\n`
   to one paragraph per line. The reverse projection
   (`adfExtractText` in `core/pkg/jira/add_comment.go`) walks the
   ADF tree and joins text leaves with `\n`; complex blocks
   (mentions, codeBlock, tables) are documented best-effort and
   lossy. Pattern: when an external API forces a heavy on-wire
   shape, hide it inside the adapter and expose the simplest seam
   types the consumer needs. M8.2 tool authors never see ADF.

3. **Field whitelist as constructor option, fail-closed default,
   rejection BEFORE the network** — `WithFieldWhitelist(fields ...string)`
   in `core/pkg/jira/options.go` configures the closed set of
   Atlassian field IDs `Client.UpdateFields` may write. A nil/empty
   whitelist refuses ALL writes
   (`core/pkg/jira/client.go::Client.fieldWhitelistContains`).
   Rejection is synchronous and precedes the HTTP exchange — the
   source-grep AC `TestUpdateFields_WhitelistEnforcement_SourceGrep`
   in `core/pkg/jira/update_fields_test.go` pins the order
   (`fieldWhitelistContains` index < `c.do(` index in the source
   file). Rationale: the adapter is the platform-facing security
   boundary; the M8.2 role authority matrix layers on top of, not
   instead of, this transport guard. Reusable for any future
   capability-restricted adapter call (e.g. a future Linear adapter
   with whitelisted state transitions).

4. **Single shared wire decoder for issue payloads** —
   `issueWire.toIssue()` in `core/pkg/jira/get_issue.go` is the
   canonical projection from Atlassian's nested
   `{id, key, fields:{summary, status, assignee, reporter, …}}`
   shape onto the typed `Issue` surface. Both `Client.GetIssue` and
   `Client.Search` (via `searchWire.Issues []issueWire`) share the
   one decoder; no per-call duplication. The raw
   `Fields map[string]json.RawMessage` is preserved verbatim so
   callers driving custom-field extraction never lose fidelity.
   Pattern: when one wire shape feeds multiple operations, hoist the
   decoder onto the wire type itself; the operation files stay
   one-screen tall.

5. **APIError with `kind endpointKind` tag for endpoint-disambiguated
   Unwrap** — `core/pkg/jira/errors.go` ships ONE `APIError` type
   carrying an unexported `kind endpointKind` (generic / issue /
   search). `APIError.Unwrap` consults `kind` so 404 maps to
   `ErrIssueNotFound` only on issue-keyed endpoints and 400 maps to
   `ErrInvalidJQL` only on `/search/jql`. Avoids the per-endpoint
   error-type sprawl an Atlassian SDK would impose — five sentinels
   ([`ErrInvalidAuth`], [`ErrIssueNotFound`], [`ErrFieldNotWhitelisted`],
   [`ErrInvalidJQL`], [`ErrRateLimited`]) cover every M8.2-callable
   discriminant. Reusable when an HTTP API multiplexes the same
   status code onto different semantics across endpoints.

6. **`BasicAuthSource` interface mirrors slack `TokenSource`** —
   `core/pkg/jira/client.go::BasicAuthSource` is a per-call resolver
   shaped exactly like `core/pkg/messenger/slack/client.go::TokenSource`.
   `StaticBasicAuth{Email, Token}` is the trivial static impl for
   tests + bootstrap; production wires the resolver to the M3.4.b
   secrets interface so token rotation is observable. Rationale: the
   M7.1.c.c lesson "per-call resolver beats process-global static"
   for tenant-scoped values applies here too — Atlassian API tokens
   rotate per-account, and a resolver-shaped interface makes
   rotation transparent to call sites.

7. **Layered source-grep ACs over runtime-mock observation** —
   Two layers of negative-AC tests use source-grep + `os.ReadFile`
   mutation discipline (per M7.1.c.a's "hollow runtime-mock AC pin"
   anti-pattern). Iter-1 review surfaced that a single negative
   substring list is brittle (a kv-key rename like `, "auth_kv",
token` would slip past `"token"`-only forbidden lists), so the
   pin layers two scans:
   - File-level forbidden list (narrow, structural): patterns that
     should never appear ANYWHERE in `client.go` outside comments
     (`"api_token"`, `"request_body"`, `"response_body"`,
     `req.Body`).
   - Logger.Log call-site scan (rigorous): `extractLoggerCalls`
     paren-balances every `c.cfg.logger.Log(...)` block in the
     source and asserts NONE contains forbidden value-leak shapes
     (`, token)`, `, email)`, `, body)`, `, req.Body`, `+token`,
     `+email`, etc.). Catches the "rename the kv key, leak the
     value" regression class the file-level scan would miss.

   Companion runtime test
   `TestDo_LoggerRedactsAtTransportErrorPath` uses unique probe
   sentinels (`REDACTION_PROBE_TOKEN_M81_DO_NOT_LOG` /
   `REDACTION_PROBE_EMAIL_M81_DO_NOT_LOG` / a tenant-text probe in
   the API-error envelope) and asserts the runtime logger never
   surfaced any of them. Generic test credentials would only catch
   verbatim-equality regressions; probe-shape sentinels catch
   subtler kv leaks.

   `TestUpdateFields_WhitelistEnforcement_SourceGrep` scans
   `update_fields.go` and asserts the source-order
   `fieldWhitelistContains` index < `c.do(` index — a refactor
   that moves the whitelist check after the network call fails the
   test even though the runtime "happy paths still pass". A small
   `stripGoComments` helper in `client_test.go` removes line +
   block comments so allowed mentions inside docstrings (e.g. the
   "redaction discipline" doc paragraph naming `"token"` and
   `"body"`) do not false-positive.

   Lesson on iteration: the original M8.1 PR shipped only the
   file-level scan; iter-1 review caught that rename-the-key
   defeats it. The fix is layered + probe-sentinels + adversarial
   test scope, not a wider-but-still-substring forbidden list.
