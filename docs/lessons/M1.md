## 2026-05-17 — M1.3.a: `peer.ask` + `peer.reply` request-reply pair

**PR**: [#160](https://github.com/vadimtrunov/watchkeepers/pull/160)
**Merged**: 2026-05-17 (squash → phase2 @ `c2457d7`)

### Context

Shipped the first leaf of the M1.3.\* peer.\* tool family. New
`core/pkg/peer/` package exposes a `Tool` orchestrator with
`Ask(ctx, AskParams) (AskResult, error)` and
`Reply(ctx, ReplyParams) error` built-ins composed over the M1.1.c
`k2k.Lifecycle` (mints conversation + Slack channel), the M1.2
`keepclient.ListPeers` resolver (target ↔ watchkeeper id / role name),
the M1.1.a `k2k.Repository` extended with `AppendMessage` /
`WaitForReply` (cond-var-driven on the in-memory adapter, polling on
the Postgres adapter), and the M3.5.a `capability.Broker.ValidateForOrg`
gate (per-tenant, per-scope deny-by-default). Storage layer extension:
migration `030_k2k_messages.sql` adds the `k2k_messages` table with
RLS keyed off `watchkeeper.org`; matching shell-script RLS gate
extended in-place (blocks w-z).

### Pattern

**Per-call resolver beats process-global static for tenant-scoped
auth tokens.** `peer.Tool` accepts a `CapabilityValidator` interface
satisfied by `*capability.Broker.ValidateForOrg` directly. Production
wiring passes the broker; tests pass a hand-rolled `fakeCapability`
recording calls + decisions. The resolver-shaped seam mirrors the
M3.5.a "per-call resolver over process-global static" discipline:
binding a token to a tenant via `IssueForOrg` + validating via
`ValidateForOrg` keeps multi-tenant deployments safe even if a future
contributor confuses two adjacent calls.

**Capability gate precedes K2K state mutation.** `Tool.Ask` runs the
`capability.Broker.ValidateForOrg(token, "peer:ask", orgID)` check
BEFORE `Lifecycle.Open` mints a conversation row, and BEFORE the
target resolver burns a `keepclient.ListPeers` round-trip. A failed
gate surfaces as `ErrPeerCapabilityDenied` (chained with the
underlying `capability.Err*` sentinel via `fmt.Errorf("%w: %w", ...)`)
without any persistent side effect. `Tool.Reply` mirrors the
discipline: capability gate → `Repository.Get` (existence + open
check) → `Repository.AppendMessage`. Mirrors the M1.1.c "fail-fast
precedes Slack" pattern at the peer-tool layer.

**Two-pass target resolver: uuid first, role second, case-fold the
role.** `Tool.resolvePeer` walks the active-peer list twice. Pass 1
exact-matches `target` against `Peer.WatchkeeperID` so a target that
parses as a watchkeeper id resolves verbatim. Pass 2 case-insensitive
matches against `Peer.Role` so the operator vocabulary "Lead" /
"strategist" / "STRATEGIST" all resolve to the same role. Neither
pass caches — every `Ask` re-fetches the active list so a
freshly-spawned peer resolves on its first try. Test-pinned by
`TestTool_Ask_TargetByRoleResolves` (role lookup hits the resolver
and proceeds to `WaitForReply` rather than failing at
`ErrPeerNotFound`).

**Cond-var-driven `WaitForReply` on the in-memory adapter with an
atomic `timedOut` flag, NOT a fixed-clock deadline check.** The
naive shape "compare `r.now()` against a deadline in the wakeup
loop" deadlocks under a fixture clock (the M1.1.a `fixedClock`
helper) because `r.now()` never advances. The implementation uses a
real `time.NewTimer(timeout)` AND an `atomic.Bool timedOut` flag
flipped by the waker goroutine on timer expiry; the main goroutine
re-checks the flag after each `cv.Wait` so the wakeup loop
terminates even when the repository's stamping clock is frozen.
The flag is also checked BEFORE the first `Wait` so a synchronous-
scan miss against an already-expired timer surfaces
`ErrWaitForReplyTimeout` without parking. Test-pinned by
`TestMemoryRepository_WaitForReply_SinceCursorIsExclusive`
(fixedClock + `since` exactly equal to the reply's `CreatedAt`
forces the timeout path).

**`since` cursor is strictly exclusive (`m.CreatedAt.After(since)`).**
A reply stamped at exactly `since` does NOT satisfy the wait. The
peer-tool layer captures `since := t.now().UTC()` BEFORE
`AppendMessage(direction=request)` so the request's own `CreatedAt`
is necessarily ≥ since — the request is not its own reply.
Test-pinned by `TestMemoryRepository_WaitForReply_SinceCursorIsExclusive`.

**Defensive deep-copy of `Body []byte` at every boundary AND in the
return value.** `Tool.Ask` and `Tool.Reply` each `make + copy` the
caller-supplied `Body` before calling `AppendMessage`; the
in-memory `MemoryRepository.AppendMessage` deep-copies AGAIN before
storing (a double-defense). `Tool.Ask` also deep-copies the returned
`Message.Body` before assembling `AskResult.ReplyBody` so a future
storage-layer optimisation that aliases the held byte slice cannot
bleed up the call chain. Test-pinned by
`TestMemoryRepository_AppendMessage_DefensiveCopyOfBody` +
`TestTool_Ask_DefensiveCopyOfBody`. Mirrors the M1.1.a participants
deep-copy contract.

**Postgres `WaitForReply` ships as a polling fallback with a
configurable interval, NOT as LISTEN/NOTIFY at M1.3.a.** The
M1.3.a AC pins "Postgres uses LISTEN/NOTIFY on a per-conversation
channel, with a polling fallback"; this leaf ships only the polling
fallback (default 100ms, override via
`PostgresRepository.WithPollInterval` for fast tests). The
`Repository.WaitForReply` signature is identical for both impls so
a future LISTEN/NOTIFY layer drops in without a wire-shape break.
The polling SQL is a single `SELECT ... WHERE conversation_id = $1
AND direction = 'reply' AND created_at > $2 ORDER BY created_at ASC
LIMIT 1` so the index `(conversation_id, created_at)` from
migration 030 keeps the planner cost flat as message volume grows.
Expiry uses a real `time.NewTimer(timeout)` rather than a deadline
derived from `r.now()` — a fixed-clock test fixture (or a custom
stamping clock injected for replay debugging) cannot pin the wait
indefinitely (iter-1 codex Major fix). The stamping clock drives
`created_at` only; the wait timeout runs on wall-clock time.

**Message `body` is stored as `bytea`, NOT `text`.** The
`peer.Tool` API documents `Body []byte` as opaque bytes — a future
M1.3.c subscription delivery may legitimately carry a serialised
protobuf or any non-UTF-8 payload. A `text` column would either
reject invalid UTF-8 at the SQL layer or round-trip corrupted
content. Migration 030 ships `body bytea NOT NULL CHECK
(length(body) > 0)`; the Go layer binds `[]byte` directly via pgx
(no `string()` coercion). Iter-1 codex Critical fix.

**`AppendMessage` enforces `status='open'` atomically via
`INSERT ... SELECT FROM k2k_conversations WHERE status='open'`.**
The in-memory adapter already gated on status under the write
lock; the Postgres adapter originally relied on the caller's
preceding `Get` to gate, which is racy — a concurrent
`Lifecycle.Close` between `Get` and `AppendMessage` would slip a
reply into an archived row. The new SQL shape uses an
`INSERT ... SELECT` subquery; when the parent conversation is not
in `status='open'`, the subquery returns zero rows, the INSERT
inserts zero rows, the `RETURNING created_at` produces
`pgx.ErrNoRows`, and the adapter translates that to
`ErrAlreadyArchived`. The peer-tool layer's existing `errors.Is`
branch on `ErrAlreadyArchived` translates to
`ErrPeerConversationClosed` unchanged. Iter-1 codex Major fix.

**Resolver miss MUST short-circuit BEFORE Lifecycle.Open mints a
row.** `Tool.Ask` runs `resolvePeer` immediately after the
capability gate and BEFORE `Lifecycle.Open`. An unknown target
surfaces as `ErrPeerNotFound` without any state mutation; the
alternative ("open the conversation, then fail when the target
doesn't exist") would leave orphan `k2k_conversations` rows that
the M1.7 archive-on-summary reaper would have to handle. Mirrors
M7.1.b's "fail-fast precedes audit/state side effects" discipline.
Test-pinned by `TestTool_Ask_UnknownTargetSurfacesErrPeerNotFound`
(asserts `Lifecycle.Open` calls = 0 after a resolver miss).

**Built-in `Source = "built-in"` constant reserved across the
M1.3.\* family.** `peer.BuiltinSourceName = "built-in"` is exported
so the M1.3.b / .c / .d siblings can stamp the same source name on
their manifests without re-declaring the literal. `peer.ask` /
`peer.reply` ride `toolregistry.DryRunModeScoped` (the lead's-DM
reroute lands in M9.4.c) and a minimum-viable JSON schema. Future
M9.3 signing will fill `Manifest.Signature` at build time; M1.3.a
leaves it empty per the `NoopSignatureVerifier` contract.

**Source-grep AC bans `keeperslog.` and `.Append(` in `ask.go` /
`reply.go`.** The peer-tool layer is the call surface; the K2K
audit taxonomy (`k2k_message_sent` / `k2k_conversation_opened` /
etc.) is M1.4's seam. The tests `TestAsk_NoAuditOrKeeperslogReferences`

- `TestReply_NoAuditOrKeeperslogReferences` read the source file,
  strip line comments, and assert neither token appears in executable
  code. Mirrors M1.1.b's
  `TestChannels_NoAuditOrKeeperslogReferences` and M1.1.c's

  source-grep discipline is uniform across every K2K + peer-tool
  surface.

**`keepclient.Client` consumed via a narrow `Lister` seam, NOT
verbatim.** `Tool.Ask` accepts a `Lister` interface declaring
the single `ListPeers(ctx, ListPeersRequest)` method. Production
wiring satisfies it with `*keepclient.Client`; tests inject a
hand-rolled `fakePeerLister` that records calls + returns a
controllable peer slice without standing up a Keep HTTP server.
Mirrors the M1.1.c `k2k.SlackChannels` "narrow seam at the
consumer" discipline.

## 2026-05-17 — M1.2: `keepclient.ListPeers` + `GET /v1/peers`

**PR**: [#158](https://github.com/vadimtrunov/watchkeepers/pull/158)
**Merged**: 95c99288056a58c3cf2eabf2f3edc1120d28a748 on 2026-05-17

### Context

Shipped the M1.2 leaf that the M1.3.\* peer.\* tool family will consume.
The endpoint `GET /v1/peers` returns every active watchkeeper joined
with its pinned `manifest_version` (description / language / tools)
and parent `manifest` (role display_name). The client method
`keepclient.Client.ListPeers(ctx, ListPeersRequest{Limit int})` mirrors
the wire shape. The leaf deliberately ships NO server-side role /
language / capability filters — M1.3.d's `peer.RoleFilter` applies
those in-memory over the unfiltered list so every M1.3.\* consumer
sees the same "all active peers at this instant" snapshot.

### Pattern

**Server-side endpoint mirrors the `/v1/watchkeepers` list shape, NOT
a freshly-invented surface.** `GET /v1/peers` reuses the
`{items, next_cursor}` envelope, the `?limit=` bounds (default 50,
inclusive cap 200, reject 0/negative/oversize/non-numeric with 400
`invalid_request`), the `ORDER BY created_at DESC` discipline, and
the `writeError` / `writeJSON` helpers. Reuse keeps the
keepclient-side decoder identical to `ListWatchkeepers` and means a
future seek-pagination follow-up lands in one place across both
endpoints. The route is wired identically in `server.go` and
`export_test.go`'s `NewRouterWithRunner` so the test seam stays
exhaustively exercisable without a real pgx pool.

**Filters live client-side per the M1.3.d resolver contract, NOT in
the server query.** M1.3.d's roadmap entry pins `peer.RoleFilter` as
an in-memory filter (`Resolve(ctx, filter RoleFilter) ([]WatchkeeperID, error)`
fans out over `keepclient.list_peers` and applies the filter
in-memory). Reproducing role / language / capability filters at the
SQL layer would duplicate the predicate across the server query and
the client resolver, with two consequences: any predicate drift
(case-sensitivity, whitespace handling) becomes a wire-version break;
and every M1.3.\* consumer would need to know whether "filter X is
server-side or client-side" — a leaky abstraction. The server query
is intentionally minimal: `WHERE wk.status = 'active'` plus INNER
joins. M1.3.d does the rest in Go.

**INNER joins skip unpinned rows; LEFT joins would surface
half-spawned watchkeepers.** The handler INNER-joins `watchkeeper` to
`manifest_version` on `active_manifest_version_id` and to `manifest`
via `mv.manifest_id`. A row with `status='active'` but
`active_manifest_version_id = NULL` (a rare wiring bug, or a future
M3.5 spawn-in-progress state) is SKIPPED rather than surfaced with
empty role / description / capabilities. The test
`TestListPeers_SQLFiltersActiveAndJoinsManifest` pins the JOIN type
verbatim so a future refactor that switched to `LEFT JOIN` for
"defensive null-tolerance" cannot silently regress the contract.

**Manifest join follows `mv.manifest_id`, NOT `wk.manifest_id`
(iter-1 codex P2 fix).** Migration 002's docblock spells out that
the composite-FK invariant "`watchkeeper.active_manifest_version_id`
references a `manifest_version` whose `manifest_id` matches
`watchkeeper.manifest_id`" is DEFERRED to Phase 2 and is not
enforced at the SQL layer today. If a write path (legacy or
malformed) lands a row where the two ids disagree, joining
`manifest` on `wk.manifest_id` would surface a peer whose `role`
(from `m.display_name`) and `description` / `language` /
`capabilities` (from `mv.*`) belong to two different manifests — an
impossible peer record that would mislead the M1.3.a target
resolver. Threading the manifest join through `mv.manifest_id`
keeps every projected row internally consistent: all four
manifest-derived fields come from the SAME manifest version the
watchkeeper is currently bound to. Test-pinned by the SQL-contract
assertion that bans `m.id = wk.manifest_id` verbatim.

**`capabilities` is extracted from `manifest_version.tools` jsonb
server-side, NOT round-tripped as raw JSON.** The wire shape pins
`capabilities` as a string array so consumers do not need to know
the `tools` schema (`[{"name":"<tool>"}]` per the M1 seed
migrations). The extraction lives in `extractCapabilities([]byte)`
with two defensive layers. First, malformed JSON or a non-array top
level returns an empty slice rather than 500ing the whole endpoint
— a manifest authoring bug should not blank out every other
well-formed peer. Second, a tool object whose `name` field is
missing or blank is SKIPPED, not appended as the empty string, so
the M1.3.d capability filter is free of spurious empty-string
matches. Both defences are test-pinned
(`TestListPeers_MalformedToolsYieldsEmptyCapabilities`,
`TestListPeers_SkipsBlankAndMissingToolNames`).

**Empty arrays on the wire are `[]`, NOT `null`.** Both the
top-level `items` envelope and per-row `capabilities` must serialise
as `[]` when empty. Server-side this is achieved by
`make([]peerRow, 0, limit)` plus `make([]string, 0)` in
`extractCapabilities`. Client-side `ListPeers` defends the contract
a second time: after Decode, a nil `Items` is normalized to an empty
slice and every per-row nil `Capabilities` is normalized likewise.
The defensive pair protects M1.3.d's `for ... range` loops from a
nil-dereference panic if a future server-side bug regresses to
emitting `null` for either shape. Test-pinned both ends
(`TestListPeers_EmptyItemsArrayNotNull`,
`TestListPeers_EmptyCapabilitiesEmittedAsArray`,
`TestClient_ListPeers_EmptyItemsNonNil`,
`TestClient_ListPeers_CapabilitiesNonNil`).

**`availability` is a string enum reserved for future M1.5 / M1.6
extensions.** Phase 2 ships a single value
(`peerAvailabilityAvailable = "available"`) because the SQL filter
`status = 'active'` is the only availability signal we have. The
column is exported so that M1.5 (token-budget exhaustion →
throttled) and M1.6 (escalation pending → escalated) can introduce
new enum values without a wire-shape break. Both ends define the
constant (`server.peerAvailabilityAvailable` plus
`keepclient.PeerAvailabilityAvailable`) so consumers can compare
with `==` rather than against a string literal, mirroring how
M1.1.a's status enum is exposed.

**Client `Limit: 0` omits the query parameter, NOT `?limit=0`.** A
naive `strconv.Itoa(0)` would put `?limit=0` on the wire, which the
server's `invalid_request` validator rejects as out-of-range. The
branch `if req.Limit > 0` gates the `q.Set("limit", ...)` call so
the zero-value Request struct lands the same wire shape as "do not
send limit at all", letting the server apply its default 50.
Test-pinned by `TestClient_ListPeers_ZeroLimitOmitsQueryParam` so a
future contributor "simplifying" the conditional cannot regress the
contract.

**Defensive client-side bounds mirror the server's bounds.** The
client method rejects `Limit < 0` and `Limit > 200` synchronously
via `ErrInvalidRequest` before any network round-trip, mirroring
`ListWatchkeepers`'s `maxWatchkeeperListLimit` discipline. This
documents the client contract in-code (a caller reading
`read_peers.go` sees the cap without consulting the server source)
and saves a 400 round-trip on obvious bugs.

**Source-grep AC bans `keeperslog.` and `.Append(` in
`handlers_peers.go`.** This handler is a read projection — the M1.4
audit subscriber owns the K2K event taxonomy (`k2k_message_sent`,
`k2k_conversation_opened`, etc.). The test
`TestHandleListPeers_NoAuditOrKeeperslogReferences` reads the source
file, strips line comments, and asserts neither token appears in
executable code. Mirrors M1.1.b's
`TestChannels_NoAuditOrKeeperslogReferences` and M1.1.c's
`TestLifecycle_NoAuditOrKeeperslogReferences` so the M1-family
source-grep discipline is uniform across every read + write surface.

**Forward-compatible wire field `next_cursor` ships as `null`.** The
response envelope's `next_cursor *string` is reserved for a future
seek-pagination follow-up — the same forward-compat pattern as
`/v1/watchkeepers`. Today every response carries `null` so a future
server-side pagination cursor can land without a wire-shape break;
clients that match against `resp.NextCursor != nil` keep working
across the upgrade.

## 2026-05-17 — M1.1.c: `wk channel reveal` CLI + K2K lifecycle wiring

**PR**: [#157](https://github.com/vadimtrunov/watchkeepers/pull/157)
**Merged**: f5407654be831e71d218cc25a80de0f98a61dd5e on 2026-05-17

### Context

Stitched M1.1.a's K2K conversation domain to M1.1.b's Slack channel
primitives behind one lifecycle entry point in `core/pkg/k2k/`. New
seams: `k2k.SlackChannels` (a narrow three-method interface satisfied
by `*slack.Client`), `k2k.Lifecycle` (the
`Open(ctx, OpenParams) (Conversation, error)` /
`Close(ctx, id, reason) error` composer), `k2k.DeriveChannelName(id)`
(the `k2k-<8-hex-prefix>` derivation), and
`k2k.Repository.BindSlackChannel(ctx, id, channelID)` (the third
write method on the persistence seam, atomic vs `Close` + duplicate-
bind). The `wk channel reveal --user <slack-user-id>
<conversation-id>` CLI lands behind a `channelResolver`
test-injection seam; the production composition opens a pgxpool +
runs `db.WithScope` to look up the bound channel id under the
operator's org-scoped RLS claim, then forwards to
`*slack.Client.RevealChannel`. Compile-time wiring assertion lives in
`core/internal/keep/k2k_wiring/` (mirroring `retire_wiring` /
`approval_wiring`).

### Pattern

**Narrow three-method `SlackChannels` interface, NOT
`*slack.Client` everywhere.** The M1.1.b extension surface added FOUR
methods to `*slack.Client` (`CreateChannel`, `InviteToChannel`,
`ArchiveChannel`, `RevealChannel`); the lifecycle layer consumes only
the first three because `RevealChannel` is owned by the
`wk channel reveal` CLI — there is no K2K Open() / Close() call that
reveals. Encoding only the three methods the lifecycle needs keeps
the test fake under 80 lines and prevents a future refactor of
RevealChannel's signature from forcing every lifecycle test to update
its mock. The compile-time assertion `var _ k2k.SlackChannels =
(*slack.Client)(nil)` lives in `core/internal/keep/k2k_wiring/wiring.go`
so a missing method (renamed Slack-side, added k2k-side) fails the
build at the wiring boundary, not at the first production run.

**`BindSlackChannel` is a SEPARATE Repository method, NOT a
`SlackChannelID` field on `OpenParams`.** The conversation ID has to
exist BEFORE the Slack channel name can be derived (the name is
`k2k-<id-prefix>`), and the Slack channel cannot be provisioned
before the conversation row mints the id. So Open() is two-phase:
`Repo.Open` mints the row + id → lifecycle derives the name + calls
Slack → `Repo.BindSlackChannel(id, channel_id)` stamps the column.
The alternative — accepting a `SlackChannelID` on `OpenParams` —
would force every caller to pre-derive the channel name + call Slack,
which inverts the lifecycle layer's reason for existing. The bind
method uses the same atomic conditional-UPDATE shape as
`Repo.Close` (`WHERE id = $1 AND status = 'open' AND
slack_channel_id IS NULL`) so a concurrent duplicate-bind matches zero
rows and the follow-up probe distinguishes `ErrConversationNotFound`
/ `ErrAlreadyArchived` / `ErrSlackChannelAlreadyBound` without
parsing the error chain.

**Bind precedes invite (iter-1 codex Major fix).** The original
`Lifecycle.Open` order was `CreateChannel → InviteToChannel →
BindSlackChannel`. Iter-1 codex review caught a race window: a
concurrent `Lifecycle.Close` racing between `CreateChannel` and the
bind step would archive the row while `Close`'s "skip ArchiveChannel
when SlackChannelID empty" branch left the live Slack channel
orphaned and unreachable from any persisted state. The fix swaps to
`CreateChannel → BindSlackChannel → InviteToChannel`: the row carries
the bound channel id from the moment any reader sees the row, so a
concurrent `Close` after `CreateChannel` correctly archives both the
row AND the Slack channel. `InviteToChannel` can still fail
post-bind, but the row+channel pair is consistent at that point — a
follow-up `Close` archives both correctly. The test
`TestLifecycle_Open_BindPrecedesInvite` pins the ordering via a
probing fake `InviteToChannel` that re-queries the repo from inside
the call stack and asserts the SlackChannelID is already bound at
invite time.

**Trust model is operator-trusted by env, mirroring the existing
`wk` Keep-token pattern (iter-1 codex Major fix).** Iter-1 codex
review raised that `wk channel reveal` forges an org-scoped
`auth.Claim` from `WATCHKEEPER_OPERATOR_ORG_ID` without a verified
credential, so an operator with DSN access could pivot tenants by
swapping the env var. The trust model is documented explicitly in
`run_channel_prod.go`'s docblock: `wk` is the operator-trusted CLI
per `core/cmd/wk/main.go`'s package doc — every Keep-backed
subcommand already trusts `WATCHKEEPER_OPERATOR_TOKEN` from env.
M1.1.c's `wk channel reveal` does not go through Keep (M1.1.c
intentionally ships no K2K HTTP endpoint — that's an M1.4+
follow-up), so it builds its own org claim from env. Tracked as a
follow-up: route `wk channel reveal` through a Keep HTTP endpoint
that derives the claim from the operator token, mirroring the
existing `wk inspect` / `wk list` pattern. The current posture is no
weaker than the existing operator-trusted `wk` surfaces.

**Resolver returns Status alongside ChannelID; CLI fail-fasts on
archived rows (iter-1 codex Minor fix).** The original `channelResolver`
interface returned only the `slack_channel_id` string, so the CLI
could not distinguish "open row with bound channel" from "archived
row with bound channel" — every reveal on an archived row burned a
Slack `conversations.invite` call before surfacing Slack's
`is_archived` code. The fix widens the seam to
`ResolveSlackChannel(ctx, id) (channelID, status string, err error)`
so the CLI gates on `status == "archived"` and returns exit 1 with a
domain-level diagnostic without contacting Slack. Mirrors the M1
fail-fast-precedes-Slack discipline from the M1.1.b lesson.

**`Lifecycle.NewLifecycle` panics on nil `Repo` / `Slack` deps.**
Mirrors the saga-step `New<Step>Step` and M1.1.a's
`NewPostgresRepository` / `NewMemoryRepository` discipline. The panic

diagnostic + exits with usage code 2 rather than stack-tracing.

**Fail-fast precedes Slack side effects.** `Lifecycle.Open`
short-circuits on a pre-cancelled context BEFORE
`Repo.Open` (which would otherwise mint a useless row whose
counterpart Slack channel never existed). `Lifecycle.Close`
short-circuits on `uuid.Nil` and `Repo.Get` BEFORE
`Slack.ArchiveChannel` so an unknown id never burns a Slack
rate-limit token. The pattern matches M1.1.b's "whitespace-trim +
fail-fast precedes Slack on every entry point" — Slack calls are the
expensive boundary; cheap validators run first.

**Orphan-row policy: failed Slack call leaves the row open with
empty `slack_channel_id`.** If `Slack.CreateChannel` returns an
error after `Repo.Open` succeeds, the lifecycle layer DOES NOT
auto-archive the orphan row. Auto-archive would couple the lifecycle
to the M1.4 audit subscriber's "no half-emitted lifecycle event"
discipline AND double-archive on a successful retry from the same
caller. Instead, the orphan is documented in the godoc + the
`Lifecycle.Close` path is orphan-aware: when the row has empty
`SlackChannelID`, Close skips the `ArchiveChannel` call and still
transitions the row to `archived`. The M1.7
archive-on-summary writer is the canonical orphan reaper.

## 2026-05-17 — M1.1.b: Slack adapter channel primitives

**PR**: [#156](https://github.com/vadimtrunov/watchkeepers/pull/156)
**Merged**: a3aa91e4a31696d788f2823c82bb859151ffd0bd on 2026-05-17

### Context

Extended `core/pkg/messenger/slack/` with the four channel-lifecycle
primitives the M1.1.c K2K wiring layer will call:
`CreateChannel(name, isPrivate)`, `InviteToChannel(channelID, userIDs)`,
`ArchiveChannel(channelID)`, `RevealChannel(channelID, userID)`. All
four are methods on `*slack.Client` (NOT on the portable
`messenger.Adapter` interface) — mirroring the
`OpenIMChannel` / `ConversationsHistory` / `CreateApp` extension
pattern that lets a Slack-specific seam ride on the same Client
without bloating the cross-platform Adapter surface. M1.1.b
deliberately ships ONLY the capability surface — no K2K consumer
calls (M1.1.c), no audit emission, no token-budget enforcement.

### Pattern

**Slack-extension methods land on `*Client`, NOT on `messenger.Adapter`.**
The portable Adapter interface fixed the six-method shape in M4.1; a
new Slack-specific channel primitive would force every future
non-Slack adapter (Discord, IRC, SMS, …) to either implement a
Slack-shaped call or return `messenger.ErrUnsupported`. The K2K
domain in `core/pkg/k2k/` is by design Slack-only at this milestone,
so M1.1.c will consume `*slack.Client` directly (per-call resolver or
typed dependency) rather than route through `messenger.Adapter`. The
compile-time assertion `var _ messenger.Adapter = (*Client)(nil)` in
`adapter_assertion_test.go` still holds — the new methods are
additive Slack-side extensions.

**Idempotent `CreateChannel` via opportunistic `conversations.list`
recovery — kind-filtered AND archive-filtered (iter-1 codex P2 fix).**
Slack's `conversations.create` does NOT echo the existing channel id
on a `name_taken` collision — it surfaces `error: "name_taken"` and
the caller must look up the existing id via `conversations.list`. The
implementation issues the create call first, catches
`ErrChannelNameTaken` via `errors.Is`, then paginates
`conversations.list` filtered to the same channel-kind (private vs
public) as the original request. Two filters apply to the page scan:
(1) kind-aware — a private create that collides with a same-name
PUBLIC channel does NOT silently bind to the public id (which would
flip the K2K conversation's privacy posture); (2) archive-aware — an
ARCHIVED same-name same-kind hit is SKIPPED, because Slack still
rejects `conversations.create` with `name_taken` against an archived
holder, but binding the K2K consumer to an archived channel id would
surface as `ErrIsArchived` on the very next `InviteToChannel` call.
When BOTH archived and non-archived same-name hits exist on the same
page (a workspace pattern after channel-recycling), the non-archived
id wins regardless of page order. When the recovery list call returns
zero NON-ARCHIVED matches the composite error carries BOTH the
original `ErrChannelNameTaken` sentinel AND the diagnostic "name X
not found among non-archived private_channel entries" so the caller
can pick a fresh name. The recovery path burns a tier-2 rate-limit
token on `conversations.list`, documented in the per-method
doc-block.

**Idempotent close / invite via Slack-side "already in target state"
codes — with a batch-size guard on `already_in_channel` (iter-1 codex
P1 fix).** `ArchiveChannel` translates `already_archived` to nil
unconditionally — Slack's archive endpoint takes a single channel id,
there is no all-or-nothing concern. `InviteToChannel` translates
`already_in_channel` to nil ONLY when the post-trim batch size is 1
(single-user invite OR a slice that trimmed down to one entry).
Slack's `conversations.invite` is documented as all-or-nothing per
call: when ANY user in a multi-user batch is already a member, Slack
returns `already_in_channel` AND does NOT invite the OTHER listed
users. Silencing that to nil would let the K2K Open() fan-out report
success while leaving some bots outside the channel. The post-trim
count rides out of `joinUserIDs(ids) (csv string, count int)` so the
trimming and the safety decision share a single source of truth.
`RevealChannel` is always single-user by construction so it always
takes the safe translation path. The sentinels
(`ErrAlreadyArchived`, `ErrAlreadyInChannel`) are still EXPORTED for
callers using the low-level `Client.Do` surface — the per-method
idempotent translation is a M1.1.b boundary contract, not a
package-wide rewrite.

**`Client.RevealChannel` is a thin single-user wrapper over
`InviteToChannel`, NOT a separate Slack endpoint.** Both ride
`conversations.invite` on the wire. Splitting the Go surface mirrors
the M1.1 AC vocabulary (`InviteToChannel(channelID, userIDs)` for the
bot-fan-out at K2K Open(); `RevealChannel(channelID, userID)` for the
human-opt-in CLI) so the M1.1.c lifecycle layer keeps a 1:1
call-site → primitive mapping. RevealChannel delegates to
InviteToChannel with a single-element slice, which means the
`liftInviteError` table stays in one place and every documented Slack
code lifts identically across the two entry points.

**Whitespace-trim + fail-fast precedes Slack on every entry point.**
Mirrors the M1.1.a `participants` validator (`strings.TrimSpace`
before accepting). `CreateChannel("   ", true)` returns
`ErrInvalidChannelName` synchronously without burning a tier-2
rate-limit token on `conversations.create`. `InviteToChannel(
chan, ["   ", ""])` returns `messenger.ErrUserNotFound` synchronously
when every entry trims to empty. `RevealChannel(chan, " ")` likewise
fails-fast with `messenger.ErrUserNotFound`. The shared
`joinUserIDs` helper drops whitespace-only entries while joining the
CSV that `conversations.invite` expects, so a `[" U1 ", "", "U2"]`
input lands on the wire as `"U1,U2"`.

**Slack error code lifting goes through `APIError.Unwrap` + per-method
`liftXxxError`.** New sentinels (`ErrChannelNameTaken`,
`ErrInvalidChannelName`, `ErrAlreadyArchived`, `ErrIsArchived`,
`ErrAlreadyInChannel`, `ErrCantInviteSelf`) ride on
`APIError.Unwrap`'s switch so `errors.Is(err, ErrXxx)` works on every
[*APIError] returned from `Client.Do`, regardless of which
per-method lift was applied. The per-method lifts
(`liftCreateChannelError`, `liftInviteError`, `liftArchiveError`,
`liftListError`) add Slack-method context and translate Slack codes
that map to PORTABLE `messenger.*` sentinels (e.g. `channel_not_found`
→ `messenger.ErrChannelNotFound`) so adapter callers can match
without importing the slack package. Mirrors `liftHistoryError` /
`liftOpenIMError` from M4.2.b/c.

**Rate-limiter tier registry extended in-place.** Added three new
entries to `defaultMethodTiers`: `conversations.create` (Tier2),
`conversations.invite` (Tier3), `conversations.archive` (Tier2). The
existing tier-bucket bookkeeping handles the new methods without code
changes — the registry is the single source of truth, mirrored at
the call-site by the `conversationsXxxMethod` constants that pin the
URL path and the registry key together via the compiler.

**Source-grep AC bans `keeperslog.` + `.Append(` in `channels.go`.**
M1.1.b is pure adapter surface; audit emission and token-budget
accounting are M1.1.c's concern. The test
`TestChannels_NoAuditOrKeeperslogReferences` reads the source file
verbatim, strips line comments, and asserts neither token appears in
executable code. Mirrors the M7.1.c.\* source-grep AC discipline so a
future contributor adding audit emission inside the primitive layer
trips a fast-failing test before the call-site change can ride out of
review.

## 2026-05-17 — M1.1.a: K2K conversation domain + Postgres persistence

**PR**: [#155](https://github.com/vadimtrunov/watchkeepers/pull/155)
**Merged**: f6a28f445304037ea07247c6ec6915a6f0edb981 on 2026-05-17

### Context

Established the K2K conversation domain at `core/pkg/k2k/`: the
`Conversation` projection, a `Repository` interface seam, an
in-memory `MemoryRepository` (production-ready for dev / smoke loops
and the sole test surface), and a Postgres-backed `PostgresRepository`
wired against a per-call `Querier` (typically the `pgx.Tx` from
`core/internal/keep/db.WithScope`). Matching migration `029_k2k_conversations.sql`
adds the table with FORCE-RLS keyed off `watchkeeper.org` (the same
GUC migration 013 introduced for manifest tenancy), per-FK + partial
indexes, and CHECK constraints on the status enum + non-empty subject
/ participants invariants. M1.1.a deliberately ships ONLY the storage
layer — no Slack adapter calls (M1.1.b), no lifecycle wiring (M1.1.c),
no audit emission (M1.4), no budget enforcement (M1.5).

### Pattern

**In-memory + Postgres Repository pair shipped together, but only the
in-memory impl carries Go-side tests.** Prior art (M7.1.a saga DAO,
M2.1.b `keepers_log`, M9.4.b proposal store) settled on "ship in-memory
DAO + tests with consumer; Postgres adapter is a follow-up paired with
consumer wiring". M1.1.a's AC explicitly demands "in-memory + Postgres
impls" — resolved by shipping a thin Postgres adapter behind a
compile-time `var _ Repository = (*PostgresRepository)(nil)` assertion
plus extending `scripts/migrate-schema-test.sh` with RLS assertions for
the new table. The Go-side test surface stays entirely on the in-memory
store (where the lifecycle, defensive-copy, concurrent-increment, and
per-org isolation behaviours are exhaustively pinned); the Postgres
adapter's correctness is exercised end-to-end by the M1.1.c lifecycle
wiring + the shell-script RLS gate, not by Go test-container
scaffolding at the leaf-package boundary.

**PostgresRepository takes a per-call `Querier`, NOT a `pgxpool.Pool`
(iter-1 fix).** Iter-1 review (codex P1) caught a load-bearing
correctness issue: wiring the adapter over a `*pgxpool.Pool` would
defeat RLS entirely. The pool checks out an arbitrary backend
connection per statement, so a `SET LOCAL ROLE wk_*_role` issued on
connection A is invisible to the next statement that lands on
connection B. Every Open / Get / List would either return zero rows
(USING NULL) or be rejected at WITH CHECK. The fix: introduce a
narrow `Querier` interface (`Query`, `QueryRow`, `Exec`) satisfied
by both `pgx.Tx` and `pgxpool.Pool`, and require the production
wiring to construct the adapter over the `pgx.Tx` obtained from
`core/internal/keep/db.WithScope`. This mirrors the same per-call
seam discipline as `keep/server/handlers_read.scopedRunner`. The
package doc-block + adapter doc-block now spell out the wiring
contract explicitly so the M1.1.c lifecycle helper cannot regress.

**`correlation_id` is `uuid` on both sides of the seam (iter-1 fix).**
Iter-1 review (codex P2, critic confirm) caught a contract drift:
the Go struct + `OpenParams` declared `CorrelationID` as `string`
("opaque id"), but the SQL column was `uuid NULL`. A caller passing a
non-UUID string (e.g. `"saga-7"`) would pass in-memory tests and fail
in production with `invalid input syntax for type uuid`. Fix: change
the Go field + param type to `uuid.UUID`, use `uuid.Nil` as the
"unset" sentinel, and pass through pgx as a `*uuid.UUID` so
`uuid.Nil` maps to SQL NULL via the standard nilable-pointer idiom
(no `NULLIF($n, '')::uuid` trickery). The contract is now symmetric
across both impls.

**Nullable columns scanned into pointer targets, not magic-value
sentinels (iter-1 fix).** Iter-1 critic flagged a hack — using
`COALESCE(closed_at, '0001-01-01'::timestamptz)` plus a year-greater-
than-1 probe to detect SQL NULL. The fix uses pgx's standard
nilable-type idiom: `*time.Time`, `*uuid.UUID`, and `*string` scan
targets. A NULL column scans as a nil pointer, dereferenced into the
Conversation field only when non-nil. The projection logic is
centralised in a private `scanConversation(pgx.Row)` helper used by
both `Get` and `List` so a future schema delta touches one place.

**FORCE-RLS keyed off the `watchkeeper.org` GUC, identical shape to
migration 013.** The K2K table inherits the manifest-tenant RLS
pattern exactly — `nullif(current_setting('watchkeeper.org', true), '')::uuid`
in both USING and WITH CHECK so an unset GUC evaluates to SQL NULL
and the policy fails closed (zero rows visible, no INSERT permitted).
Three per-role policies (`wk_org_role` / `wk_user_role` /
`wk_agent_role`) mirror the row-set the manifest policies already grant
through. This keeps the per-tenant isolation contract uniform across
every tenant-scoped table in the schema; future K2K consumers can rely
on `core/internal/keep/db.WithScope` to establish the GUC before
calling the Postgres adapter, with no per-call tenant threading.

**Atomic `IncTokens` via `UPDATE ... WHERE status = 'open' RETURNING
tokens_used`.** A concurrent increment-vs-close race is handled by
Postgres row-level locking — the conditional UPDATE matches zero rows
when the row has transitioned to `archived` and the adapter surfaces
`ErrAlreadyArchived` after a follow-up status probe. The in-memory
adapter mirrors the contract by holding the write-lock for the full
read-modify-write under one acquisition (a 16-goroutine concurrency
test pins the "no lost update" invariant; a naive RLock-read +
Lock-write split would silently lose increments). The
`RETURNING tokens_used` shape lets the M1.5 enforcement layer compare
against `token_budget` without a follow-up `Get` round-trip.

**Defensive deep-copy of `Participants` on every boundary.** The
in-memory adapter clones the caller-supplied participants slice on
`Open`, returns clones on `Get` / `List`, and tests pin both directions
of the contract (caller mutation post-`Open` must not bleed; caller
mutation of a `Get` result must not bleed). Mirrors `cloneProposal` in
`core/pkg/approval/proposalstore.go`. The Postgres adapter inherits the
contract structurally — pgx serializes the slice once at QueryRow
time, so a caller-side mutation between `Open` returning and pgx
serializing cannot bleed, but a future contributor adding a code path
that holds the slice across the call must apply the same defensive
copy.

**Fail-fast validation precedes mutex / persistence in every `Open` /
`IncTokens` path.** The validator chain runs BEFORE any lock acquire
and BEFORE any SQL roundtrip. Mirrors the M7.1.b "fail-fast precedes
audit/state side effects" discipline. Validators surface typed
sentinels (`ErrEmptyOrganization`, `ErrEmptySubject`,
`ErrEmptyParticipants`, `ErrInvalidTokenBudget`,
`ErrInvalidTokenDelta`, `ErrAlreadyArchived`) so callers can branch on
`errors.Is` rather than parsing the error chain. The whitespace-only
participant check (`strings.TrimSpace`) prevents a `"   "` entry from
smuggling past the empty-check and into a downstream `InviteToChannel`
call.

**Shell-script RLS gate extended in-place, not duplicated.** The new
RLS assertions for `k2k_conversations` (cross-tenant SELECT
invisibility, cross-tenant INSERT rejection, FK violation, unset-GUC
fail-closed, CHECK constraint enforcement on participants / subject /
status) land in `scripts/migrate-schema-test.sh` as blocks (q) through
