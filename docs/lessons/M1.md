## 2026-05-17 — M1.4: Standard K2K event taxonomy in `keeperslog`

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

Shipped the M1.4 ROADMAP item: the six closed-set K2K event-type
constants (`k2k_conversation_opened`, `k2k_conversation_closed`,
`k2k_message_sent`, `k2k_message_received`, `k2k_over_budget`,
`k2k_escalated`) plus a typed `audit.Emitter` seam the K2K lifecycle
layer (`k2k.Lifecycle.Open` / `.Close`) and the peer.\* tool layer
(`peer.Tool.Ask` / `.Reply`) call to publish keeperslog rows without
importing `keeperslog` directly (the M1.3.a/b/c/d source-grep AC bans
`keeperslog.` and `.Append(` in the call-site files). New package
`core/pkg/k2k/audit/` ships the constants, the `Emitter` interface
with four typed event-shaped methods (`EmitConversationOpened` /
`EmitConversationClosed` / `EmitMessageSent` / `EmitMessageReceived`),
the closed-set event structs (`ConversationOpenedEvent` /
`ConversationClosedEvent` / `MessageSentEvent` /
`MessageReceivedEvent`), and the production `Writer` wrapping a
`keeperslog.Writer`-shaped `Appender`. The M1.5 / M1.6 leaves will
emit `k2k_over_budget` / `k2k_escalated` against the same constants
when they ship their respective enforcement / escalation seams; M1.4
defines the constants now so the taxonomy is a single source of
truth.

### Pattern

**Audit emission flows through a typed seam, never through inlined
`keeperslog.Append`.** Every M1.3.\* prior-art entry pins a
source-grep AC that bans `keeperslog.` and `.Append(` in the
peer.\* tool files; M1.1.c pins the same ban on `k2k/lifecycle.go`.
M1.4 lands the legitimate audit emitter as `core/pkg/k2k/audit`
behind an `Emitter` interface. The consumer files
(`k2k/lifecycle.go`, `peer/ask.go`, `peer/reply.go`) gain an
`audit` import (NOT `keeperslog`) and call typed methods
(`EmitConversationOpened`, `EmitMessageSent`) so the bans stay
green. The source-grep test files
(`TestAsk_NoAuditOrKeeperslogReferences`,
`TestReply_NoAuditOrKeeperslogReferences`,
`TestLifecycle_NoAuditOrKeeperslogReferences`) keep working because
they grep for `keeperslog.` and `.Append(` substrings — the
peer-tool layer never types either.

**`audit.Emitter` is an OPTIONAL `Deps` field in both
`k2k.LifecycleDeps` and `peer.Deps`.** Mirrors M1.3.c's `EventBus`
optional-dep discipline: every M1.1.c-era / M1.3.a-era wiring path
(production + tests) stayed valid because `Auditor: nil` is the
no-op default. Adding `Auditor` as a fifth REQUIRED dep in
`peer.NewTool` would have broken every existing test fixture (16
files) without buying back any safety — an audit miss is an
observability bug, not a correctness bug. `TestPeerAuditorOptional`

- `TestLifecycle_Open_NilAuditorIsNoop` pin this.

**Audit emission failure NEVER short-circuits the caller's
success.** Both `Lifecycle.Open` / `.Close` and `Tool.Ask` /
`.Reply` discard the audit emit's error via `_, _ =
auditor.Emit*(...)`. The persisted K2K state (the row, the Slack
channel, the message append) is the load-bearing surface; an audit
write is observability. Propagating an audit failure would (a)
mask the underlying success — the caller's downstream code thinks
the K2K state was not mutated when it was — and (b) introduce a
fragility on a non-load-bearing dep (Keep being momentarily
unreachable would block every peer tool call). The
`audit.Writer`'s underlying `keeperslog.Writer` already logs the
emit failure via `WithLogger`; the diagnostic chain stays intact
without coupling the call surface to the audit surface.
`TestLifecycle_Open_AuditEmitFailureDoesNotPropagate` +
`TestTool_Ask_AuditEmitFailureDoesNotPropagate` +
`TestTool_Reply_AuditEmitFailureDoesNotPropagate` pin this on every
emit site.

**Conversation lifecycle owns
`k2k_conversation_opened`/`k2k_conversation_closed`; peer-tool
layer owns `k2k_message_sent`/`k2k_message_received`.** The
roadmap AC enumerates the six event types but does not pin the
emission seams. The natural split is by ownership: the lifecycle
layer mints + archives conversations (so it emits the lifecycle
events); the peer-tool layer appends messages (so it emits the
message events). `peer.Tool.Close` therefore performs NO audit
emit at the peer layer — the underlying `k2k.Lifecycle.Close`
already emits `EventConversationClosed` for it. This split keeps
each layer's responsibility narrow and prevents a double-emit
when an integration test exercises both layers.

**Reply emits the sender side; Ask emits the recipient side of
the same reply.** `peer.Tool.Reply` is called by the replier and
emits `EventMessageSent(direction=reply)` from the replier's
viewpoint; `peer.Tool.Ask` blocks on `WaitForReply` and, when its
wait unblocks, emits `EventMessageReceived(direction=reply,
sender=replier, recipient=acting-asker)` from the asker's
viewpoint. A subscriber joins the two halves on `conversation_id`

- `message_id`. This matches the dual-perspective discipline an
  audit consumer needs (the audit row records BOTH "I sent" and
  "someone received" for the same wire event) without forcing the
  replier or the asker to know about the other side's audit
  emission. Test-pinned at
  `TestTool_Ask_EmitsMessageSentForRequestAppend` (asserts one
  `MessageSent` row + one `MessageReceived` row for a complete
  round-trip).

**Body bytes NEVER reach the audit payload (PII discipline).**
Every M1.3.\* prior-art entry pins "no payload bytes ever reach
the returned error or any diagnostic surface". M1.4 extends this
to the audit payload: `MessageSentEvent` / `MessageReceivedEvent`
carry `MessageID` + `ConversationID` + `Direction` + `CreatedAt`

- sender/recipient ids — but NEVER the body bytes. The persisted
  `k2k_messages.body` column is the source of truth; an audit
  consumer needing the body content reads the message row directly,
  joining on `message_id`. The `audit.Writer` constructs the
  payload from a closed-set `map[string]any` of typed primitives
  (strings, uuid.UUID.String(), int64) so a future contributor
  adding a body field there fails code review with a visible diff.
  `TestEmitMessageSent_HappyPath` pins the absence of `body` /
  `subject` / `payload` keys; `TestEmitConversationOpened_HappyPath`
  pins the absence of `role` / `display_name` (no participant
  metadata leakage at the conversation-opened boundary either).

**`close_summary` is NOT carried in the audit payload.** The
M1.3.b `peer.Close` flow writes `Conversation.CloseSummary` as
operator-supplied free text on the row. The
`EventConversationClosed` payload carries `CloseReason` (the
stable closed-set code like `peer.close`) but NOT `close_summary`
— an audit consumer needing the summary joins on the persisted
row. Two reasons: (a) summaries can be lengthy + arbitrary
operator text, inflating every audit row; (b) the M1.7
archive-on-summary writer's responsibility is to cross-link
summary content into a Keep knowledge chunk, not to duplicate it
into every closed-event audit row.
`TestEmitConversationClosed_HappyPath` asserts the absence of
`close_summary` from the payload map.

**Defensive deep-copy of `Participants` at the audit-payload
boundary.** The `ConversationOpenedEvent.Participants` slice
flows verbatim from the persisted `Conversation.Participants` (a
deep copy made by the repository); the `audit.Writer` makes a
SECOND copy via `cloneStrings(evt.Participants)` before stamping
it onto the `map[string]any` payload. Two copies, not redundant:
the first protects the in-flight emit from caller-side mutation;
the second protects the recorded payload from the recorded-event
slice (the same `[]string` referenced from two places in the
emit's local scope). Mirrors the M1.3.d "two-copy" body
discipline on broadcast.
`TestEmitConversationOpened_DefensiveCopyOfParticipants` +
`TestLifecycle_Open_DefensiveCopyOfParticipantsToAuditPayload`
pin this from both ends.

**`EventOverBudget` + `EventEscalated` ship as constants without
emit helpers.** The M1.4 ROADMAP AC enumerates all six event
types but the corresponding emission sites belong to M1.5 (budget
enforcement) and M1.6 (escalation saga). Shipping the constants
now in `core/pkg/k2k/audit/events.go` keeps the taxonomy as a
single source of truth — M1.5/M1.6 will join the taxonomy rather
than minting a parallel string. `TestEventTypes_Closed` asserts
the six-element shape AND defensive-copies the slice on every
`EventTypes()` call so a caller mutating the slice does not bleed
back into the package's view.

**Lifecycle.Close re-`Get`s the row to populate `closed_at` from
the persisted column.** Stamping the audit event's `ClosedAt`
from a wall-clock `time.Now()` would have introduced a tiny
skew between the audit row and the persisted column (multi-ms
on a busy box). Re-reading the row via `Repo.Get` after the
`Repo.Close` transition guarantees the audit `ClosedAt` is
byte-identical to the column. A failed `Get` on the post-close
read is logged but does NOT short-circuit the audit emit — the
emit is skipped (no audit row is better than a wrong audit row).
Test-pinned at `TestLifecycle_Close_EmitsConversationClosedAuditRow`.

**Saga-replay double-close emits NO duplicate
`EventConversationClosed`.** A second `Lifecycle.Close` on an
already-archived row returns `ErrAlreadyArchived` from
`Repo.Close`; the emit branch is GATED behind the success of
`Repo.Close`, so the second close skips emission. The
race-winner's emit owns the persisted audit row; emitting again
would inflate every saga-replay path's audit chain.
`TestLifecycle_Close_NoEmitOnDoubleClose` pins this.

## 2026-05-17 — M1.3.d: `peer.broadcast` fan-out + `peer.RoleFilter`

**PR**: [#168](https://github.com/vadimtrunov/watchkeepers/pull/168)
**Merged**: 2026-05-17 (squash → phase2 @ `0661f7e`)

### Context

Shipped the fourth leaf of the M1.3.\* peer.\* tool family. New
`core/pkg/peer/broadcast.go` exposes `Tool.Broadcast(ctx,
BroadcastParams) (BroadcastResult, error)` composing the new M1.3.d
`peer.RoleFilter` resolver (`core/pkg/peer/filter.go`) with M1.3.a
`Tool.Ask` under a bounded worker pool. The filter applies in memory
over the M1.2 `keepclient.ListPeers` snapshot (roles + languages
case-insensitive, capabilities set-superset + case-sensitive,
`ExcludeSelf` toggle); the broadcast layer fans out via a buffered-
semaphore-bounded goroutine pool (default 8). Partial failures land
in `BroadcastResult.Targets[*].Err` without aborting siblings.

### Pattern

**`RoleFilter.validate()` rejects an empty filter BEFORE any
`ListPeers` round-trip.** The roadmap AC pins
`ErrPeerRoleFilterEmpty` for the no-criteria case because a wildcard
fan-out is almost always a caller bug (token-budget runaway + Slack
DM stampede). The validator also rejects whitespace-only entries on
any axis — silently broadening to "every peer on this axis" is the
same hazard with a less obvious surface. `ExcludeSelf` alone does
NOT make a filter non-empty. Test-pinned by
`TestRoleFilter_Validate_EmptyFilterRejected` +
`TestRoleFilter_Validate_WhitespaceEntryRejected`.

**Capability gate precedes `FilterResolver.Resolve` precedes the
fan-out.** Resolution order: `ctx.Err → validate inputs → validate
filter → capability gate (peer:broadcast) → resolve targets → empty
result surfaces ErrPeerNoTargets → fan-out`. Per-target Ask
capability check runs INSIDE Ask; its rejection lands on
`BroadcastResultTarget.Err` without aborting siblings — observability
beats fail-fast at the per-target level. The broadcast-scope token
and the per-target Ask token are separately plumbed because the
broker mints scope-bound tokens, not a single multi-scope token.

**Bounded worker pool via buffered-semaphore channel; slot acquired
in the dispatching goroutine BEFORE `go func()` spawn.** A 100-target
broadcast with `Concurrency=8` never exceeds 8 spawned-and-running
Ask goroutines. The `countingFilterResolver` + `blockingLister`
test-time pair pins this: the lister blocks every Ask's
`resolvePeer` round-trip on a shared `gate` channel + tracks
in-flight via `atomic.Int32` with a peak counter; the test asserts
`peak ≤ bound` even when all 100 targets dispatch. Right-sizing the
semaphore down to `len(targets)` when `Concurrency > len(targets)`
avoids over-allocating the channel buffer. Test-pinned by
`TestTool_Broadcast_WorkerPoolBoundEnforced`.

**ctx-cancel during dispatch surfaces per-target errors, NOT a
top-level error.** The dispatching goroutine acquires the semaphore
under `select{ case sem<-{}: case <-ctx.Done(): }`; on the cancel
branch the remaining targets land with `ctx.Err()` in their
`BroadcastResultTarget.Err`, then the function waits on the
WaitGroup so the returned slice is settled. Aggregate return is
`(result, nil)` so the caller walks the slice to find both
ctx-canceled targets AND completed targets. Ctx cancellation BEFORE
the resolver runs returns `ctx.Err()` as a top-level error (no
targets dispatched).

**Fire-and-forget = "skip WaitForReply", NOT "drop the reply
body".** (iter-1 codex P1 fix.) The first iteration routed the
per-target fan-out through `Tool.Ask` with a positive
`PerTargetTimeout` and dropped the returned `ReplyBody` at the
broadcast boundary. Codex iter-1 caught the regression: that still
blocks every broadcast leg on `Repository.WaitForReply`, so
broadcasts to peers that never call `peer.Reply` either time out
per-leg (reporting `ErrPeerTimeout` against successfully sent
requests) or stall the fan-out for the full `PerTargetTimeout` per
slowest target. Corrected shape: the worker calls
`Lifecycle.Open` + `Repository.AppendMessage` directly (the
`broadcastSendOne` helper) — NO WaitForReply — so a broadcast leg
returns as soon as the conversation is minted and the request
message is persisted. The recipient may still emit a `peer.Reply`
later; an operator can open the conversation via the persisted
`BroadcastResultTarget.ConversationID` to inspect any reply via
`k2k.Repository.WaitForReply` / `k2k.Repository.Get`. A future
M1.3.e leaf can add an optional reply aggregator without breaking
the M1.3.d return shape.

**Avoid the N+1 `ListPeers` round-trip by bypassing `Tool.Ask`.**
(iter-1 codex P2 fix.) The first iteration's worker called
`Tool.Ask(target=peer.WatchkeeperID)`, which re-entered
`Tool.resolvePeer` and triggered a fresh `keepclient.ListPeers`
call per target. A 100-target broadcast therefore performed
1 + 100 = 101 `ListPeers` round-trips, and the per-target lookups
could return inconsistent results under active-set churn mid-
broadcast. The corrected worker takes the already-resolved
`keepclient.Peer` from `FilterResolver.Resolve` and drives
`Lifecycle.Open` directly with `Participants: []string{actingID,
target.WatchkeeperID}`. The fan-out is now single-snapshot in
cardinality + content. The `broadcastSendOne(ctx, actingID, orgID,
subject, body, target, correlationID) (uuid.UUID, error)` helper
makes the seam substitutable and is pinned by the partial-failure
test (`partialFailureLifecycle` driving one target's Open to a
sentinel error).

**Defensive deep-copy of `Body` happens ONCE at the broadcast
layer before fan-out; the worker copies AGAIN per target.** Two
copies, not redundant: the broadcast-layer copy isolates the caller
from in-flight workers (mutating `body` after `Broadcast` returns
must not bleed); the per-target copy (inside `broadcastSendOne`)
isolates each persisted message body from every sibling worker
that shares the broadcast-layer copy. The property: no goroutine
ever observes a slice another goroutine can write to. Test-pinned
by `TestTool_Broadcast_DefensiveCopyOfBody` (uses a `recordingRepo`
wrapper that snapshots every `AppendMessage.Body` at hand-off).

**`FilterResolver` is a separate seam from `Lister`; the default
resolver wraps `Lister`.** Hoisting `FilterResolver` to a named
interface (`Resolve(ctx, filter, actingID) ([]Peer, error)`) lets
the bounded-pool test inject a deterministic target set
(`countingFilterResolver`) while INDEPENDENTLY exercising the
per-Ask in-flight bound (via `blockingLister`). The default-mode
`defaultFilterResolver` wraps the supplied `Lister` so production
wiring that only sets `Deps.PeerLister` still gets a working
broadcast surface — `NewTool` falls back to
`resolverFromLister(deps.PeerLister)` when `Deps.FilterResolver` is
nil. Mirrors the M1.3.c "narrow seam at the consumer" discipline.

**Non-`Available` peers dropped at the resolver, not at the
broadcast layer.** The resolver enforces the closed-set rule: any
peer whose `Availability != PeerAvailabilityAvailable` is dropped
BEFORE the broadcast layer sees it. A future "throttled" /
"escalated" state therefore stays out of every fan-out without a
client release. Test-pinned by
`TestRoleFilter_NonAvailablePeersSkipped`.

**`Capabilities` filter is set-superset + case-sensitive; `Roles` /
`Languages` are case-insensitive.** Capability ids are machine-
minted (exact-match identifiers); set-superset semantic (peer must
declare ALL required capabilities) lets a broadcast targeting "peers
that can both ask AND reply" surface a precise hit list. Roles +
languages are operator-facing free text, so case-insensitive match
is the operator-friendly default. Test-pinned by
`TestRoleFilter_Capabilities_SetSupersetMatch` +
`TestRoleFilter_Capabilities_CaseSensitive` +
`TestRoleFilter_Roles_CaseInsensitiveMatch`.

**Stable sort by `WatchkeeperID` at the resolver guarantees
deterministic ordering downstream.** Pinning the sort at
`defaultFilterResolver.Resolve` decouples observability (operator
dashboards, audit subscribers in M1.4, test assertions) from the
lister's wire order. Test-pinned by
`TestRoleFilter_StableSortByWatchkeeperID`.

**Helper extraction keeps `Tool.Broadcast` under the gocyclo limit.**
Iter-1 self-review: the pre-commit gocyclo gate (threshold 15)
rejected an inlined Broadcast with cyclomatic complexity 19. Hoisted
`validateBroadcastParams(ctx, params) error` (the eight fail-fast
checks) and `resolveConcurrency(requested, targets) int` (the
default-of-8 + right-size logic) as private helpers. The orchestrator
now reads top-to-bottom as "validate → capability → resolve →
fan-out".

**Source-grep AC bans `keeperslog.` and `.Append(` in `broadcast.go`
and `filter.go`.** Same shape as M1.3.a / .b / .c surfaces. The
peer-tool layer is the call surface; the K2K audit taxonomy belongs
to M1.4. `TestBroadcast_NoAuditOrKeeperslogReferences` and
`TestFilter_NoAuditOrKeeperslogReferences` pin this.

## 2026-05-17 — M1.3.c: `peer.subscribe` event-stream seam

**PR**: [#165](https://github.com/vadimtrunov/watchkeepers/pull/165)
**Merged**: 2026-05-17 (squash → phase2 @ `0395dac`)

### Context

Shipped the third leaf of the M1.3.\* peer.\* tool family. New
`core/pkg/peer/subscribe.go` exposes `Tool.Subscribe(ctx,
SubscribeParams) (<-chan Event, CancelFunc, error)` that gates on the
`peer:subscribe` capability + a self-subscription rule, then forwards
to a new `peer.EventBus` seam. The bus has two implementations: an
in-memory `MemoryEventBus` (used by tests + dev/smoke loops) and a
Postgres `PostgresEventBus` that uses `LISTEN/NOTIFY` over a new
`watchkeeper.peer_events` table behind a `peer_event_published`
trigger (migration `033_peer_events.sql` — number bumped to 033
because 031 is already claimed by `031_k2k_close_summary.sql` from
M1.3.b and `031_manifest_version_metadata.sql` from M3.3, and 032
is claimed by M7.1.a's `032_watchkeepers_role_id.sql` landing
concurrently with this leaf; the M1.3.b "pick the next free
integer rather than reusing a slot documented in an older roadmap
entry" discipline applies). The migrate-schema-test bash harness gains a new block (ab)
pinning the column shape, the trigger registration, a per-tenant
INSERT under `wk_org_role`, and a cross-tenant RLS rejection.

### Pattern

**`EventBus.Subscribe` returns `(<-chan Event, CancelFunc, error)`
not `(chan Event, error)`.** The third return separates the
subscription-lifecycle signal (CancelFunc) from the event-delivery
channel. A naive shape "return a chan you close" would either (a)
force the caller to ALSO supply a ctx for cancellation and race the
close on ctx-cancel-then-CancelFunc, or (b) collapse the two into a
single chan whose close-direction is ambiguous (did the bus close it
because the subscription ended? did the consumer close it to
unsubscribe? what about a double-close panic?). The three-return
shape makes the contract explicit: the bus owns the channel close;
the caller owns the cancel signal; ctx-cancel and CancelFunc both
reach the same idempotent teardown via a `sync.Once` guard.
Test-pinned at `core/pkg/peer/eventbus_memory_test.go`
(`TestMemoryEventBus_CtxCancelClosesChannel` +
`TestMemoryEventBus_CancelFuncClosesChannel` + double-cancel no-op).

**Per-subscription `sync.RWMutex` serialises send + close on the
delivery channel; the bus mutex is released BEFORE the fan-out
non-blocking-send.** First iteration held the bus mutex only across
the subscriber-snapshot phase, then ran the per-subscriber
non-blocking send outside the mutex. The Go race detector immediately
flagged a write-to-closed channel: the watchdog goroutine could
`close(s.out)` while a concurrent `Publish` was inside
`select{ case s.out <- ... : default: }`. Holding the bus mutex
during the fan-out would have serialised every publisher → every
subscriber, defeating the point of fan-out. Per-subscription RWMutex
(`trySend` takes RLock; `close` takes WLock + flips a `closed` bool

- closes the channel) lets concurrent publishers fan out into
  different subscriptions in parallel while guaranteeing the close
  cannot race with a send. The `closed` bool inside the RLock lets a
  `trySend` racing the WLock acquire observe "closed → drop, do not
  panic" via the same code path as "buffer full → drop, bump counter".
  Test-pinned by `TestMemoryEventBus_ConcurrentPublishSubscribe` (16
  publishers × 4 subscribers under -race must report `ok` with zero
  drops). The mistake was a real cost — the first `go test -race ./...`
  surfaced 15 separate `WARNING: DATA RACE` entries; a naive review
  (no race-detector) would have shipped the bug.

**Slow-consumer drop policy: bounded buffer + atomic counter +
non-blocking send.** The roadmap AC pins "bounded buffer + counter
on dropped events". `MemoryEventBus.Publish` does `select{ case s.out
<- ev: default: b.dropped.Add(1) }`; the same shape lives in the
Postgres adapter's drainer goroutine. Two correctness implications:
(1) a slow consumer cannot wedge the publisher (head-of-line-blocking
inversion: one slow subscriber would otherwise starve every
publisher), (2) the dropped-event counter is bus-wide and atomic so
an operator can compare two snapshots to detect a drop-rate spike
without per-subscription bookkeeping. The bounded-buffer size is
configurable via `WithMemoryBufferSize`; default is 16 (large enough
for a single-GC-pause burst, small enough that a wedged consumer
cannot dominate the heap). Test-pinned at
`TestMemoryEventBus_SlowConsumerDropsAndCounts` (4-slot buffer + 16
publishes ⇒ 4 delivered + 12 in the counter).

**Cancel-leak test counts goroutines, not subscriptions.** The
M1.3.c AC pins "cancel-leak test pins zero goroutines after 100
subscribe/cancel cycles". The most direct shape is
`runtime.NumGoroutine` snapshots before / after the 100-cycle loop;
a strict `after <= before` would fail under the test-harness's own
scheduler goroutines (and any concurrent t.Parallel siblings). The
test allows a +2 slack and a 2-second settle deadline with periodic
`runtime.GC()` so the watchdog goroutine + the per-subscription
close-path goroutine have a chance to exit before the assertion
fires. Asserting on a private `activeSubscriptions()` counter would
have been more precise but tied the test to package internals;
goroutine-counting is the AC-defined surface. Test-pinned at
`TestMemoryEventBus_CancelLeakPinsZeroGoroutines`.

**Self-subscription gate at M1.3.c; cross-peer waits for M1.4.**
The roadmap AC names the filter shape as
`{TargetWatchkeeperID, EventTypes []string}` but does not pin the
authorisation model. Two reasonable shapes — (a) admit any target
under a `peer:subscribe` capability, (b) require the acting
watchkeeper to be in the same conversation as the target's events.
M1.3.c lands the simplest workable rule: empty target subscribes to
every event in the tenant (the M1.4 audit subscriber consumes this
way); non-empty target MUST equal the acting watchkeeper id
(self-subscription only). Cross-peer subscription requires a richer
participant-membership gate which IS M1.4's responsibility (the
audit subscriber owns the conversation participant set). The
`ErrPeerSubscriptionPermission` sentinel surfaces a cross-peer
attempt WITHOUT leaking whether the foreign peer exists.
Test-pinned at `TestTool_Subscribe_CrossPeerTargetRejected` +
`TestTool_Subscribe_SelfSubscriptionAllowed` +
`TestTool_Subscribe_EmptyTargetSubscribesToAll`.

**Postgres adapter splits `Querier` (Publish-side, per-tx) from
`PoolAcquirer` (Subscribe-side, long-lived).** The M1.1.a
`k2k.PostgresRepository` holds a single `Querier` (typically a
`pgx.Tx` from `db.WithScope`) so its statements run under the
caller's `SET LOCAL ROLE` + `SET LOCAL watchkeeper.org`. The
`PostgresEventBus` cannot use the same shape for `Subscribe`
because a `LISTEN` bind survives only on the connection that issued
it — a per-statement pool checkout would drop the bind on the next
checkout. The adapter splits the seam in two: the Publish-side keeps
the `Querier` per-tx discipline; the Subscribe-side takes a
`PoolAcquirer` that yields a long-lived `ListenerConn` (with `Exec`,
`WaitForNotification`, `Query`, `Release`). Production wiring
satisfies `PoolAcquirer` with a thin wrapper over `*pgxpool.Pool`;
the production wiring runs `SET LOCAL ROLE` + `SET LOCAL
watchkeeper.org` over the acquired connection ONCE before issuing
`LISTEN peer_events` so the same session GUC drives both the LISTEN
bind and the subsequent SELECT-on-notify drainer. The seam shape
lets tests inject a hand-rolled fake (`fakeListenerConn` over a
`chan Notification`) without standing up a real Postgres + LISTEN
backend.

**Migration `033_peer_events.sql` uses `jsonb` payload + per-row
NOTIFY trigger.** Three sub-decisions: (1) `payload jsonb NOT NULL
DEFAULT '{}'::jsonb` — `jsonb` over `bytea` because the future M1.4
audit consumer + any operator-facing dashboard will want indexed
payload queries (`payload ->> 'foo'`); the empty-object default
preserves backwards compatibility with a publisher that omits the
body. (2) `peer_event_published` plpgsql function fires AFTER INSERT
FOR EACH ROW + emits `pg_notify('peer_events', NEW.id::text)` — the
id-in-payload shape lets a future drainer SELECT a single row
directly instead of polling every row stamped after the cursor (the
M1.3.c drainer uses the cursor-poll shape because a missed
notification must still degrade to "next notify catches up"). (3)
RLS policies key off `watchkeeper.org` GUC with the same
`nullif(..., '')::uuid` cast as migrations 029 + 030 so an unset
GUC is fail-closed; an unset GUC reads as SQL NULL and the policy
USING + WITH CHECK both reject.

**Postgres `jsonb` column INSERT uses `string(payload) + ::jsonb`,
not raw `[]byte`** (iter-1 self-review fix). First iteration passed
`event.Payload []byte` directly as the bound parameter. pgx v5
encodes `[]byte` as `bytea` by default — the Postgres server then
rejects the INSERT (`column "payload" is of type jsonb but
expression is of type bytea`). Prior art
(`handlers_write.handleLogAppend`, M9.4.a) uses `string(payload)`

- explicit `::jsonb` cast in the SQL literal; the M1.3.c adapter
  mirrors that exactly. Empty / nil payloads coerce to the canonical
  empty-object literal `"{}"` so the `NOT NULL` + `DEFAULT
'{}'::jsonb` column does not surface a constraint violation on a
  publisher that omits the body. Test-pinned by
  `TestPostgresEventBus_Publish_HappyPath_RunsInsert` (asserts SQL
  contains `::jsonb` + args[4] is the JSON string) and
  `TestPostgresEventBus_Publish_EmptyPayloadCoercesToEmptyObject`.

**Tuple cursor `(created_at, id) > ($cursor_ts, $cursor_id)` for
the LISTEN/NOTIFY drainer** (iter-1 codex P1 + P2 fix). First
iteration anchored the drainer's initial cursor at app-side
`time.Now()` and advanced it via `created_at > $since`. Two
correctness bugs surfaced under codex review: (1) a row committed
AFTER Subscribe but with a Postgres `now()` value OLDER than the
drainer's app-side `time.Now()` (long-running publisher tx, or
clock skew across publisher / subscriber hosts) gets silently
filtered out when its notification arrives — `created_at > $since`
is false; (2) two rows sharing a `created_at` (Postgres `now()` is
transaction-scoped, so two concurrent inserts may legitimately
stamp the same timestamp; or two inserts from a single tx) — the
strictly-greater filter drops one of them on the next poll. Fix:
prime the cursor by querying `MAX(created_at)` + matching id once
synchronously AFTER LISTEN binds (any row committed before LISTEN
is observable to a fresh SELECT), then advance the cursor via the
tuple comparator `(created_at, id) > ($2, $3)` with the matching
`ORDER BY created_at ASC, id ASC`. The tuple comparator is the
SQL-standard lexicographic operator that Postgres evaluates
correctly across the composite. Empty-table initial state anchors
at `(time.Time{}, uuid.Nil)` so the first real row is strictly
greater. Test-pinned by the existing
`TestPostgresEventBus_SubscribeAcquiresAndIssuesLISTEN` +
`TestPostgresEventBus_CtxCancelClosesChannelAndReleasesConn` (the
prime-the-cursor Query path runs over the fake; an empty rowset is
the canonical empty-table case).

**Source-grep AC bans `keeperslog.` and `.Append(` in
`subscribe.go`.** Same shape as M1.1.b's
`TestChannels_NoAuditOrKeeperslogReferences`, M1.1.c's
`TestLifecycle_NoAuditOrKeeperslogReferences`, M1.3.a's
`TestAsk_NoAuditOrKeeperslogReferences` /

executable code so the M1-family source-grep discipline stays
uniform across every K2K + peer-tool surface.

**`EventBus` is an OPTIONAL Dep in `peer.Tool`.** The M1.3.a /
M1.3.b `Tool` constructor panics on every nil Dep (`PeerLister`,
`Lifecycle`, `Repository`, `Capability`). Adding `EventBus` as a

### Pattern

**Separate `close_reason` from `close_summary`.** The pre-existing
`close_reason` (M1.1.a, migration 029) is the lifecycle-layer
rationale: a stable closed-set code (the M1.6 escalation auto-archive
sentinel, the M1.7 archive-on-summary writer's stable reason). The
new `close_summary` (M1.3.b, migration 031) is the human-readable
one-liner the operator supplies to `peer.Close`. Folding the two into
a single column would have either forced the M1.6 escalation timeout
to invent a free-text rationale (it has none) or forced the M1.7
archive-on-summary writer to parse a column that means two different
things in two different lifecycle paths. Two columns, two writers,
two contracts — the M1.7 archive-on-summary writer cross-links the
summary into the Keep knowledge chunk; M1.4 audit emission joins on
the reason. Test-pinned at `core/pkg/peer/close_test.go:50` (asserts
`CloseReason == "peer.close"` AND `CloseSummary == operator text` are
distinct after a successful close).

**`Tool.Close` short-circuits BEFORE Lifecycle.Close on an
already-archived row WITH a non-empty `close_summary` to preserve
the first close's summary AND recovers a half-completed close (archive
succeeded, summary write failed) by writing the supplied summary when
the existing `close_summary` is empty.** Idempotent double-close is a
hard AC of M1.3.b. The naive shape "call Lifecycle.Close, get
ErrAlreadyArchived, return nil" returns nil correctly but ALSO calls
`Repository.SetCloseSummary` on the second close, overwriting the
FIRST close's summary. The flow runs `Repository.Get` first; when the
observed status is `StatusArchived` AND CloseSummary is non-empty, the
flow returns nil WITHOUT calling either Lifecycle.Close or
SetCloseSummary so the first summary persists. When the observed
status is `StatusArchived` AND CloseSummary is empty (a prior close
crashed mid-flow after archiving but before writing the summary), the
flow skips Lifecycle.Close (already done) but DOES call
SetCloseSummary so the caller's summary lands on the row — recovering
a half-completed close without a manual reaper (iter-1 codex P1 +
iter-1 critic B2 fix). The saga-replay race (Get sees open,
Lifecycle.Close loses the race) is also a no-op: an
`ErrAlreadyArchived` from Lifecycle.Close translates to nil and
SetCloseSummary is NOT called (the race-winner owns the persisted
summary). Test-pinned by
`TestTool_Close_IdempotentDoubleClose_PreservesFirstSummary`,
`TestTool_Close_RecoversSummaryOnArchivedButEmptyRow`, and
`TestTool_Close_IdempotentOnAlreadyArchivedRace`.

**Cross-tenant defensive check on `Repository.Get` mirrors RLS
semantics in the in-memory adapter.** Production Postgres wiring
already filters by tenant via the `watchkeeper.org` RLS GUC (an unset
/ mismatched GUC fail-closes to ErrConversationNotFound). The
in-memory adapter does NOT scope `Get` by org so a tool call carrying
a foreign-tenant conversation id would slip past Get and reach the
participant gate (where it would coincidentally also fail because the
acting watchkeeper is unlikely to be a participant — but the
diagnostic chain would leak existence). `Tool.Close` adds an explicit
`conv.OrganizationID != params.OrganizationID → ErrPeerConversationNotFound`
check immediately after Get so the in-memory + any future non-RLS
Repository impl honour the same boundary, and the foreign-tenant
caller observes "not found" rather than "permission denied". Iter-1
critic B1 fix. Test-pinned by `TestTool_Close_CrossTenantSurfacesNotFound`.

**Participant-membership gate uses exact case-sensitive match;
diagnostic chain does NOT leak the participant list.** The acting
watchkeeper id must appear verbatim in the persisted
`Conversation.Participants` slice (which the M1.1.a Repository.Open
already rejects whitespace-only entries from). A case-insensitive
match would admit a typo'd id that happens to differ only in case;
the participant set is platform-minted, not operator-facing free
text, so case-folding is not warranted. The
`ErrPeerClosePermission` error message carries the conversation id
only — leaking the full participant slice in the diagnostic chain
would surface workforce-composition info to a non-participant caller
(an LLM hallucinating a target id, a malicious cross-conversation
poke). Test-pinned by `TestTool_Close_NonParticipantRejected`
(asserts the error message contains neither the asker's id nor the
target peer's id).

**Capability gate precedes Get; Get precedes participant gate; gate
precedes Lifecycle.Close.** The resolution order is fixed:
`validate → capability → Get → participant check → idempotent
short-circuit → Lifecycle.Close → SetCloseSummary`. The capability
gate runs BEFORE Get so a cross-tenant token never burns a storage
round-trip (mirrors M1.3.a's "fail-fast precedes K2K state mutation"
discipline). The participant gate runs AFTER Get because it needs the
persisted participants slice as the source of truth (the caller
cannot supply the participants — they live on the row). The
idempotent-archived short-circuit runs AFTER the participant gate so
a non-participant attempting to close an archived conversation also
surfaces `ErrPeerClosePermission` (rather than nil) — a closed
conversation is still owned by its participants. Test-pinned by
`TestTool_Close_CapabilityDenied` (gate hit → 0 Lifecycle.Close
calls), `TestTool_Close_UnknownConversation` (0 Lifecycle.Close
calls), `TestTool_Close_NonParticipantRejected` (0 Lifecycle.Close
calls).

**`SetCloseSummary` is a separate `Repository` method, not a new
parameter on `Repository.Close`.** The K2K `Repository.Close` was
established by M1.1.a as the lifecycle-layer state transition; adding
a `summary` argument there would have forced both the M1.6
escalation auto-archive AND the M1.7 archive-on-summary writer to
either pass an empty string or invent a summary they don't have. A
separate `SetCloseSummary` method preserves the existing contract,
keeps the Postgres adapter's `Close` UPDATE statement unchanged, and
lets the peer-tool layer compose the two writes in the order
required (close transitions status → summary writes onto the
archived row). Mirrors M1.3.a's "extend the repository surface
purely additively rather than mutating existing signatures" discipline.

**`SetCloseSummary` returns `ErrConversationNotArchived` on an open
row.** Writing a summary onto a still-open conversation is a
programmer bug at the call site (a forgotten Lifecycle.Close call).
The sentinel is distinct from `ErrAlreadyArchived` so the caller can
distinguish "wrong direction" (open row received a write meant for
archived) from "duplicate close" (archived row received a second
transition). The Postgres adapter uses the same conditional-UPDATE +
probe shape as `Close` / `BindSlackChannel` so the sentinel
selection is uniform across the M1.1.a state-transition surface.
Test-pinned by
`TestMemoryRepository_SetCloseSummary_OpenRowSurfacesNotArchived`.

**Lifecycle seam extension stays narrow.** The peer-tool layer's
`Lifecycle` interface gains `Close(ctx, id, reason) error` alongside
the existing `Open`. The seam mirrors the upstream
`k2k.Lifecycle.Close` signature 1:1 so production wiring passes
`*k2k.Lifecycle` verbatim; tests inject a `fakeLifecycle` that
routes through the in-memory repo's `Repository.Close`. The fake
optionally short-circuits Lifecycle.Close with an injected error so
test cases can pin "lifecycle error must not write summary" without
implementing a full lifecycle fake. Mirrors M1.3.a's "narrow seam at
the consumer" discipline.

**Source-grep AC bans `keeperslog.` and `.Append(` in `close.go`.**
Same shape as M1.1.b's `TestChannels_NoAuditOrKeeperslogReferences`,
M1.1.c's `TestLifecycle_NoAuditOrKeeperslogReferences`, and M1.3.a's
`TestAsk_NoAuditOrKeeperslogReferences` /
`TestReply_NoAuditOrKeeperslogReferences`. The peer-tool layer is
the call surface; the K2K `k2k_conversation_closed` audit taxonomy
is M1.4's seam. `TestClose_NoAuditOrKeeperslogReferences` reads the
source file, strips line comments, and asserts neither token appears

uniform across every K2K + peer-tool surface.

extended in-place (blocks w-z).

### Pattern

**Per-call resolver beats process-global static for tenant-scoped
auth tokens.** `peer.Tool` accepts a `CapabilityValidator` interface
satisfied by `*capability.Broker.ValidateForOrg` directly. Production
wiring passes the broker; tests pass a hand-rolled `fakeCapability`
recording calls + decisions. The resolver-shaped seam mirrors the
M3.5.a "per-call resolver over process-global static" discipline:
binding a token to a tenant via `IssueForOrg` + validating via
`ValidateForOrg` keeps multi-tenant deployments safe even if a future
contributor confuses two adjacent calls.

**Capability gate precedes K2K state mutation.** `Tool.Ask` runs the
`capability.Broker.ValidateForOrg(token, "peer:ask", orgID)` check
BEFORE `Lifecycle.Open` mints a conversation row, and BEFORE the
target resolver burns a `keepclient.ListPeers` round-trip. A failed
gate surfaces as `ErrPeerCapabilityDenied` (chained with the
underlying `capability.Err*` sentinel via `fmt.Errorf("%w: %w", ...)`)
without any persistent side effect. `Tool.Reply` mirrors the
discipline: capability gate → `Repository.Get` (existence + open
check) → `Repository.AppendMessage`. Mirrors the M1.1.c "fail-fast
precedes Slack" pattern at the peer-tool layer.

**Two-pass target resolver: uuid first, role second, case-fold the
role.** `Tool.resolvePeer` walks the active-peer list twice. Pass 1
exact-matches `target` against `Peer.WatchkeeperID` so a target that
parses as a watchkeeper id resolves verbatim. Pass 2 case-insensitive
matches against `Peer.Role` so the operator vocabulary "Lead" /
"strategist" / "STRATEGIST" all resolve to the same role. Neither
pass caches — every `Ask` re-fetches the active list so a
freshly-spawned peer resolves on its first try. Test-pinned by
`TestTool_Ask_TargetByRoleResolves` (role lookup hits the resolver
and proceeds to `WaitForReply` rather than failing at
`ErrPeerNotFound`).

**Cond-var-driven `WaitForReply` on the in-memory adapter with an
atomic `timedOut` flag, NOT a fixed-clock deadline check.** The
naive shape "compare `r.now()` against a deadline in the wakeup
loop" deadlocks under a fixture clock (the M1.1.a `fixedClock`
helper) because `r.now()` never advances. The implementation uses a
real `time.NewTimer(timeout)` AND an `atomic.Bool timedOut` flag
flipped by the waker goroutine on timer expiry; the main goroutine
re-checks the flag after each `cv.Wait` so the wakeup loop
terminates even when the repository's stamping clock is frozen.
The flag is also checked BEFORE the first `Wait` so a synchronous-
scan miss against an already-expired timer surfaces
`ErrWaitForReplyTimeout` without parking. Test-pinned by
`TestMemoryRepository_WaitForReply_SinceCursorIsExclusive`
(fixedClock + `since` exactly equal to the reply's `CreatedAt`
forces the timeout path).

**`since` cursor is strictly exclusive (`m.CreatedAt.After(since)`).**
A reply stamped at exactly `since` does NOT satisfy the wait. The
peer-tool layer captures `since := t.now().UTC()` BEFORE
`AppendMessage(direction=request)` so the request's own `CreatedAt`
is necessarily ≥ since — the request is not its own reply.
Test-pinned by `TestMemoryRepository_WaitForReply_SinceCursorIsExclusive`.

**Defensive deep-copy of `Body []byte` at every boundary AND in the
return value.** `Tool.Ask` and `Tool.Reply` each `make + copy` the
caller-supplied `Body` before calling `AppendMessage`; the
in-memory `MemoryRepository.AppendMessage` deep-copies AGAIN before
storing (a double-defense). `Tool.Ask` also deep-copies the returned
`Message.Body` before assembling `AskResult.ReplyBody` so a future
storage-layer optimisation that aliases the held byte slice cannot
bleed up the call chain. Test-pinned by
`TestMemoryRepository_AppendMessage_DefensiveCopyOfBody` +
`TestTool_Ask_DefensiveCopyOfBody`. Mirrors the M1.1.a participants
deep-copy contract.

**Postgres `WaitForReply` ships as a polling fallback with a
configurable interval, NOT as LISTEN/NOTIFY at M1.3.a.** The
M1.3.a AC pins "Postgres uses LISTEN/NOTIFY on a per-conversation
channel, with a polling fallback"; this leaf ships only the polling
fallback (default 100ms, override via
`PostgresRepository.WithPollInterval` for fast tests). The
`Repository.WaitForReply` signature is identical for both impls so
a future LISTEN/NOTIFY layer drops in without a wire-shape break.
The polling SQL is a single `SELECT ... WHERE conversation_id = $1
AND direction = 'reply' AND created_at > $2 ORDER BY created_at ASC
LIMIT 1` so the index `(conversation_id, created_at)` from
migration 030 keeps the planner cost flat as message volume grows.
Expiry uses a real `time.NewTimer(timeout)` rather than a deadline
derived from `r.now()` — a fixed-clock test fixture (or a custom
stamping clock injected for replay debugging) cannot pin the wait
indefinitely (iter-1 codex Major fix). The stamping clock drives
`created_at` only; the wait timeout runs on wall-clock time.

**Message `body` is stored as `bytea`, NOT `text`.** The
`peer.Tool` API documents `Body []byte` as opaque bytes — a future
M1.3.c subscription delivery may legitimately carry a serialised
protobuf or any non-UTF-8 payload. A `text` column would either
reject invalid UTF-8 at the SQL layer or round-trip corrupted
content. Migration 030 ships `body bytea NOT NULL CHECK
(length(body) > 0)`; the Go layer binds `[]byte` directly via pgx
(no `string()` coercion). Iter-1 codex Critical fix.

**`AppendMessage` enforces `status='open'` atomically via
`INSERT ... SELECT FROM k2k_conversations WHERE status='open'`.**
The in-memory adapter already gated on status under the write
lock; the Postgres adapter originally relied on the caller's
preceding `Get` to gate, which is racy — a concurrent
`Lifecycle.Close` between `Get` and `AppendMessage` would slip a
reply into an archived row. The new SQL shape uses an
`INSERT ... SELECT` subquery; when the parent conversation is not
in `status='open'`, the subquery returns zero rows, the INSERT
inserts zero rows, the `RETURNING created_at` produces
`pgx.ErrNoRows`, and the adapter translates that to
`ErrAlreadyArchived`. The peer-tool layer's existing `errors.Is`
branch on `ErrAlreadyArchived` translates to
`ErrPeerConversationClosed` unchanged. Iter-1 codex Major fix.

**Resolver miss MUST short-circuit BEFORE Lifecycle.Open mints a
row.** `Tool.Ask` runs `resolvePeer` immediately after the
capability gate and BEFORE `Lifecycle.Open`. An unknown target
surfaces as `ErrPeerNotFound` without any state mutation; the
alternative ("open the conversation, then fail when the target
doesn't exist") would leave orphan `k2k_conversations` rows that
the M1.7 archive-on-summary reaper would have to handle. Mirrors
M7.1.b's "fail-fast precedes audit/state side effects" discipline.
Test-pinned by `TestTool_Ask_UnknownTargetSurfacesErrPeerNotFound`
(asserts `Lifecycle.Open` calls = 0 after a resolver miss).

**Built-in `Source = "built-in"` constant reserved across the
M1.3.\* family.** `peer.BuiltinSourceName = "built-in"` is exported
so the M1.3.b / .c / .d siblings can stamp the same source name on
their manifests without re-declaring the literal. `peer.ask` /
`peer.reply` ride `toolregistry.DryRunModeScoped` (the lead's-DM
reroute lands in M9.4.c) and a minimum-viable JSON schema. Future
M9.3 signing will fill `Manifest.Signature` at build time; M1.3.a
leaves it empty per the `NoopSignatureVerifier` contract.

**Source-grep AC bans `keeperslog.` and `.Append(` in `ask.go` /
`reply.go`.** The peer-tool layer is the call surface; the K2K
audit taxonomy (`k2k_message_sent` / `k2k_conversation_opened` /
etc.) is M1.4's seam. The tests `TestAsk_NoAuditOrKeeperslogReferences`

- `TestReply_NoAuditOrKeeperslogReferences` read the source file,

**`keepclient.Client` consumed via a narrow `Lister` seam, NOT

Mirrors the M1.1.c `k2k.SlackChannels` "narrow seam at the
consumer" discipline.

## 2026-05-17 — M1.2: `keepclient.ListPeers` + `GET /v1/peers`

**PR**: [#158](https://github.com/vadimtrunov/watchkeepers/pull/158)
**Merged**: 95c99288056a58c3cf2eabf2f3edc1120d28a748 on 2026-05-17

### Context

Shipped the M1.2 leaf that the M1.3.\* peer.\* tool family will consume.
The endpoint `GET /v1/peers` returns every active watchkeeper joined
with its pinned `manifest_version` (description / language / tools)
and parent `manifest` (role display_name). The client method
`keepclient.Client.ListPeers(ctx, ListPeersRequest{Limit int})` mirrors
the wire shape. The leaf deliberately ships NO server-side role /
language / capability filters — M1.3.d's `peer.RoleFilter` applies
those in-memory over the unfiltered list so every M1.3.\* consumer
sees the same "all active peers at this instant" snapshot.

### Pattern

**Server-side endpoint mirrors the `/v1/watchkeepers` list shape, NOT
a freshly-invented surface.** `GET /v1/peers` reuses the
`{items, next_cursor}` envelope, the `?limit=` bounds (default 50,
inclusive cap 200, reject 0/negative/oversize/non-numeric with 400
`invalid_request`), the `ORDER BY created_at DESC` discipline, and
the `writeError` / `writeJSON` helpers. Reuse keeps the
keepclient-side decoder identical to `ListWatchkeepers` and means a
future seek-pagination follow-up lands in one place across both
endpoints. The route is wired identically in `server.go` and
`export_test.go`'s `NewRouterWithRunner` so the test seam stays
exhaustively exercisable without a real pgx pool.

**Filters live client-side per the M1.3.d resolver contract, NOT in
the server query.** M1.3.d's roadmap entry pins `peer.RoleFilter` as
an in-memory filter (`Resolve(ctx, filter RoleFilter) ([]WatchkeeperID, error)`
fans out over `keepclient.list_peers` and applies the filter
in-memory). Reproducing role / language / capability filters at the
SQL layer would duplicate the predicate across the server query and
the client resolver, with two consequences: any predicate drift
(case-sensitivity, whitespace handling) becomes a wire-version break;
and every M1.3.\* consumer would need to know whether "filter X is
server-side or client-side" — a leaky abstraction. The server query
is intentionally minimal: `WHERE wk.status = 'active'` plus INNER
joins. M1.3.d does the rest in Go.

**INNER joins skip unpinned rows; LEFT joins would surface
half-spawned watchkeepers.** The handler INNER-joins `watchkeeper` to
`manifest_version` on `active_manifest_version_id` and to `manifest`
via `mv.manifest_id`. A row with `status='active'` but
`active_manifest_version_id = NULL` (a rare wiring bug, or a future
M3.5 spawn-in-progress state) is SKIPPED rather than surfaced with
empty role / description / capabilities. The test
`TestListPeers_SQLFiltersActiveAndJoinsManifest` pins the JOIN type
verbatim so a future refactor that switched to `LEFT JOIN` for
"defensive null-tolerance" cannot silently regress the contract.

**Manifest join follows `mv.manifest_id`, NOT `wk.manifest_id`
(iter-1 codex P2 fix).** Migration 002's docblock spells out that
the composite-FK invariant "`watchkeeper.active_manifest_version_id`
references a `manifest_version` whose `manifest_id` matches
`watchkeeper.manifest_id`" is DEFERRED to Phase 2 and is not
enforced at the SQL layer today. If a write path (legacy or
malformed) lands a row where the two ids disagree, joining
`manifest` on `wk.manifest_id` would surface a peer whose `role`
(from `m.display_name`) and `description` / `language` /
`capabilities` (from `mv.*`) belong to two different manifests — an
impossible peer record that would mislead the M1.3.a target
resolver. Threading the manifest join through `mv.manifest_id`
keeps every projected row internally consistent: all four
manifest-derived fields come from the SAME manifest version the
watchkeeper is currently bound to. Test-pinned by the SQL-contract
assertion that bans `m.id = wk.manifest_id` verbatim.

**`capabilities` is extracted from `manifest_version.tools` jsonb
server-side, NOT round-tripped as raw JSON.** The wire shape pins
`capabilities` as a string array so consumers do not need to know
the `tools` schema (`[{"name":"<tool>"}]` per the M1 seed
migrations). The extraction lives in `extractCapabilities([]byte)`
with two defensive layers. First, malformed JSON or a non-array top
level returns an empty slice rather than 500ing the whole endpoint
— a manifest authoring bug should not blank out every other
well-formed peer. Second, a tool object whose `name` field is
missing or blank is SKIPPED, not appended as the empty string, so
the M1.3.d capability filter is free of spurious empty-string
matches. Both defences are test-pinned
(`TestListPeers_MalformedToolsYieldsEmptyCapabilities`,
`TestListPeers_SkipsBlankAndMissingToolNames`).

**Empty arrays on the wire are `[]`, NOT `null`.** Both the
top-level `items` envelope and per-row `capabilities` must serialise
as `[]` when empty. Server-side this is achieved by
`make([]peerRow, 0, limit)` plus `make([]string, 0)` in
`extractCapabilities`. Client-side `ListPeers` defends the contract
a second time: after Decode, a nil `Items` is normalized to an empty
slice and every per-row nil `Capabilities` is normalized likewise.
The defensive pair protects M1.3.d's `for ... range` loops from a
nil-dereference panic if a future server-side bug regresses to
emitting `null` for either shape. Test-pinned both ends
(`TestListPeers_EmptyItemsArrayNotNull`,
`TestListPeers_EmptyCapabilitiesEmittedAsArray`,
`TestClient_ListPeers_EmptyItemsNonNil`,
`TestClient_ListPeers_CapabilitiesNonNil`).

**`availability` is a string enum reserved for future M1.5 / M1.6
extensions.** Phase 2 ships a single value
(`peerAvailabilityAvailable = "available"`) because the SQL filter
`status = 'active'` is the only availability signal we have. The
column is exported so that M1.5 (token-budget exhaustion →
throttled) and M1.6 (escalation pending → escalated) can introduce
new enum values without a wire-shape break. Both ends define the
constant (`server.peerAvailabilityAvailable` plus
`keepclient.PeerAvailabilityAvailable`) so consumers can compare
with `==` rather than against a string literal, mirroring how
M1.1.a's status enum is exposed.

**Client `Limit: 0` omits the query parameter, NOT `?limit=0`.** A
naive `strconv.Itoa(0)` would put `?limit=0` on the wire, which the
server's `invalid_request` validator rejects as out-of-range. The
branch `if req.Limit > 0` gates the `q.Set("limit", ...)` call so
the zero-value Request struct lands the same wire shape as "do not
send limit at all", letting the server apply its default 50.
Test-pinned by `TestClient_ListPeers_ZeroLimitOmitsQueryParam` so a
future contributor "simplifying" the conditional cannot regress the
contract.

**Defensive client-side bounds mirror the server's bounds.** The
client method rejects `Limit < 0` and `Limit > 200` synchronously
via `ErrInvalidRequest` before any network round-trip, mirroring
`ListWatchkeepers`'s `maxWatchkeeperListLimit` discipline. This
documents the client contract in-code (a caller reading
`read_peers.go` sees the cap without consulting the server source)
and saves a 400 round-trip on obvious bugs.

**Source-grep AC bans `keeperslog.` and `.Append(` in
`handlers_peers.go`.** This handler is a read projection — the M1.4
audit subscriber owns the K2K event taxonomy (`k2k_message_sent`,
`k2k_conversation_opened`, etc.). The test
`TestHandleListPeers_NoAuditOrKeeperslogReferences` reads the source
file, strips line comments, and asserts neither token appears in
executable code. Mirrors M1.1.b's
`TestChannels_NoAuditOrKeeperslogReferences` and M1.1.c's
`TestLifecycle_NoAuditOrKeeperslogReferences` so the M1-family
source-grep discipline is uniform across every read + write surface.

**Forward-compatible wire field `next_cursor` ships as `null`.** The
response envelope's `next_cursor *string` is reserved for a future
seek-pagination follow-up — the same forward-compat pattern as
`/v1/watchkeepers`. Today every response carries `null` so a future
server-side pagination cursor can land without a wire-shape break;
clients that match against `resp.NextCursor != nil` keep working
across the upgrade.

## 2026-05-17 — M1.1.c: `wk channel reveal` CLI + K2K lifecycle wiring

**PR**: [#157](https://github.com/vadimtrunov/watchkeepers/pull/157)
**Merged**: f5407654be831e71d218cc25a80de0f98a61dd5e on 2026-05-17

### Context

Stitched M1.1.a's K2K conversation domain to M1.1.b's Slack channel
primitives behind one lifecycle entry point in `core/pkg/k2k/`. New
seams: `k2k.SlackChannels` (a narrow three-method interface satisfied
by `*slack.Client`), `k2k.Lifecycle` (the
`Open(ctx, OpenParams) (Conversation, error)` /
`Close(ctx, id, reason) error` composer), `k2k.DeriveChannelName(id)`
(the `k2k-<8-hex-prefix>` derivation), and
`k2k.Repository.BindSlackChannel(ctx, id, channelID)` (the third
write method on the persistence seam, atomic vs `Close` + duplicate-
bind). The `wk channel reveal --user <slack-user-id>
<conversation-id>` CLI lands behind a `channelResolver`
test-injection seam; the production composition opens a pgxpool +
runs `db.WithScope` to look up the bound channel id under the
operator's org-scoped RLS claim, then forwards to
`*slack.Client.RevealChannel`. Compile-time wiring assertion lives in
`core/internal/keep/k2k_wiring/` (mirroring `retire_wiring` /
`approval_wiring`).

### Pattern

**Narrow three-method `SlackChannels` interface, NOT
`*slack.Client` everywhere.** The M1.1.b extension surface added FOUR
methods to `*slack.Client` (`CreateChannel`, `InviteToChannel`,
`ArchiveChannel`, `RevealChannel`); the lifecycle layer consumes only
the first three because `RevealChannel` is owned by the
`wk channel reveal` CLI — there is no K2K Open() / Close() call that
reveals. Encoding only the three methods the lifecycle needs keeps
the test fake under 80 lines and prevents a future refactor of
RevealChannel's signature from forcing every lifecycle test to update
its mock. The compile-time assertion `var _ k2k.SlackChannels =
(*slack.Client)(nil)` lives in `core/internal/keep/k2k_wiring/wiring.go`
so a missing method (renamed Slack-side, added k2k-side) fails the
build at the wiring boundary, not at the first production run.

**`BindSlackChannel` is a SEPARATE Repository method, NOT a
`SlackChannelID` field on `OpenParams`.** The conversation ID has to
exist BEFORE the Slack channel name can be derived (the name is
`k2k-<id-prefix>`), and the Slack channel cannot be provisioned
before the conversation row mints the id. So Open() is two-phase:
`Repo.Open` mints the row + id → lifecycle derives the name + calls
Slack → `Repo.BindSlackChannel(id, channel_id)` stamps the column.
The alternative — accepting a `SlackChannelID` on `OpenParams` —
would force every caller to pre-derive the channel name + call Slack,
which inverts the lifecycle layer's reason for existing. The bind
method uses the same atomic conditional-UPDATE shape as
`Repo.Close` (`WHERE id = $1 AND status = 'open' AND
slack_channel_id IS NULL`) so a concurrent duplicate-bind matches zero
rows and the follow-up probe distinguishes `ErrConversationNotFound`
/ `ErrAlreadyArchived` / `ErrSlackChannelAlreadyBound` without
parsing the error chain.

**Bind precedes invite (iter-1 codex Major fix).** The original
`Lifecycle.Open` order was `CreateChannel → InviteToChannel →
BindSlackChannel`. Iter-1 codex review caught a race window: a
concurrent `Lifecycle.Close` racing between `CreateChannel` and the
bind step would archive the row while `Close`'s "skip ArchiveChannel
when SlackChannelID empty" branch left the live Slack channel
orphaned and unreachable from any persisted state. The fix swaps to
`CreateChannel → BindSlackChannel → InviteToChannel`: the row carries
the bound channel id from the moment any reader sees the row, so a
concurrent `Close` after `CreateChannel` correctly archives both the
row AND the Slack channel. `InviteToChannel` can still fail
post-bind, but the row+channel pair is consistent at that point — a
follow-up `Close` archives both correctly. The test

**Resolver returns Status alongside ChannelID; CLI fail-fasts on

## 2026-05-17 — M1.1.b: Slack adapter channel primitives

**PR**: [#156](https://github.com/vadimtrunov/watchkeepers/pull/156)
**Merged**: a3aa91e4a31696d788f2823c82bb859151ffd0bd on 2026-05-17

### Context

Extended `core/pkg/messenger/slack/` with the four channel-lifecycle
primitives the M1.1.c K2K wiring layer will call:
`CreateChannel(name, isPrivate)`, `InviteToChannel(channelID, userIDs)`,
`ArchiveChannel(channelID)`, `RevealChannel(channelID, userID)`. All
four are methods on `*slack.Client` (NOT on the portable
`messenger.Adapter` interface) — mirroring the
`OpenIMChannel` / `ConversationsHistory` / `CreateApp` extension
pattern that lets a Slack-specific seam ride on the same Client
without bloating the cross-platform Adapter surface. M1.1.b
deliberately ships ONLY the capability surface — no K2K consumer
calls (M1.1.c), no audit emission, no token-budget enforcement.

### Pattern

**Slack-extension methods land on `*Client`, NOT on `messenger.Adapter`.**
The portable Adapter interface fixed the six-method shape in M4.1; a
new Slack-specific channel primitive would force every future
non-Slack adapter (Discord, IRC, SMS, …) to either implement a
Slack-shaped call or return `messenger.ErrUnsupported`. The K2K
domain in `core/pkg/k2k/` is by design Slack-only at this milestone,
so M1.1.c will consume `*slack.Client` directly (per-call resolver or
typed dependency) rather than route through `messenger.Adapter`. The
compile-time assertion `var _ messenger.Adapter = (*Client)(nil)` in
`adapter_assertion_test.go` still holds — the new methods are
additive Slack-side extensions.

**Idempotent `CreateChannel` via opportunistic `conversations.list`
recovery — kind-filtered AND archive-filtered (iter-1 codex P2 fix).**
Slack's `conversations.create` does NOT echo the existing channel id
on a `name_taken` collision — it surfaces `error: "name_taken"` and
the caller must look up the existing id via `conversations.list`. The
implementation issues the create call first, catches
`ErrChannelNameTaken` via `errors.Is`, then paginates
`conversations.list` filtered to the same channel-kind (private vs
public) as the original request. Two filters apply to the page scan:
(1) kind-aware — a private create that collides with a same-name
PUBLIC channel does NOT silently bind to the public id (which would
flip the K2K conversation's privacy posture); (2) archive-aware — an
ARCHIVED same-name same-kind hit is SKIPPED, because Slack still
rejects `conversations.create` with `name_taken` against an archived
holder, but binding the K2K consumer to an archived channel id would
surface as `ErrIsArchived` on the very next `InviteToChannel` call.
When BOTH archived and non-archived same-name hits exist on the same
page (a workspace pattern after channel-recycling), the non-archived
id wins regardless of page order. When the recovery list call returns
zero NON-ARCHIVED matches the composite error carries BOTH the
original `ErrChannelNameTaken` sentinel AND the diagnostic "name X
not found among non-archived private_channel entries" so the caller
can pick a fresh name. The recovery path burns a tier-2 rate-limit
token on `conversations.list`, documented in the per-method
doc-block.

**Idempotent close / invite via Slack-side "already in target state"
codes — with a batch-size guard on `already_in_channel` (iter-1 codex
P1 fix).** `ArchiveChannel` translates `already_archived` to nil
unconditionally — Slack's archive endpoint takes a single channel id,
there is no all-or-nothing concern. `InviteToChannel` translates
`already_in_channel` to nil ONLY when the post-trim batch size is 1
(single-user invite OR a slice that trimmed down to one entry).
Slack's `conversations.invite` is documented as all-or-nothing per
call: when ANY user in a multi-user batch is already a member, Slack
returns `already_in_channel` AND does NOT invite the OTHER listed
users. Silencing that to nil would let the K2K Open() fan-out report
success while leaving some bots outside the channel. The post-trim
count rides out of `joinUserIDs(ids) (csv string, count int)` so the
trimming and the safety decision share a single source of truth.
`RevealChannel` is always single-user by construction so it always
takes the safe translation path. The sentinels
(`ErrAlreadyArchived`, `ErrAlreadyInChannel`) are still EXPORTED for
callers using the low-level `Client.Do` surface — the per-method
idempotent translation is a M1.1.b boundary contract, not a
package-wide rewrite.

**`Client.RevealChannel` is a thin single-user wrapper over
`InviteToChannel`, NOT a separate Slack endpoint.** Both ride
`conversations.invite` on the wire. Splitting the Go surface mirrors
the M1.1 AC vocabulary (`InviteToChannel(channelID, userIDs)` for the
bot-fan-out at K2K Open(); `RevealChannel(channelID, userID)` for the
human-opt-in CLI) so the M1.1.c lifecycle layer keeps a 1:1
call-site → primitive mapping. RevealChannel delegates to
InviteToChannel with a single-element slice, which means the
`liftInviteError` table stays in one place and every documented Slack
code lifts identically across the two entry points.

**Whitespace-trim + fail-fast precedes Slack on every entry point.**
Mirrors the M1.1.a `participants` validator (`strings.TrimSpace`
before accepting). `CreateChannel("   ", true)` returns
`ErrInvalidChannelName` synchronously without burning a tier-2
rate-limit token on `conversations.create`. `InviteToChannel(
chan, ["   ", ""])` returns `messenger.ErrUserNotFound` synchronously
when every entry trims to empty. `RevealChannel(chan, " ")` likewise
fails-fast with `messenger.ErrUserNotFound`. The shared
`joinUserIDs` helper drops whitespace-only entries while joining the
CSV that `conversations.invite` expects, so a `[" U1 ", "", "U2"]`
input lands on the wire as `"U1,U2"`.

**Slack error code lifting goes through `APIError.Unwrap` + per-method
`liftXxxError`.** New sentinels (`ErrChannelNameTaken`,
`ErrInvalidChannelName`, `ErrAlreadyArchived`, `ErrIsArchived`,
`ErrAlreadyInChannel`, `ErrCantInviteSelf`) ride on
`APIError.Unwrap`'s switch so `errors.Is(err, ErrXxx)` works on every
[*APIError] returned from `Client.Do`, regardless of which
per-method lift was applied. The per-method lifts
(`liftCreateChannelError`, `liftInviteError`, `liftArchiveError`,
`liftListError`) add Slack-method context and translate Slack codes
that map to PORTABLE `messenger.*` sentinels (e.g. `channel_not_found`
→ `messenger.ErrChannelNotFound`) so adapter callers can match
without importing the slack package. Mirrors `liftHistoryError` /
`liftOpenIMError` from M4.2.b/c.

**Rate-limiter tier registry extended in-place.** Added three new
entries to `defaultMethodTiers`: `conversations.create` (Tier2),
`conversations.invite` (Tier3), `conversations.archive` (Tier2). The
existing tier-bucket bookkeeping handles the new methods without code
changes — the registry is the single source of truth, mirrored at
the call-site by the `conversationsXxxMethod` constants that pin the
URL path and the registry key together via the compiler.

**Source-grep AC bans `keeperslog.` + `.Append(` in `channels.go`.**
M1.1.b is pure adapter surface; audit emission and token-budget
accounting are M1.1.c's concern. The test
`TestChannels_NoAuditOrKeeperslogReferences` reads the source file
verbatim, strips line comments, and asserts neither token appears in
executable code. Mirrors the M7.1.c.\* source-grep AC discipline so a
future contributor adding audit emission inside the primitive layer
trips a fast-failing test before the call-site change can ride out of
review.

## 2026-05-17 — M1.1.a: K2K conversation domain + Postgres persistence

**PR**: [#155](https://github.com/vadimtrunov/watchkeepers/pull/155)
**Merged**: f6a28f445304037ea07247c6ec6915a6f0edb981 on 2026-05-17

### Context

Established the K2K conversation domain at `core/pkg/k2k/`: the
`Conversation` projection, a `Repository` interface seam, an
in-memory `MemoryRepository` (production-ready for dev / smoke loops
and the sole test surface), and a Postgres-backed `PostgresRepository`
wired against a per-call `Querier` (typically the `pgx.Tx` from
`core/internal/keep/db.WithScope`). Matching migration `029_k2k_conversations.sql`
adds the table with FORCE-RLS keyed off `watchkeeper.org` (the same
GUC migration 013 introduced for manifest tenancy), per-FK + partial
indexes, and CHECK constraints on the status enum + non-empty subject
/ participants invariants. M1.1.a deliberately ships ONLY the storage
layer — no Slack adapter calls (M1.1.b), no lifecycle wiring (M1.1.c),
no audit emission (M1.4), no budget enforcement (M1.5).

### Pattern

**In-memory + Postgres Repository pair shipped together, but only the
in-memory impl carries Go-side tests.** Prior art (M7.1.a saga DAO,
M2.1.b `keepers_log`, M9.4.b proposal store) settled on "ship in-memory
DAO + tests with consumer; Postgres adapter is a follow-up paired with
consumer wiring". M1.1.a's AC explicitly demands "in-memory + Postgres
impls" — resolved by shipping a thin Postgres adapter behind a
compile-time `var _ Repository = (*PostgresRepository)(nil)` assertion
plus extending `scripts/migrate-schema-test.sh` with RLS assertions for
the new table. The Go-side test surface stays entirely on the in-memory
store (where the lifecycle, defensive-copy, concurrent-increment, and
per-org isolation behaviours are exhaustively pinned); the Postgres
adapter's correctness is exercised end-to-end by the M1.1.c lifecycle
wiring + the shell-script RLS gate, not by Go test-container
scaffolding at the leaf-package boundary.

**PostgresRepository takes a per-call `Querier`, NOT a `pgxpool.Pool`
(iter-1 fix).** Iter-1 review (codex P1) caught a load-bearing
correctness issue: wiring the adapter over a `*pgxpool.Pool` would
defeat RLS entirely. The pool checks out an arbitrary backend
connection per statement, so a `SET LOCAL ROLE wk_*_role` issued on
connection A is invisible to the next statement that lands on
connection B. Every Open / Get / List would either return zero rows
(USING NULL) or be rejected at WITH CHECK. The fix: introduce a
narrow `Querier` interface (`Query`, `QueryRow`, `Exec`) satisfied
by both `pgx.Tx` and `pgxpool.Pool`, and require the production
wiring to construct the adapter over the `pgx.Tx` obtained from
`core/internal/keep/db.WithScope`. This mirrors the same per-call
seam discipline as `keep/server/handlers_read.scopedRunner`. The
package doc-block + adapter doc-block now spell out the wiring
contract explicitly so the M1.1.c lifecycle helper cannot regress.

**`correlation_id` is `uuid` on both sides of the seam (iter-1 fix).**
Iter-1 review (codex P2, critic confirm) caught a contract drift:
the Go struct + `OpenParams` declared `CorrelationID` as `string`
("opaque id"), but the SQL column was `uuid NULL`. A caller passing a
non-UUID string (e.g. `"saga-7"`) would pass in-memory tests and fail
in production with `invalid input syntax for type uuid`. Fix: change
the Go field + param type to `uuid.UUID`, use `uuid.Nil` as the
"unset" sentinel, and pass through pgx as a `*uuid.UUID` so
`uuid.Nil` maps to SQL NULL via the standard nilable-pointer idiom
(no `NULLIF($n, '')::uuid` trickery). The contract is now symmetric
across both impls.

**Nullable columns scanned into pointer targets, not magic-value
sentinels (iter-1 fix).** Iter-1 critic flagged a hack — using
`COALESCE(closed_at, '0001-01-01'::timestamptz)` plus a year-greater-
than-1 probe to detect SQL NULL. The fix uses pgx's standard
nilable-type idiom: `*time.Time`, `*uuid.UUID`, and `*string` scan
targets. A NULL column scans as a nil pointer, dereferenced into the
Conversation field only when non-nil. The projection logic is
centralised in a private `scanConversation(pgx.Row)` helper used by
both `Get` and `List` so a future schema delta touches one place.

**FORCE-RLS keyed off the `watchkeeper.org` GUC, identical shape to
migration 013.** The K2K table inherits the manifest-tenant RLS
pattern exactly — `nullif(current_setting('watchkeeper.org', true), '')::uuid`
in both USING and WITH CHECK so an unset GUC evaluates to SQL NULL
and the policy fails closed (zero rows visible, no INSERT permitted).
Three per-role policies (`wk_org_role` / `wk_user_role` /
`wk_agent_role`) mirror the row-set the manifest policies already grant
through. This keeps the per-tenant isolation contract uniform across
every tenant-scoped table in the schema; future K2K consumers can rely
on `core/internal/keep/db.WithScope` to establish the GUC before
calling the Postgres adapter, with no per-call tenant threading.

**Atomic `IncTokens` via `UPDATE ... WHERE status = 'open' RETURNING
tokens_used`.** A concurrent increment-vs-close race is handled by
Postgres row-level locking — the conditional UPDATE matches zero rows
when the row has transitioned to `archived` and the adapter surfaces
`ErrAlreadyArchived` after a follow-up status probe. The in-memory
adapter mirrors the contract by holding the write-lock for the full
read-modify-write under one acquisition (a 16-goroutine concurrency
test pins the "no lost update" invariant; a naive RLock-read +
Lock-write split would silently lose increments). The
`RETURNING tokens_used` shape lets the M1.5 enforcement layer compare
against `token_budget` without a follow-up `Get` round-trip.

**Defensive deep-copy of `Participants` on every boundary.** The
in-memory adapter clones the caller-supplied participants slice on
`Open`, returns clones on `Get` / `List`, and tests pin both directions
of the contract (caller mutation post-`Open` must not bleed; caller
mutation of a `Get` result must not bleed). Mirrors `cloneProposal` in
`core/pkg/approval/proposalstore.go`. The Postgres adapter inherits the
contract structurally — pgx serializes the slice once at QueryRow
time, so a caller-side mutation between `Open` returning and pgx
serializing cannot bleed, but a future contributor adding a code path
that holds the slice across the call must apply the same defensive
copy.

**Fail-fast validation precedes mutex / persistence in every `Open` /
`IncTokens` path.** The validator chain runs BEFORE any lock acquire
and BEFORE any SQL roundtrip. Mirrors the M7.1.b "fail-fast precedes
audit/state side effects" discipline. Validators surface typed
sentinels (`ErrEmptyOrganization`, `ErrEmptySubject`,
`ErrEmptyParticipants`, `ErrInvalidTokenBudget`,
`ErrInvalidTokenDelta`, `ErrAlreadyArchived`) so callers can branch on
`errors.Is` rather than parsing the error chain. The whitespace-only
participant check (`strings.TrimSpace`) prevents a `"   "` entry from
smuggling past the empty-check and into a downstream `InviteToChannel`
call.

**Shell-script RLS gate extended in-place, not duplicated.** The new
RLS assertions for `k2k_conversations` (cross-tenant SELECT
invisibility, cross-tenant INSERT rejection, FK violation, unset-GUC
fail-closed, CHECK constraint enforcement on participants / subject /
status) land in `scripts/migrate-schema-test.sh` as blocks (q) through
(v), re-using the `org_a_id` / `org_b_id` fixtures seeded by the
earlier (l-prep) manifest-RLS block. Keeps the migration test matrix
in one place and lets a future tenant-scoped table land its RLS
assertions next to the existing ones rather than spawning a parallel
test harness.
