**Merged**: 2026-05-16 (squash commit `374ae5a`)

### Context

Closed Â§10 Phase B item B3 from `docs/ROADMAP-phase1.md`: "M4 rate-limiter load test â€” tier-2 burst + sustained limits honored under simulated load." The existing `core/pkg/messenger/slack/ratelimiter_test.go` already covered race-cleanliness at modest concurrency (32g Ã— 4 calls = 128 attempts, `TestRateLimiter_ConcurrentAllow`), per-tier isolation, refill cadence on a fake clock, and Wait ctx semantics â€” but had no assertion specifically targeting tier-2 (the load-relevant tier for `users.list`, `conversations.list`, and the `apps.manifest.*` family from M4.2.d), and no assertion that exercised the Wait blocking path against a real clock under contended throughput. ### Pattern

**Gated load-test file shape mirrors the gated bench-test file shape (B1 prior art)**: top-of-file `//go:build loadtest` tag + header doc-block naming BOTH the `make` target and the raw `go test -tags=loadtest ...` invocation so the file is self-documenting for a future reader who cannot ask the build system. The companion Makefile target lists `-tags=loadtest -run=TestRateLimiterLoad -bench=BenchmarkRateLimiter_Tier2 -benchmem` and intentionally omits `-race`: the race detector adds ~5Ã— wall-clock overhead and would invalidate the throughput projection the real-clock Wait assertion gates on. Race coverage for the same RateLimiter code path lives in the default `go-test` target via `TestRateLimiter_ConcurrentAllow`. Pattern: any test whose ASSERTION is a wall-clock measurement must run without `-race`; race coverage for the same code path belongs in a separate unit test on the default path. (`core/pkg/messenger/slack/ratelimiter_load_test.go:1-41`, `Makefile:130-145`)

**Burst-honored assertion uses 4Ã— the unit-test concurrency to make over-grant races dominate signal**: the unit test (`TestRateLimiter_ConcurrentAllow`) drives 32g Ã— 4 = 128 attempts against a 50-burst bucket. Doubling either dimension would not change the assertion shape â€” the load test pushes 128g Ã— 100 = 12 800 attempts against a 20-burst bucket (640Ã— overshoot). The ratio matters: any over-grant from a race in `bucket.take` would surface as `allowed > 20`, and the 640Ã— overshoot guarantees the assertion fails LOUD (an over-grant of even a single token shows clearly against the 20-token budget) rather than within statistical noise. Pattern: a load-shaped assertion's signal-to-noise ratio is `(load attempts) / (budget)`. Tune the load until that ratio is â‰¥ 100Ã—. (`ratelimiter_load_test.go:72-106`)

**Pin the assertion against the SHIPPED default method-tier registry, not against an override**: `TestRateLimiterLoad_Tier2BurstHonored` deliberately uses `users.list` (which is `Tier2` per the package's `defaultMethodTiers` map) instead of overriding an arbitrary method to Tier2. The precondition `if got := rl.Tier(method); got != Tier2` makes the dependency explicit â€” a future change to the registry (e.g. demoting `users.list` to Tier3) flips the test red and forces the contract to be re-justified in the same PR. Pattern: load tests pinned against override values silently survive registry drift; load tests pinned against shipped defaults catch that drift early. (`ratelimiter_load_test.go:92-95`)

**Sustained-rate-across-multiple-windows is a strictly stronger assertion than single-window burst, and the float-stability precondition fails LOUD on future tier drift**: a refill bug that adds Sustained Ã— 2 tokens per Window would still be Burst-capped within any single Window snapshot â€” so the burst assertion would pass. Asserting per-Window grant equals `min(Sustained, Burst)` for N consecutive Windows (5 in this test) gates the refill cadence independently of the bucket cap. The generalisation `want := min(Sustained, Burst)` is correct even when callers later set `Sustained != Burst` (the leaky-bucket model caps the refilled token count at `Burst`). Iter-1 surfaced a related fragility: tier-2's default 20-req-per-60-s budget has refillRate 20/60 â‰ˆ 0.333..., and the product (rate Ã— Window-seconds) HAPPENS to round to exactly 20.0 under float64 arithmetic; a future budget update to e.g. Sustained=22 would produce 21.999... and silently flip the per-Window assertion red. The test now opens with an `if math.Abs(refilled - Sustained) > 1e-9` guard that fatals with a clear "test design needs adjustment for the new budget" message â€” a future drift surfaces as a test-design TODO, not a flake. Pattern: when an assertion's correctness depends on a float-arithmetic property of the inputs, encode the property as a precondition that fails LOUD with remediation guidance, not as an implicit assumption. (`ratelimiter_load_test.go:108-196`)

**Real-clock Wait throughput assertion needs a tolerance band on MEASURED elapsed, NOT the configured constant**: `TestRateLimiterLoad_Tier2WaitConcurrentThroughput` runs 32 goroutines calling `rl.Wait` for 3 s wall-clock (15 Windows Ã— 200ms â€” iter-1 caught that a tighter 1.5 s budget = 7.5 Windows could flake 15-20 % under noisy-neighbor load on 2-vCPU CI runners; doubling the window count halves the jitter contribution). The projection is computed against the MEASURED `time.Since(start)` elapsed, not the configured `loadDuration` constant, so a runtime that returns from the ctx timer slightly early or late shifts the bound to match â€” the assertion stays bias-free regardless of scheduler jitter. The Â±15 % tolerance band is asymmetric in meaning: the upper bound is HARD (any result strictly above is a budget violation â€” the leaky-bucket model forbids it), the lower bound is SOFT (a result far below suggests timer-rounding overhead OR a fairness regression where one goroutine starves the rest). Two distinct error messages tell a future failure-investigator immediately which half tripped. Pattern: real-clock throughput assertions need (a) measured-elapsed projection (b) asymmetric error messages because over- and under-shoot mean different bugs. (`ratelimiter_load_test.go:198-302`)

**Fake clock for budget reasoning, real clock for blocking-path reasoning**: the Allow assertions use a fake clock because the test reasons about discrete refill events the test fully controls â€” substituting a fake clock there is the right call. The Wait assertion uses a real clock because Wait's implementation uses `time.NewTimer` to schedule its sleep, and substituting a fake clock there would mock out the very behaviour under test (does Wait actually block the right amount of time, does the runtime wake waiters fairly). The split is principled: fake clock for "what would the budget grant given exact elapsed time" and real clock for "does the production scheduler honour that budget under contended wakeups." Pattern: when a code path's correctness depends on the wall-clock runtime (timers, sleep cadence, scheduler fairness), a fake clock test cannot stand in for a real-clock test of the same path. (`ratelimiter_load_test.go:204-211`)

## 2026-05-04 â€” M4.1: portable adapter interfaces avoid platform leaks via Metadata maps

**PR**: [#41](https://github.com/vadimtrunov/watchkeepers/pull/41)
**Merged**: 2026-05-04 (squash sha `eeb4049`)

### Pattern

When defining a Go interface that abstracts over multiple platforms (messengers,
databases, secret stores, etc.), ALL platform-specific fields belong in an opaque
`Metadata map[string]string` or `map[string]any`, not as named struct fields.
M4.1's `MessengerAdapter` keeps the cross-platform spine (`Message`,
`IncomingMessage`, `BotProfile`, `UserQuery`, `User`) plain â€” every Slack-specific
concept (`thread_ts`, `channel_type`, `enterprise_id`) lives in `Metadata`.
Reviewers verify zero platform-specific identifiers leak into source via `grep`
against the suspected platform's API vocabulary. The M4.2 (Slack) author then
populates `Metadata` at integration boundaries; future adapters (Discord, Teams)
populate the same `Metadata` key conventions without interface churn. Apply when
defining the next adapter interface (`RuntimeAdapter`, `LLMProvider`).

## 2026-05-04 â€” M4.2.b: typed map envelopes lie about wire format; assert raw bytes

**PR**: [#43](https://github.com/vadimtrunov/watchkeepers/pull/43)
**Merged**: 2026-05-04 (squash sha `872b8a1`)

### Pattern

When an HTTP request body is constructed from a `map[string]string`, EVERY value
goes on the wire as a JSON string â€” even fields the upstream API expects as
integer/bool/etc. The first M4.2.b PR shipped Slack's `status_expiration` as
`"1234567890"` (string), which Slack rejects as `invalid_profile`. Tests that
decode into `json.Number` are insufficient because `json.Number` is
`type Number string` â€” `Int64()` succeeds on a quoted string value too, giving
false confidence. Assertions MUST check the raw wire bytes: required substring
`"status_expiration":1234567890` (number literal) AND banned substring
`"status_expiration":"1234567890"` (quoted form). When a typed envelope can't
carry the right shape, switch to `map[string]any` and parse-on-store. Apply when
implementing any HTTP wire adapter against a typed external API.

## 2026-05-04 â€” M4.2.c.1: ack BEFORE handler dispatch decouples external timeouts from local latency

**PR**: [#44](https://github.com/vadimtrunov/watchkeepers/pull/44)
**Merged**: 2026-05-04 (squash sha `9f2c6ba`)

### Pattern

When an external protocol mandates an ack within N seconds (Slack Socket Mode = 3s),
ack ON RECEIPT before invoking any user-supplied handler. The handler runs AFTER ack,
fire-and-forget. Coupling ack timing to handler completion is wrong because: (a) handler
latency is unknowable (could be a slow DB call, an LLM round-trip, etc.); (b) at-least-once
semantics live at the protocol layer â€” Slack will redeliver if we crash before ack, so the
ack window is solely about "did the wire receive the event," not "did we successfully process
it." Test discipline: pin the contract by running a deliberately slow handler (1.5s+) and
asserting the ack arrives within Nms regardless. Apply when next implementing any
event-protocol consumer (webhook receiver, Kafka consumer with manual commit, etc.).

## 2026-05-04 â€” M4.2.c.2: per-session ctx is required to unblock blocking reads on reconnect

**PR**: [#45](https://github.com/vadimtrunov/watchkeepers/pull/45)
**Merged**: 2026-05-04 (squash sha `6d3ac20`)

### Pattern

When a streaming consumer reconnects (Socket Mode disconnect envelope, WS transport
error, pong-timeout), the parent `runCtx` is still live but the CURRENT connection
must be torn down. A naive `conn.Read()` in a pump goroutine parks indefinitely
because the server holds the connection open after sending its disconnect frame
waiting for the client close. Solution: wrap each session in
`context.WithCancel(runCtx)` and cancel `sessCtx` on every reconnect transition.
The pump's `conn.Read(sessCtx)` or its `select`-with-`ctx.Done` unblocks promptly.
Without it, reconnects either deadlock or leak the pump goroutine. Apply when
implementing any reconnect-resilient streaming consumer (WebSocket, SSE, gRPC stream).

## 2026-05-04 â€” M4.2.d.1: Slack `apps.manifest.create` modern shape uses OBJECT-typed manifest, not string-encoded JSON

**PR**: [#46](https://github.com/vadimtrunov/watchkeepers/pull/46)
**Merged**: 2026-05-04 (squash sha `a7ef1b5`)

### Pattern

When implementing an HTTP wire adapter, always re-read the target API's CURRENT
documentation rather than relying on prior intuition. The legacy
`apps.manifest.update` (form-urlencoded) wrapped the manifest as a STRING-typed
field carrying inner JSON. The modern JSON-bodied `apps.manifest.create` expects
the manifest as a regular object field: `{"manifest": {...}}`. The byte-level
wire-format discipline from M4.2.b applies â€” assert `"manifest":{` (object
literal) required AND `"manifest":"{` (escaped-string-encoded inner) banned. The
typed `manifestSettingKey{name, kind}` registry catches future drift where new
fields might land with the wrong JSON type. Apply when adapting any external API
surface that has multi-version contracts.

## 2026-05-04 â€” M4.2.d.2: token-sink callback beats interface change for adapterâ†’secrets handoff

**PR**: [#47](https://github.com/vadimtrunov/watchkeepers/pull/47)
**Merged**: 2026-05-04

### Pattern

When an adapter must hand secrets (OAuth tokens, API keys) to the application's
secrets store, but the existing `secrets.SecretSource` interface is read-only and
adding a `secrets.SecretSink` would be out-of-scope, use a per-Client functional
option that takes a callback: `WithInstallTokenSink(func(ctx, AppID, WorkspaceRef,
InstallTokens) error)`. The callback runs inside the operation's ctx (cancellation
propagates), the missing-sink case fails synchronously before any external call
(no transient leak window), and the caller wires whatever secret-store they have
â€” vault, SSM, env, in-memory test fake â€” without forcing an interface change.
Companion pattern: per-call params resolver via `WithInstallParamsResolver(...)`
for read-side secrets when the input type lacks a Metadata bag (here
`messenger.WorkspaceRef`). Apply when adding an adapter method that needs to
either read secrets-by-key or write secrets without contract gymnastics.

## 2026-05-04 â€” M4.3: write-out JSON file is the right Phase-1 bridge when secrets interface is read-only

**PR**: [#48](https://github.com/vadimtrunov/watchkeepers/pull/48)
**Merged**: 2026-05-04

### Pattern

When an existing `secrets.SecretSource` interface is read-only and adding a
write side would be out-of-scope (interface change blast radius), bridge the
gap with a structured-JSON write-out file (mode `0o600`) the operator pipes
into their secrets store via `jq | vault kv put` (or analogous). Pair this
with a future-proof callback option on the adapter (`WithCreateAppCredsSink`)
so the file-output sink is just a Phase-1 binding â€” a vault-backed sink drops
in later without script rewrite. Companion verification: a binary-grep test
that builds the binary with `-trimpath` + `CGO_ENABLED=0` and asserts no
`xox[bpe]-` / `xapp-` patterns appear. Apply when an adapter must hand secrets
to the application but the secrets contract isn't ready for write-side yet.

## 2026-05-04 â€” M4.4: documenting a security gap requires a TRACKED follow-up item, not "see review"

**PR**: [#49](https://github.com/vadimtrunov/watchkeepers/pull/49)
**Merged**: 2026-05-04

### Pattern

When a security fix is deferred (e.g., cross-tenant authorization in M4.4
deferred pending `auth.Claim` refactor), the deferral comment must point at a
CONCRETE, UNCHECKED, TRACKED roadmap/backlog item â€” not at the review thread,
the milestone label, or "future work". Pointers like "see M4.4 review" become
dead references the moment the milestone is marked `[x]`. Reviewers verify
both: that the deferral is honest (gap named, threat described, fix shape
sketched) AND that there is a discoverable line item a future contributor can
pick up. Companion: also surface the operator-deployment posture in
`DEVELOPING.md` so ops/SRE see it without reading source code. Apply when
shipping any code with a known security posture different from the eventual
production posture.
