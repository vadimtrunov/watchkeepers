### Review iter-1 fixes

Two parallel reviewers ran on `rdd/M10.3-compose-grafana`:

  OpenAI codex CLI; `omc` wrapper not installed; the first attempt
  with `--base main` + custom prompt was rejected by codex CLI 0.130.0

- `general-purpose` agent dispatched with critic framing (the
  `oh-my-claudecode:critic` plugin is not installed in this
  environment, so the closest equivalent was used — same precedent
  as M10.1 and M10.2).

Operator chose "All findings". Union of 19 valid findings (20 from
critic + 2 from codex, minus the 2 critic dupes that overlapped with
codex on the DSN-password concern, minus 1 critic finding flagged as
factually-wrong: critic #8 claimed migrations 017/024 do not exist,
but `deploy/migrations/017_watchmaster_manifest_seed.sql` and
`deploy/migrations/024_coordinator_manifest_seed.sql` both ship in
the repo today). Each fix lands on the same branch before commit;
regression-pinned by a named test in the
`TestIter1_*` family or by a rewritten existing test.

- **codex-P1 + critic-#1 — DSN password URL-encoding + visibility.**
  The original compose embedded `${POSTGRES_PASSWORD}` directly in
  `postgres://...` DSN strings (env-block on both `migrate` and
  `keep`). Two failure modes: (a) URI-reserved characters in the
  password (`@:/?#`) break the DSN at parse time, so many generated
  passwords from password managers would crash both services; (b)
  the DSN with cleartext password is visible to anyone with
  `docker inspect` daemon access. Fix:
  `deploy/migrate-entrypoint.sh` now assembles the DSN INSIDE the
  container from component env vars
  (`POSTGRES_USER`+`POSTGRES_PASSWORD`+`POSTGRES_HOST`+`POSTGRES_PORT`+`POSTGRES_DB`)
  with an awk-based URL-encoder for the password and user, AND it
  reads the password from `/run/secrets/postgres_password` via the
  docker-secrets mount instead of an env var. The compose
  `secrets:` block (new) declares both `postgres_password` and
  `keep_token_signing_key` as file-mounted secrets. Pinned by
  `TestIter1_PostgresPasswordViaDockerSecret`. For the keep
  service, see codex-P2 + critic-#2 below.

- **codex-P2 — `.env` leaks into Docker build context.**
  The repo had no `.dockerignore`; every `docker compose build`
  uploaded the operator's `.env` (containing the freshly-provisioned
  signing key + admin password) to the Docker daemon as part of the
  build context. On a remote daemon this leaks credentials off the
  workstation; even locally, the daemon-side cache retains the
  upload. Fix: new `.dockerignore` at repo root excludes `.env*`,
  `secrets/`, `.git/`, build outputs, IDE noise, and the docs tree
  (Dockerfiles only `COPY` deploy/migrations + entrypoint scripts
  by name, so the broader docs surface is pure dead-weight).

- **critic-#1 + critic-#2 — KEEP_TOKEN_SIGNING_KEY visible via
  `docker inspect`.** Extended `core/internal/keep/config/config.go`
  with a `envValue(name)` helper that recognises `<name>_FILE` env
  vars and reads the file contents (trimming trailing newline) when
  the `_FILE` variant is set. Compose now passes
  `KEEP_TOKEN_SIGNING_KEY_FILE=/run/secrets/keep_token_signing_key`
  alongside a `secrets:` mount instead of the cleartext base64 env. Same
  fallback shape works for `KEEP_DATABASE_URL_FILE` for operators
  who prefer to mount a pre-assembled DSN. Pinned by
  `TestIter1_TokenSigningKey_FileFallback` +
  `TestIter1_DatabaseURL_FileFallback` +
  `TestIter1_DatabaseURL_FileMissing` (config_test.go) and
  `TestIter1_KeepTokenSigningKeyViaDockerSecret` +
  `TestIter1_PostgresPasswordViaDockerSecret` (compose_test.go).

- **critic-#3 — agent-stub services exited 0 and confused
  `compose up --wait`.** The iter-0 stub commands ran
  `echo "..."; exit 0` with `restart: "no"`, so the four agent stubs
  appeared as `Exited (0)` within seconds. Operators (and any
  future `compose up --wait` gate) cannot distinguish "intentional
  stub" from "crashed". Fix: each stub now runs
  `echo "..."; exec sleep infinity` AND is gated behind the new
  `profiles: [stubs]` profile so default `docker compose up` does
  not start them. Operators probe the future shape with
  `docker compose --profile stubs up <name>` — the stub stays in
  `Running` state with the banner readable via `compose logs`.
  Pinned by `TestIter1_AgentStubsGatedByProfile` +
  `TestIter1_DataPlaneServicesNotGatedByProfile`.

- **critic-#4 — `Dockerfile.migrate` built goose without
  `CGO_ENABLED=0`.** A goose binary built CGO-enabled from
  `golang:1.26-alpine` copied into `alpine:3.20` could fail at
  runtime if it resolved libc symbols absent in the smaller runtime
  image. Fix: prepended `CGO_ENABLED=0` to the `go install`
  invocation so the binary is statically linked, mirroring the
  `deploy/Dockerfile.keep` pattern.

- **critic-#5 — Prometheus + Grafana bound to all interfaces.**
  Default `ports: ["9090:9090"]` and `["3000:3000"]` bind to
  `0.0.0.0`; the un-authenticated PromQL API leaks to anything on
  the same dev-laptop network. Fix: explicit `127.0.0.1:` prefix on
  every published port (`prometheus`, `grafana`, AND `keep` — the
  Keep service's own API requires auth but its `/metrics` mount is
  intentionally outside the auth wall and was likewise exposed).
  Operators with a deliberate reverse proxy override per-host.
  Pinned by `TestIter1_PrometheusAndGrafanaBoundToLocalhost`.

- **critic-#6 — `grafana.depends_on.prometheus: service_started`.**
  Prometheus has no native healthcheck; first scrape + config parse
  can take several seconds. Grafana's datasource provisioning may
  probe a not-yet-ready Prometheus and mark the datasource
  "unhealthy" on first load. Fix: added a `wget -qO-
http://localhost:9090/-/ready` healthcheck to the `prometheus`
  service and switched grafana's depends_on to
  `condition: service_healthy`. Pinned by
  `TestIter1_GrafanaDependsOnPrometheusHealthy`.

- **critic-#7 — LLM token-spend dashboard panel grouped by
  unbounded `agent_id`.** `sum by (agent_id, kind, provider)` works
  for a single agent but balloons cardinality (and Prometheus
  query memory) once N agents > ~50. Fix: default sum-by drops
  `agent_id`; a new `$agent_id` template variable provides
  cardinality-bounded drill-down via the panel matcher
  `agent_id=~"$agent_id"`. Pinned by
  `TestIter1_DashboardLLMPanelBoundsCardinality`.

- **critic-#9 — dashboard `schemaVersion: 39` lagged Grafana 11.x.**
  Bumped to `40`; pinned by
  `TestIter1_DashboardSchemaVersionGrafana11`. Grafana 11.x runs
  an in-memory auto-migration when loading older schemas; the
  on-disk version stayed stale because `allowUiUpdates: false`
  prevents writes back. Updating the pin silences the log line on
  every grafana restart.

- **critic-#10 — `.env.example` placeholder for
  `KEEP_TOKEN_SIGNING_KEY` was not valid base64.** Operators who
  copied `.env.example` to `.env` and ran `compose up` without
  editing the literal `replace-with-base64-encoded-key-of-at-least-32-bytes`
  hit a config-load error rather than the documented
  fail-fast `${VAR:?...}` message. Fix: the key is no longer in
  `.env` at all — it is mounted via the
  `keep_token_signing_key` docker secret file. The `.env.example`
  now documents the `openssl rand -base64 48 >
secrets/keep_token_signing_key` provisioning step.

- **critic-#11 — Grafana 11.3.0 close to EOL.** Bumped pin to
  `grafana/grafana:11.5.4` (current LTS as of 2026-05).

- **critic-#15 — `repoRoot` non-relocatable under sandboxed test
  runners.** Added a `WK_REPO_ROOT` env override; the
  `runtime.Caller(0)` fallback stays for the common case.
  Sandboxed runners (bazel, `go test -C`) export
  `WK_REPO_ROOT=/path/to/repo` before invoking the deploy test
  package.

- **critic-#14 — migrate-entrypoint did not log completion verb.**
  Added a trailing `echo "migrate: complete (${verb})"` so
  `compose logs migrate` is grep-friendly when the M10.4 smoke
  triage runs against a stuck `up`.

- **critic-#16 — `GF_SECURITY_ADMIN_PASSWORD` defaulted to
  `admin`.** Now REQUIRED via `${VAR:?...}` substitution — same
  pattern the `POSTGRES_PASSWORD`-replacing-secret-mount uses for
  the Postgres credential. The `admin/admin` default was a
  foot-gun: the `grafana_data` volume persists the rotated
  password silently, so the next operator inheriting the stack saw
  the rotated value and could not log in.

- **critic-#17 — `ca-certificates` dead-weight in
  `Dockerfile.migrate`.** The migrate sidecar connects to Postgres
  over the docker bridge with `sslmode=disable`; the trust store
  has no verification work to do. Dropped the `apk add` step.

- **critic-#18 — em-dash in `${VAR:?msg}` substitution messages.**
  Replaced with ASCII `--` so terminals without UTF-8 handling
  render the diagnostic cleanly.

- **critic-#19 — datasource uid duplicated in every panel.** Added
  a `datasource` template variable defaulting to `prometheus`;
  every panel now uses `"datasource": { "type": "prometheus",
"uid": "${datasource}" }`. A future user-side datasource swap
  (e.g. Cortex, Mimir, Thanos) is a single template-variable
  change. Pinned by `TestIter1_DashboardDatasourceTemplating`.

- **critic-#20 — `DOCKER_COMPOSE ?=` Makefile override
  undocumented.** Added a paragraph above the variable naming the
  hyphenated v1 binary as the legacy escape hatch + the v2 syntax
  requirements (`condition: service_completed_successfully`,
  `secrets:` mount) the compose file assumes.

- **critic-#12 + critic-#13 — documentation nits.** Documented the
  15/15 scrape/eval cadence choice in this lesson (#12) and the
  `--force-recreate` re-migration behaviour in the compose file
  comment block (#13).

### Context

`docs/ROADMAP-phase1.md` §M10.3 asks for a finalized compose stack
covering six named services (core, keep, postgres, watchmaster, sample
coordinator, dev Slack socket bridge) AND a Grafana starter dashboard.
Phase 1 Definition of Done §7 pins the contract:
"`docker-compose up` brings Phase 1 online with no manual steps beyond
secret provisioning."

Today only two of those six services have buildable binaries (`keep`,
which is fully wired with `/metrics` after M10.1, and `core` /
`watchkeeper`, which is still a placeholder that prints text and
exits). The long-running harness supervisor that hosts agents lands in
a follow-up. Two paths were considered:

1. Drop the un-wired services from compose until binaries exist.
   Default `compose up` is clean but the future deployment shape is
   invisible to operators and runbooks.
2. Pin all six services NOW with the un-wired ones as loud one-shot
   stubs naming the follow-up M-id. Same shape as M10.2's
   `notWiredExit` exit-3 stubs and M10.1's pre-pinned Prometheus name
   set.

Path (2) won, mirroring prior art. Default `make compose-up` boots the
data plane (postgres → migrate → keep) + observability plane
(prometheus + grafana) AND prints the four stub services' follow-up
M-id banners on first `up`.

The PR therefore lands: top-level `docker-compose.yml` (9 services,
3 named volumes, 1 bridge network), `deploy/Dockerfile.migrate`
(multi-stage build of `pressly/goose v3.27.0` matching the Makefile's
`GOOSE_VERSION` pin), `deploy/migrate-entrypoint.sh` (POSIX wrapper
that injects `$DATABASE_URL` into goose's CLI shape),
`deploy/observability/prometheus.yml` (single scrape job for
`keep:8080/metrics`),
`deploy/observability/grafana/provisioning/{datasources,dashboards}/*`
(provisioned Prometheus datasource + file-provider dashboard
loader), `deploy/observability/grafana/dashboards/watchkeeper-phase1.json`
(starter dashboard with one panel per `wkmetrics.expectedFamilies`
plus the default process collectors), `.env.example` (operator
template for the four required secrets), eight Makefile targets
(`compose-{up,up-d,build,down,down-volumes,logs,ps}`), and a Go test
package `core/internal/deploy/compose_test.go` (six tests pinning the
service set, dependency graph, scrape contract, dashboard
coverage, and datasource uid).

### Pattern

1. **Contract-stub services name the follow-up M-id in their own
   stdout.** Compose declares `core` / `watchmaster` / `coordinator` /
   `slack-bridge` as `image: alpine:3.20` services whose `command`
   prints `<service>: contract stub — <description> in M10.3.<b|c>` and
   exits 0 (`docker-compose.yml:178-242`). Operators reading
   `compose logs` on a fresh `up` see the future deployment shape
   AND the grep target (`M10.3.b`, `M10.3.c`) that names the wiring
   trail. Same pattern as M10.2's
   `runStubs.go` / `notWiredExit(stderr, label, followup)` (exit-3
   contract) and M10.1's pre-pinned Prometheus name set — pin the
   contract NOW, wire the call site later.

2. **The Grafana dashboard's PromQL coverage is contract-tested
   against `wkmetrics.expectedFamilies`**
   (`core/internal/deploy/compose_test.go:TestDashboardCoversExpectedFamilies`).
   The test walks every panel's `targets[].expr` in
   `watchkeeper-phase1.json` and asserts each of the seven
   `watchkeeper_*` metric stems appears in at least one PromQL string.
   The `expectedFamilies` list is intentionally duplicated from
   `core/pkg/wkmetrics/wkmetrics_test.go` rather than extracted to a
   shared constant — a metric rename has to touch BOTH lists, with
   the test name making the intent explicit. This catches the silent
   regression where a renamed Prometheus family blanks a panel
   without surfacing a runtime error.

3. **The dashboard `uid` and the Grafana datasource `uid` are pinned
   by separate tests so a rename is two contract failures, not one.**
   `TestDashboardUIDStable` (`uid: watchkeeper-phase1`) pins the
   permalink shape `/d/watchkeeper-phase1/...` operators bookmark;
   `TestGrafanaDatasourceUIDStable` (`uid: prometheus`) pins the
   reference every panel's `datasource.uid` field uses. Renaming
   either uid without updating the other side would orphan every
   panel; the two distinct test names make the failure mode
   self-describing.

4. **Migration sidecar over auto-migration on Keep boot.** A separate
   one-shot `migrate` service (`docker-compose.yml:65-78` +
   `deploy/Dockerfile.migrate`) runs `goose -dir /migrations postgres
"$DATABASE_URL" up` and exits 0, gated by `postgres`'s healthcheck
   passing. Keep depends_on migrate with
   `condition: service_completed_successfully` so the HTTP server
   never serves a request against a stale schema. The alternative
   (auto-migrate on Keep boot) would couple the schema lifecycle to
   the application lifecycle — two-image rollbacks, blue/green
   deploys, and the M10.4 backup/restore drill all want migrations
   as a discrete artefact. The shape mirrors the existing
   `make migrate-up` / `make migrate-status` targets (which use the
   same goose version pin) so dev + compose run identical
   migration code paths.

5. **goose is built from source in a multi-stage Dockerfile pinned to
   the same `GOOSE_VERSION` constant the Makefile uses** (`deploy/Dockerfile.migrate`).
   Using `pressly/goose:3` from an external registry would either
   pull a moving `:3` tag or require a separate version pin — both
   risk drift from the Makefile's `GOOSE_VERSION ?= v3.27.0`.
   `go install ...@v3.27.0` in the build stage gives a single source
   of truth: bump the Makefile constant, bump the Dockerfile
   constant, both code paths move in lockstep.

6. **Compose `$$` escape is required for shell variables in
   `command:`** (`docker-compose.yml:200, 215`). The contract-stub
   services use a shell variable to capture the manifest UUID:
   `manifest_id="$${COORDINATOR_MANIFEST_ID:-...}"`. Without the
   double-dollar, Compose interpolates `${manifest_id}` at parse time
   (finds it empty in env, emits a warning, and substitutes an empty
   string into the rendered command). The `$$` escape passes a
   literal `$` through to the shell, where the `:-default` syntax
   resolves at runtime against either the operator's `.env` override
   or the documented manifest-seed UUID. Tested end-to-end: a one-off
   `docker compose run --rm coordinator` prints the resolved
   `00000000-0000-0000-0000-000000000024` UUID, not an empty string.

7. **`.env.example` uses `${VAR:?explanation}` in compose to fail
   fast on missing secrets** (`docker-compose.yml:54, 108, 109`).
   `compose config` exits non-zero with the explanation string for
   any of `POSTGRES_PASSWORD` / `KEEP_TOKEN_SIGNING_KEY` /
   `KEEP_TOKEN_ISSUER` if absent. The alternative (silent default
   like `${POSTGRES_PASSWORD:-changeme}`) would boot a stack with a
   well-known credential and surface the bug only at first real
   request. The explanation string names `.env.example` so the
   operator sees the fix path in the same diagnostic.

8. **POSIX-sh entrypoint with explicit `# shellcheck shell=sh`
   pragma** (`deploy/migrate-entrypoint.sh:13`). The runtime image
   (`alpine:3.20`) ships ash, not bash; shellcheck's default bash
   mode would flag `[ ]` tests and unbraced `$VAR` as bashism style
   nags (SC2292 / SC2250). The pragma tells shellcheck to apply POSIX
   rules and the SC2250 fix (braces) is taken anyway because it's
   strictly more portable. Same precedent as
   `scripts/test-migrate.sh` which declares its bash dependency at
   the shebang.

9. **`.prettierignore` carries `.env*` because prettier has no
   parser for KEY=VALUE shell-style files** (`.prettierignore`).
   Without the ignore, `pnpm exec prettier --check .` fails on
   `.env.example`. The grandfathered-docs comment block in the same
   file is the precedent.

## 2026-05-14 — M10.2: unified `wk` operator CLI subsuming `wk-tool`

**PR**: #145
**Merged**: c6cb672 on 2026-05-14

### Context

`docs/ROADMAP-phase1.md` §M10.2 asks for a single operator CLI covering
fifteen noun-groups (spawn / retire / list / inspect / logs /
tail-keepers-log / tool / tools sources / notebook / personality /
language / budget / approvals) plus a uniform `make wk CMD="…"` shortcut.
Following the M10.1 single-PR precedent, the operator chose the umbrella
shape over the natural a/b/c decomposition; the PR therefore lands the
full CLI contract in one go (`core/cmd/wk/`), absorbs the M9.5 / M9.6 /
M9.7 / M9.4.b operator surface from the retired `wk-tool` binary, and
pins the noun-group dispatcher + flag shapes for every command. Where a
shipped backend exists today (Keep HTTP surface for spawn / retire /
list / inspect / logs / tail / personality / language / budget show;
local-FS for notebook show / forget / export / import; `wk-tool`
migrations for tool local-install / rollback / hosted export / share)
the command is fully wired. Where the backend seam has not landed yet
(tool list / tool hosted list+show / tools sources \* / notebook
archive / notebook list-archives / budget set / approvals \*) the
command returns exit-3 with the
follow-up M-id named verbatim in the diagnostic — same pattern as
M10.1's stable Prometheus name-set pinned before every call site
landed.

`wk-tool` is removed in the same PR: `core/cmd/wk-tool/` is deleted and
the four migrated subcommands keep their flag shapes verbatim so
runbook snippets stay short — only the invocation prefix changes
(`wk-tool local-install …` → `wk tool local-install …`). The
`tools-local-install` make target preserves its name (M9.5 runbook
compatibility) and now forwards through `wk tool local-install` under
the hood.

### Review iter-1 fixes

Two parallel reviewers ran on `rdd/M10.2-wk-cli`:

- `codex review --base main --title "M10.2 wk CLI"` (real OpenAI codex
  CLI; `omc` wrapper not installed). Output captured to
  `/tmp/codex-review-M10.2.txt`.
- `general-purpose` agent dispatched with critic framing (the
  `oh-my-claudecode:critic` plugin is not installed in this
  environment, so the closest equivalent was used — same precedent
  as M10.1).

Operator chose "All findings". Union of 26 findings was applied on
the same branch before commit; each fix is regression-pinned by a
named test in the `TestIter1_*` family or by a rewritten existing
test.

- **C1 (critic) — `wk retire` had a legacy no-archive escape hatch.**
  The original `runRetire` called `UpdateWatchkeeperStatus(..., "retired")`
  when `--archive-uri` was absent; that path is the M6.2.c synchronous
  retire tool's seam, which intentionally elides the archive URI.
  Exposing it through the unified operator CLI was a one-line escape
  from the M7.2.b/c archive-on-retire invariant. Fix: `--archive-uri`
  is now REQUIRED (`run_watchkeeper.go:106-124`); the call site
  unconditionally rides through `UpdateWatchkeeperRetired`. Regression
  test `TestIter1_Retire_MissingArchive_UsageError` pins the gate.

- **C2 (critic) — `--reason` missing from every write subcommand
  outside `tool`.** The deleted wk-tool enforced `--reason` for every
  state-changing call; the unified CLI initially exposed spawn,
  retire, personality set, language set with no audit text. Fix:
  `--reason` is now REQUIRED on spawn/retire/personality set/language
  set (`run_watchkeeper.go:54-78,106-128`, `run_manifest.go:226-243`)
  AND echoed in the operator-facing summary so it lands in shell
  transcripts even when the Keep endpoint has no native reason field.
  Regression: `TestIter1_Spawn_MissingReason_UsageError`,
  `TestIter1_Retire_MissingReason_UsageError`.

- **M1 (critic) — read-merge-write had no version_no race guard.**
  Two concurrent `wk personality set` calls both computed
  `mv.VersionNo + 1`; one PUT lost the race and surfaced the
  keepclient's terse `ErrConflict`. Fix
  (`run_manifest.go:269-275`): detect `errors.Is(err,
keepclient.ErrConflict)` and emit a tailored "concurrent edit
  detected — a competing manifest version landed first; re-run the
  command" diagnostic. Pinned by `TestIter1_PersonalitySet_ConflictHint`.

- **M2 (critic) — legacy empty `system_prompt` rows blocked
  personality/language set.** The keepclient's PUT pre-flight rejects
  empty SystemPrompt; the copy-every-field merge shipped the prior
  row's value verbatim. Fix: optional `--system-prompt` flag on
  manifest set commands (`run_manifest.go:218,222,261-264`) so an
  operator can repair the legacy row in the same PUT. Override is
  write-scoped (no `show`-side counterpart).

- **M3, M4 (critic) + codex P3 — help text drift.** `wk --help` claimed
  `wk budget show [--scope ...]` but the implementation required
  `--agent --from --to [--grain]`; help also claimed `wk notebook show
... [--limit N]` but `runNotebookShow` registers no `--limit`.
  Codex P3: Makefile header advertised non-existent per-noun-group
  targets. Fix: `writeUsage` rewritten end-to-end to match every
  subcommand's actual flag set (`run.go:74-130`); Makefile header
  retracted to "composite shortcut is the SOLE general entry point",
  with an iter-1-m1 caveat about CMD word-splitting added separately.

- **M5 (critic) — SSE tail vulnerable to short writes.**
  Original `runTailKeepersLog` wrote directly to `*os.File`; a
  transient EAGAIN (e.g. terminal Ctrl-S) would kill the stream.
  Fix: wrap stdout in `bufio.NewWriter` and flush per frame
  (`run_logs.go:120-159`). The bufio.Writer retries short writes
  internally; flush-per-frame preserves interactive responsiveness.

- **M6 (critic) — `wk logs` accepted malformed positionals + negative
  limits.** A non-UUID `<wk-id>` silently produced zero matches against
  the client-side actor filter, indistinguishable from a real empty
  result; `--limit -1` reached `keepclient.LogTail` and surfaced the

### Pattern

1. **Hierarchical noun-group dispatcher beats flat subcommand list.**
   `core/cmd/wk/run.go` switches on the first arg (noun-group) and
   delegates to a per-noun-group dispatcher (`runTool`, `runNotebook`,
   `runPersonality`, …) which switches on the second arg (verb). The
   shape mirrors the ROADMAP text verbatim (`wk tool hosted export …`)
   and keeps each dispatcher small (≤60 LoC). A flat dispatcher (which
   M9.5's `wk-tool` used) does not scale past ~6 subcommands without
   either a generated parser or a third-party CLI lib.

2. **Centralise the Keep-client construction.** Every Keep-backed
   subcommand resolves the bearer token + base URL through one helper
   (`core/cmd/wk/keep.go` `withKeepClient(env, stderr, label, f)`).
   Missing env vars become exit-2 with a SINGLE diagnostic naming both
   `WATCHKEEPER_KEEP_BASE_URL` and `WATCHKEEPER_OPERATOR_TOKEN` (mirror
   the M9.5 accumulated-missing-flag pattern); the closure shape keeps
   each subcommand's error path one line. The token is intentionally
   ENV-only — `ps -ef` would leak it from argv.

3. **Exit code 3 = "contract present, backend pending"** (`run.go`
   `notWiredExit(stderr, label, followup)`). The CLI's contract surface
   is pinned NOW even where the seam is missing — every stub diagnostic
   names the follow-up M sub-item that will wire it (e.g. `M10.2.b` for
   `tools sources sync` / `notebook archive`, `M10.2.c` for `approvals
pending` / `budget set`). Operator runbooks can already reference the
   final CLI shape; the failure mode is loud and self-describing rather
   than silent or 0-stubbed. Mirror the M10.1 pattern (Prometheus name
   set defined up front; call sites land later).

4. **`make wk CMD="<noun-group> <verb> [flags]"` is the single
   composite shortcut.** Operators don't need a per-subcommand make
   target for every leaf — one wrapper that forwards CMD verbatim to
   `./bin/wk` covers every present and future verb. The Makefile keeps
   the legacy `tools-local-install` target name for runbook compatibility
   (M9.5 referenced it; M10.2 changes the underlying binary but not the
   target shape). A future `tools-rollback`, `tools-hosted-export`, etc.
   target are intentionally NOT in the Makefile — `make wk CMD="…"` is
   the supported general path; named targets only exist where a legacy
   runbook already references them.

5. **Read-merge-write for manifest field mutations**
   (`core/cmd/wk/run_manifest.go` `runManifestSet`). Personality and
   language `set` commands resolve <wk-id> → ManifestID → current
   ManifestVersion via two GETs, copy EVERY field of the in-memory
   ManifestVersion into a PutManifestVersionRequest with `version_no+1`
   and the target column overwritten, then PUT. The merge intentionally
   copies every field because the server has no PATCH API today; a
   missing field would let the jsonb column defaults overwrite a
   previously-set value. The closure shape (`apply
func(*ManifestVersion, string)`) keeps personality/language set
   one-line per field.

6. **Migrate-with-prefix-rename preserves all M9.5 tests** (`run_test.go`).
   The M9.5 / M9.6 / M9.4.b test bodies stayed verbatim under the new
   binary; only the subcommand args were prefixed with `"tool"` and one
   nested case with `"tool", "hosted", "export"`. Constants
   `dataDirEnvKey` / `localSourcesEnvKey` / `defaultLocalSource` /
   `errMissingFlags` / `jsonlPublisher` / `trustHintResolver` /
   `singleSourceLookup` / `hostedSourceLookup` / `shareSourceLookup`
   moved verbatim into `core/cmd/wk/run_tool.go`. The two redaction-harness
   tests (`TestRun_LocalInstall_JSONLStdoutOmitsFolderContentCanary`,
   `TestRun_Share_TokenCanaryNotEchoedToStdoutOrStderr`) carry forward
   exactly because they pin a runtime invariant that survives any
   prefix rename.

7. **`wk logs <wk>` is client-side filter, by design** (`run_logs.go`).
   Keep's `/v1/keepers-log` has no per-actor query parameter as of
   M10.2; the CLI requests up to 1000 events via `--limit` (server
   clamps) and drops rows whose `actor_watchkeeper_id != <wk>`. A
   future M10.2.d follow-up can add a server-side filter when keepers-log
   volume makes the 1000-row client filter inefficient. The default
   limit was sized to comfortably exceed a dev-workspace day of events
   while staying under Keep's documented `maxLogLimit` clamp.

8. **`wk tail-keepers-log` is the only long-lived subcommand**
   (`run_logs.go`). It uses `keepclient.Subscribe`'s SSE stream and
   prints each event frame's payload as one JSON line on stdout.
   Heartbeat frames are silently dropped by the keepclient scanner;
   the stream stays open until ctx is cancelled (SIGINT/SIGTERM,
   plumbed via `main.go`'s `signal.NotifyContext`). Both clean EOF
   (`io.EOF`) and a local close (`ErrStreamClosed`) terminate with
   exit 0 — only an unexpected error path emits exit 1.

9. **Notebook subcommands respect `$WATCHKEEPER_DATA`, not
   `$WATCHKEEPER_DATA_DIR`** (`run_notebook.go` package doc).
   `core/pkg/notebook/path.go`'s `agentDBPath` reads `WATCHKEEPER_DATA`
   verbatim from `os.Getenv`; the CLI does NOT shadow this (an attempt
   to push the toolregistry's `WATCHKEEPER_DATA_DIR` here would
   silently misroute the SQLite file). Documented honestly in the
   noun-group help and in this lesson so a future operator reading
   only the CLI help finds the rule without re-deriving it. The
   `notebook archive` + `list-archives` subcommands stub to exit-3
   today because they additionally need an `archivestore.Storer`
   configured via the Keep service's config — operator-side wiring
   for that storer lands in M10.2.b.

10. **`tools-local-install` make target is preserved through the
    rename** (`Makefile`). The wk-tool → wk migration changes the
    underlying binary; the Makefile target name stays so M9.5's
    `make tools-local-install FOLDER=… REASON=… OPERATOR=…
WATCHKEEPER_DATA_DIR=…` runbook snippet keeps working. The target
    body now calls `./bin/wk tool local-install` instead of
    `./bin/wk-tool local-install` and depends on `wk-build` instead
    of `wk-tool-build`. No operator-facing flag changes.

### Pattern

1. **Nil-safe recorders are the single contract carrier for cross-cutting
   instrumentation** (`core/pkg/wkmetrics/wkmetrics.go:272-336`,
   methods `RecordTokenSpend`, `ObserveLLMRequest`,
   `RecordToolInvocation`, `SetEventBusDepth`, `SetRateLimitRemaining`,
   `ObserveHTTPRequest`, `RecordOutboxPublished`, `Instrument`). Every
   record-* method short-circuits on `m == nil`, so callers can hold a
   `*wkmetrics.Metrics` field and call it unconditionally. The diff at
the outbox call sites becomes one-line additions
(`core/internal/keep/publish/outbox.go:225, 251, 257`) with no
`if m != nil` noise. The contract is named explicitly in the
   package doc so a reviewer reading a future call site finds the rule
   without re-deriving it.

2. **The full Phase-1 metric name set is pinned NOW, not when each
   call site lands** (`core/pkg/wkmetrics/wkmetrics.go:16` namespace
   constant + `wkmetrics.go:171-256` collector declarations). Renaming a
   Prometheus metric or label is a Grafana-dashboard-breaking change;
   pinning the contract at the package-creation point means future
   instrumentation sub-items (LLM token spend, tool invocations,
   rate-limit headroom) plug into a stable name/label surface without
   re-litigating naming. The metric-family list is duplicated in
   `wkmetrics_test.go:expectedFamilies` and asserted by
   `TestNew_RegistersExpectedFamilies` — a drift catches loudly in CI.

3. **Typed-enum label values beat source-grep AC.** Iter-1 review M4
   asked for a source-grep that would assert no free-form outcome
   strings landed in `WithLabelValues(...)` call sites. The cleaner
   answer is `type Outcome string` + `OutcomeSuccess` / `OutcomeError`
   typed constants (`wkmetrics.go:23-34`). Recorder signatures accept
   the typed enum; a free-form `"ok"` is a compile error. The
   string aliases at line 39-43 stay so dynamic label composition
   (e.g. label values read from external config) still works.

4. **Vec metrics do not appear in `Registry().Gather()` until at least
   one label combination has been touched**
   (`core/pkg/wkmetrics/wkmetrics_test.go:54-79` `touchAllFamilies`
   helper). Tests asserting "the family is registered" MUST first
   touch a label combination per family; otherwise the assertion
   fails on a freshly-built Metrics. The helper centralises the
   touch list so adding a metric family is one append rather than a
   per-test edit.

5. **HTTP latency middleware uses the static route template AND wraps
   the auth middleware**, not just the handler
   (`core/internal/keep/server/server.go:80-104`
   `NewRouterWithMetrics`). Iter-1 reversed the wrap order:
   `instrument(route, authed(h))` not `authed(instrument(route, h))`.
   This ensures 401 responses still register in the histogram so
   operators see auth-failure volume on the same panel as the
   happy-path traffic. The static route template (e.g.
   `"/v1/manifests/{manifest_id}"`) is the `route` label, NOT the
   raw URL path — cardinality stays bounded regardless of path
   parameters.

6. **Backwards-compat preserved by additive constructors, not signature
   changes** (`core/internal/keep/server/server.go:55-58, 80-89`
   `NewRouter` → `NewRouterWithMetrics`, `New` → `NewWithMetrics`).
   The original `NewRouter(v, pool, reg, heartbeat)` and
   `New(cfg, pool, v, reg)` are now thin wrappers that pass `nil`
   metrics. This kept the M2.7.\* test surface (handlers_read_test.go,
   handlers_write_test.go, handlers_subscribe_test.go,
   server_test.go) at zero changes. The router imports
   `wkmetrics` directly — the direction is allowed
   (`server → wkmetrics`); `wkmetrics` MUST NOT learn about HTTP
   server internals.

7. **`/metrics` mounted OUTSIDE the auth wall + operator opt-out for
   process collectors** (`core/internal/keep/server/server.go:73-75`
   mux.Handle precedes the `if v != nil` auth block;
   `wkmetrics.go:66-79` `Options.DisableProcessCollectors`).
   Prometheus scrapers typically authenticate by network ACL or
   sidecar; forcing a JWT here would prevent the dashboard stack
   from working out of the box. The deployment trade-off is that
   default `ProcessCollector` + `GoCollector` leak process-level
   info; operators behind a hostile network set
   `Options.DisableProcessCollectors: true` AND/OR ACL the listener.
   Contract regression-pinned by
   `TestNewRouterWithMetrics_MountsMetricsOutsideAuthWall` and
   `TestIter1_NewWithOptions_DisableProcessCollectors`.

8. **Structured logging adopts `log/slog` (stdlib), zero new logging
   dependency** (`core/pkg/wklog/wklog.go`). The package is a thin
   wrapper providing: (a) JSON handler factory with `subsystem`
   baked in via `slog.Logger.With`; (b) correlation-id injection via
   a typed `context.Context` key + custom `correlationHandler` so the
   id appears as a TOP-LEVEL JSON field, not a nested group; (c)
   env-driven level config with `WK_LOG_LEVEL_<UPPERCASE_SUBSYSTEM>`
   override and `WK_LOG_LEVEL` fallback; (d) loud one-shot warning on
   unrecognised level strings so a typo surfaces at boot rather than
   silently downgrading. Tests inject deterministic `LevelLookup`
   functions instead of `os.Setenv` so parallel sub-tests do not
   race on the global env (`wklog_test.go:staticLookup`).
   **Limitation pinned in iter-1**: `WithGroup` nests correlation_id
   under the group due to slog group semantics — documented honestly
   in the package doc + `TestWithGroup_CorrelationIDNestsUnderGroup`
   regression.

9. **The outbox success counter is committed atomically with the
   transaction** (`core/internal/keep/publish/outbox.go:209-260`).
   Iter-1 review C2 / Codex-P1b: bumping a Prometheus counter
   inside a transaction that can still roll back is a silent data-
   corruption pattern. The fix is to collect successful row IDs
   locally during the loop and emit ONE `OutcomeSuccess` sample per
   row AFTER `tx.Commit` returns nil. On Commit failure or upstream
   tick error, emit ONE batch-level `OutcomeError` instead of
   per-row over-counting.
