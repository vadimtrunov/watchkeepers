## 2026-05-16 — M5 DoD bullet B5: cross-cutting M5 Notebook-scenarios suite driving `BuildTurnRequest` + `NewWiredRuntime` end-to-end

**PR**: [#152](https://github.com/vadimtrunov/watchkeepers/pull/152)
**Merged**: 2026-05-16 (squash commit `a8f03c0`)

### Context

Closed §10 Phase B item B5 from `docs/ROADMAP-phase1.md` and the two remaining §M5 verification bullets: "Auto-injection: seed Notebook with a lesson, start a new turn, observe lesson content in the prompt window" and "Auto-reflection: force a tool error, check that a `lesson` entry appears with `active_after` 24h in the future and `lesson_learned` is in Keeper's Log". Each bullet had unit-level coverage already (M5.6.d `TestBuildTurnRequest_*` for the recall-into-System wiring; M5.6.b/c `wired_runtime_test.go` + M5.6.f `reflection_lifecycle_e2e_test.go` for the reflector + keepers-log edges). The B5 delta is a single cross-cutting scenario file that drives the production composition — `llm.BuildTurnRequest(ctx, manifest, query, embedder, supervisor)` for the inject path and `runtime.NewWiredRuntime(fake, WithToolErrorReflector(reflector), WithKeepersLogWriter(writer), WithAgentID(...))` for the reflect path — so a regression in option-application order or post-loader assembly cannot slip past per-package unit tests. Three top-level tests (two bullet scenarios + one option-order independence guard added in iter-1), no shared state across tests, deliberately orthogonal to M5.6.f's shared-state harness. Total runtime ≈ 100 ms.

### Pattern

**A B-series cross-cutting verification file co-locates with the existing test helpers, not in a new package**: B2 placed `chain_integration_test.go` in `core/internal/m3chain/`; B3 placed `ratelimiter_load_test.go` in `core/pkg/messenger/slack/` next to the rate limiter; B4 placed `m5-sandbox-scenarios.test.ts` in `harness/test/` next to the wire-boundary unit tests; B5 placed `notebook_scenarios_e2e_test.go` in `core/pkg/runtime/` next to the M5.6.b/c/f helpers (`recordingKeepClient` at `wired_runtime_test.go:343`, `newFakeAgentRuntime` at `wired_runtime_test.go:36`, `errToolBoom` at `wired_runtime_test.go:93`, `recallAllLessons` + `fixedClock` + `reflectorAgentID` at `tool_error_reflector_test.go:31-60`). The shared `package runtime_test` declaration lets the new file reuse private helpers without exporting a single test-only API. Pattern: a B-series file lives where the helpers it would otherwise duplicate already live; if the only reason to introduce a new package is "to give the scenario its own home", reuse the existing test package instead. (`core/pkg/runtime/notebook_scenarios_e2e_test.go:1-2`)

**A bullet's wire boundary is whatever production callers actually invoke, not a unit underneath it**: B4's wire boundary was `runHarness(stdin, stdout, signal, provider)` consuming NDJSON because the M5 sandbox bullets are TS-resident behaviours; B5's wire boundaries are `llm.BuildTurnRequest(ctx, manifest, query, embedder, supervisor)` and `runtime.NewWiredRuntime(inner, opts...)` because the M5 Notebook bullets are Go-resident behaviours surfaced through Go function calls (no harness-side JSON-RPC method handles "build a turn request" or "wire a runtime"). The B5 reflect test uses `WithKeepersLogWriter` on the wired runtime rather than `WithKeepersLog` on the reflector directly — same observable behaviour as the M5.6.f harness, but it pins the M5.6.c constructor's writer-into-reflector decoration; a future regression that stopped the decoration would slip past M5.6.f but fail B5. Caveat the lesson cannot drop: the over-fakeAgentRuntime composition is still a test composition (no production caller wires a fake), so this pin is "M5.6.c option-application discipline" rather than "exact production boot path". (`core/pkg/runtime/notebook_scenarios_e2e_test.go:88-168` inject path + `:205-352` reflect path)

**Discriminator-field assertions are Fatalf, not Errorf**: the inject test pins `req.Metadata[llm.MetadataKeyRecalledMemoryStatus] == llm.RecalledMemoryStatusApplied` BEFORE asserting on `req.System` content, and uses `t.Fatalf` so a silent demotion (`disabled_topk_zero` / `agent_not_registered` / `embed_error` / `recall_error` / `no_matches`) stops the test at the informative line instead of cascading into three near-identical content-missing errors. Same shape on the `RecalledMemoryHeader` check — a missing header means the block was never rendered, so subject / content checks below cannot possibly pass and would only add noise. Iter-1 finding (critic m#1): the original draft used `Errorf` and the lesson claimed "fails LOUD on the right line" — `Errorf` doesn't stop, so the claim was false. Pattern: discriminator-first ordering is necessary but not sufficient; the discriminator MUST be `Fatalf` because its failure invalidates every downstream assertion. (`core/pkg/runtime/notebook_scenarios_e2e_test.go:144-158`)

**Cross-store + cross-field assertions over single-store cardinality**: the reflect test asserts (a) `entry.EvidenceLogRef == keepEventID` linking the lesson row to its keepers-log event, (b) the keepers-log `Payload` JSON contains the forced tool name + err message (not just `EventType == "lesson_learned"`), and (c) `entry.ToolVersion == "v1.0.0"` matching the value the caller passed on the `ToolCall` (M5.6.e hot-load needs-review machinery keys off this field). Pre-iter-1 the test pinned only the lesson's existence + cooling-off bound; iter-1 reviewers (codex m#5, critic m#3) flagged that a regression emitting an empty / wrong payload, or dropping the ToolVersion from `composeLessonEntry`, would pass count-only assertions. The assertion phase lives in two helpers (`assertB5KeepersLogLessonLearnedEvent`, `assertB5NotebookLessonRow`) to keep gocyclo under the 15 ceiling lefthook enforces. Pattern: when a scenario writes through N stores that are supposed to be linked, every link AND every per-row field that downstream behaviour keys off MUST be in the assertion phase — counts alone miss the "wrote the rows but broke the link" failure mode; extract helpers per store to keep the test top-level under gocyclo. (`core/pkg/runtime/notebook_scenarios_e2e_test.go:278-356`)

**Binding-time ties to production constants beat coupled literals in test code**: the reflect test's 24h cooling-off assertion uses `runtime.DefaultCoolingOff` — the exported production constant — rather than a local `b5DefaultCoolingOff = 24 * time.Hour`. A coupled literal that "happens to match" the production default passes today but silently weakens to "asserts whatever the test code says" if the production default ever moves; a binding-time tie via the exported constant means a future bump in production fails the test in lockstep with no manual sync. Iter-1 finding (critic Major #2): the original draft declared the coupled literal AND the lesson claimed it "asserts against the reflector's surfaced default" — the claim was false because the production constant was unexported. Fix landed in two halves: export `runtime.DefaultCoolingOff` (production change) + delete the local literal (test change). Pattern: every numeric / string constant the bullet text pins MUST be sourced from the production package via an exported identifier; if no exported identifier exists, export one — the test is the reason the constant is part of the public surface. (`core/pkg/runtime/notebook_scenarios_e2e_test.go:283` + `core/pkg/runtime/tool_error_reflector.go:14-23`)

**Option-order independence is a separate sub-test, not an assumption**: `NewWiredRuntime`'s contract is "options may be supplied in any order; the post-options pass threads the writer into the reflector regardless". The primary auto-reflection scenario exercises only one permutation (reflector-then-writer); a regression that made the constructor order-sensitive while preserving that exact permutation would slip through. The fix is a second scenario (`TestM5Verification_B5_AutoReflection_WiredRuntimeOptionOrderIndependent`) that flips the order and pins the same `EvidenceLogRef` cross-reference under the reversed permutation. Iter-1 finding (codex Major #2): without the reverse-order test the order-independence claim is unenforceable. Pattern: every "option order does not matter" or "either ordering yields the same wiring" docstring claim needs a dedicated test that asserts the property under the non-canonical ordering — otherwise the claim is wishful and the canonical-ordering test cannot catch the regression that would invalidate it. (`core/pkg/runtime/notebook_scenarios_e2e_test.go:355-425`)

### Iter-1 review patterns (codex + critic in parallel)

Two reviewers ran in parallel on the same branch; the union of their findings drove the test-file rewrite and the production-side `DefaultCoolingOff` export. Concrete deltas from iter-1: time anchor moved from a hard-coded `time.Date(2026, 5, 16, ...)` to `time.Now().Truncate(time.Second)` (critic Major #1 — the literal read as calendar-bound code even though the math is relative); local `b5DefaultCoolingOff` constant deleted in favour of an exported `runtime.DefaultCoolingOff` (critic Major #2); reverse-order option permutation added as a new sub-test (codex Major #2); keepers-log assertion grew payload-field checks (codex m#5 — `EventType` alone proves "some event" not "the event for this failure"); reflect test grew a `ToolVersion` assertion (critic m#3 — lateral coverage gap relative to M5.6.f); status-key + header checks moved from `Errorf` to `Fatalf` (critic m#1). The "constants over inline literals" pattern from the original draft was dropped as table-stakes (critic Nit #3 holistic) and folded into the broader binding-time-ties pattern above.

- Files: `core/pkg/runtime/notebook_scenarios_e2e_test.go` (3 scenario tests, ~100 ms wall-clock); `core/pkg/runtime/tool_error_reflector.go:14-23` (exported `DefaultCoolingOff`).

## 2026-05-16 — M5 DoD bullet B4: cross-cutting M5 sandbox-scenarios suite driving `runHarness` end-to-end

**PR**: [#151](https://github.com/vadimtrunov/watchkeepers/pull/151)
**Merged**: 2026-05-16 (squash commit `c8b7e7e`)

### Context

Closed §10 Phase B item B4 from `docs/ROADMAP-phase1.md` and three of the six M5 verification bullets in one suite: "A runaway test tool is killed by the wall-clock limit", "A tool that tries undeclared network access is rejected by the capability broker", and "Replacing `LLMProvider` with `FakeProvider` runs the full harness suite without code changes outside the provider package". Each bullet already had unit-level coverage (`invokeTool.test.ts:143-164` wall-clock cases, `worker-broker.test.ts:62-108` `gateToolInvocation` net.connect denials, `llm-conformance.test.ts` swap-without-touching-core suite); the B4 delta is a single cross-cutting scenario file that drives the harness's actual NDJSON stdio loop (`runHarness`) end-to-end so a regression in dispatcher/wire layering can never slip past per-package unit tests. Three describe blocks, four tests total (one per bullet plus a structural import-discipline guard for bullet 3). Total runtime ≈ 70 ms — the wall-clock kill is the longest at ~50 ms because the isolated-vm budget is the assertion subject.

### Pattern

**A "verification bullet" closure file lives next to the unit tests but drives the full wire path, not the unit**: prior art on B2 (`core/internal/m3chain/chain_integration_test.go`) and B3 (`core/pkg/messenger/slack/ratelimiter_load_test.go`) shows the same shape — one file per closed acceptance bullet (or bullet-cluster), each test wires existing units end-to-end through the production boundary the bullet's text actually references. For M5 the wire boundary is `runHarness(stdin, stdout, signal, provider)` consuming NDJSON; every assertion in B4 feeds requests through an in-memory `Readable`, collects responses + notifications through a `CollectingWritable`, and parses NDJSON back out. Pattern: when a §Verification line says "X happens through Y", the scenario test MUST drive Y, not the unit Y is built from. A unit test that asserts the same outcome with `Y`'s callee invoked directly is necessary (it pins the unit) but not sufficient (it cannot catch a regression in `Y`'s composition). (`harness/test/m5-sandbox-scenarios.test.ts:1-49` header + imports)

**`ToolErrorCode` values are application-range (-32000..-32099) by design and ride on the wire `error.code` field verbatim — assert on the typed enum, not on the magic number**: the M5.3.b dispatcher lifts `MethodError.code` (carrying the `ToolErrorCode` enum value) into the JSON-RPC error envelope without translation, so the wire response observed at the stdio boundary carries `error.code === ToolErrorCode.ToolTimeout` (-32001) for the wall-clock scenario and `error.code === ToolErrorCode.ToolCapabilityDenied` (-32003) for the undeclared-net scenario. (Code-table source of truth: `harness/src/invokeTool.ts:47-62`.) Asserting against the enum constant instead of the literal numeric code makes a future code-table reshuffle a typecheck failure, not a silently-skipping runtime mismatch. The `error.message` regex is the human-readable sibling that pins the deny reason (`/net\.connect denied/i` + `/api\.example\.com:443/`) — together they prove the gating decision's `reason` field round-trips through the wire. (`harness/test/m5-sandbox-scenarios.test.ts:165-167`, `:230-232`)

**Bullet-1 + Bullet-2 share a module-global ACL gate that needs explicit `beforeEach`/`afterEach` reset**: the harness's `setManifest` JSON-RPC handler writes through `setActiveToolset()` into a module-level singleton in `manifest.ts`. The `invokeTool` handler then reads from that same singleton via `getActiveToolset()` to enforce the M5.5.b.a ACL gate; without explicit reset the singleton bleeds across test files (and across the `vitest --reporter=verbose` run order). Use `__resetActiveToolsetForTests()` symmetrically in BOTH `beforeEach` AND `afterEach` so a failed test does not poison the next case AND a failed teardown does not poison an unrelated downstream suite. Prior art: `invokeTool.test.ts:25-31` uses the same symmetric reset pattern. Pattern: a singleton wired via the wire boundary needs a wire-shaped reset, but tests get to reach for the back-door `__resetActiveToolsetForTests` because the wire interface only supports "replace toolset" (never "tear down"). (`harness/test/m5-sandbox-scenarios.test.ts:112-118`)

**Worker-gate denial elapsed budget asserts the "never paid the fork cost" invariant cheaply**: `gateToolInvocation` is a pure synchronous function (`harness/src/worker/broker.ts:55-100`) the dispatcher calls BEFORE `spawnWorker` so a deny costs ~0 ms; a future regression that accidentally spawned the worker first would push the elapsed time past the ~20-50 ms cold-fork floor on the current Node runtime. A 500 ms upper-bound budget on the scenario gives ~10× headroom for a slow 2-vCPU CI runner's readline + JSON.parse + dispatcher overhead + cold isolated-vm module load while still rejecting any regression that smuggled a fork into the deny path. (Iter-1 widened this from 250 ms after a reviewer flagged the cold-import floor; the deny-without-fork discipline survives because ~50 ms fork-cost remains well inside the 500 ms band.) Asserting on elapsed wall-clock is what makes the "the broker rejected without spawning" property mechanically checkable from the outside — without it, the deny-without-spawn discipline would be enforced only by code review. (`harness/test/m5-sandbox-scenarios.test.ts:235-240`; ADR §0001 `docs/adr/0001-worker-substrate.md` records the host-side gating decision.)

**The "swap without touching core" structural half guards SOURCE-LEVEL imports, NOT the runtime module graph**: the M5 bullet "Replacing `LLMProvider` with `FakeProvider` runs the full harness suite without code changes outside the provider package" has a runtime half (FakeProvider drives complete + countTokens + reportCost + stream round-trip through `runHarness`) and a structural half (this scenario file's OWN imports must not name the real adapter, directly or through a re-exporting barrel). The structural half cannot be a runtime "no real adapter loaded" claim: `runHarness` itself eagerly imports `claude-code-provider.ts` from `harness/src/index.ts`, so the adapter class is ALWAYS in the module graph. The roadmap bullet is about source-level provider-swap mechanics, not transitive runtime loads. The guard is a self-reflexive meta-test that reads the file's own source via `fileURLToPath(import.meta.url) + readFileSync` and asserts the import specifiers contain no `claude-code-provider` literal AND no `llm/index.js` barrel path (the barrel re-exports `ClaudeCodeProvider`, so a barrel import would smuggle the real adapter symbol into the file's bindings even though the path string does not match the leaf). Anchor the regex at line-start (`/^\s*(?:import|export)[^"';]*from\s+["']([^"']+)["']/gm`) so prose mentions inside docstrings stay false-negatives. Iter-1 fix: codex + critic both flagged that the first-draft regex missed barrel indirection AND was satisfied by importing `FakeProvider` from `../src/llm/index.js`; the fix is two-fold (import from leaf `../src/llm/fake-provider.js` + `../src/llm/types.js`, AND reject barrel paths in the guard). Pattern: when a structural-discipline assertion claims X, walk the failure modes that would defeat X (re-export indirection, comment false-positive, indirect imports through default-export) and ENCODE each one as a discrete rejection in the guard. (`harness/test/m5-sandbox-scenarios.test.ts:375-419`)

**Bucket-shaped wire-assertions beat exact total-count assertions when the wire surface has documented growth axes**: the FakeProvider scenario observes ~9 NDJSON lines on stdout for a complete + countTokens + reportCost + stream + hello + shutdown sequence (1× `harness/ready` boot notification + 2× `stream/event` notifications + 6× id-bearing responses). A `expect(lines).toHaveLength(9)` would fail LOUD on a regression of the bullet under test BUT would also fail on a benign future expansion (e.g., a new `harness/metric` heartbeat notification) — conflating the two failure modes. Better shape: bucket the lines by category at parse time and assert per-bucket counts (`responsesByID.size === 6`, `streamNotifications.length === 2`, `readyNotifications.length >= 1`). Each bucket assertion pins the contract for one wire axis without coupling to the rest. Iter-1 fix: a reviewer pointed out the brittleness of the strict total; the bucket-shaped form was adopted on the same branch. Pattern: when a scenario's expected output spans N orthogonal wire axes, assert per-axis (or per-bucket) rather than on the sum — the sum overcouples and a future axis-3 expansion fails an unrelated axis-1 assertion. (`harness/test/m5-sandbox-scenarios.test.ts:316-339`)

## 2026-05-04 — M5.1: portable runtime interface decouples from wire schema via post-loader projection

**PR**: [#54](https://github.com/vadimtrunov/watchkeepers/pull/54)
**Merged**: 2026-05-04

### Pattern

When defining a runtime-facing type that mirrors a wire-format type (here
`runtime.Manifest` mirrors `keepclient.ManifestVersion`), do not import the wire
type directly. Define a runtime-local type that represents the POST-LOADER shape —
`system_prompt` already templated with personality+language, `tools` decoded from
raw JSON to `[]string`, `authority_matrix` projected to the access-control shape
the runtime actually consults. The mapping (M5.5 loader) owns the projection.
Decoupling means a future Keep schema change does not ripple into the runtime, and
the runtime interface stays portable across providers (Claude Code, FakeProvider,
future LLMs). Companion: alias-import the package (`agentruntime "core/pkg/runtime"`)
since `runtime` clashes with stdlib. Apply when extending any boundary type that
lives both in storage and in computation — store layer carries wire shape, compute
layer carries logical shape, loader layer is the projection.

## 2026-05-04 — M5.2: streaming interface design — callback + Subscription.Stop mirrors message-bus pattern across LLMs

**PR**: [#55](https://github.com/vadimtrunov/watchkeepers/pull/55)
**Merged**: 2026-05-04

### Pattern

When designing a portable streaming interface (LLM provider, websocket adapter, SSE
consumer), the right shape is `Stream(ctx, req, handler) (Subscription, error)` where
handler is a callback `func(ctx, event) error` and Subscription has only `Stop() error`.
Avoid returning channels at the interface boundary — they leak goroutine ownership and
make Stop semantics ambiguous. Avoid raw iterators — they require the caller to manage
backpressure. The callback + Stop pattern matches what M5.1 (`AgentRuntime.Subscribe`)
and M4.1 (`messenger.Adapter.Subscribe`) already use, so the M5.10 conformance harness
can drive any of them through a single test scaffolding. Companion: when a streaming
type carries multiple event shapes (text delta vs tool-call JSON fragment), prefer
separate fields per shape over reusing one — field overload reads cleanly to the author
but trips up reviewers and integrators. Apply when the next streaming consumer arrives.

- Files: `core/pkg/llm/provider.go`, `core/pkg/llm/errors.go`, `core/pkg/llm/doc.go`
- Docs: `docs/ROADMAP-phase1.md` §M5.2.

## 2026-05-04 — M5.3.a: JSON-RPC notification semantics — id-omitted is NOT id=null

**PR**: [#56](https://github.com/vadimtrunov/watchkeepers/pull/56)
**Merged**: 2026-05-04

### Pattern

When implementing a JSON-RPC server, distinguish three id states correctly:
(1) id present + non-null → request, server MUST reply with matching id;
(2) id present + null → request with null id, server MUST reply with id=null
(also valid for unparseable-JSON responses per §5.1);
(3) id ABSENT → notification, server MUST NOT reply at all (even on unknown
method or handler error). The common mistake is treating cases (2) and (3) as
the same — they're not. Cases (2) and (3) require different parser branches and
different handler-loop dispositions. Test discipline: pin all four notification
cases (known method, unknown method, handler throws, malformed JSON) with
captured-stdout assertions to lock the contract. Apply when implementing any
RPC framing where the spec distinguishes request from notification.

- Files: `harness/src/jsonrpc.ts`, `harness/src/dispatcher.ts`, `harness/test/jsonrpc.test.ts`, `harness/test/dispatcher.test.ts`

## 2026-05-05 — M5.3.b.a: pure-JS isolated-vm sandbox via invokeTool RPC

**PR**: [#57](https://github.com/vadimtrunov/watchkeepers/pull/57)
**Merged**: pending

### Context

First half of M5.3.b — the pure-JS tool invocation path. Adds a JSON-RPC method
`invokeTool` that compiles caller-supplied source as the body of `function(input){…}`
in a fresh `isolated-vm` Isolate per call, with wall-clock timeout and memory ceiling.
No host I/O capabilities (no `require`, `process`, `fetch`); the worker-process I/O-gated
path lands in M5.3.b.b.

### Pattern

Pick `evalClosure` over the `compileScript` + `Reference.apply` pair when the round-trip
is "compile a function, call it once, copy result back" — single API call covers compile,
inject positional args (`arguments: { copy: true }`), enforce wall-clock budget on the C++
side (`timeout`), and copy the return through a structured-clone boundary
(`result: { copy: true }`). Always dispose the Isolate in `finally`. Translate
isolated-vm's message-string error reporting via `/timed out/i` and
`/exceeded its memory limit/i` regexes to typed `MethodError` codes; isolated-vm does NOT
expose dedicated subclasses for timeout/OOM at v6.1.2. Application-range JSON-RPC error
codes (-32000 down) widen via a single explicit cast at the `MethodError` site rather than
extending the protocol-level `JsonRpcErrorCode` union — keeps the protocol band clean.

### Anti-pattern

Do NOT vendor isolated-vm without first checking that the test runner's Node ABI has
prebuilds. Native compile failures on CI for missing-ABI cases are silent until the
`ts-ci` job runs. We landed on `^6.1.2` because Node 24 ABI 137 prebuilds shipped for
linux-x64 + darwin-arm64 — verify before pinning.

- Files: `harness/src/invokeTool.ts`, `harness/src/methods.ts`, `harness/test/invokeTool.test.ts`, `harness/package.json`

## 2026-05-06 — M5.3.b.b.a: ADR 0001 selects child_process.fork as worker substrate

**PR**: [#58](https://github.com/vadimtrunov/watchkeepers/pull/58)
**Merged**: pending

### Context

First half of M5.3.b's I/O-capable tool path needed a substrate decision before
implementation could proceed. Three options were on the table: `worker_threads`
(same OS process, V8-isolated), `child_process.fork` (separate Node process,
IPC channel), `child_process.spawn` (separate Node process, stdio framing).
The pure-JS path landed in M5.3.b.a inside `isolated-vm`; this ADR governs the
I/O-capable sibling.

### Pattern

When picking a Node sandboxing substrate for caller-supplied code with declared
I/O capabilities, prefer **OS-process isolation** (`child_process.fork`) over
in-process V8 isolation (`worker_threads`). Same-process execution carries
upstream-documented risks (isolated-vm README explicitly recommends keeping
isolates in a different Node.js process for resilience). The fork option
preserves Node-style IPC (no second framing layer) AND gives true OS isolation
for blast-radius control.

**Critical caveat — verify before relying on**: `child_process.fork` IPC uses
**JSON serialization by default**. Structured-clone semantics (`Date`, typed
arrays, `undefined`-valued keys, `Map`) require explicitly opting in via
`serialization: 'advanced'` on the `fork()` options (per Node `child_process`
docs). The harness MUST set this flag. The contrast between fork and spawn
collapses if `serialization: 'advanced'` is forgotten — both lose `Date` and
typed arrays in that case.

The Nygard ADR format (`Context / Decision / Considered alternatives /
Consequences / References`) is now the project convention — `docs/adr/`
directory + `docs/adr/README.md` index established by this TASK.

### References

- Files: `docs/adr/0001-worker-substrate.md`, `docs/adr/README.md`, `harness/README.md`
- Docs: `docs/ROADMAP-phase1.md` §M5.3.b.b.a; Node `child_process` docs §`serialization`

## 2026-05-06 — M5.3.b.b.b: capability declaration schema and gating policy types

**PR**: [#59](https://github.com/vadimtrunov/watchkeepers/pull/59)
**Merged**: pending

### Context

Worker-process I/O capability declaration and gating decisions require a stable schema before wiring into the dispatcher (M5.3.b.b.d) or spawning workers (M5.3.b.b.c). This TASK defines the zod-based wire contract for the four I/O axes (filesystem read/write, network egress, env vars, child-process spawn) and gating policy shape without implementing dispatch logic or spawn integration.

### Pattern

**Front-load schema definition before runtime wiring.** Define the capability and policy shapes as standalone, test-covered zod schemas in a dedicated module (`harness/src/capabilities.ts`) before gluing them into the dispatcher. This keeps each downstream PR (M5.3.b.b.d, M5.3.b.b.c) focused on a single concern—type safety is already proven, validation tests already passed—and prevents each integrator from re-deriving the schema or inventing local validation. The schema module stays pure (no I/O, no side effects), making it easy to unit-test and to import from multiple downstream sites without coupling them to dispatch or spawn logic.

## 2026-05-06 — M5.4.a: Sandbox guardrails — wall-clock timeout + output-byte cap

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

Implemented Go-side process-control sandbox guardrails via `exec.CommandContext` + time-based timeout + output-byte wrapping in `core/pkg/runtime/sandbox.go`. **Race-safe idempotent kill via `sync.Once` + `atomic.Value.CompareAndSwap`.** When a process can be killed from multiple goroutines (wall-clock timer firing, context cancellation, output-cap watcher all reaching `cmd.Process.Kill()` simultaneously), use `sync.Once` to ensure `Kill()` itself is called once (OS-level idempotent), then guard the `TermReason` field write with `atomic.Value.CompareAndSwap` so only the first goroutine wins the semantic labeling. `exec.CommandContext` itself will SIGKILL on context cancel — the explicit watcher's label write must race-safely beat it.

## 2026-05-06 — M5.4.b: Sandbox guardrails — CPU-time + memory-ceiling rlimits

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

Implemented Go-side process resource limits via `unix.Prlimit` (RLIMIT_CPU, RLIMIT_AS) applied post-`cmd.Start()`. Complements M5.4.a (wall-clock + output-cap). CPU-time limit enforcement surfaces as SIGXCPU; memory ceiling as SIGKILL. Platform divergence (Linux real / Darwin stub / unsupported error) factored into build-tagged files to keep syscall complexity isolated.

### Pattern

**`unix.Prlimit` applied post-Start, not SysProcAttr pre-Start.** Go stdlib doesn't expose CPU/AS limits as first-class `SysProcAttr` fields. Post-Start approach: call `unix.Prlimit(pid, RLIMIT_CPU, ...)` after `cmd.Start()` returns (child PID known), before any timers arm. On error: kill child, wrap, return. Simpler than forking to C or using unsafe pointer manipulation. Hard-limit (`Rlimit.Max == Rlimit.Cur`) prevents the child from raising its own soft limit at runtime — stricter than `RLIMIT_INFINITY` but appropriate for harness-spawned subprocesses that don't need escalation.

**Build-tagged file split isolates platform divergence.** `sandbox_rlimit_linux.go` (real `unix.Prlimit`), `sandbox_rlimit_darwin.go` (stub returning nil — wall-clock + output-cap from M5.4.a still apply), `sandbox_rlimit_unsupported.go` (returns `ErrUnsupportedPlatform` only when caller actually requested limits). Stub non-enforcement is fine when "no enforcement" was the Phase-1 baseline; don't emit error on unsupported platforms unless limits were explicitly configured.

**Test-helper self-spawn pattern for memory-ceiling.** When a test needs a subprocess that allocates >1GB (to trigger RLIMIT_AS), re-exec the test binary itself with an env var guard (`SANDBOX_TEST_HELPER=memhog`). Child branch in `init()`-style guard does the allocation + `os.Exit(0)`. Avoids the `go build` complication of a separate test binary; works portably on any platform that has the test binary.

**`t.Setenv` + `t.Parallel()` panic detection.** Per Go testing docs, `t.Setenv` panics if the test or any ancestor called `t.Parallel()`. Caught at code review (Linux CI would have crashed at runtime). When introducing `t.Setenv`, audit the test and parents for `t.Parallel()` and remove it if both are needed.

**Best-effort signal classification.** SIGXCPU → `cpu_time`; SIGKILL with `MemoryCeilingBytes>0` → `memory_ceiling`. Heuristic — kernel can fire SIGKILL for unrelated reasons (operator OOM kill); attribution is conditioned on the configured limit. Document the heuristic so callers don't over-trust the label.

### Anti-pattern

Do NOT call `db.Close()` BEFORE removing from the registry. If a concurrent `Open(agentID)` slips in between Close-call and registry-remove, it could see a closed handle. Order is: `Lock → delete from map → Unlock → Close db → return any close error`.

**Cross-language seams legitimately double surface.** The bidirectional JSON-RPC contract between TS harness and Go stdio host introduces parallel code paths: TS `parseResponse` + `RpcClient` (pending-request map, id-correlated replies); Go `core/pkg/harnessrpc/Host` + `Register` + `Run(ctx, in, out)` (request dispatch, response write); TS tests + Go tests + Go integration test (end-to-end `echo` method via `io.Pipe`). This PR is +1383 LOC across 5 files (exceeds the 1000-LOC target for AC8 by 38%). The overshoot is structurally inherent to first-cross-language-seam introduction. Subsequent methods on this seam (e.g., `notebook.remember` in M5.5.d.a.b) reuse the framing — per-method delta will be much smaller. **File count stays ≤6 (strictly enforced); LOC overshoot on foundational cross-language PRs is acceptable with reviewer accept-note.**

**NDJSON framing: one atomic Write per line.** Writing `line` and `\n` in two separate `io.Writer.Write()` calls on `io.Pipe` is a race under concurrent goroutines — bytes interleave, the host parses garbage. **The fix is `Write(append(line, '\n'))`** — one atomic call. This is the framing contract: one NDJSON line = one Write call. Reusable rule for any future stdio-framed protocol in this codebase.

**`atomic.Pointer[error]` for first-error capture in a loop.** When a long-running loop must capture the FIRST write/IO error and exit cleanly without serializing all writers, `atomic.Pointer[error]` paired with a top-of-loop nil-check is cheaper than a mutex+Cond. Lock-free; observable without polling; preserves per-iteration optimism on the read side. Pattern reusable for future stdio servers (e.g., M6 supervisor wiring).

**Trust model documentation as a substitute for code complexity.** When a peer is a child process under your control (harness spawned by core), unbounded per-request goroutines are acceptable — you do not need backpressure semaphores or rate limiters. **Document the trust assumption explicitly** so future readers do not add unnecessary defenses.

### Anti-pattern

Silent error swallow on the write side (`_ = writeLine(resp)`). Justification "the next read will EOF" fails when the write fd is broken but the read fd is healthy — the loop happily consumes more requests, drops more responses, every caller's promise hangs. **Capture write errors via `atomic.Pointer` and exit `Run` cleanly.**

### Pattern: one-turn parity via SDK-iterator interception + interrupt

When adapting an "all-inclusive" SDK — one that owns multi-turn execution
and tool dispatch internally — to a one-turn `LLMProvider` contract,
intercept the SDK iterator's first assistant message that contains
`tool_use` blocks, capture those blocks, then call `iter.interrupt()` and
return. The SDK never gets a chance to route the tool call to its own
handler. A sentinel-tagged error in a stub MCP handler acts as
belt-and-suspenders: if the interrupt arrives too late (the SDK has already
started its own handler dispatch before the interrupt is processed), the
stub raises a recognisable error rather than silently running a no-op
tool. This pattern generalises to any "SDK wants to own the loop, we want
a single turn" adapter: (1) start the SDK iterator, (2) scan events for
the first assistant message with tool blocks, (3) interrupt before the SDK
re-enters its handler, (4) return the captured blocks upstream.

See `harness/src/llm/tool-bridge-interceptor.ts` and
`harness/src/llm/tool-bridge-mcp-stub-server.ts` for the implementation.

### Pattern: deliberate-stub gap surfaced via `skipMethods` in a parameterised conformance suite

When one implementation of a multi-implementation interface temporarily
lacks a method, extend the conformance case's descriptor struct with an
optional `skipMethods?: readonly Method[]` field and use vitest's
`it.skipIf` per-test gating instead of dropping the case from the suite
entirely. The gap stays visible to every maintainer who reads the suite
(the `skipMethods` annotation is self-documenting), the conformance proof
continues to run for all methods the implementation does support, and the
skipped test re-enables automatically the moment `skipMethods` is cleared.
Dropping the case entirely hides the gap; a blanket `describe.skipIf`
hides all coverage for the affected impl, not just the missing method.

Applied in `harness/test/llm-conformance.test.ts` via
`skipMethods: ["countTokens"]` on the `ClaudeAgentProvider` case.

### Pattern: spec-deviation capture in the plan's file map

When implementation diverges from a spec's proposed layout during
planning, record the deviation inline in the plan's "File map" section
(or equivalent) rather than blocking on a spec amendment round-trip.
The plan becomes the authoritative layout record once implementation
begins; the spec may be reconciled later. A one-line note pointing back
to the spec section keeps the audit trail intact without the overhead of
a formal amendment. Reusable for any spec → plan → implementation flow
where small structural refinements are normal and the spec is
design-time, not implementation-time, ground truth.
