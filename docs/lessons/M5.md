## 2026-05-04 — M5.1: portable runtime interface decouples from wire schema via post-loader projection

**PR**: [#54](https://github.com/vadimtrunov/watchkeepers/pull/54)
**Merged**: 2026-05-04

### Pattern

When defining a runtime-facing type that mirrors a wire-format type (here
`runtime.Manifest` mirrors `keepclient.ManifestVersion`), do not import the wire
type directly. Define a runtime-local type that represents the POST-LOADER shape —
`system_prompt` already templated with personality+language, `tools` decoded from
raw JSON to `[]string`, `authority_matrix` projected to the access-control shape
the runtime actually consults. The mapping (M5.5 loader) owns the projection.
Decoupling means a future Keep schema change does not ripple into the runtime, and
the runtime interface stays portable across providers (Claude Code, FakeProvider,
future LLMs). Companion: alias-import the package (`agentruntime "core/pkg/runtime"`)
since `runtime` clashes with stdlib. Apply when extending any boundary type that
lives both in storage and in computation — store layer carries wire shape, compute
layer carries logical shape, loader layer is the projection.

- Files: `core/pkg/runtime/runtime.go`, `core/pkg/runtime/doc.go`, `core/pkg/runtime/errors.go`

## 2026-05-04 — M5.2: streaming interface design — callback + Subscription.Stop mirrors message-bus pattern across LLMs

**PR**: [#55](https://github.com/vadimtrunov/watchkeepers/pull/55)
**Merged**: 2026-05-04

### Pattern

When designing a portable streaming interface (LLM provider, websocket adapter, SSE
consumer), the right shape is `Stream(ctx, req, handler) (Subscription, error)` where
handler is a callback `func(ctx, event) error` and Subscription has only `Stop() error`.
Avoid returning channels at the interface boundary — they leak goroutine ownership and
make Stop semantics ambiguous. Avoid raw iterators — they require the caller to manage
backpressure. The callback + Stop pattern matches what M5.1 (`AgentRuntime.Subscribe`)
and M4.1 (`messenger.Adapter.Subscribe`) already use, so the M5.10 conformance harness
can drive any of them through a single test scaffolding. Companion: when a streaming
type carries multiple event shapes (text delta vs tool-call JSON fragment), prefer
separate fields per shape over reusing one — field overload reads cleanly to the author
but trips up reviewers and integrators. Apply when the next streaming consumer arrives.

### References

- Files: `core/pkg/llm/provider.go`, `core/pkg/llm/errors.go`, `core/pkg/llm/doc.go`
- Docs: `docs/ROADMAP-phase1.md` §M5.2.

## 2026-05-04 — M5.3.a: JSON-RPC notification semantics — id-omitted is NOT id=null

**PR**: [#56](https://github.com/vadimtrunov/watchkeepers/pull/56)
**Merged**: 2026-05-04

### Pattern

When implementing a JSON-RPC server, distinguish three id states correctly:
(1) id present + non-null → request, server MUST reply with matching id;
(2) id present + null → request with null id, server MUST reply with id=null
(also valid for unparseable-JSON responses per §5.1);
(3) id ABSENT → notification, server MUST NOT reply at all (even on unknown
method or handler error). The common mistake is treating cases (2) and (3) as
the same — they're not. Cases (2) and (3) require different parser branches and
different handler-loop dispositions. Test discipline: pin all four notification
cases (known method, unknown method, handler throws, malformed JSON) with
captured-stdout assertions to lock the contract. Apply when implementing any
RPC framing where the spec distinguishes request from notification.

### References

- Files: `harness/src/jsonrpc.ts`, `harness/src/dispatcher.ts`, `harness/test/jsonrpc.test.ts`, `harness/test/dispatcher.test.ts`
- Docs: `docs/ROADMAP-phase1.md` §M5.3.a.

## 2026-05-05 — M5.3.b.a: pure-JS isolated-vm sandbox via invokeTool RPC

**PR**: [#57](https://github.com/vadimtrunov/watchkeepers/pull/57)
**Merged**: pending

### Context

First half of M5.3.b — the pure-JS tool invocation path. Adds a JSON-RPC method
`invokeTool` that compiles caller-supplied source as the body of `function(input){…}`
in a fresh `isolated-vm` Isolate per call, with wall-clock timeout and memory ceiling.
No host I/O capabilities (no `require`, `process`, `fetch`); the worker-process I/O-gated
path lands in M5.3.b.b.

### Pattern

Pick `evalClosure` over the `compileScript` + `Reference.apply` pair when the round-trip
is "compile a function, call it once, copy result back" — single API call covers compile,
inject positional args (`arguments: { copy: true }`), enforce wall-clock budget on the C++
side (`timeout`), and copy the return through a structured-clone boundary
(`result: { copy: true }`). Always dispose the Isolate in `finally`. Translate
isolated-vm's message-string error reporting via `/timed out/i` and
`/exceeded its memory limit/i` regexes to typed `MethodError` codes; isolated-vm does NOT
expose dedicated subclasses for timeout/OOM at v6.1.2. Application-range JSON-RPC error
codes (-32000 down) widen via a single explicit cast at the `MethodError` site rather than
extending the protocol-level `JsonRpcErrorCode` union — keeps the protocol band clean.

### Anti-pattern

Do NOT vendor isolated-vm without first checking that the test runner's Node ABI has
prebuilds. Native compile failures on CI for missing-ABI cases are silent until the
`ts-ci` job runs. We landed on `^6.1.2` because Node 24 ABI 137 prebuilds shipped for
linux-x64 + darwin-arm64 — verify before pinning.

### References

- Files: `harness/src/invokeTool.ts`, `harness/src/methods.ts`, `harness/test/invokeTool.test.ts`, `harness/package.json`
- Docs: `docs/ROADMAP-phase1.md` §M5 → M5.3 → M5.3.b.a

## 2026-05-06 — M5.3.b.b.a: ADR 0001 selects child_process.fork as worker substrate

**PR**: [#58](https://github.com/vadimtrunov/watchkeepers/pull/58)
**Merged**: pending

### Context

First half of M5.3.b's I/O-capable tool path needed a substrate decision before
implementation could proceed. Three options were on the table: `worker_threads`
(same OS process, V8-isolated), `child_process.fork` (separate Node process,
IPC channel), `child_process.spawn` (separate Node process, stdio framing).
The pure-JS path landed in M5.3.b.a inside `isolated-vm`; this ADR governs the
I/O-capable sibling.

### Pattern

When picking a Node sandboxing substrate for caller-supplied code with declared
I/O capabilities, prefer **OS-process isolation** (`child_process.fork`) over
in-process V8 isolation (`worker_threads`). Same-process execution carries
upstream-documented risks (isolated-vm README explicitly recommends keeping
isolates in a different Node.js process for resilience). The fork option
preserves Node-style IPC (no second framing layer) AND gives true OS isolation
for blast-radius control.

**Critical caveat — verify before relying on**: `child_process.fork` IPC uses
**JSON serialization by default**. Structured-clone semantics (`Date`, typed
arrays, `undefined`-valued keys, `Map`) require explicitly opting in via
`serialization: 'advanced'` on the `fork()` options (per Node `child_process`
docs). The harness MUST set this flag. The contrast between fork and spawn
collapses if `serialization: 'advanced'` is forgotten — both lose `Date` and
typed arrays in that case.

The Nygard ADR format (`Context / Decision / Considered alternatives /
Consequences / References`) is now the project convention — `docs/adr/`
directory + `docs/adr/README.md` index established by this TASK.

### References

- Files: `docs/adr/0001-worker-substrate.md`, `docs/adr/README.md`, `harness/README.md`
- Docs: `docs/ROADMAP-phase1.md` §M5.3.b.b.a; Node `child_process` docs §`serialization`

## 2026-05-06 — M5.3.b.b.b: capability declaration schema and gating policy types

**PR**: [#59](https://github.com/vadimtrunov/watchkeepers/pull/59)
**Merged**: pending

### Context

Worker-process I/O capability declaration and gating decisions require a stable schema before wiring into the dispatcher (M5.3.b.b.d) or spawning workers (M5.3.b.b.c). This TASK defines the zod-based wire contract for the four I/O axes (filesystem read/write, network egress, env vars, child-process spawn) and gating policy shape without implementing dispatch logic or spawn integration.

### Pattern

**Front-load schema definition before runtime wiring.** Define the capability and policy shapes as standalone, test-covered zod schemas in a dedicated module (`harness/src/capabilities.ts`) before gluing them into the dispatcher. This keeps each downstream PR (M5.3.b.b.d, M5.3.b.b.c) focused on a single concern—type safety is already proven, validation tests already passed—and prevents each integrator from re-deriving the schema or inventing local validation. The schema module stays pure (no I/O, no side effects), making it easy to unit-test and to import from multiple downstream sites without coupling them to dispatch or spawn logic.

## 2026-05-06 — M5.3.b.b.c: worker spawn + JSON-RPC transport

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

Worker-process lifecycle (spawn, ready handshake, graceful terminate, crash detection) and JSON-RPC request–response correlation over an IPC duplex required a transport layer that is both spec-compliant and robust to channel failures. Pattern emerged: wrap synchronous send calls in try/catch on both notification and response paths, and treat malformed response envelopes as terminal rejections rather than parse-errors-with-hanging-awaiters.

### Pattern

**Wrap every synchronous transport send in try/catch; reject (not parse-fire) on malformed jsonrpc version.** JSON-RPC 2.0 notifications are fire-and-forget — synchronous throws on `channel.send` must not propagate to the caller. Catch and log locally. On the response path, if the envelope is malformed (missing/wrong `jsonrpc` field, unknown `error` shape), do not fire a parse-error event and leave the pending-id awaiter hanging — instead, reject the matching pending entry immediately via `pending.reject(error)`. Both rules surfaced in `harness/src/worker/transport.ts` (wrap at lines 108, 118, 155; malformed-version reject at 154) after review cycle 1. Patterns apply to any JSON-RPC duplex (stdio, websocket, IPC), not just this substrate.

### References

- Files: `harness/src/worker/{spawn,transport,bootstrap}.ts`, `harness/test/worker-spawn.test.ts`
- Docs: `docs/adr/0001-worker-substrate.md` §Consequences

## 2026-05-06 — M5.3.b.b.d: worker dispatcher + capability broker (B1 ACE caught at review)

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

Wiring `tool.kind === 'worker'` into `harness/src/invokeTool.ts` with capability gating exposed a class of bug specific to JSON-RPC dispatchers that spawn child processes: ANY field in the wire-validated `Params` shape that ends up as a fork-time argument (binary path, args, env override) is a wire ACE surface. Capability gating cannot protect against it because gating runs AFTER spawn arguments are chosen.

### Pattern

**Spawn-target injection — defence: drop the field structurally, don't try to validate it.** When designing dispatcher Params for a tool runner that spawns processes, do NOT include `bootstrapPath` / `argv0` / `envOverride` / `cwd` in the wire-validated type. Either remove the field entirely (`validateParams` returns `InvalidParams` if the JSON contains `"spawnOptions"` or similar — even with `null`), or split into two types: a wire-callable subset and an internal-only superset reachable only via a typed in-process function. The internal entry point should be named to broadcast its contract (`runWorkerToolUnsafeInternal_TEST_ONLY`) and ideally guarded by a depcruise/eslint-no-restricted-imports rule that bans imports outside the dispatcher module and tests. Reviewer caught this in PR review; the fix was a structural drop of `WorkerTool.spawnOptions` plus 3 regression tests asserting `validateParams` rejects the field on the typed handler, the JSON-RPC handleLine path, and even with `null`. Apply when adding ANY tool runner that spawns subprocesses.

### References

- Files: `harness/src/invokeTool.ts` (wire validation + dispatcher branch + try/catch around terminate in finally), `harness/src/worker/broker.ts` (gateToolInvocation + ToolOperation tagged union; exact-match path contract documented)
- Docs: `docs/adr/0001-worker-substrate.md` §Decision (capabilities frozen at spawn — but spawn arguments themselves are the ACE surface)

## 2026-05-06 — M5.3.c.a: Auto-derive zod tool schemas from Tool Manifest at harness boot

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

Tool execution requires runtime validation of input payloads against the manifest spec. Rather than maintaining separate schema definitions (one in the manifest DDL, another in zod), we synthesized the zod schema from the manifest at harness boot via a `deriveToolSchemas` helper function. This eliminates duplication and guarantees that validation logic cannot drift from the declared contract.

### Pattern

**Strict-by-default cascades from manifest layer to derived schemas.** When synthesizing a runtime zod schema from a manifest spec, the natural approach is to leave the synthesized `z.object({})` non-strict (preserving forward-compatibility for "extra payload fields"). This is a foot-gun: an input like `{ message: 'hi', extra: 'sneak' }` for a tool whose spec only declares `message` would silently pass validation, drop `extra`, and reach the tool — exactly the failure mode the manifest contract claims to prevent. The fix is one character (`.strict()` on the synthesized object) plus locking the behaviour in tests (unknown-key rejection + optional-key wrong-type rejection). The principle: **strict posture must cascade from the manifest layer down to every schema derived from it.** "Additive extensibility" belongs in the manifest source-of-truth (declare the property), not in unchecked runtime payloads.

### References

- Files: `harness/src/toolManifest.ts` (lines 178–195 — `buildSchemaForSpec` and `.strict()` on derived objects), `harness/test/toolManifest.test.ts` (optional-key strict-rejection + unknown-key rejection assertions)
- Docs: `docs/ROADMAP-phase1.md` §M5.3.c.a; companion to M5.3.b.b.b (schema-only modules ship `.strict()` everywhere) and M5.3.b.b.d (defence-in-depth at every wire boundary)

## 2026-05-06 — M5.3.c.b: LLMProvider wrapper: parameterize model/system-prompt/context from Manifest

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — Phase 7a

### Context

M5.1 established that `runtime.Manifest` is a post-loader projection of the wire schema.
M5.3.c.b extends this principle by adding request builders (`BuildCompleteRequest`,
`BuildStreamRequest`, `BuildCountTokensRequest`) that consolidate the field mapping at
the request boundary, so every concrete `Provider` implementation reads from the same
logical shape without duplicating projection logic.

### Pattern

Manifest→Request builders are the right place to lock the projection contract before
any concrete provider exists. The builders map required fields (`Model`, `System`,
reserved-key `Language` → `Metadata["language"]`) and compose functional-options
(last-write-wins). Critical detail: `Metadata` is defensively copied per caller safety,
but `[]Message` is forwarded by reference (callers must not mutate). Table-driven tests
across all three builders prove parity in 30 lines vs. 90 in copy-paste form. Pattern
principle: boundary-layer builders are where field mapping rules become testable.

## 2026-05-06 — M5.3.c.c.a: Add TS LLMProvider interface + FakeProvider mirroring Go contract

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — Phase 7a

### Context

Establish the TypeScript-side `LLMProvider` portable interface mirroring the Go contract
from M5.3.c (post-loader projection of model + system prompt + messages + tools;
`complete` / `stream` / `countTokens` / `reportCost` methods; discriminated `Role` /
`FinishReason` / `StreamEventKind` string-literal unions). Ship a `FakeProvider` test
stand-in with synchronous stream dispatch and validation-before-recording semantics.

### Pattern

Mirror Go's nominal-typing for single-value domains via branded intersection types
(`type Model = string & { __brand: "llm.Model" }`). Use `as const` arrays for enum-like
unions (`Role`, `FinishReason`, `StreamEventKind`) so both compile-time `typeof X[number]`
and runtime iteration use one source. Mirror Go's error sentinels via a single `LLMError`
class with discriminated `code` field; static factories (`LLMError.invalidPrompt(msg?)`,
etc.) match Go's construction ergonomics. Mirror Go's `*FakeProvider` synchronous contract
verbatim — validation errors (non-empty model, non-empty messages, non-null handler, etc.)
take precedence over canned errors; stream dispatch is deterministic (`for-of await`)
before the `stream()` call resolves. Provider interface stays a pure interface (zero
runtime footprint); contract exercised through `FakeProvider implements LLMProvider`.
Under `verbatimModuleSyntax: true`, exporting interfaces via `export type { … }` forces
consumers to use `import { type LLMProvider, FakeProvider, LLMError }` — document as a
TS-codebase-specific convention.

### References

- Files: `harness/src/llm/{types,errors,provider,fake-provider,index}.ts`, `harness/test/llm-fake-provider.test.ts`
- Docs: `docs/ROADMAP-phase1.md` §M5.3.c; counterpart Go implementation in M5.3.c.b (manifest request builders) and Go core in `core/pkg/llm/{provider,fake_provider_test}.go`

## 2026-05-06 — M5.3.c.c.b: Implement ClaudeCodeProvider adapter (default impl) with unit tests

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

Implemented the TypeScript adapter binding the LLM harness to the Anthropic SDK (`@anthropic-ai/sdk@^0.94.0`). The adapter must handle runtime-facing portable types (e.g., `maxTokens: number | undefined`) that use `0` to signal "provider default", but the SDK requires explicit numeric values and rejects `0` with a BadRequestError. Also discovered SDK API-shape deviations from the spec (e.g., `OverloadedError` not exported; new stop_reasons `pause_turn` / `refusal` added post-design).

### Pattern

For portable-zero semantics at adapter boundaries: when a typed field is required and must allow "use provider default", do NOT rely on truthiness checks like `req.maxTokens ?? DEFAULT`. Instead, implement a small resolver helper (`resolveMaxTokens(v: number | undefined): number => v && v > 0 ? v : DEFAULT`) that collapses both `undefined` AND `0` to the SDK's required value. Pin with a unit test asserting the SDK spy's actual field value, not just the call's success. Example: `expect(anthropic.messages.create).toHaveBeenCalledWith(expect.objectContaining({ max_tokens: DEFAULT }))` for both `undefined` and `0` inputs.

For SDK API-shape discovery: before drafting acceptance criteria, consult the SDK's context7 docs or verify the actual export list. Known deviations for `@anthropic-ai/sdk@^0.94.0`: `OverloadedError` is `APIError(status: 529)` (handle via catch-all branch); new stop_reasons (`pause_turn`, `refusal`) added since M5.2 design (use `default: return 'stop'` for forward-compat). Stream cancellation uses `stream.controller.abort()`; guard the call count with idempotent logic (`if (!this._stopRan) { stream.controller.abort(); this._stopRan = true }`).

For validation precedence testing: do NOT just verify the final result; prove the SDK was never contacted. Use spy assertions like `.not.toHaveBeenCalled()` on each negative case (model invalid → SDK silent; messages empty → SDK silent; etc.). Test all three SDK-bound methods (`complete`, `stream`, `countTokens`) with the same precedence order.

- Files: `harness/src/llm/claude-code-provider.ts`, `harness/test/llm-claude-code-provider.test.ts`, `harness/src/llm/fake-provider.ts` (validation pattern reference)
