## 2026-05-09 — M7.1.a: Saga skeleton: spawn/ package, state machine, saga-state DAO + migration

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

M7.1.a introduced `core/pkg/spawn/saga` with a step-based orchestrator and state-persistence DAO, plus the SQL migration for saga-state storage. The package defines a generic `Step` interface so M7.1.b–.e (Slack interaction, App provisioning, Notebook provision, runtime launch) can plug concrete steps without modifying saga core. Migration `019_spawn_sagas.sql` ships alongside the in-memory DAO so M7.1.b can wire a Postgres adapter without migration churn.

### Pattern

**Step-interface seam for multi-step external flows**: A saga's `Run` loop consumes `[]Step` where each step is `{Name() string, Execute(ctx) error}`. The orchestrator owns state persistence (UpdateStep BEFORE Execute) and audit emit (started BEFORE Execute, completed/failed BEFORE return). Future PRs add concrete steps as struct types — no modification to saga core. Reusable for retire saga (M7.2) and any multi-step flow.

**Audit-emit-before-return ordering generalizes M6.3.e to saga steps**: Every state transition's audit row emits BEFORE the boundary call (Execute / return). On `Step.Execute` error: `MarkFailed` → emit `saga_failed` → return error. Inverting any of the three creates a silent-data-loss defect. Pattern extends M6.3.e's "stream emit before user handler" to the saga domain.

**PII discipline closed-set payload (M2b.7 + M6.3.f wire-shape)**: Saga audit payloads are restricted to three keys (`saga_id`, `step_name`, `last_error_class`) and a regression-test grep-asserts the JSON keys. Step authors cannot leak raw step params or `error.Error()` strings into audit — payload constructors live in saga core, not Step impls.

**Migration-shipped-with-skeleton**: Per M6.3.b "ship in-memory DAO with consumer", M7.1.a ships the migration alongside the in-memory DAO so M7.1.b wires Postgres without migration churn.

## 2026-05-09 — M7.1.b: Slack interaction handler: approval action → Keeper's Log event → saga kickoff

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

M7.1.b wired Slack-approval-action → Keeper's Log audit event → spawn-saga kickoff using the M7.1.a step-interface seam. New seam `SpawnKickoff` interface in approval-package + concrete `SpawnKickoffer` impl in spawn-package + production-wiring helper. The saga runs with empty `[]Step{}` in this PR; concrete steps land in M7.1.c–.e.

### Pattern

**Per-tool branch on PendingApproval.ToolName**: When an approval flow needs to dispatch differently per tool, the cleanest seam is to branch the dispatcher on the closed-set `pending.ToolName` and route `propose_spawn` to a new `SpawnKickoff` seam while leaving the existing `Replayer` path for other tools. Mirrors the M6.3.b two-seam pattern (Replayer + Renderer); keeps the new dispatch path independently testable.

**manifest_version_id flows through ParamsJSON, not the PendingApproval struct**: When the upstream tool already serialised its params snapshot into `pending.ParamsJSON`, downstream branches that need a specific field (here `manifest_version_id`) parse it from the JSON envelope rather than expanding the `PendingApproval` struct. Path B (parse + typed sentinel `ErrMissingManifestVersionID`) keeps the diff inside the M7.1.b file set; path A (extend `PendingApproval` + DAO surface) was rejected because it would force changes to the M6.3.b storage contract for a transient need.

**Audit-emit-before-state-write generalises across the seam**: `SpawnKickoffer.Kickoff` emits `manifest_approved_for_spawn` BEFORE `Insert` BEFORE `Run([]Step{})`. The audit row is canonical even when state-persistence fails (M6.3.e + M7.1.a pattern). Test `TestSpawnKickoff_InsertError_AuditAlreadyEmitted` pins this — Insert error → audit row IS persisted, saga DAO has no row.

**Local seam re-declaration over cross-file unexported interface**: When an impl file needs an interface type already declared in a sibling file in the same package, prefer re-declaring locally (`kickoffLogAppender` in spawnkickoff.go alongside `keepersLogAppender` in slack*app.go) with a compile-time drift-pin assertion `var * kickoffLogAppender = keepersLogAppender(nil)`. Reader of one file sees the contract; build catches signature drift.

**Exact-equality vocabulary pin over HasPrefix tautology**: When guarding event-type vocabulary against future renames, use `if eventType != "manifest_approved_for_spawn"` rather than `!HasPrefix(et, "manifest_approved_for_spawn")` (which is tautological for the literal string). Keep `HasPrefix` checks against COLLIDING prefixes (`llm_turn_cost`, `saga_`) as defense-in-depth.

## 2026-05-09 — M7.1.c.a: CreateApp saga step + watchkeeper secrets column + migration

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

M7.1.c.a wired the first concrete saga.Step impl (CreateApp) bridging the M6.1.b SlackAppRPC privileged RPC to a new watchkeeper.slack_app_creds Postgres table via the existing M4.2.d sink-adapter contract. Introduced saga.SpawnContext value type for cross-step parameter passing and stabilized manifest_version_id + agent_id access across future steps.

### Pattern

**Sink-to-DAO bridge owns watchkeeperID, not appID**: A saga step that bridges an OUT-OF-BAND credential delivery (M4.2.d sink) to a persistence DAO must close over the saga's primary key (watchkeeperID from SpawnContext), NOT the credential's own assigned id (creds.AppID). Re-create scenarios change AppID; watchkeeper id is stable across the saga's lifetime. The DAO column layout reflects this: watchkeeper_id is PK, app_id is a column.

**SpawnContext as saga-level cross-step seam**: Future M7.1.x steps need read access to manifest_version_id, agent_id, and the dispatcher's Claim. Define a typed value (saga.SpawnContext) + accessors (WithSpawnContext / SpawnContextFromContext) in the saga package — NOT in the spawn package — so step impls have a stable contract that does not couple them to step-specific constructors.

**Source-grep AC pinning for negative AC ("does NOT call X")**: When an AC asserts that a unit must NOT call some API, the test that pins it cannot be a runtime mock — even an injected fake passes vacuously if the unit reaches the API via a different path or is later refactored. The robust pin is os.ReadFile on the unit's source + strings.Contains outside //-prefixed lines. Mutation-tested (adding the forbidden call must cause the test to fail).

**Migration ships with deferred encryption-at-rest comment**: Per the M6.3.b "ship in-memory DAO with consumer" rhythm, M7.1.c.a's migration uses text columns for credentials with an explicit -- TODO: encryption-at-rest deferral comment. The Phase-2 secrets pass will rotate to encrypted bytea. Keeps the M7.1.c.a PR scope tight.

### Anti-pattern

Hollow runtime-mock AC pins. The iter-1 reviewer caught a keeperslog.New(keep) that was constructed and immediately discarded (\_ = ...) — the recordingKeepClient could never observe the call because the writer was never injected. The fix replaced it with a source-grep mutation-tested assertion. Lesson: any AC pin that depends on observation through an injected dependency must verify the dependency reaches the unit-under-test; otherwise prefer source-grep.

## 2026-05-10 — M7.3.c: Per-step Compensate implementations + fault-injection harness

**PR**: #125
**Merged**: 4dede7d on 2026-05-10

### Context

M7.3.c lands the per-step `Compensate` implementations under M7.3
(Robustness) on top of the M7.3.b
[`saga.Compensator`](../../core/pkg/spawn/saga/compensator.go) seam.
All five M7.1.c–.e concrete steps gain a `Compensate(ctx)` method;
each forward-side step file additionally introduces ONE rollback
seam interface in the same file:

- [`CreateAppStep`](../../core/pkg/spawn/createapp_step.go) gains
  [`SlackAppTeardown`](../../core/pkg/spawn/createapp_step.go) (best-
  effort Slack-side teardown — Phase-1 reality: Slack does NOT
  expose a public `apps.delete`, so the production wrapper wipes
  the local `slack_app_creds` row + records the abandoned `app_id`
  to a follow-up reconciler queue an operator can drain manually).
- [`OAuthInstallStep`](../../core/pkg/spawn/oauthinstall_step.go) gains
  [`OAuthInstallRevoker`](../../core/pkg/spawn/oauthinstall_step.go)
  (Slack `auth.revoke` of the bot/user tokens). Compensate ALSO
  calls a new
  [`WatchkeeperSlackAppCredsDAO.WipeInstallTokens`](../../core/pkg/spawn/watchkeeper_creds.go)
  AFTER the revoke (or after a revoke failure) — wipe-always
  ordering keeps the encrypted columns consistent with a
  platform-side install state that may already be inconsistent.
  A companion
  [`WatchkeeperSlackAppCredsDAO.WipeAll`](../../core/pkg/spawn/watchkeeper_creds.go)
  exists for the [`SlackAppTeardown`](../../core/pkg/spawn/createapp_step.go)
  production wrapper's full-row deletion.
- [`BotProfileStep`](../../core/pkg/spawn/botprofile_step.go) gains
  an EXPLICIT no-op Compensate (returns nil, never touches the
  setter). Distinct from the M7.3.b "non-Compensator silently
  skipped" pattern: the audit chain emits a positive
  `saga_step_compensated` row for `bot_profile` so an operator
  reading the rollback chain sees the explicit "we considered the
  bot_profile rollback and intentionally no-opped" signal.
- [`NotebookProvisionStep`](../../core/pkg/spawn/notebookprovision_step.go) gains
  [`NotebookProvisionArchiver`](../../core/pkg/spawn/notebookprovision_step.go)
  (archive-not-delete + flag-for-review). Compensate validates the
  returned URI via [`net/url.Parse`](../../core/pkg/spawn/notebookprovision_step.go) + non-empty
  scheme, reusing the M7.2.b sentinels [`ErrEmptyArchiveURI`](../../core/pkg/spawn/notebookarchive_step.go) /
  [`ErrInvalidArchiveURI`](../../core/pkg/spawn/notebookarchive_step.go).
- [`RuntimeLaunchStep`](../../core/pkg/spawn/runtimelaunch_step.go) gains
  [`RuntimeTeardown`](../../core/pkg/spawn/runtimelaunch_step.go)
  (runtime stop + cost-record finalisation). Forwards both
  watchkeeperID + manifestVersionID to the seam.

The fault-injection harness in
[`fault_injection_test.go`](../../core/pkg/spawn/fault_injection_test.go)
wires all five concrete steps + recording fakes per Compensator-
side seam, then table-drives the failure index across steps 0..4.
For each case it asserts: (a) the failing step is NOT compensated,
(b) every step earlier in forward order IS compensated in REVERSE
order, (c) every step later in forward order receives NO Execute
call, and (d) the BotProfile Setter call count never exceeds the
forward-Execute count (no rollback-side setter dispatch). A
companion test pins the M7.3.b kickoffer-side
`manifest_rejected_after_spawn_failure` emit using a one-step
stub-failing saga (so the harness does not need to plumb
`OAuthCode` through the kickoffer's [`saga.SpawnContext`](../../core/pkg/spawn/saga/spawn_context.go)
seed — that's a separate M7.1.x concern).

### Pattern

1. **Per-saga state contract: SpawnContext + DAO, NEVER
   receiver-stash** — every concrete `Compensate` body sources
   `watchkeeperID` from
   [`saga.SpawnContextFromContext`](../../core/pkg/spawn/saga/spawn_context.go),
   then dispatches a single seam call that internally reads
   per-saga forward-state (the freshly-minted `app_id`, the
   encrypted bot token, the per-watchkeeper notebook file path)
   from the M7.1.c.a `slack_app_creds` DAO row keyed by the same
   `watchkeeperID`. The DAO IS the durability layer the M7.3.b
   lesson #1 named ("on a DAO with its own durability story,
   mirroring the M2b `slack_app_creds` row"). The
   [`SpawnContext`](../../core/pkg/spawn/saga/spawn_context.go)
   stays a tiny per-call ctx-value (no new outbox value type
   needed); the receiver-stash anti-pattern stays banned. Pinned
   by [`TestPerStepCompensate_ReadsSpawnContextFromCtx`](../../core/pkg/spawn/compensators_test.go)
   which source-greps every step file's `Compensate` function
   body for the `saga.SpawnContextFromContext(ctx)` call. Pattern:
   when a per-step Compensate needs per-saga forward-state, route
   the lookup through the existing keyed-DAO row; the
   `SpawnContext` carries only the saga-level handle, not the
   per-step state itself.

2. **Wipe-always ordering with revoke-error precedence** — the
   M7.3.c [`OAuthInstallStep.Compensate`](../../core/pkg/spawn/oauthinstall_step.go)
   ALWAYS dispatches [`WatchkeeperSlackAppCredsDAO.WipeInstallTokens`](../../core/pkg/spawn/watchkeeper_creds.go)
   AFTER the [`OAuthInstallRevoker.Revoke`](../../core/pkg/spawn/oauthinstall_step.go)
   call — even when revoke FAILS — so a subsequent saga retry
   never observes stale ciphertext for a token Slack already
   ignores. The revoke error is the LOAD-BEARING return; a wipe
   error surfaces ONLY when revoke succeeded (otherwise the wipe
   error is masked so the operator sees the genuine platform-
   side root cause). Mirrors the M7.3.b "kickoffer best-effort
   emit does not shadow runner error" pattern at the per-step
   layer. Pinned by
   [`TestOAuthInstallStep_Compensate_RevokeFails_WipeStillRuns_ErrorIsRevoke`](../../core/pkg/spawn/compensators_test.go).
   Pattern: when a Compensate body needs to do TWO best-effort
   actions (one platform-side, one DAO hygiene) and the platform
   action is the operator's "what failed?" signal, run BOTH
   regardless of the platform-action outcome AND surface ONLY the
   platform error in the wrap chain.

3. **Explicit no-op Compensator over silent skip** — the
   M7.3.b lesson #1 doc-block presented BotProfile as the
   canonical example of a step that does NOT implement
   Compensator (silently skipped on the rollback walk). M7.3.c
   intentionally DEVIATES: the
   [`BotProfileStep.Compensate`](../../core/pkg/spawn/botprofile_step.go)
   returns nil unconditionally (no SpawnContext read, no setter
   call), and the
   [`var _ saga.Compensator = (*BotProfileStep)(nil)`](../../core/pkg/spawn/botprofile_step.go)
   compile-time assertion is present so the saga emits a
   POSITIVE `saga_step_compensated` row for `bot_profile`. The
   audit-chain reader sees the explicit "we considered the
   compensation and no-opped" signal instead of having to reason
   about absence (which a non-Compensator would produce).
   Trade-off: one extra audit row per saga rollback that crosses
   step 2; benefit: operator-visible terminal accounting of which
   compensators fired. Pattern: when a step's rollback is a
   genuine no-op (the compensation is moot because the
   later-in-reverse rollback chain makes the side-effect
   inaccessible), prefer an EXPLICIT no-op `Compensate` over
   omitting `Compensator` — the audit chain's positive row is
   strictly more informative than the silent skip.

4. **Archive-not-delete + flag-for-review with URI shape
   validation** — the
   [`NotebookProvisionStep.Compensate`](../../core/pkg/spawn/notebookprovision_step.go)
   dispatches [`NotebookProvisionArchiver.ArchiveAndFlagForReview`](../../core/pkg/spawn/notebookprovision_step.go)
   and validates the returned URI is non-empty + parses via
   [`net/url.Parse`](../../core/pkg/spawn/notebookprovision_step.go)
   with a non-empty scheme. The validation REUSES the M7.2.b
   sentinels [`ErrEmptyArchiveURI`](../../core/pkg/spawn/notebookarchive_step.go) /
   [`ErrInvalidArchiveURI`](../../core/pkg/spawn/notebookarchive_step.go) —
   the same shape contract M7.2.b's retire-saga
   [`NotebookArchiveStep`](../../core/pkg/spawn/notebookarchive_step.go)
   already pins. The archive operation is "archive-not-delete":
   the seam contract forbids deletion of the source notebook file
   (the M2b.7 `notebook_entry_remembered` rows are durable
   records of the failed spawn that an operator might need for
   post-mortem). Pattern: when a rollback path's archive contract
   spans two saga families (spawn rollback + retire saga), reuse
   the existing shape sentinels rather than minting per-saga
   variants — operators reading the audit chain see a uniform
   `last_error_class` vocabulary regardless of which saga
   triggered the archive.

5. **At-least-once teardown forwards manifestVersionID +
   watchkeeperID** — the
   [`RuntimeTeardown`](../../core/pkg/spawn/runtimelaunch_step.go)
   seam takes BOTH ids even though the production wrapper
   currently uses only `watchkeeperID` (the runtime is keyed
   off it; the cost-aggregator's join is per-watchkeeper). The
   `manifestVersionID` is forwarded so a future teardown impl
   that needs to drain version-pinned cost rows can branch on it
   without re-resolving the saga. The seam contract documents
   at-least-once semantics explicitly (mirrors the M7.1.e
   `LaunchRuntime` doc-block); a repeat Compensate against an
   already-stopped runtime returns nil (idempotent). Pinned by
   [`TestRuntimeLaunchStep_Compensate_HappyPath_ForwardsBothIDs`](../../core/pkg/spawn/compensators_test.go).
   Pattern: when a future-extension knob lives on the
   `SpawnContext` AND today's seam impl ignores it, plumb the
   knob THROUGH the seam interface anyway — adding it later
   would force every existing fake / production wrapper to
   re-implement the signature.

6. **DAO Wipe surface idempotent on missing row** — the new
   [`WatchkeeperSlackAppCredsDAO.WipeInstallTokens`](../../core/pkg/spawn/watchkeeper_creds.go) /
   [`WipeAll`](../../core/pkg/spawn/watchkeeper_creds.go) methods
   return nil on a missing row (treat as already-wiped). The
   M7.3.b best-effort rollback chain explicitly allows
   double-Compensate (an operator's transient retry of a failed
   saga); a Wipe surface that returned `ErrCredsNotFound` would
   surface a hard failure on the second call and the rollback
   chain would emit a `saga_compensation_failed` audit row for
   what is functionally a clean state. Pinned by the
   [`TestOAuthInstallStep_Compensate_HappyPath_RevokeThenWipeAndReturnsNil`](../../core/pkg/spawn/compensators_test.go)
   case (after the first Wipe leaves the install-tokens map
   empty, a second Compensate call would reach the same clean
   state — the test's `_, _, _, _, _, ok := dao.GetInstallTokens(...)`
   assertion validates absence rather than a specific success
   sentinel). Pattern: every DAO method on the rollback path
   MUST be idempotent on missing rows; a hard-error-on-missing
   contract is incompatible with the M7.3.b best-effort
   discipline.

7. **Compensator-required dep panics at construction** — every
   step's `New*Step` constructor now panics on a nil
   Compensator-side dep
   ([`Teardown`](../../core/pkg/spawn/createapp_step.go),
   [`Revoker`](../../core/pkg/spawn/oauthinstall_step.go),
   [`Archiver`](../../core/pkg/spawn/notebookprovision_step.go),
   [`Teardown`](../../core/pkg/spawn/runtimelaunch_step.go)) —
   the M7.1.c.a panic-on-nil discipline extends to the M7.3.c
   rollback seams. Surfacing the wiring gap at boot (rather than
   on first rollback) is strictly better: a missing dep would
   otherwise reach the M7.3.b `safeCompensate` panic harness on
   the rollback path and class onto the
   [`LastErrorClassCompensatePanic`](../../core/pkg/spawn/saga/compensator.go)
   sentinel, hiding the wiring bug behind a runtime audit row.
   Pinned by per-step constructor-panic tests
   ([`TestNewCreateAppStep_PanicsOnRequiredFields`](../../core/pkg/spawn/createapp_step_test.go),
   [`TestNewOAuthInstallStep_PanicsOnRequiredFields`](../../core/pkg/spawn/oauthinstall_step_test.go),
   [`TestNewNotebookProvisionStep_PanicsOnNilArchiver`](../../core/pkg/spawn/notebookprovision_step_test.go),
   [`TestNewRuntimeLaunchStep_PanicsOnNilTeardown`](../../core/pkg/spawn/runtimelaunch_step_test.go);
   BotProfile has no Compensator-side dep — the no-op needs no
   seam). Pattern: a step that satisfies `saga.Compensator` MUST
   panic on nil Compensator-side deps at construction; the
   safeCompensate harness is for genuine impl panics, not for
   wiring gaps.

8. **Cross-step harness via `saga.Runner.Run` direct dispatch** —
   the [`fault_injection_test.go`](../../core/pkg/spawn/fault_injection_test.go)
   harness drives `saga.Runner.Run` DIRECTLY (not via
   [`SpawnKickoffer.Kickoff`](../../core/pkg/spawn/spawnkickoff.go))
   with a pre-seeded `SpawnContext` carrying a non-empty
   `OAuthCode`. The kickoffer path's `OAuthCode` plumbing is a
   separate M7.1.x concern (the kickoffer doesn't seed it
   today; production wiring will plumb it through the M6.3.b
   approval card / OAuth callback). Bypassing the kickoffer
   keeps the harness focused on the rollback chain — every
   step's `Compensate` runs through the saga runner's
   reverse-walk + `context.WithoutCancel` + `safeCompensate`
   harness without coupling the test to an unrelated
   kickoffer-input gap. The kickoffer-side
   `manifest_rejected_after_spawn_failure` emit is pinned by a
   companion
   [`TestKickoffer_RejectionEmit_FiresOnStep0Failure`](../../core/pkg/spawn/fault_injection_test.go)
   case using a one-step stub-failing saga (no `OAuthCode`
   needed). Pattern: when a cross-step integration harness
   needs to exercise the runner's rollback chain AND a separate
   kickoffer-side emit, separate the two concerns into distinct
   tests — the runner-side harness drives the steps directly,
   the kickoffer test uses a minimum-viable failing step. The
   alternative (forcing the kickoffer to seed every step's
   per-call inputs) couples the harness to unrelated wiring
   gaps and amplifies test breakage when those gaps are filled.

9. **PII canary harness on per-step Compensate wrap** — every
   per-step Compensate failure-path test injects a synthetic
   canary substring (`piiCanary`) into the seam's returned error
   and asserts: (a) the canary IS preserved by `%w` (otherwise
   `errors.Is` wouldn't match — defensive assertion), AND (b)
   the step-internal config (creds, app_id, approval token,
   app_name) does NOT leak into the wrap chain. Mirrors the
   M7.1.c.\* PII canary harness pattern. Pinned by
   [`TestCreateAppStep_Compensate_PIICanary_NeverLeaksAppNameOrApprovalToken`](../../core/pkg/spawn/compensators_test.go)
   and analogous cases for the other steps. Pattern: a
   per-step PII-discipline test MUST assert BOTH directions —
   "the wrap preserves the underlying error string" AND "the
   wrap does NOT inject step-internal config substrings" —
   because asserting only the second direction would tolerate
   a regression that drops the underlying error entirely
   (silent observability loss).

10. **Source-grep AC: per-step Compensate BODY MUST NOT call
    `keeperslog.Append`** — a forbidden-substring scan over the
    five step files (`createapp_step.go`,
    `oauthinstall_step.go`, `botprofile_step.go`,
    `notebookprovision_step.go`, `runtimelaunch_step.go`)
    pinned by [`TestPerStepCompensate_DoesNotEmitToKeepersLog`](../../core/pkg/spawn/compensators_test.go).
    The scan slices each file to the `Compensate(...)` function
    body only via the shared
    [`compensateFunctionBody`](../../core/pkg/spawn/compensators_test.go)
    helper — a whole-file scan would also trip on legitimate
    forward-path emits and would miss a regression that adds a
    non-keeperslog audit name to a Compensate body. The audit
    chain belongs to the saga runner; a Compensate that emits
    its own audit row would double-emit on the rollback path AND
    would couple the step's blast radius to the keeperslog wire
    vocabulary. Pattern: a load-bearing source-grep AC pin MUST
    scope its scan window to the function body it claims to
    enforce; whole-file scans are a "looks pinned but isn't"
    anti-pattern (iter-1 critic Critical against the M7.3.c
    pre-iter-1 implementation that scanned whole files).

### Iter-1 patterns (codex + critic review)

1. **Failed-step partial-success surfaces deferred to a future
   reconciler (codex Major #1/#2/#3)** — codex flagged that the
   M7.3.b "failed step is NOT compensated" discipline collides
   with the spawn-saga reality that
   [`SlackAppRPC.CreateApp`](../../core/pkg/spawn/createapp_step.go)
   AND [`SlackAppInstaller.InstallApp`](../../core/pkg/spawn/oauthinstall_step.go)
   both perform the externally-visible side effect (Slack `apps.manifest.create` /
   `oauth.v2.access`) BEFORE the credentials sink runs. A
   sink-failure path therefore returns non-nil from `Execute`
   even though the platform side has already mutated; the
   runner skips the step's `Compensate`, and the orphaned
   Slack App / live install survives. The new
   [`SlackAppTeardown`](../../core/pkg/spawn/createapp_step.go) /
   [`OAuthInstallRevoker`](../../core/pkg/spawn/oauthinstall_step.go)
   seam signatures take only `watchkeeperID` — the in-process
   `app_id` / token bundle that would need to be cleaned up is
   not threaded through, so even an in-Execute fail-fast
   cleanup against the existing seams could not delete the
   platform-side artefact (the DAO row was never persisted).
   M7.3.c ACCEPTS this limitation as a documented Phase-1
   constraint: a future M7.3.d-or-M7.4 reconciler will land BOTH
   (a) widened seam signatures
   (`Teardown.TeardownAppByID(ctx, watchkeeperID, knownAppID)` /
   `Revoker.RevokeWithToken(ctx, watchkeeperID, knownToken)`) AND
   (b) in-Execute fail-fast cleanup on the partial-success
   branches of CreateApp / OAuthInstall, OR will defer the
   cleanup to a sweep-style M7.3.d-or-M7.4 reconciler that polls
   the `slack_app_creds` rows for orphaned `app_id` values
   without the corresponding install-tokens columns. Pattern:
   when the "fail-fast precedes side effect" discipline
   (M7.1.c.\* / M7.3.b lesson) does NOT actually keep the
   partial-success surface narrow for a specific step (because
   the platform call is the side effect AND it precedes any
   client-side fail-fast), document the residual surface
   explicitly in the step file's doc-block + the lesson entry
   rather than ship half-baked in-Execute cleanup that can't
   fully recover. The fault-injection harness in
   [`fault_injection_test.go`](../../core/pkg/spawn/fault_injection_test.go)
   intentionally exercises only top-level Execute failures
   (BEFORE the platform call) for the same scoping reason —
   exhaustive partial-success coverage requires the widened
   seams that land with M7.3.d-or-M7.4.

   Naming convention: every reference to this deferred follow-up
   uses the verbatim string `M7.3.d-or-M7.4 reconciler` so an
   `rg` search returns ALL surfaces in one query — see
   [`createapp_step.go`](../../core/pkg/spawn/createapp_step.go),
   [`oauthinstall_step.go`](../../core/pkg/spawn/oauthinstall_step.go),
   [`fault_injection_test.go`](../../core/pkg/spawn/fault_injection_test.go),

## 2026-05-10 — M7.3.b: Compensation infrastructure in saga package + manifest-rejected emit on rollback

**PR**: #124
**Merged**: aed3902 on 2026-05-10

### Context

M7.3.b shipped the rollback foundation under M7.3 (Robustness). The
[`saga`](../../core/pkg/spawn/saga/) package grows an OPTIONAL
[`Compensator`](../../core/pkg/spawn/saga/compensator.go) interface
that step authors may satisfy alongside [`saga.Step`](../../core/pkg/spawn/saga/saga.go);
when a later step's `Execute` returns non-nil, [`saga.Runner.Run`](../../core/pkg/spawn/saga/saga.go)
walks every previously-successful step in REVERSE forward order and
calls `Compensate` on each implementer (silently skipping non-
implementers). Three new closed-set audit event types ride the
M7.1.a `saga_*` vocabulary:
[`saga_step_compensated`](../../core/pkg/spawn/saga/compensator.go),
[`saga_compensation_failed`](../../core/pkg/spawn/saga/compensator.go),
and the saga-level summary
[`saga_compensated`](../../core/pkg/spawn/saga/compensator.go) emitted
after every failure path (zero per-step rows on a first-step
failure; len(executed) per-step rows otherwise). The spawn-side
[`SpawnKickoffer.Kickoff`](../../core/pkg/spawn/spawnkickoff.go) now
emits a single
[`manifest_rejected_after_spawn_failure`](../../core/pkg/spawn/spawnkickoff.go)
audit row AFTER `runner.Run` returns non-nil; the row carries the
FIRST-call's `saga_id` / `manifest_version_id` / `watchkeeper_id`
(via [`saga.IdempotentInsertResult.Existing`](../../core/pkg/spawn/saga/dao.go))
plus the bot's `agent_id` and the `tok-<6-prefix>` redacted
approval token. No concrete Compensate impls land in M7.3.b — the
M7.1.c.a CreateApp / M7.1.c.b.b OAuthInstall / M7.1.c.c BotProfile /
M7.1.d NotebookProvision / M7.1.e RuntimeLaunch step set is
wire-compatible because non-Compensator steps are silently skipped
on the rollback walk. M7.3.c lands the per-step Compensate
implementations + a fault-injection harness.

### Pattern

1. **Optional interface seam over mandatory subtype** — the
   M7.3.b rollback contract MUST NOT force every saga step to
   declare a Compensate method. Many concrete steps have nothing
   to roll back: M7.1.c.c
   [`BotProfileStep`](../../core/pkg/spawn/botprofile_step.go)'s
   only side effect is a watchkeeper-row write that the M7.1.c.a
   CreateApp.Compensate teardown of the Slack App makes moot
   anyway; M7.2.c
   [`MarkRetiredStep`](../../core/pkg/spawn/markretired_step.go)
   is the terminal step in its saga and never has a successor
   that could fail and trigger its rollback. The optional
   [`Compensator`](../../core/pkg/spawn/saga/compensator.go)
   interface lets these steps stay AS-IS — the runner uses a
   type-assertion (`step.(Compensator)`) at the rollback walk and
   silently skips non-implementers (no `saga_step_compensated`
   row, no `saga_compensation_failed` row, no panic). Pattern:
   when a behavioural extension applies to a SUBSET of an
   existing interface's implementers, model it as an OPTIONAL
   sibling interface over a mandatory base extension; the runtime
   type-assertion at dispatch time is the seam, not a
   compile-time inheritance forcing every consumer to provide a
   no-op stub.

2. **Best-effort rollback discipline at the audit-chain layer** —
   when a [`Compensator.Compensate`](../../core/pkg/spawn/saga/compensator.go)
   returns non-nil the runner emits
   [`saga_compensation_failed`](../../core/pkg/spawn/saga/compensator.go)
   AND continues with the remaining (earlier-in-forward-order)
   compensations rather than aborting the chain. Stopping
   mid-rollback would leave the operator with a half-rolled-back
   saga where the failed-compensation step's audit row is the
   only signal AND the still-pending compensations are silent —
   strictly worse than the best-effort variant where every
   successful-step compensation has its own
   `saga_step_compensated` or `saga_compensation_failed` audit
   row. The original step error stays the
   [`Runner.Run`](../../core/pkg/spawn/saga/saga.go) return value;
   compensation outcomes are audit-chain-only. Pattern: when a
   chain of cleanup actions is partial-failure-tolerant, the
   audit chain (not the function return value) carries the
   per-element outcome; the chain-driver's error contract reports
   only the original cause.

3. **Saga-level summary row emitted unconditionally on the
   failure path** — even when zero per-step compensations fire
   (a first-step failure leaves `executed` empty), the runner
   still emits
   [`saga_compensated`](../../core/pkg/spawn/saga/compensator.go)
   so a downstream consumer can pin "rollback chain complete"
   without counting per-step rows. The summary row is the
   audit-chain analogue of M7.1.a's
   [`saga_completed`](../../core/pkg/spawn/saga/saga.go) — every
   terminal saga (success or failure) gets exactly one
   "we're done" row regardless of how many per-step rows
   precede it. Pattern: a chain-driver's audit chain MUST carry
   a saga-level terminal row even on the empty-chain path —
   downstream consumers pin completion on the summary, not on
   the count of per-step rows.

4. **Distinct-default sentinel for compensate-failure last_error_class** —
   the existing [`LastErrorClassDefault`](../../core/pkg/spawn/saga/saga.go)
   sentinel (`step_execute_error`) classes a `Step.Execute`
   failure; the M7.3.b
   [`LastErrorClassCompensateDefault`](../../core/pkg/spawn/saga/compensator.go)
   sentinel (`step_compensate_error`) classes a
   `Compensator.Compensate` failure. The two surfaces share a
   resolver (the new
   [`resolveLastErrorClassWithDefault`](../../core/pkg/spawn/saga/saga.go)
   helper, which takes the fallback as a parameter so the
   `LastErrorClassed` walk stays DRY) but emit DIFFERENT
   defaults so an operator filtering the audit chain by
   `last_error_class` can distinguish "step blew up" from
   "step's rollback blew up" without parsing the surrounding
   `event_type`. Pattern: when two failure surfaces share a
   resolution algorithm but represent semantically distinct
   outcomes, factor the algorithm and customize the fallback per
   call site — DON'T overload one default and force consumers to
   read the surrounding context to disambiguate.

5. **Reverse-walk is positional, not Compensator-presence-aware**
   — the runner's reverse loop iterates `executed[len-1..0]`
   regardless of which entries implement
   [`Compensator`](../../core/pkg/spawn/saga/compensator.go). A
   step that does NOT implement the interface is still appended
   to `executed` so the loop's positional logic stays correct;
   the per-step Compensate dispatch checks `step.(Compensator)`
   at iteration time. Pre-iter design alternative: filter
   `executed` to only Compensator implementers before iterating
   — rejected because it forces a second pass over the slice
   AND because a future Compensator added on a step previously
   non-Compensator would fail to compose with the partial
   `executed` snapshot. Pattern: compensation-walk slices carry
   ALL successfully-executed steps, regardless of compensator
   presence; the seam check happens at dispatch, not at slice
   construction.

6. **Kickoffer-side rejection emit is best-effort and
   error-preserving** — the spawn kickoffer's new
   [`manifest_rejected_after_spawn_failure`](../../core/pkg/spawn/spawnkickoff.go)
   append is wrapped in a `_, _ = ...` discard so a transient
   keeperslog outage that drops the rejection row MUST NOT
   shadow the original `runner.Run` error. The operator's
   wrap-chain answer to "what failed?" stays the saga's step
   error chain (`spawn: kickoff: run saga: <step err>`),
   regardless of whether the rejection-row append succeeded.
   The pin in
   [`TestSpawnKickoff_RejectionAppendFails_OriginalRunErrorPreserved`](../../core/pkg/spawn/spawnkickoff_test.go)
   asserts both directions: `errors.Is(err, stepErr) == true`
   AND `errors.Is(err, appendErr) == false`. Pattern: an
   audit-only side effect on a failure path MUST be silenced
   in the wrap chain; the original failure is the operator's
   load-bearing signal, the audit emit is best-effort
   instrumentation.

7. **Rejection-emit ids sourced from `result.Existing`, not
   second-call args** — the M7.3.a iter-1 lesson #1
   ("stuck-saga catch-up branch") held forward verbatim into
   M7.3.b: the rejection event MUST source `saga_id` /
   `manifest_version_id` / `watchkeeper_id` from
   [`saga.IdempotentInsertResult.Existing`](../../core/pkg/spawn/saga/dao.go)
   so a catch-up rejection (where the second-call's args differ
   from the persisted FIRST-call ids) names the original saga
   on the audit chain rather than a self-contradictory mix.
   The M7.3.b implementation reuses the same `emitSagaID` /
   `emitManifestVersionID` / `emitWatchkeeperID` locals
   already populated from `result.Existing` for the
   `manifest_approved_for_spawn` emit, so the rejection row
   stays consistent with the approval row by construction.
   Pattern: when an insert-or-replay DAO surface discards
   caller-supplied ids on dedup, EVERY subsequent audit row in
   the same handler MUST source ids from `Existing.*`, not
   from the caller's parallel locals. Reuse the locals
   populated for the first audit row to enforce the invariant
   at the call site.

8. **Manifest-DAO mutation deferred to a downstream consumer** —
   the roadmap text "the kickoffer emits
   `manifest_rejected_after_spawn_failure` so the Manifest is
   marked rejected and the operator is surfaced" reads two
   ways: (a) emit the audit row AND ALSO mutate the Manifest
   DAO, or (b) emit the audit row and let a downstream
   consumer (a future M7.4-or-equivalent reconciler) mutate
   the Manifest. M7.3.b ships (b) — the kickoffer's blast
   radius stays at the keeperslog seam; the Manifest-DAO write
   is deferred to a follow-up so this PR's surface stays
   narrow per the roadmap's "no concrete Compensate impls —
   foundation for M7.3.c" wording. Pinned by the source-grep AC
   in
   [`TestSpawnKickoffSourceCode_NoManifestDAOWrite`](../../core/pkg/spawn/spawnkickoff_test.go)
   that scans non-comment lines for forbidden substrings
   (`ManifestDAO`, `manifestDAO`, `MarkManifestRejected`,
   `RejectManifest`, `core/pkg/manifest`). Pattern: when a
   roadmap's outcome verb ("Manifest is marked rejected")
   could land at the emitter OR at a downstream consumer,
   default to the consumer-side write and defend the boundary
   with a source-grep AC; consolidating the surface keeps the
   M7-family kickoffers' "audit-only" character intact for the
   M7.3.c per-step Compensate implementations to lean on.

9. **Existing concurrent-replay test deflaked via blocking-step
   barrier** — the M7.3.a iter-1 catch-up branch
   ([`SpawnKickoffer.Kickoff`](../../core/pkg/spawn/spawnkickoff.go)
   on `Status == SagaStatePending`) made the M7.3.a-shipped
   `TestSpawnKickoff_ConcurrentReplays_ExactlyOneRun` test
   non-deterministic on a zero-step saga: the saga transitioned
   pending → completed inside `Runner.Run` without ever
   visiting `in_flight`, so concurrent goroutines that won the
   InsertIfAbsent mutex AFTER the winner's Insert but BEFORE
   the winner's MarkCompleted observed `Status == pending` and
   fell into the catch-up branch (multiple `manifest_approved_for_spawn`
   rows + multiple `Get` calls). Verified flaky on `main` via
   a `git worktree` checkout: `-count=10` produced 10 / 10
   failures with varying counts. M7.3.b's deflake replaces the
   zero-step saga with a one-blocking-step saga + barrier:
   goroutine #1 acquires InsertIfAbsent, runs the saga, hits
   the blocking step, signals `entered`, then waits on
   `release`; the test polls `len(insertIfAbsentSeq)` until
   all 16 calls land (all 15 others observe Status=in_flight
   on the locked DAO and route through the true-replay
   branch); the test then closes `release` to let the winner
   complete. Pattern: a concurrent test that races a
   state-machine transition needs an explicit barrier holding
   the state at the desired observation point — relying on
   "the goroutines schedule in the order I expect" is
   undefended against the M7.3.a-iter-1-style branch addition
   that widens the prior race window. Future M7.x test authors:
   when a kickoffer branches on `Existing.Status`, every
   concurrent test of that kickoffer needs a deterministic
   anchor at the desired Status; the blocking-step barrier is
   the lightest-weight form.

### Iter-1 patterns (critic review)

1. **Compensator's per-saga state contract is `SpawnContext`,
   NEVER receiver-stash (critic Major #1)** — the pre-iter-1
   doc-block at
   [`compensator.go`](../../core/pkg/spawn/saga/compensator.go)
   invited step authors to "stash any per-saga forward-state on
   the receiver" so a future M7.3.c CreateApp.Compensate could
   read the freshly-minted Slack `app_id`. That advice directly
   contradicts the M7.1.a [`saga.Step`](../../core/pkg/spawn/saga/saga.go)
   contract ("the same step instance MAY serve multiple sagas
   concurrently — MUST NOT carry per-saga state"). Two distinct
   sagas hitting the SAME step instance concurrently would race
   the receiver-stash; a M7.3.a-iter-1 catch-up Run on
   `Status == pending` would Compensate against stale ids from
   the prior run. The fix rewrites the doc-block to point at the
   `SpawnContext` / `RetireResult` ctx-outboxes as the canonical
   per-saga state channel; M7.3.c authors land their forward
   values on the outbox (or on a DAO with its own durability
   story, mirroring the M2b `slack_app_creds` row), and
   Compensate reads them back via the same outbox. Pattern: when
   a stateful contract collision lurks between two doc-blocks on
   sibling interfaces, fix the contradiction at the doc-block
   layer BEFORE the contract is locked in by concrete impls; a
   future-maintainer reading either doc in isolation must reach
   the same conclusion.

2. **`Runner.compensate` runs under `context.WithoutCancel`
   (critic Major #2)** — pre-iter-1 the rollback walk passed the
   parent ctx straight into every
   [`Compensator.Compensate`](../../core/pkg/spawn/saga/compensator.go)
   call. A request-bound parent that fired Cancel mid-saga (HTTP
   timeout on the Slack approval card; operator-initiated abort;
   dispatcher tear-down) would poison every Compensate with
   `context.Canceled` — the entire rollback chain would
   uniformly fail, leaking Slack Apps and OAuth tokens. The fix
   derives `compensateCtx := context.WithoutCancel(ctx)` (Go
   1.21+) before the rollback loop in
   [`saga.go`](../../core/pkg/spawn/saga/saga.go); per-Compensate
   timeouts belong to the step author (mirrors the M7.1.c.\*
   "Execute owns its own retry / timeout policy" pattern). The
   audit chain still uses the ORIGINAL `ctx` for the per-step
   audit emits so trace identifiers propagate. Pinned by
   [`TestRunner_Compensate_PreCancelledParentCtx_StillDrivesRollback`](../../core/pkg/spawn/saga/compensator_test.go).
   Pattern: when a chain-driver dispatches cleanup on a path
   triggered BY the parent's cancellation, the cleanup ctx MUST
   be derived via `context.WithoutCancel` — propagating the
   cancelled parent makes the rollback uniformly fail, which is
   strictly worse than the parent's failure mode it was
   designed to remediate.

3. **Distinct sentinels for `context.Canceled` /
   `DeadlineExceeded` / panic on the Compensate path (critic
   Major #3 + missing-edge-case)** — pre-iter-1
   [`resolveCompensateLastErrorClass`](../../core/pkg/spawn/saga/compensator.go)
   collapsed every untyped error onto the
   [`LastErrorClassCompensateDefault`](../../core/pkg/spawn/saga/compensator.go)
   sentinel `step_compensate_error`; the M7.3.b lesson #4 sold
   the M7.1.a-vs-M7.3.b "execute vs compensate" disambiguation
   but in practice every rollback failure landed under the
   same sentinel until M7.3.c step authors plumbed
   `LastErrorClassed`. The fix folds three additional buckets
   onto the resolver:
   [`LastErrorClassCompensateContextCanceled`](../../core/pkg/spawn/saga/compensator.go)
   (`compensate_context_cancelled`),
   [`LastErrorClassCompensateContextDeadline`](../../core/pkg/spawn/saga/compensator.go)
   (`compensate_context_deadline`), and
   [`LastErrorClassCompensatePanic`](../../core/pkg/spawn/saga/compensator.go)
   (`compensate_panic`). Resolution priority: typed
   `LastErrorClassed` wins over the stdlib-context branches,
   so a step author who wraps `context.Canceled` under a typed
   sentinel keeps their step-specific class. A buggy
   panicking Compensate is recovered into a typed
   [`errPanicDuringCompensate`](../../core/pkg/spawn/saga/saga.go)
   value via a `defer/recover` harness (`safeCompensate` in
   `saga.go`), classed onto the panic-bucket sentinel WITHOUT
   ever serialising the recovered value (M2b.7 PII discipline —
   panicking impls could carry secrets in the recovered value).
   Pattern: when an error-classification helper has an opaque-
   default fallback, audit-chain queryability degrades to "two
   buckets". Decompose the fallback's uniform branch into a
   small fixed set of `errors.Is`-resolved buckets BEFORE
   shipping the contract; a closed-set of N+1 sentinels is
   strictly more useful than a single default with a comment
   that says "M7.3.c authors will type their errors".

4. **Best-effort rejection-emit failure surfaces structured
   `slog.Warn` (critic Minor #1)** — pre-iter-1 the kickoffer's
   `_, _ = k.logger.Append(...)` discarded any Append error,
   leaving "the kickoffer attempted the rejection emit and got
   an error" invisible at the call site (the keeperslog Writer's
   own diagnostic sink covers it but is opaque to the kickoffer
   interface). The fix wraps the discard in
   `if _, appendErr := k.logger.Append(...); appendErr != nil`
   and emits a `slog.WarnContext` carrying `saga_id`, `agent_id`,
   and a closed-set `err_class=manifest_rejected_after_spawn_failure_emit_dropped`
   key. The slog default handler is opaque in unit tests but
   production wires a structured handler for ops. Pattern:
   best-effort discards on a kickoffer's audit-failure surface
   need a structured-log emit at the call site even when the
   downstream sink already logs the failure internally — the
   call-site signal is what an ops investigation reaches first.

5. **Concurrency test polling has a 5s watchdog +
   `time.Sleep(time.Millisecond)` instead of `runtime.Gosched()`
   (critic Minor #2)** — pre-iter-1
   `TestSpawnKickoff_ConcurrentReplays_ExactlyOneRun`'s polling
   loop used `runtime.Gosched()` (a hint, not a yield) without a
   timeout cap. A regression that deadlocks the catch-up branch
   would hang the test forever rather than fail fast. The fix
   adds `deadline := time.After(5 * time.Second)` with a
   `select { case <-deadline: t.Fatalf(...); case <-time.After(time.Millisecond): }`
   inner step. Pattern: concurrent-test polling that races a
   state-machine transition needs both a watchdog (fast-fail on
   regression) and a deterministic yield (`time.Sleep` over
   `runtime.Gosched`) — `Gosched` is a scheduler hint that the
   runtime may ignore.

6. **`recordingDAO` records seq for every DAO method, not just
   the M7.3.a subset (critic Minor #3)** — pre-iter-1 the test
   fake delegated `UpdateStep` / `MarkCompleted` / `MarkFailed`
   to `inner` without seq capture, so a future regression that
   re-orders MarkFailed-vs-the-compensation-walk couldn't be
   pinned through the DAO. The fix adds `updateStepSeq` /
   `markCompletedSeq` / `markFailedSeq` slices on the recording
   DAO and ticks them on every dispatch. Tooling-only, no
   behaviour change; future tests pin ordering via the new
   counters.

7. **Source-grep AC widens against `manifest_status` /
   `Manifest.Status` / generic `UpdateManifest` /
   `manifest_versions` (critic Minor #6)** — pre-iter-1
   [`TestSpawnKickoffSourceCode_NoManifestDAOWrite`](../../core/pkg/spawn/spawnkickoff_test.go)
   only flagged `ManifestDAO` / `MarkManifestRejected` / etc.;
   a regression that imported a Manifest mutation under a
   non-flagged identifier would slip through. The fix extends
   the forbidden-substring list to cover the closed-set of
   plausible Manifest-DAO surfaces. Pattern: source-grep ACs
   defending a "this file does not touch X" boundary need to
   enumerate the closed set of identifiers a regression could
   reach X through; partial enumeration leaves a future
   reviewer-bypassable hole.

8. **`retire_*` prefix-collision pin for the new event_type
   (critic missing-edge-case)** — pre-iter-1
   `TestEventTypeManifestRejectedAfterSpawnFailure_PrefixCollisionFree`
   defended against `llm_turn_cost_*` / `saga_*` / existing
   `manifest_*` events but not the M7.2.a `retire_*` family. A
   regression renaming the event to `retire_manifest_rejected_*`
   would silently feed bogus rows into the retire-side
   audit-chain consumer. Fix adds `retire_` to the prefix-check.

9. **`safeCompensate` recovers panics into a typed error +
   keeps the goroutine alive to emit the saga-level summary
   (critic missing-edge-case)** — pre-iter-1 a buggy
   panicking Compensate would tear down the saga goroutine and
   skip the [`EventTypeSagaCompensated`](../../core/pkg/spawn/saga/compensator.go)
   summary row, leaving the operator with a half-rolled-back
   audit chain and no "rollback complete" terminal signal. The
   fix wraps each Compensate dispatch in a `defer/recover`
   harness; the recovered panic surfaces as a typed
   [`errPanicDuringCompensate`](../../core/pkg/spawn/saga/saga.go)
   that classes onto the panic-bucket sentinel. The audit
   chain's "we attempted N rollbacks; here's the terminal row"
   semantics survive a buggy step author. Pattern: a
   chain-driver's terminal summary row needs a panic-recovery
   harness around every dispatch; without it, the bug class
   "step impl panicked" silently degrades the audit chain's
   load-bearing terminal-row guarantee.

## 2026-05-10 — M7.3.a: Idempotency keys on spawn + retire saga kickoffers

**PR**: #123
**Merged**: dd42059 on 2026-05-10

### Context

M7.3.a shipped the idempotency-dedup contract for the `spawn` and
`retire` saga kickoffers. The forward-saga dispatcher mints a fresh
candidate `sagaID` + `watchkeeperID` via `uuid.New()` on every
Approve-button click; pre-M7.3.a a Slack retry / double-click would
have produced two sagas, two `manifest_approved_for_spawn` audit
rows, and two Slack Apps. M7.3.a routes both kickoffers through a
new
[`SpawnSagaDAO.InsertIfAbsent(ctx, id, manifestVersionID, idempotencyKey)`](../../core/pkg/spawn/saga/dao.go)
seam keyed by the upstream `approval_token`; a duplicate token
returns the existing saga row (no error) and the kickoffer emits a
distinct `*_replayed_*` audit event without re-running
[`saga.Runner.Run`](../../core/pkg/spawn/saga/saga.go).

### Pattern

1. **Dedup at the storage layer with a partial UNIQUE index, not at
   the application layer** — `idempotency_key text NULL` carries a
   `CREATE UNIQUE INDEX … WHERE idempotency_key IS NOT NULL`
   ([`023_spawn_sagas_idempotency_key.sql`](../../deploy/migrations/023_spawn_sagas_idempotency_key.sql))
   so the SQL planner enforces "at most one row per non-NULL key"
   without blocking legacy NULL-keyed rows. Application-layer dedup
   (e.g. SELECT-then-INSERT) loses the race under concurrent
   approvals; the partial UNIQUE index makes "exactly one INSERT
   wins" the storage-side guarantee. Pattern: when a dedup contract
   is meaningful only for a non-empty optional column, use a partial
   UNIQUE index — full UNIQUE works on Postgres (NULLs are distinct)
   but loses the explicit "this column has dedup semantics only when
   non-NULL" intent.

2. **Result-struct over (Saga, bool, error) triple** — the new
   [`saga.IdempotentInsertResult{Inserted bool, Existing Saga}`](../../core/pkg/spawn/saga/dao.go)
   surfaces the insert-vs-replay branch as a typed field rather than
   a positional bool. Two reasons: a future addition (e.g. a
   `Conflicted bool` for the same-id-different-key wiring-bug
   branch) lands as a new field without breaking the signature; and
   the call site reads as `if !result.Inserted { … }` rather than
   `if !inserted { … }` — the former survives a future refactor that
   reorders the return values, the latter does not. Pattern:
   multi-valued returns whose semantic-shape might grow → struct;
   stable two-of-the-same → triple. The M7.3.a kickoffers
   ([spawnkickoff.go](../../core/pkg/spawn/spawnkickoff.go),
   [retirekickoff.go](../../core/pkg/spawn/retirekickoff.go)) both
   branch on `result.Inserted` — the field name is the contract.

3. **M7.1.b "audit-emit precedes Insert" invariant relaxes at the
   kickoffer surface (per-step invariant unchanged)** — pre-M7.3.a
   the kickoffer emitted `manifest_approved_for_spawn` BEFORE the
   DAO Insert; on transient Insert failure the audit row was the
   canonical "we tried" signal. M7.3.a inverts this at the
   kickoffer-only level: `InsertIfAbsent` MUST run first because the
   event*type (`\*\_approved*_`vs`\_\_replayed_\*`) depends on the
insert-vs-replay branch, which the DAO call decides. A transient
`InsertIfAbsent` error short-circuits with NO audit row — the
operator's retry on the same idempotency key dedups cleanly. The
saga-step-level "audit-step-started precedes Execute" invariant
in [`saga.Runner.Run`](../../core/pkg/spawn/saga/saga.go) is
unchanged. Pattern: invariants that depend on "we don't know yet"
at the time of audit-emit need to relax when downstream DAO
semantics introduce a deterministic insert-vs-replay split. The
dispatcher's `appendReplayFailed` audit row at
[`dispatcher.go:316`](../../core/pkg/messenger/slack/inbound/approval/dispatcher.go)
   covers the operator-visible failure surface; the kickoffer's
   silence about the audit chain on a transient DAO error is
   internal-only, not user-visible.

4. **Empty idempotency_key rejected at TWO layers** — the DAO
   surface ([`saga.ErrEmptyIdempotencyKey`](../../core/pkg/spawn/saga/dao.go))
   AND the kickoffer fail-fast (empty `approvalToken` →
   [`spawn.ErrInvalidKickoffArgs`](../../core/pkg/spawn/spawnkickoff.go))
   both reject empty values. Defense-in-depth rationale: the
   partial UNIQUE-WHERE-NOT-NULL index allows multiple NULL rows
   (Postgres semantics), so an empty-key bypass would silently
   double-create on retry. The kickoffer guard catches the wiring
   bug at the boundary (faster signal); the DAO guard catches a
   future direct caller that bypasses the kickoffer. Pattern:
   when a value is structurally load-bearing across N layers, each
   layer enforces its own shape contract — the M7.2.c
   "defense-in-depth empty-URI gate at three layers" lesson held
   forward to the M7.3.a empty-key gate.

5. **Distinct audit event_type for the replay path, not a payload
   field on the original** — `manifest_approval_replayed_for_spawn`
   /
   `retire_approval_replayed_for_watchkeeper` are wholly new
   event types, not a `was_replay: true` field on the original
   `manifest_approved_for_spawn` /
   `retire_approved_for_watchkeeper`. Reasoning: a downstream
   consumer ("how many fresh spawns happened today?") branches on
   wire `event_type` (a closed-set string match), not on payload
   shape; commingling the two semantics under one event_type would
   force every consumer to grow a payload-field branch the
   M2b.7-derived "closed-set keys + grep-asserted vocabulary" PII
   discipline was designed to avoid. The replay payload carries TWO
   replay-only fields (`saga_id`, `previous_status`) the original
   event lacks, so the wire shape is genuinely distinct anyway.
   Pattern: when two operationally-distinct outcomes ride the same
   audit chain, model them as two event_types — the SQL `WHERE
event_type = …` filter is the consumer's first-class index.

6. **Same-id-with-fresh-key as a wiring-bug branch, not as a
   double-replay** — `InsertIfAbsent` rejects "same supplied id,
   different idempotency_key" with a wrapped error matching the
   shape of the legacy
   [`Insert`'s](../../core/pkg/spawn/saga/dao.go) duplicate-id
   error. Reasoning: the dispatcher mints `sagaID` via `uuid.New()`
   per call (not per-token), so a repeat sagaID under a fresh
   approval_token is structurally a wiring bug — surfacing it as
   the same shape the SQL `id` PRIMARY KEY would surface keeps
   "the type system encodes the transition rule the column name
   already promised" (M7.2.c lesson #1 held forward). The
   alternative — silently treating same-id-different-key as a
   replay — would mask the bug. Pattern: idempotency-aware
   surfaces still surface caller wiring bugs; "the dedup key
   matched, just route through" is wrong when the candidate id
   collides under a different key.

7. **Concurrent-replay test (16 goroutines, exactly-one-insert)
   pinned at TWO layers** — the DAO test
   ([`TestMemorySpawnSagaDAO_InsertIfAbsent_Concurrent_ExactlyOneInsert`](../../core/pkg/spawn/saga/dao_memory_test.go))
   pins the storage-side single-INSERT guarantee; the kickoffer
   test
   ([`TestSpawnKickoff_ConcurrentReplays_ExactlyOneRun`](../../core/pkg/spawn/spawnkickoff_test.go))
   pins the application-side single-Run guarantee (the audit chain
   sees exactly one `manifest_approved_for_spawn` and 15 replays;
   the recording DAO sees exactly one `Get` from the runner's
   first action). Both are race-detector-required. Pattern: a
   concurrency invariant that spans storage AND application layers
   needs a test at each layer — a single layer's pin can pass
   while a regression at the other layer (e.g. a kickoffer that
   loops on a `result.Inserted == false` branch and re-emits) goes
   undetected.

### Iter-1 patterns (codex + critic review)

1. **Stuck-saga catch-up branch (codex iter-1 Critical)** — pre-iter-1
   the kickoffer's replay branch always returned `nil` without
   running the saga; if the original Kickoff's audit-append failed
   AFTER `InsertIfAbsent` succeeded, the row was orphaned in
   `pending` forever AND the upstream `pending_approvals.Resolve`
   state-machine had already flipped the token to `approved`, so no
   future approval flow could complete the saga. The fix splits the
   replay branch on `result.Existing.Status`:
   `Status == SagaStatePending` falls through to a CATCH-UP path that
   emits the missed `manifest_approved_for_spawn` row AND calls
   `Runner.Run` for the existing saga id; non-pending status is the
   true replay (skip Run). Pattern: an idempotency-keyed insert that
   precedes a downstream side effect MUST recover the side-effect
   chain on retry when the prior call interrupted between them — a
   plain "key matched, return nil" replay branch silently strands
   work-in-progress. The catch-up uses
   [`Existing.ManifestVersionID`](../../core/pkg/spawn/saga/dao.go) +
   [`Existing.WatchkeeperID`](../../core/pkg/spawn/saga/dao.go)
   so the resumed run picks up the FIRST-call's identifiers, not the
   discarded second-call candidates.

2. **Replay payload sources ALL ids from `Existing`, not new-call
   args (codex iter-1 Major)** — pre-iter-1 the replayed-event
   payload accepted the SECOND-call's `manifestVersionID` /
   `watchkeeperID` and emitted them verbatim, even though the DAO's
   `InsertIfAbsent` discards those values on a key-collision. A
   retried approval supplying different ids would have produced a
   self-contradictory audit row (`saga_id` from the first call,
   `manifest_version_id` from the second). The fix: both
   [`manifestApprovalReplayedPayload`](../../core/pkg/spawn/spawnkickoff.go) and
   [`retireApprovalReplayedPayload`](../../core/pkg/spawn/retirekickoff.go)
   now accept ONLY `existingSagaID` / `existingManifestVersionID` /
   `existingWatchkeeperID` from the caller; the kickoffer Kickoff
   passes [`result.Existing.*`](../../core/pkg/spawn/saga/dao.go)
   into them. Pattern: when the DAO discards caller-supplied
   identifiers on a dedup, the audit-emit must source the same
   values the DAO actually persisted, never a parallel copy from
   the caller's local variables.

3. **`Saga.WatchkeeperID` field + `InsertIfAbsent`'s 4th uuid arg
   (codex iter-1 + critic K2)** — pre-iter-1 the spawn-side replay
   payload could not emit the FIRST-call's watchkeeperID because the
   saga DAO row carried no such field; downstream consumers of the
   replay event had no path back to "which bot got assigned to this
   approved Manifest" without a separate `watchkeeper_slack_app_creds`
   join (which only exists for sagas that reached the M7.1.c.a
   CreateApp step). The fix extends
   [`saga.Saga`](../../core/pkg/spawn/saga/dao.go) with a
   `WatchkeeperID` field, threads `watchkeeperID` through the
   `InsertIfAbsent` signature, and persists it on the row.
   Migration 023 grows a `watchkeeper_id uuid NULL` column to back
   the projection. The retire-side replay payload already had a
   `watchkeeper_id` key; the spawn-side now does too — closing the
   cross-kickoffer payload-shape asymmetry critic K2 flagged.

4. **Whitespace-only key normalises to empty at the DAO layer
   (codex iter-1 Minor #4)** — `strings.TrimSpace` runs before the
   `ErrEmptyIdempotencyKey` check in
   [`MemorySpawnSagaDAO.InsertIfAbsent`](../../core/pkg/spawn/saga/dao_memory.go);
   a `"  \t\n  "` sentinel cannot smuggle a bypass past the check
   even though it would have hashed distinctly under a naive equality
   test. The kickoffer-side fail-fast in
   [`SpawnKickoffer.Kickoff`](../../core/pkg/spawn/spawnkickoff.go)
   runs the same trim for an early signal at the seam. The DAO does
   NOT silently trim the key for storage — a non-whitespace key is
   persisted verbatim — so the dedup index sees the caller's exact
   bytes, only the empty-check is permissive.

5. **Typed `ErrIdempotencyIndexInconsistent` sentinel for
   DAO-internal-bug branch (critic K5)** — pre-iter-1 the
   "idempotency map points at a missing row" branch returned a bare
   `fmt.Errorf` whose only signal was a string substring; pinning it
   in tests required `strings.Contains` rather than `errors.Is`. The
   fix mints
   [`saga.ErrIdempotencyIndexInconsistent`](../../core/pkg/spawn/saga/dao.go)
   at the same level as `ErrSagaNotFound` and
   `ErrEmptyIdempotencyKey`. The branch is unreachable through the
   public surface in the in-memory DAO; a future Postgres adapter
   that surfaces a transient FK-cleanup race would propagate the
   typed sentinel cleanly through the kickoffer's wrap chain.

6. **Approval-package `SpawnKickoff` seam doc rewritten for the
   M7.3.a contract (codex iter-1 Minor #3)** — the seam interface in
   [`approval/spawn_kickoff.go`](../../core/pkg/messenger/slack/inbound/approval/spawn_kickoff.go)
   pre-iter-1 said "implementations MUST emit
   `manifest_approved_for_spawn` BEFORE inserting the saga row";
   that's now false. The doc-block rewrite spells out the M7.3.a
   invariants explicitly: implementations persist via
   `InsertIfAbsent`, emit either the approved-event OR the replayed-
   event (never both), and short-circuit Runner.Run on replay of an
   advanced saga. Future alternative implementations of the seam
   cannot accidentally satisfy the comment while violating the
   idempotency design.

7. **Migration prior-art claim corrected to point at migration 012
   (critic K1)** — the migration's doc-block originally claimed the
   partial-UNIQUE-WHERE-NOT-NULL pattern "matches the convention
   established for similarly-shaped optional UNIQUE columns in the
   M2 keep-side schema", but a grep of `deploy/migrations/` shows
   the M2 schema (`human_slack_user_id_key` in migration 012) uses a
   regular UNIQUE and leans on Postgres-native NULL-distinctness
   instead. The fix re-attributes the precedent: regular UNIQUE
   would functionally suffice on Postgres, but the partial form is
   chosen here so the dedup intent is explicit at the index
   definition. Pattern: when a migration doc claims prior art, grep
   the migrations dir to verify; a misattribution misleads future
   maintainers more than no claim at all.

**Merged**: c337814 on 2026-05-10

### Context

M7.2.c shipped the second concrete `saga.Step` of the retire family
([`MarkRetiredStep`](../../core/pkg/spawn/markretired_step.go))
along with the database column, server-side handler extension, and
keepclient method needed to persist the M7.2.b-produced

reads [`saga.SpawnContext`](../../core/pkg/spawn/saga/spawn_context.go)

- [`saga.RetireResult`](../../core/pkg/spawn/saga/retire_result.go)
  off ctx and dispatches via a thin
  [`WatchkeeperRetirer`](../../core/pkg/spawn/markretired_step.go)

  NEW
  [`keepclient.Client.UpdateWatchkeeperRetired`](../../core/pkg/keepclient/write_watchkeeper.go)
  method. The keep server's `PATCH /v1/watchkeepers/{id}/status`
  handler accepts an optional `archive_uri` body field paired with
  `status:"retired"` and persists it atomically inside the same
  scoped tx (migration
  [`022_watchkeepers_archive_uri.sql`](../../deploy/migrations/022_watchkeepers_archive_uri.sql)).
  The legacy M6.2.c synchronous `RetireWatchkeeper` Watchmaster tool
  keeps its `watchmaster_retire_watchkeeper_*` audit chain and
  continues to call
  [`keepclient.Client.UpdateWatchkeeperStatus`](../../core/pkg/keepclient/write_watchkeeper.go)
  with `status="retired"` and NO `archive_uri` — wire-compatible with
  the migration (column is `text NULL`).

### Pattern

1. **Sibling client method over signature-extending the existing
   one** — `UpdateWatchkeeperRetired(ctx, id, archiveURI)` is a NEW
   method on [`keepclient.Client`](../../core/pkg/keepclient/write_watchkeeper.go),
   not a third positional argument added to the existing
   `UpdateWatchkeeperStatus(ctx, id, status)`. Rationale: the
   pending→active call sites cannot accidentally smuggle a stray
   archive_uri (the type system rejects it); the M7.2.c saga step's
   call site reads as a single intent ("mark retired with an archive
   URI") without a sentinel non-empty check; and a future migration
   that requires archive_uri on retire flips the contract on the new
   method alone, leaving the legacy
   [`UpdateWatchkeeperStatus`](../../core/pkg/keepclient/write_watchkeeper.go)
   behaviour unchanged for the M6.2.c synchronous tool's benefit.
   Pattern: when a new optional parameter is meaningful only for a
   subset of an existing endpoint's transitions, prefer a sibling
   method to a wider signature; the type system encodes the
   transition rule the existing method's name already promised.

2. **Server-side optional column field with paired-status pre-tx
   gate** — `updateWatchkeeperStatusRequest` grew an
   `archive_uri,omitempty` field; the parser rejects the
   (`status:"active"`, `archive_uri:non-empty`) combination
   pre-tx with 400 `invalid_request` BEFORE the SELECT FOR UPDATE
   step touches the row. The body's combinatorial space shrinks to
   {`status:"active"` + no archive_uri, `status:"retired"` + optional
   archive_uri}; any other combination is structurally a wiring bug.
   Pattern: when a new optional body field is meaningful for ONE
   transition only, gate the (field, status) tuple at the parser
   layer — defending the SQL UPDATE branch alone leaves a window
   where a misuse could persist a value on the wrong path before a
   future maintainer notices the asymmetry.

3. **Two SQL UPDATE branches over a single conditional** — the
   server's active→retired switch arm contains TWO `tx.Exec` calls:
   one when `body.ArchiveURI != ""` (binds `archive_uri = $2`); one
   when it's empty (no second binding, column stays NULL). A single
   query with `archive_uri = COALESCE($2, archive_uri)` was
   rejected as it would silently keep a previous (impossible-by-
   schema-CHECK) archive_uri on a re-mark, which is harder to reason
   about than two explicit branches. Pattern: when the SQL update
   shape diverges by a presence/absence of one column, prefer two
   short queries to one conditional — duplicate text is cheaper to
   audit than a parameter that means "leave alone OR overwrite"
   depending on its value.

4. **Bare-text sentinel for clean wrap-chain composition (M7.2.b
   lesson held forward)** — [`ErrMissingArchiveURI`](../../core/pkg/spawn/markretired_step.go)
   is minted with bare text (`errors.New("missing ArchiveURI on
RetireResult")`) so the wrap chain reads
   `"spawn: mark_retired step: missing ArchiveURI on RetireResult"`
   rather than the double-prefixed
   `"spawn: mark_retired step: spawn: mark_retired step: ..."`
   shape. Reaffirms the M7.2.b iter-1 strengthening across the
   M7.2.c sentinels.

5. **Reuse sentinels across the M7.2.x retire-step family** —
   [`MarkRetiredStep`](../../core/pkg/spawn/markretired_step.go)
   reuses [`ErrMissingSpawnContext`](../../core/pkg/spawn/createapp_step.go)
   - [`ErrMissingAgentID`](../../core/pkg/spawn/createapp_step.go)
   - [`ErrMissingRetireResult`](../../core/pkg/spawn/notebookarchive_step.go)
   - [`ErrCredsNotFound`](../../core/pkg/spawn/watchkeeper_creds.go)
     without introducing per-step duplicates. The new sentinel
     [`ErrMissingArchiveURI`](../../core/pkg/spawn/markretired_step.go)
     names a STRUCTURALLY-distinct case (outbox is present but the
     `ArchiveURI` field is empty — different from
     `ErrMissingRetireResult` which covers the missing-OUTBOX case),
     so the saga-family one-sentinel-per-error-class rule is
     preserved. Pattern: when a step adds a sentinel, verify it covers
     a class that no existing M7-family sentinel already names; reuse
     first, mint only when the case is irreducible.

6. **Lean-seam source-grep AC mirrored from M7.1.d/.e/M7.2.b with
   `core/pkg/keepclient` added** — the closed-set forbidden-substring
   list for `markretired_step.go` rejects `keeperslog.`, `.Append(`,
   `core/pkg/notebook`, `core/pkg/archivestore`, `core/pkg/messenger`,
   `core/pkg/runtime`, `core/pkg/keepclient`, and
   `WatchkeeperSlackAppCredsDAO`. Adding `core/pkg/keepclient` to
   the M7.2.b list pins the M7.2.c-specific "step does not see the
   keep client" claim — that wiring belongs to the production
   [`WatchkeeperRetirer`](../../core/pkg/spawn/markretired_step.go)
   wrapper. Same enforcement shape as M7.1.d/.e/M7.2.b; the
   keepclient addition follows the principle that any direct
   substrate import in the step file is a regression worth
   surfacing as a build-time signal.

7. **Pointer-stored outbox lets the producer's write reach the
   consumer's read across one ctx (M7.2.b lesson held forward)** —
   the M7.2.b NotebookArchive step writes
   `result.ArchiveURI = uri` through the same `*RetireResult`
   pointer the M7.2.c MarkRetired step reads via
   [`saga.RetireResultFromContext`](../../core/pkg/spawn/saga/retire_result.go).
   The pointer is seeded once per saga at the Kickoffer
   (`ctx = saga.WithRetireResult(ctx, &saga.RetireResult{})`); within
   one saga, the [`saga.Runner`](../../core/pkg/spawn/saga/saga.go)
   dispatches the two steps sequentially (NotebookArchive →
   MarkRetired), so the producer's write is visible on the
   consumer's read without any synchronisation primitive. Across
   distinct sagas, each ctx carries its own pointer — no shared
   state. The M7.2.c happy-path test pins the URI pass-through
   (`call.archiveURI == result.ArchiveURI`); the missing-URI case
   pins the consumer-side guard. Pattern: a context-pointer outbox
   that participates in cross-step writes works because step
   ordering is deterministic — adding an out-of-order step or a
   parallel step-fan-out family in the future would invalidate this
   pattern and require a sync primitive.

8. **Defense-in-depth empty-URI gate at three layers** — the saga
   step rejects empty `result.ArchiveURI` via
   [`ErrMissingArchiveURI`](../../core/pkg/spawn/markretired_step.go);
   the keepclient method rejects empty `archiveURI` via
   `ErrInvalidRequest`; the keep server's parser rejects
   whitespace-only `archive_uri` via 400 `invalid_request`. None of
   the three is sufficient alone — the saga step covers the
   in-process wiring bug (M7.2.b producer published nothing); the
   client guard covers a future direct caller bypass; the server
   guard covers an arbitrary HTTP client. M7.2.b's iter-1
   reviewers caught a similar layering gap on the URI-shape side
   (parse-via-`net/url` only at the producer); M7.2.c carries the
   defense-in-depth lesson into the persistence layer. Pattern:
   when a value is structurally load-bearing across N layers,
   each layer enforces its own shape contract — silently trusting
   "an upstream layer already validated" is the regression vector.

9. **Wire-shape compatibility for legacy callers via `omitempty`** —
   the keepclient's `updateWatchkeeperStatusRequest` struct grew an
   optional `ArchiveURI` field tagged `json:"archive_uri,omitempty"`;
   the existing pending→active and (M6.2.c) active→retired-without-
   archive call sites continue to send their pre-M7.2.c wire shape
   verbatim — `omitempty` ensures the new field never materialises
   in their bodies. The server's
   [`DisallowUnknownFields`](../../core/internal/keep/server/handlers_write.go)
   decoder accepts both shapes because the field is now part of the
   struct's known set. Pattern: when extending a wire schema with an
   optional field, both ends MUST learn the field but the legacy
   senders MUST keep their pre-extension wire shape — anything else
   forces a coordinated rollout. Pinned by
   [`TestClient_UpdateWatchkeeperStatus_OmitsEmptyArchiveURI`](../../core/pkg/keepclient/write_watchkeeper_test.go).

10. **Deferred wiring of M6.2.c into the kickoffer is the next
    milestone** — the roadmap line "wires M6.2.c retire tool through
    the saga" is interpreted as "ships the seams + DB column the

    [`RetireWatchkeeper`](../../core/pkg/spawn/retire_watchkeeper.go)

### Iter-2 patterns (codex + critic review)

1. **Custom JSON-decode helper for absent/explicit-null/string
   trichotomy (codex C1, Major)** — the iter-1 server upgrade from
   plain `string` to `*string` closed the present-but-empty bypass
   but left a structurally distinct state on the wire: explicit
   JSON null. Go's `encoding/json` decodes `"archive_uri": null` to
   a nil `*string`, which the iter-1 `body.ArchiveURI != nil` guard
   treated as ABSENT — so `{"status":"active","archive_uri":null}`
   skipped the active-with-archive guard and
   `{"status":"retired","archive_uri":null}` silently fell into the
   legacy no-archive branch. Iter-2 fix: an unexported
   `optionalArchiveURI{Present, Null, Value}` value type with
   `UnmarshalJSON` on the pointer receiver records that the key
   was in the body (`Present`) AND distinguishes `null` from a
   string literal (`Null`). Parser logic becomes
   `Present && Null → reject; Present && !Null → validate; !Present
→ legacy no-archive path`. Pattern: when a HTTP body needs to
   distinguish "key omitted" from "key present with null value",
   a `*T` field is insufficient — Go's stdlib folds both into nil.
   Mint a small typed helper with explicit Present/Null/Value
   bits; the cost is one type and one `UnmarshalJSON`, the win is
   a closed-set state machine that the parser can pattern-match.

2. **RFC 3986 absolute-URI gate at every load-bearing layer (codex
   C2, Major)** — the wire contract documents `archive_uri` as an
   RFC 3986 URI with a non-empty scheme, but iter-1 only enforced
   non-blank. `net/url.Parse` tolerates path-only inputs like
   `"garbage"` or `"../../tmp"` (it returns a `*URL` whose
   `Scheme` is empty and `Path` is the input verbatim), so a
   direct PATCH caller could land arbitrary garbage on the
   column. Iter-2 fix: `url.Parse(s); !u.IsAbs()` rejects every
   schemeless input across the saga step (already pre-validated
   via M7.2.b's `ErrInvalidArchiveURI`),
   [`UpdateWatchkeeperRetired`](../../core/pkg/keepclient/write_watchkeeper.go)
   (the keepclient seam closest to a non-saga caller), and the
   server parser (the HTTP boundary). Pattern: when a wire
   contract names "RFC 3986 with non-empty scheme", code MUST
   enforce `u.IsAbs()` not just `u != nil` or `err == nil` —
   `net/url.Parse` returns valid `*URL` values for inputs that
   fail the RFC's `absolute-URI` production. Pin schemes via a
   table-driven positive test (`file://`, `s3://`, `gs://`,
   synthetic `test://`) so the gate accepts arbitrary schemes
   rather than a hardcoded allowlist that would reject a future
   archivestore backend.

3. **Explicit `archive_uri = NULL` stamp + storage-layer CHECK
   constraint (codex C3, Minor)** — the iter-1 no-archive branch
   omitted the `archive_uri` column from the UPDATE, which would
   preserve any stale non-NULL value on a row that legitimately
   cannot reach this branch (the state machine forbids
   retired→active so re-mark is impossible — but only via the
   handler; a direct DB write could violate the invariant). Iter-2
   fix: (a) the no-archive UPDATE explicitly writes
   `SET ..., archive_uri = NULL`, and (b) migration 022 adds
   `CHECK (archive_uri IS NULL OR status = 'retired')` so the
   invariant holds at the storage layer regardless of how the row
   got mutated. Pattern: when a column is conditionally meaningful
   (only valid when a sibling column has a particular value),
   enforce the invariant at the storage layer with a CHECK
   constraint AND have the handler explicitly stamp the conditional
   column rather than relying on "we never reach this branch with
   a stale value" — the CHECK protects against direct DB writes,
   the explicit stamp is belt-and-suspenders for the handler.

4. **Symmetric pointer-typed body field client-vs-server (critic
   M3, Major)** — `keepclient.updateWatchkeeperStatusRequest.ArchiveURI`
   is `*string` symmetric with the server's `optionalArchiveURI`
   semantics (omitted-by-default; only the
   [`UpdateWatchkeeperRetired`](../../core/pkg/keepclient/write_watchkeeper.go)
   method materialises a non-nil pointer with a pre-validated
   URI). Iter-1 had the field as plain `string` + `omitempty`,
   which folded "absent" and "intentionally empty" into the same
   encoded form — a future caller minting
   `body{Status: "retired", ArchiveURI: someMaybeEmpty}` would
   serialize the empty-string case to absent and silently take
   the no-archive server branch. Pattern: when one end of a wire
   contract has trichotomy semantics (absent / null / string),
   the other end's request struct MUST use a pointer-typed field
   even if its only production caller pre-validates non-empty —
   asymmetry is a future-maintainer regression vector.

5. **Doc-vs-reality review of the diff summary (critic M2,
   Major)** — the original M7.2.c diff summary said
   "production wiring composes a Retirer wrapping
   keepclient.UpdateWatchkeeperRetired", but the
   [`retire_wiring/wiring.go`](../../core/internal/keep/retire_wiring/wiring.go)
   +11 LOC are docstring + the `Steps` field plumbing already
   shipped in M7.2.a — there is NO `WatchkeeperRetirer`
   implementation, NO `MarkRetiredStep` wired into Steps, NO
   smoke test. The lesson narrative correctly named this as
   deferred (pattern #10) but the diff summary misled the codex
   - critic reviewers' first reads. Pattern: when summarising a
     PR for parallel reviewers, distinguish "lesson narrative
     promises" (forward-looking; deferred to a follow-up) from
     "PR ships" (the actual diff); a misleading summary costs
     review time on both sides and weakens the iter-1 review's
     accuracy. The fix here was to clarify the deferred-wrapper
     status in the iter-2 pattern set + reaffirm pattern #10's
     deferred-integration narrative.

## 2026-05-09 — M7.2.b: NotebookArchive saga step + RetireResult outbox value type

**PR**: #120
**Merged**: f6aa80c on 2026-05-09

### Context

M7.2.b added the first concrete `saga.Step` of the retire family
(`NotebookArchiveStep`) bridging
[`notebook.ArchiveOnRetire`](../../core/pkg/notebook/archive_on_retire.go)
into the M7.2.a retire kickoffer's step slot. The step dispatches
through a thin `NotebookArchiver` seam keyed off
[`saga.SpawnContext.AgentID`](../../core/pkg/spawn/saga/spawn_context.go)
(the watchkeeper-being-retired) and publishes the resulting archive
URI onto a NEW saga-level value type,
[`saga.RetireResult`](../../core/pkg/spawn/saga/retire_result.go), so
the future M7.2.c MarkRetired step can persist the URI onto the
watchkeeper row. The retire kickoffer was extended with a
single line — `ctx = saga.WithRetireResult(ctx, &saga.RetireResult{})`
— directly after the existing `WithSpawnContext` seed. The step
itself is lean — no `notebook` import, no `archivestore` import, no
DAO, no audit emit — matching the M7.1.d / M7.1.e step shape exactly.

### Pattern

1. **Saga-level outbox value type for inter-step writes** —
   The roadmap's "SpawnContext-equivalent" wording for the
   archive_uri handoff naturally maps to a NEW saga-level value
   type ([`saga.RetireResult`](../../core/pkg/spawn/saga/retire_result.go))
   distinct from [`saga.SpawnContext`](../../core/pkg/spawn/saga/spawn_context.go).
   Rationale: SpawnContext is read-only input seeded once at kickoff;
   RetireResult is a write-then-read outbox steps mutate. Mixing
   them on a single struct would muddy "input bag vs output channel"
   semantics and force the spawn family to also carry retire-only
   fields. The two types coexist on the same ctx — retire kickoffer
   seeds BOTH; spawn kickoffer seeds ONLY SpawnContext (no spawn
   step needs an in-process outbox today — the cross-step values
   the spawn family produces, e.g. M7.1.c.a's `slack_app_creds`
   row, live on a DAO with their own durability story instead).
   Pattern: ctx-pointer outboxes are for in-process step-to-step
   handoff that does NOT need to survive a saga restart; values
   that DO need to survive belong on a DAO. Reusable for any
   future saga family that needs cross-step in-process outputs.

2. **Pointer-stored vs value-stored on context** — `context.WithValue`
   stores values by interface-copy: a non-pointer struct snapshotted
   on every `.Value(...)` retrieval would make Step-A's writes
   invisible to Step-B. To make inter-step writes observable,
   [`WithRetireResult`](../../core/pkg/spawn/saga/retire_result.go)
   stores a `*RetireResult`; every retrieval returns the SAME
   pointer; step authors mutate fields through it. Within one saga
   the `saga.Runner` runs steps sequentially → no concurrent writes;
   across sagas each ctx carries its OWN pointer → no shared state.
   Pattern: a context-stored value type that participates in
   inter-step writes MUST be pointer-stored, not value-stored.
   `SpawnContext` is correctly value-stored because it is read-only
   after kickoff seeding; the two patterns are deliberately distinct.

3. **Two-branch guard for the outbox-presence assertion** —
   `RetireResultFromContext` follows the idiomatic `(value, ok)`
   shape: `ok=true` reports KEY presence, not pointer non-nilness.
   A caller that explicitly stamps `nil` via `WithRetireResult(ctx,
nil)` produces `(nil, true)`. The step's guard checks BOTH
   branches (`!ok || result == nil`) and returns
   [`ErrMissingRetireResult`](../../core/pkg/spawn/notebookarchive_step.go)
   on either; without the second branch a future maintainer's
   `if !ok` simplification would let a nil-pointer Execute reach
   `result.ArchiveURI = uri` and panic. Pattern: any
   pointer-stored context-value's caller-side guard is two-branch.

4. **Step collapses substrate's partial-success tuple into clean
   success/fail (UNCONDITIONALLY on err shape)** — The substrate
   [`notebook.ArchiveOnRetire`](../../core/pkg/notebook/archive_on_retire.go)
   pairs a non-empty `uri` with a non-nil `err` on multiple
   post-Put paths (the documented LogAppend-failure path AND the
   theoretical audit-marshal path; future substrate edits could
   add more). The saga step's `NotebookArchiver` seam contract
   therefore pushes UNCONDITIONAL tuple-collapse into the
   production wrapper: wrapper returns `(uri, nil)` on FULL
   success only, returns `("", err)` on ANY failure regardless of
   which sub-path produced it. The wrapper does NOT branch on err
   shape to decide whether to forward the URI. Iter-1 critic
   finding (Cr2): the prior wording named only the LogAppend
   path and risked a wrapper author missing the audit-marshal
   path; the unconditional-collapse contract is robust to future
   substrate sub-paths. Retry-on-audit-failure is the wrapper's
   job, not the step's. Pattern: when a substrate exposes a
   partial-success tuple, the seam contract collapses it
   unconditionally on `err != nil`; do not enumerate the substrate's
   sub-paths in the seam doc-block.

5. **Empty-URI-on-success is a fail-closed wiring bug** — A
   production wrapper that returns `("", nil)` is a programmer
   error: the substrate's contract is that a successful archive
   yields a non-empty URI. The step rejects this combination with
   [`ErrEmptyArchiveURI`](../../core/pkg/spawn/notebookarchive_step.go)
   BEFORE publishing to the outbox. Without this guard the M7.2.c
   MarkRetired step would persist an empty `archive_uri` onto the
   watchkeeper row, severing the retire-trail with no audit signal.
   Pattern: when a seam's success path requires a non-empty
   primary value, validate it explicitly at the consumer — do not
   trust the upstream to honour its own contract; cheap
   fail-closed assertions catch wiring drift the type system can't.

6. **Distinct outbox per Kickoff isolates concurrent retire sagas**
   — The kickoffer seeds a FRESH `&saga.RetireResult{}` on every
   `Kickoff` call (NOT a construction-time field). Two concurrent
   retire sagas observe distinct pointers; one's write cannot leak
   into the other's outbox. Pinned by
   `TestRetireKickoff_SeedsDistinctRetireResultPerKickoff` in
   `retirekickoff_test.go`, which captures the pointer from two
   sequential Kickoffs and asserts `first != second`. Pattern:
   per-call mutable state on `ctx` MUST be allocated per-call —
   constructing it once on the receiver is the easy concurrency bug.

7. **Lean-seam source-grep AC mirrored from M7.1.d/.e with
   archivestore added** — The closed-set forbidden-substring list
   for `notebookarchive_step.go` rejects `keeperslog.`, `.Append(`,
   `core/pkg/notebook`, `core/pkg/archivestore`,
   `core/pkg/messenger`, `core/pkg/runtime`, `core/pkg/keepclient`,
   and `WatchkeeperSlackAppCredsDAO`. Adding `core/pkg/archivestore`
   - `core/pkg/keepclient` to the M7.1.d list pins the M7.2.b
     specific "step does not see the backend store nor the audit
     client" claim — both belong to the production
     [`NotebookArchiver`](../../core/pkg/spawn/notebookarchive_step.go)
     wrapper. Same enforcement shape as M7.1.d/.e; longer forbidden
     list because the retire substrate spans two backends (notebook
   - archivestore) plus the audit emit (keepclient) instead of
     one.

8. **Seam-side panic over consumer-side double-check for "must be
   non-nil" pointers** (iter-1 codex finding C3). The first cut of
   [`saga.WithRetireResult`](../../core/pkg/spawn/saga/retire_result.go)
   accepted nil and forced every consumer (today: the step's
   `Execute`) to repeat the check. Iter-1 strengthening: panic at
   the seam on nil. Tradeoff: the seam-side panic surfaces the
   wiring bug ONCE, at the misuse site (kickoffer's seed call),
   rather than letting it propagate past every step author who
   may forget the second guard branch. The step's `Execute` now
   has a single `!ok` branch; `RetireResultFromContext` returns
   a guaranteed non-nil pointer when `ok == true`. Pattern: when
   a context-pointer's "MUST be non-nil" invariant is documented,

7. **Pin ctx-propagation through a recording fake, not a discarding fake** — Iter-1 codex review caught that the original test fake's `ProvisionNotebook(_ context.Context, ...)` discarded the incoming ctx, so a future regression to `s.provisioner.ProvisionNotebook(context.Background(), ...)` (or `context.WithoutCancel(ctx)`) would still pass every other assertion. The fix records the ctx on every call and pins propagation two ways: (a) a `context.WithValue` sentinel must round-trip through to the seam (verifies value-chain integrity); (b) cancelling the caller's parent ctx AFTER the synchronous call returns must surface as `context.Canceled` on the recorded ctx (verifies cancellation-chain integrity, independent of the seam's return value). Pattern: when a step's contract is "forward the caller's ctx verbatim", the test fake MUST record the ctx — discarding it makes the contract decorative.

8. **Source-grep AC must enforce the _full_ lean-seam claim, not just audit-emit** — Iter-1 codex review caught that the original AC checked only `keeperslog.` + `.Append(`, leaving the lesson's broader "no `notebook`/DAO/messenger imports" claim unenforced. The fix extends the closed-set forbidden-substring list to also reject `core/pkg/notebook`, `core/pkg/messenger`, and `WatchkeeperSlackAppCredsDAO`. Pattern: when a lesson narrative claims multiple invariants (audit-free + substrate-free + DAO-free + platform-free), the source-grep AC must enumerate ALL of them — partial coverage lets a future maintainer add a forbidden import while still passing the test, and the lesson silently rots.

## 2026-05-09 — M7.1.c.c: BotProfile saga step + register step list in spawn kickoff wiring

**PR**: #116
**Merged**: aece000 on 2026-05-09

### Context

M7.1.c.c added the third concrete saga.Step (`BotProfileStep`) bridging `messenger.Adapter.SetBotProfile` (Slack `users.profile.set`) into the spawn flow, plus extended `spawn.SpawnKickoffer` to register a step list at construction time and seed `saga.SpawnContext` on the per-call ctx. The kickoffer's `Kickoff` signature gained `watchkeeperID uuid.UUID` and `claim saga.SpawnClaim` parameters with fail-fast validation that rejects `uuid.Nil` ids BEFORE any audit/state side effect; the dispatcher mints `uuid.New()` per call and resolves the claim per-call via the new `approval.WithSpawnClaimResolver` option (a `func(ctx, manifestVersionID) (saga.SpawnClaim, error)` so multi-tenant deployments derive the right tenant claim from the row instead of pinning a process-global value). The `approval_wiring` helper exposes `SpawnClaimResolver` + `Steps` deps so the future Slack-bot binary plugs concrete steps without re-plumbing the dispatcher.

### Pattern

1. **Step list registered at constructor; per-saga values flow through SpawnContext** — `SpawnKickoffer` accepts `Steps []saga.Step` in its deps and stamps a defensive copy at construction. On every `Kickoff` it seeds `saga.SpawnContext{ManifestVersionID, AgentID: watchkeeperID, Claim}` onto ctx BEFORE `runner.Run`, so registered steps read per-saga values via `saga.SpawnContextFromContext(ctx)` without re-plumbing each step constructor. Reusable for any saga that wires per-call values to multiple steps without forcing each step's API surface to grow.

2. **Per-call resolver beats process-global static for tenant-scoped values** — Iter-1 codex review caught a multi-tenant correctness bug: a static `WithWatchmasterClaim(saga.SpawnClaim)` option pinned one claim on the dispatcher, so a single bot serving multiple orgs would replay every approval under the same authority tuple. The fix replaces the static option with `WithSpawnClaimResolver(func(ctx, manifestVersionID) (saga.SpawnClaim, error))`. The resolver runs INSIDE the dispatch path, derives the tenant claim from the per-call manifestVersionID, and returns errors that surface as `approval_replay_failed`. Pattern: when a value depends on per-request state (tenant id, request scope), wire it as a resolver func, NOT as a constructor-time static value — even when "every deployment ships one tenant" is true today.

3. **Optional Setter seam decouples step from messenger client construction** — `BotProfileSetter` interface (`SetBotProfile(ctx, watchkeeperID, profile)`) hides the per-watchkeeper bot-token decryption + per-call client clone. The step does not import `messenger/slack` directly; the production wiring layer composes a setter that decrypts the M7.1.c.b.b token, builds a per-call client, and forwards. Same pattern as `SlackAppInstaller` from M7.1.c.b.b — keep the saga step lean, push platform-specific wiring to the production composition layer.

4. **Empty profile is a no-op, not a panic** — `messenger.BotProfile{}` (entirely empty) is a documented adapter-level no-op; the step accepts it without panic at construction. This matches the real-world need where a deployment without a personality profile still completes the saga rather than fail-closing.

5. **Backward-compatible step-list default** — `Steps` is an _optional_ `[]saga.Step` on `SpawnKickoffDeps`; nil keeps the M7.1.b zero-step behaviour (single `manifest_approved_for_spawn` + single `saga_completed`). Pattern: when a long-lived seam grows a new optional field, ship it with a documented zero-value contract so existing callers stay green without a coordinated rollout.

6. **Defensive deep copy on reference-typed step config** — Iter-1 codex review caught that `BotProfileStep` stored `messenger.BotProfile` by shallow copy, leaving `Metadata` (map) and `AvatarPNG` (slice) aliased to caller-owned memory. Under concurrent saga runs OR post-construction caller mutations, this would race AND change what gets sent to Slack without going back through construction. The fix added a `cloneBotProfile` helper called both in `NewBotProfileStep` (snapshot) and in `Execute` (fresh per-call copy to the Setter). Pattern: when a step's "immutable configuration" claim covers a value type with internal map / slice fields, defensively deep-copy in the constructor AND clone again on dispatch — mirrors `NewCreateAppStep`'s `Scopes` defensive copy.

7. **Fail-fast validation precedes audit-emit + state-write** — Iter-1 codex review caught that `SpawnKickoffer.Kickoff` accepted `uuid.Nil` for `sagaID` / `manifestVersionID` / `watchkeeperID` and only failed downstream after emitting `manifest_approved_for_spawn` + inserting the saga row. The fix added an explicit `ErrInvalidKickoffArgs` sentinel and validation guard BEFORE the audit Append. Pattern: a malformed input that's a programmer / wiring bug (not a runtime fault) should leave NO audit row and NO persisted state — the audit row should always reflect "we tried and failed downstream", never "we accepted garbage and persisted it".

## 2026-05-09 — M7.1.c.b.b: OAuthInstall saga step + encrypted bot-token storage using secrets crypto primitive

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

M7.1.c.b.b added the second concrete saga.Step (`OAuthInstallStep`) in the spawn flow. It bridges `messenger.Client.InstallApp` (M4.2.d.2 OAuth `oauth.v2.access` exchange) to encrypted token storage in `watchkeeper.slack_app_creds` via the M7.1.c.b.a `secrets.AESGCMEncrypter`. The step pulls M7.1.c.a's stored `client_id`/`client_secret` via the resolver, receives `InstallTokens` via the sink, encrypts each token, and persists ciphertexts.

### Pattern

1. **Empty-plaintext short-circuit avoids encrypted-empty-string regression** — `encryptIfNonEmpty(ctx, plaintext)` returns `nil, nil` when `len(plaintext) == 0`. Encrypting an empty plaintext via AES-GCM produces a 28-byte ciphertext (12 nonce + 16 tag); storing that as "no refresh_token" would silently disagree with downstream `len() == 0` checks. Reusable for any optional encrypted field where "absent" must round-trip as nil/empty, not as a fixed-size ciphertext. Pinned by AC test asserting stored CT is `nil` when input is empty.

2. **Per-call resolver+sink wrap of a long-lived Client** — `messenger.Client.InstallApp` accepts `WithInstallTokenSink` / `WithInstallParamsResolver` per call (clones config). The saga step builds fresh closures per Execute that capture the SpawnContext-derived `watchkeeperID`, NOT the client-shared install state. Pattern: when an SDK Client takes per-call options, prefer per-Execute closures over construction-time wiring so step concurrency is safe and the closure scope matches the per-saga state.

3. **Highest-risk leak vector for redaction tests is the post-Encrypt-pre-DAO path** — when an Encrypt-then-Persist pipeline fails at Persist, all three plaintext tokens are still in scope on the goroutine; a future maintainer wrapping `tokens.AccessToken` into the DAO error message would slip past the existing redaction harness undetected. Pattern: redaction-grep tests MUST cover EVERY failure path, especially the post-encrypt-pre-store one, not just the obvious "encrypter error" or "missing input" paths. Iter-1 reviewer caught this gap; the iter-1 fix added a 5th case to the redaction table.

4. **Reuse sentinel errors across saga steps** — `OAuthInstallStep` reuses `ErrCredsNotFound` from M7.1.c.a (raised when the M7.1.c.a creds row is missing). Don't duplicate sentinels per step; one sentinel per error class across the M7 saga family. Reusable for M7.1.c.c BotProfileStep + M7.1.d Notebook step + M7.1.e runtime launch step.
