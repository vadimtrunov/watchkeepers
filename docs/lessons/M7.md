## 2026-05-09 — M7.1.a: Saga skeleton: spawn/ package, state machine, saga-state DAO + migration

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

M7.1.a introduced `core/pkg/spawn/saga` with a step-based orchestrator and state-persistence DAO, plus the SQL migration for saga-state storage. The package defines a generic `Step` interface so M7.1.b–.e (Slack interaction, App provisioning, Notebook provision, runtime launch) can plug concrete steps without modifying saga core. Migration `019_spawn_sagas.sql` ships alongside the in-memory DAO so M7.1.b can wire a Postgres adapter without migration churn.

### Pattern

**Step-interface seam for multi-step external flows**: A saga's `Run` loop consumes `[]Step` where each step is `{Name() string, Execute(ctx) error}`. The orchestrator owns state persistence (UpdateStep BEFORE Execute) and audit emit (started BEFORE Execute, completed/failed BEFORE return). Future PRs add concrete steps as struct types — no modification to saga core. Reusable for retire saga (M7.2) and any multi-step flow.

**Audit-emit-before-return ordering generalizes M6.3.e to saga steps**: Every state transition's audit row emits BEFORE the boundary call (Execute / return). On `Step.Execute` error: `MarkFailed` → emit `saga_failed` → return error. Inverting any of the three creates a silent-data-loss defect. Pattern extends M6.3.e's "stream emit before user handler" to the saga domain.

**PII discipline closed-set payload (M2b.7 + M6.3.f wire-shape)**: Saga audit payloads are restricted to three keys (`saga_id`, `step_name`, `last_error_class`) and a regression-test grep-asserts the JSON keys. Step authors cannot leak raw step params or `error.Error()` strings into audit — payload constructors live in saga core, not Step impls.

**Migration-shipped-with-skeleton**: Per M6.3.b "ship in-memory DAO with consumer", M7.1.a ships the migration alongside the in-memory DAO so M7.1.b wires Postgres without migration churn.

## 2026-05-09 — M7.1.b: Slack interaction handler: approval action → Keeper's Log event → saga kickoff

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

M7.1.b wired Slack-approval-action → Keeper's Log audit event → spawn-saga kickoff using the M7.1.a step-interface seam. New seam `SpawnKickoff` interface in approval-package + concrete `SpawnKickoffer` impl in spawn-package + production-wiring helper. The saga runs with empty `[]Step{}` in this PR; concrete steps land in M7.1.c–.e.

### Pattern

**Per-tool branch on PendingApproval.ToolName**: When an approval flow needs to dispatch differently per tool, the cleanest seam is to branch the dispatcher on the closed-set `pending.ToolName` and route `propose_spawn` to a new `SpawnKickoff` seam while leaving the existing `Replayer` path for other tools. Mirrors the M6.3.b two-seam pattern (Replayer + Renderer); keeps the new dispatch path independently testable.

**manifest_version_id flows through ParamsJSON, not the PendingApproval struct**: When the upstream tool already serialised its params snapshot into `pending.ParamsJSON`, downstream branches that need a specific field (here `manifest_version_id`) parse it from the JSON envelope rather than expanding the `PendingApproval` struct. Path B (parse + typed sentinel `ErrMissingManifestVersionID`) keeps the diff inside the M7.1.b file set; path A (extend `PendingApproval` + DAO surface) was rejected because it would force changes to the M6.3.b storage contract for a transient need.

**Audit-emit-before-state-write generalises across the seam**: `SpawnKickoffer.Kickoff` emits `manifest_approved_for_spawn` BEFORE `Insert` BEFORE `Run([]Step{})`. The audit row is canonical even when state-persistence fails (M6.3.e + M7.1.a pattern). Test `TestSpawnKickoff_InsertError_AuditAlreadyEmitted` pins this — Insert error → audit row IS persisted, saga DAO has no row.

**Local seam re-declaration over cross-file unexported interface**: When an impl file needs an interface type already declared in a sibling file in the same package, prefer re-declaring locally (`kickoffLogAppender` in spawnkickoff.go alongside `keepersLogAppender` in slack*app.go) with a compile-time drift-pin assertion `var * kickoffLogAppender = keepersLogAppender(nil)`. Reader of one file sees the contract; build catches signature drift.

**Exact-equality vocabulary pin over HasPrefix tautology**: When guarding event-type vocabulary against future renames, use `if eventType != "manifest_approved_for_spawn"` rather than `!HasPrefix(et, "manifest_approved_for_spawn")` (which is tautological for the literal string). Keep `HasPrefix` checks against COLLIDING prefixes (`llm_turn_cost`, `saga_`) as defense-in-depth.

## 2026-05-09 — M7.1.c.a: CreateApp saga step + watchkeeper secrets column + migration

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

M7.1.c.a wired the first concrete saga.Step impl (CreateApp) bridging the M6.1.b SlackAppRPC privileged RPC to a new watchkeeper.slack_app_creds Postgres table via the existing M4.2.d sink-adapter contract. Introduced saga.SpawnContext value type for cross-step parameter passing and stabilized manifest_version_id + agent_id access across future steps.

### Pattern

**Sink-to-DAO bridge owns watchkeeperID, not appID**: A saga step that bridges an OUT-OF-BAND credential delivery (M4.2.d sink) to a persistence DAO must close over the saga's primary key (watchkeeperID from SpawnContext), NOT the credential's own assigned id (creds.AppID). Re-create scenarios change AppID; watchkeeper id is stable across the saga's lifetime. The DAO column layout reflects this: watchkeeper_id is PK, app_id is a column.

**SpawnContext as saga-level cross-step seam**: Future M7.1.x steps need read access to manifest_version_id, agent_id, and the dispatcher's Claim. Define a typed value (saga.SpawnContext) + accessors (WithSpawnContext / SpawnContextFromContext) in the saga package — NOT in the spawn package — so step impls have a stable contract that does not couple them to step-specific constructors.

**Source-grep AC pinning for negative AC ("does NOT call X")**: When an AC asserts that a unit must NOT call some API, the test that pins it cannot be a runtime mock — even an injected fake passes vacuously if the unit reaches the API via a different path or is later refactored. The robust pin is os.ReadFile on the unit's source + strings.Contains outside //-prefixed lines. Mutation-tested (adding the forbidden call must cause the test to fail).

**Migration ships with deferred encryption-at-rest comment**: Per the M6.3.b "ship in-memory DAO with consumer" rhythm, M7.1.c.a's migration uses text columns for credentials with an explicit -- TODO: encryption-at-rest deferral comment. The Phase-2 secrets pass will rotate to encrypted bytea. Keeps the M7.1.c.a PR scope tight.

### Anti-pattern

Hollow runtime-mock AC pins. The iter-1 reviewer caught a keeperslog.New(keep) that was constructed and immediately discarded (\_ = ...) — the recordingKeepClient could never observe the call because the writer was never injected. The fix replaced it with a source-grep mutation-tested assertion. Lesson: any AC pin that depends on observation through an injected dependency must verify the dependency reaches the unit-under-test; otherwise prefer source-grep.

## 2026-05-09 — M7.2.a: Retire saga kickoff seam + RetireKickoffer + production-wiring helper

**PR**: #119
**Merged**: f98a96c on 2026-05-09

### Context

M7.2.a opened the M7.2 retire-saga family by shipping the
`RetireKickoffer` seam — the symmetric counterpart to the M7.1.b
`SpawnKickoffer` for the retire flow. The kickoffer composes the
`retire_approved_for_watchkeeper` audit event, persists a saga row via
the M7.1.a [`saga.SpawnSagaDAO.Insert`](../../core/pkg/spawn/saga/dao.go),
seeds [`saga.SpawnContext`](../../core/pkg/spawn/saga/spawn_context.go)
on the per-call `ctx`, and runs an empty step list through
[`saga.Runner.Run`](../../core/pkg/spawn/saga/saga.go) (zero-step
behaviour mirrors M7.1.b). The construction-time `Steps` slot is the
same M7.1.c.c-introduced shape; M7.2.b NotebookArchive + M7.2.c
MarkRetired land in their own milestones and populate the slot via the
production-wiring helper. The M6.2.c synchronous `RetireWatchkeeper`
Watchmaster tool is **not** modified by this PR — its existing
`watchmaster_retire_watchkeeper_*` audit chain remains the production
trigger surface; M7.2.c will route it through the kickoffer.

### Pattern

1. **Saga DAO is generic across spawn vs retire — event_type
   distinguishes the semantics, not the DAO surface.** The M7.1.a
   `saga.SpawnSagaDAO` (with its `019_spawn_sagas.sql` migration)
   was authored as the spawn-saga state-machine sink, but the row
   shape (id, manifest_version_id, status, current_step, last_error,
   timestamps) carries no spawn-specific semantics. M7.2.a reuses it
   as-is — no new DAO, no new migration, no schema fork. The
   downstream `keepers_log` consumer disambiguates spawn vs retire
   sagas via the audit `event_type` column
   (`retire_approved_for_watchkeeper` vs
   `manifest_approved_for_spawn`), not via a per-flow DAO. Reusable
   for any future kickoffer family that walks a generic saga shape.

2. **One sentinel per error class across the kickoffer family
   (M7.1.c.c lesson held forward).** The retire kickoffer reuses
   `ErrInvalidKickoffArgs` from `spawnkickoff.go` for `uuid.Nil`
   sagaID / manifestVersionID / watchkeeperID — no
   `ErrInvalidRetireKickoffArgs` shadow sentinel. The wrap message
   (`"%w: empty sagaID"`) is identical; tests `errors.Is` against the
   shared sentinel. Pattern: when a sentinel describes a closed-set
   programmer-error class (malformed kickoff args, missing
   SpawnContext, missing CredsRow), keep ONE per class regardless of
   how many sibling kickoffers / steps exhibit the class. The M7.1.c
   family already encoded this for steps; M7.2.a confirms it scales
   to kickoffers.

3. **Local seam re-declaration mirrored across kickoffer family with
   bidirectional drift-pin (M7.1.b lesson held forward, iter-1
   strengthening).** The retire kickoffer declares its own
   `retireLogAppender` 1-method interface alongside spawn kickoffer's
   `kickoffLogAppender` (which itself mirrors slack*app.go's
   `keepersLogAppender`). Drift-pinning is **bidirectional** within
   the spawn / retire pair:
   `var * retireLogAppender = kickoffLogAppender(nil)`catches a
NARROWING of`kickoffLogAppender`that would drop a method retire
still requires;`var \_ kickoffLogAppender = retireLogAppender(nil)`catches an EXPANSION of`retireLogAppender`that would add a method
spawn does not yet supply. The third hop —`kickoffLogAppender ↔ keepersLogAppender`(slack_app.go) — is
pinned in spawnkickoff.go's existing single-direction assertion
and only catches drift when both files are built in the same go
compile unit (which they are, since both live in package`spawn`).
   A reader of either kickoffer file sees the contract; the build
   catches any drift across the family. Iter-1 lesson: when shipping
   the second sibling of an interface family, pin the seam
   bidirectionally — single-direction assertions silently miss
   asymmetric drift.

4. **Distinct event-type prefix for the retire-saga family +
   defense-in-depth vocabulary tests.** The kickoff event constant
   `EventTypeRetireApprovedForWatchkeeper = "retire_approved_for_watchkeeper"`
   uses a `retire_` prefix that is collision-free with
   `manifest_*` (M7.1.b spawn family), `saga_*` (M7.1.a runner),
   `llm_turn_cost_*` (M6.3.e), `notebook_*` (M2b), and the M6.2.c
   `watchmaster_retire_watchkeeper_*` synchronous-retire family.
   Two regression tests pin the boundary in BOTH directions:
   `TestEventTypeRetireApprovedForWatchkeeper_NoPrefixCollision`
   asserts the constant value's exact text + non-collision against
   the five sibling families; `TestRetireKickoff_DoesNotEmitSpawnFamilyEvents`
   asserts NO `manifest_*` event surfaces from a retire run. The
   second test is defense-in-depth: a future maintainer who edits
   the constant to alias onto `manifest_approved_for_spawn` would
   break the second test even if the first one was somehow softened.
   Pattern: when a new audit-event family lands, pin BOTH the constant's
   exact text AND the absence of any forbidden sibling-family event
   on the wire.

5. **Closed-set 4-key payload distinguishes audit-emitter from
   action-target.** The retire kickoff payload carries 4 keys
   (`manifest_version_id`, `watchkeeper_id`, `approval_token_prefix`,
   `agent_id`) — one more than spawn's 3-key payload. The new
   `watchkeeper_id` key names the row being retired (action target);
   the existing `agent_id` key names the bot that emitted the audit
   row (Watchmaster). Spawn's payload omits the equivalent
   distinction because there's nothing to retire — the watchkeeper
   doesn't exist yet — so `agent_id` is the only relevant id. Pattern:
   when a new flow's audit row has BOTH an emitting agent AND a
   distinct action target, give them separate payload keys; conflating
   them on a single `agent_id` key would force a downstream consumer
   to disambiguate by event_type rather than payload shape.

6. **Source-grep AC enforces the lean-seam claim across the
   kickoffer family (M7.1.d/.e lesson extended).** The kickoffer is a
   composer, not a substrate caller — it imports only `keeperslog` +
   `spawn/saga` + `google/uuid`. The closed-set forbidden-substring
   list rejects `core/pkg/notebook`, `core/pkg/archivestore`,
   `core/pkg/runtime`, `core/pkg/messenger`, and
   `WatchkeeperSlackAppCredsDAO` from `retirekickoff.go`'s source
   text. A future maintainer who imports `notebook` here to call
   `notebook.ArchiveOnRetire` directly (skipping the M7.2.b step
   layer) is the regression this AC pins. Same enforcement shape as
   M7.1.d/.e source-grep ACs; the forbidden list now includes
   `archivestore` because M7.2.b's NotebookArchive step will
   consume that substrate behind a seam, and M7.2.a must not
   shortcut it.

7. **Defensive copy of `Steps` slice at construction (M7.1.c.c
   lesson held forward).** `NewRetireKickoffer` snapshots
   `deps.Steps` via `append([]saga.Step(nil), deps.Steps...)` at
   construction time. The defensive-copy pin test mutates the
   caller's slice AFTER construction and asserts the saga still runs
   the originally-registered steps; without the copy, a caller
   re-using their slice could swap a step in mid-saga. Reaffirms the
   M7.1.c.c rule across the retire kickoffer; once M7.2.b/.c land
   their concrete steps, the production wiring will populate the
   slot through the same defensive-copy seam.

8. **Production-wiring helper ships ahead with deferred call site
   (M7.1.b lesson held forward).** `core/internal/keep/retire_wiring`
   mirrors `core/internal/keep/approval_wiring` shape: a
   `ComposeRetireKickoffer(deps)` helper composes
   `keeperslog.Writer + saga.MemorySpawnSagaDAO + saga.Runner +
spawn.RetireKickoffer` and returns the kickoffer + DAO. No
   production caller exists today (the M6.2.c retire tool stays
   synchronous; M7.2.c routes it through this helper). The wiring
   helper ships now so the composition shape is pinned + smoke-tested
   before the consumer arrives. Smoke test asserts non-nil kickoffer,
   non-nil DAO, and nil-deps rejection — same shape as
   `approval_wiring/wiring_test.go`. Pattern: when a seam ships ahead
   of its consumer, pin the composition shape immediately so the
   consumer's later wiring lands as a 1-line plumb-through, not a
   ground-up wire.

### References

- Files: `core/pkg/spawn/{retirekickoff.go, retirekickoff_test.go}`,
  `core/internal/keep/retire_wiring/{wiring.go, wiring_test.go}`

## 2026-05-09 — M7.1.e: Runtime launch + intro message saga step, wiring saga to completion

**PR**: #118
**Merged**: 279de44 on 2026-05-09

### Context

M7.1.e added the fifth and final concrete `saga.Step`
(`RuntimeLaunchStep`) that drives the spawn saga to completion: a
single lean seam
(`RuntimeLauncher.LaunchRuntime(ctx, watchkeeperID, manifestVersionID, profile)`)
the production wiring backs with a wrapper that (1) projects the
watchkeeper's manifest pinned by `manifestVersionID`, (2) calls
[`runtime.AgentRuntime.Start`](../../core/pkg/runtime/runtime.go) with
the projection plus the per-deployment Personality / Language, and (3)
posts the bot's intro message via
[`messenger.Adapter.SendMessage`](../../core/pkg/messenger/messenger.go)
into the deployment-configured announce channel. The step itself is
lean — no `runtime` import, no `messenger` import, no `notebook`
import, no DAO, no audit emit — matching the M7.1.d
`NotebookProvisionStep` shape exactly. The construction-time
`RuntimeLaunchProfile{Personality, Language, IntroText}` value type
holds only scalar strings, so no defensive deep copy is required
today; if a future field grows a reference type (extra metadata map,
attachment byte slice), the M7.1.c.c `cloneBotProfile` pattern
applies. The step's successful return surfaces as the saga.Runner's
final `saga_completed` audit emit — this step is where the saga stops
needing to be told what to do next.

### Pattern

1. **Lean saga-step seam survives a 2-substrate composite step** —
   When a saga step's job is "boot the runtime AND post the intro
   message" (two distinct substrate concerns: M5.3+ harness +
   M4.2.b messenger), still define a single-method `Launcher` seam
   (`LaunchRuntime(ctx, watchkeeperID, manifestVersionID, profile) error`)
   and push BOTH substrate-specific calls
   (`runtime.Start` + `messenger.SendMessage`) to the production
   wiring layer. The step file imports only `saga` + `uuid` + `fmt` +
   `context` — same lean shape as M7.1.d `NotebookProvisionStep` even
   though the wrapper internally orchestrates two distinct subsystems.
   The lesson holds: lean-seam discipline scales from 1-substrate
   (M7.1.d notebook) to 2-substrate (M7.1.e runtime + messenger)
   without growing the step's import set. Reusable for any future
   M7.x step composing N substrate calls behind one `Doer` seam.

2. **The terminal step does NOT emit `saga_completed` itself** — The
   final step in a saga slice MUST NOT touch the audit chain even
   though it is "the one that finishes the saga". The saga.Runner
   emits `saga_completed` AFTER the final step's `Execute` returns nil
   AND `MarkCompleted` succeeds; emitting it from inside the step
   would either double-emit or leak the saga-level vocabulary into a
   step-level concern. AC7's audit-discipline grep applies to every
   step uniformly — the terminal step is not an exception. The
   M7.1.a saga.Runner contract handles saga termination; M7.1.e's
   `runtime_launch` step is the LAST step but knows nothing about
   "last".

3. **Reuse sentinels across the M7.1.c-.e saga-step family** —
   `RuntimeLaunchStep` reuses `ErrMissingSpawnContext` +
   `ErrMissingAgentID` from M7.1.c.a / M7.1.c.b.b / M7.1.c.c / M7.1.d
   without introducing per-step duplicates. Reaffirms the M7.1.c.b.b
   "Reuse sentinel errors across saga steps" lesson across the FULL
   M7.1.c-.e family: one sentinel per error class regardless of how
   many steps land. The redaction-harness AC then asserts the same
   sentinel surfaces unchanged through the wrap chain on a `launcher
returns ErrCredsNotFound` test case — pinning that the wrap stays
   compatible with `errors.Is` callers, mirroring the M7.1.d pin.

4. **PII redaction harness covers every failure path including
   ctx-cancel** — The redaction harness applies canary substrings
   (`secret-personality-CANARY`, `secret-LANG-CANARY`,
   `secret-INTRO-CANARY`) across **five** failure paths: `launcher
error`, `missing spawn context`, `nil agent id`, `launcher returns
ErrCredsNotFound`, **and `cancelled context`**. Generalizes the
   M7.1.d "every error-string-building path is covered" lesson to a
   3-field profile (vs M7.1.d's 2-field). The cancellation case is
   cheap (1 ctx + `cancel()`) and preempts a future maintainer who
   decorates the cancel branch with profile content for "richer
   diagnostics".

5. **Step list registration slot already exists; M7.1.e ships pure
   step + tests** — M7.1.c.c shipped the `Steps []saga.Step` slot on
   `approvalwiring.ApprovalDispatcherDeps` + `spawn.SpawnKickoffDeps`
   with a documented zero-value contract. M7.1.e does NOT touch
   `approval_wiring/wiring.go` — the future Slack-bot binary
   populates Steps with `[CreateApp, OAuthInstall, BotProfile,
NotebookProvision, RuntimeLaunch]` (in that order) when it takes
   over wiring. Same pattern as M7.1.d: when a long-lived seam
   already has the registration slot, downstream PRs ship pure step +
   tests without re-plumbing the wiring file; the slot's zero-value
   contract keeps M7.1.b's smoke test green without a coordinated
   rollout. With M7.1.e shipped, the M7.1.c-.e step family is now
   complete and the future Slack-bot binary's wiring is fully
   specified.

6. **Pin ctx-propagation through a recording fake, not a discarding
   fake (M7.1.d lesson held forward)** — The test fake's
   `LaunchRuntime` records the incoming ctx on every call and pins
   propagation two ways: (a) a `context.WithValue` sentinel must
   round-trip through to the seam (verifies value-chain integrity);
   (b) cancelling the caller's parent ctx AFTER the synchronous call
   returns must surface as `context.Canceled` on the recorded ctx
   (verifies cancellation-chain integrity, independent of the seam's
   return value). Pattern: when a step's contract is "forward the
   caller's ctx verbatim", the test fake MUST record the ctx —
   discarding it makes the contract decorative.

7. **Source-grep AC enforces the FULL lean-seam claim, not just
   audit-emit (M7.1.d lesson extended for runtime substrate)** —
   The closed-set forbidden-substring list rejects `keeperslog.`,
   `.Append(`, `core/pkg/runtime`, `core/pkg/messenger`,
   `core/pkg/notebook`, and `WatchkeeperSlackAppCredsDAO`. Adding
   `core/pkg/runtime` to the M7.1.d list pins the M7.1.e-specific
   "step does not import the harness substrate" claim — the
   production wrapper composes the runtime.Manifest and calls
   AgentRuntime.Start; the step itself never sees those types. Same
   enforcement shape as M7.1.d; longer forbidden list because M7.1.e
   spans two substrates instead of one.

8. **Forward saga-pinned manifest_version_id through the seam — never
   re-resolve "current active" by watchkeeper id** (iter-1 codex
   finding). The seam signature accepts `manifestVersionID uuid.UUID`
   alongside `watchkeeperID`; `Execute` reads `sc.ManifestVersionID`
   off `saga.SpawnContext` and adds a `uuid.Nil` guard
   (`ErrMissingManifestVersion`, the same sentinel M7.1.c.a's
   `CreateAppStep` already uses). Without this pin the production
   wrapper has only two bad options: re-resolve the bot's
   "current-active" manifest (no longer saga-deterministic — if the
   active manifest flips between approval and runtime launch, the
   wrong runtime config boots) or reach into `ctx` for spawn
   internals (defeats the seam shape). The happy-path test asserts
   `call.manifestVersionID == sc.ManifestVersionID` so a future
   regression that drops the field is caught at the test layer; the
   `nil_manifest_version` case in the redaction harness pins the
   guard wraps without leaking profile content. Generalises
   M7.1.c.a's `ErrMissingManifestVersion` reuse into a saga-family
   rule: any step whose external behaviour depends on manifest-version
   semantics MUST validate the field BEFORE dispatch.

9. **Document the at-least-once / idempotency contract for composite
   seams explicitly** (iter-1 codex finding + critic Skeptic
   confirmation). When a single seam batches two non-atomic side
   effects (`runtime.Start` + intro `SendMessage`), the partial-
   success path becomes the implicit retry surface: Start succeeds,
   intro post fails → step returns error → `saga.Runner` records
   `saga_failed` while the runtime is already live ("live but
   silent"). Until M7.3 ships compensations, the seam's documentation
   MUST name this limitation and require implementers to be
   idempotent across retries on the same `(watchkeeperID,
manifestVersionID)` pair (typical impl: DAO lookup detects an
   already-running runtime and either no-ops or re-announces). The
   step itself does not decide retry policy — the operator (or the
   M7.3 compensator) does. Pattern: any composite-seam step lands
   with an explicit at-least-once contract in the seam's docstring;
   silently relying on "future M7.3 will fix this" lets a maintainer
   ship a non-idempotent wrapper that strands runtimes on partial
   failures.

## 2026-05-09 — M7.1.d: Notebook provision saga step + per-agent file with personality/language

**PR**: #117
**Merged**: ba9dd0c on 2026-05-09

### Context

M7.1.d added the fourth concrete `saga.Step` (`NotebookProvisionStep`) bridging the M2b.1 [`notebook.Open`](../../core/pkg/notebook/db.go) per-agent SQLite substrate into the spawn flow. The step dispatches via a thin `NotebookProvisioner` seam keyed off [`saga.SpawnContext.AgentID`](../../core/pkg/spawn/saga/spawn_context.go); the production wiring (future Slack-bot binary) backs the seam with a wrapper that calls `notebook.Open(ctx, watchkeeperID.String(), ...)` (creating `<WATCHKEEPER_DATA>/notebook/<watchkeeperID>.sqlite` on first touch — the substrate's path validator consumes the canonical UUID text form) and seeds the agent's personality / language as foundational notebook entries via [`notebook.DB.Remember`](../../core/pkg/notebook/remember.go). The step itself is lean — no `notebook` import, no DAO, no audit emit — matching the M7.1.c.c `BotProfileStep` shape exactly. The construction-time `NotebookProfile{Personality, Language}` value type holds only scalar strings, so no defensive deep copy is required today; if a future field grows a reference type, the M7.1.c.c `cloneBotProfile` pattern applies.

### Pattern

1. **Lean saga-step seam over substrate-aware step** — When a saga step's job is "ensure resource X exists for watchkeeperID and seed Y", define a single-method `Provisioner` seam (`ProvisionNotebook(ctx, watchkeeperID, profile) error`) and push the substrate-specific work (`notebook.Open` + `Remember` + close-on-error rollback) to the production wiring layer. The step file imports only `saga` + `uuid` + `fmt` + `context` — no `notebook`, no DAO, no `messenger`. Same shape as M7.1.c.c `BotProfileSetter` and M7.1.c.b.b `SlackAppInstaller`. Reusable for any future "ensure-and-seed" step (M7.2 retire's `Archive` is the inverse-shape analogue).

2. **Closed-set step name pre-mentioned in the saga.Step docstring is the canonical identifier** — `core/pkg/spawn/saga/saga.go` already named `notebook_provision` as one of two example identifiers in [`Step` interface docstring](../../core/pkg/spawn/saga/saga.go) when M7.1.a landed. Using that exact spelling for `NotebookProvisionStepName` keeps the documentation in lock-step with the wire vocabulary; a typo would surface as a docstring-vs-impl drift on visual review and as a `current_step` DAO mismatch on integration. Pattern: when a docstring forecasts a downstream identifier, treat it as a load-bearing pin, not a placeholder.

3. **Scalar-only Profile value avoids defensive-copy cost when no reference fields exist** — `NotebookProfile{Personality string; Language string}` carries no map / slice / pointer / byte-slice fields, so a value-copy is sufficient at construction and on dispatch. The constructor stores `deps.Profile` directly; `Execute` forwards `s.profile` directly. Pattern: **only** add `cloneXxx` when the value type has at least one reference field; otherwise the helper is dead weight that future maintainers may copy-paste into another scalar-only struct. The doc-block explicitly anchors the rule + the `cloneBotProfile` migration path so a future field-growth review immediately triggers the deep-copy upgrade.

4. **Reuse sentinels across the M7.1.c.x saga-step family** — `NotebookProvisionStep` reuses `ErrMissingSpawnContext` + `ErrMissingAgentID` from M7.1.c.a / M7.1.c.b.b / M7.1.c.c without introducing per-step duplicates. Reaffirms the M7.1.c.b.b "Reuse sentinel errors across saga steps" lesson: one sentinel per error class across the M7 saga family, regardless of how many steps land. The redaction-harness AC then asserts the same sentinel surfaces unchanged through the wrap chain on a `provisioner returns ErrCredsNotFound` test case — pinning that the wrap stays compatible with `errors.Is` callers.

5. **PII redaction harness covers every failure path including ctx-cancel** — The redaction harness applies canary substrings (`secret-personality-CANARY`, `secret-LANG-CANARY`) across **five** failure paths: `provisioner error`, `missing spawn context`, `nil agent id`, `provisioner returns ErrCredsNotFound`, **and `cancelled context`**. M7.1.c.b.b's iter-1 caught a missing post-encrypt-pre-store path; the lesson generalizes: every path that builds an error string from the step must be covered. The cancellation case is cheap (1 ctx + `cancel()`) and preempts a future maintainer who decorates the cancel branch with profile content for "richer diagnostics".

6. **Step list registration slot already exists; M7.1.d ships pure step + tests** — M7.1.c.c shipped the `Steps []saga.Step` slot on `approvalwiring.ApprovalDispatcherDeps` + `spawn.SpawnKickoffDeps` with a documented zero-value contract. M7.1.d does NOT touch `approval_wiring/wiring.go` — the future Slack-bot binary populates Steps with `[CreateApp, OAuthInstall, BotProfile, NotebookProvision]` (in that order) when it takes over wiring. Pattern: when a long-lived seam already has the registration slot, downstream PRs ship pure step + tests without re-plumbing the wiring file; the slot's zero-value contract keeps M7.1.b's smoke test green without a coordinated rollout.

7. **Pin ctx-propagation through a recording fake, not a discarding fake** — Iter-1 codex review caught that the original test fake's `ProvisionNotebook(_ context.Context, ...)` discarded the incoming ctx, so a future regression to `s.provisioner.ProvisionNotebook(context.Background(), ...)` (or `context.WithoutCancel(ctx)`) would still pass every other assertion. The fix records the ctx on every call and pins propagation two ways: (a) a `context.WithValue` sentinel must round-trip through to the seam (verifies value-chain integrity); (b) cancelling the caller's parent ctx AFTER the synchronous call returns must surface as `context.Canceled` on the recorded ctx (verifies cancellation-chain integrity, independent of the seam's return value). Pattern: when a step's contract is "forward the caller's ctx verbatim", the test fake MUST record the ctx — discarding it makes the contract decorative.

8. **Source-grep AC must enforce the _full_ lean-seam claim, not just audit-emit** — Iter-1 codex review caught that the original AC checked only `keeperslog.` + `.Append(`, leaving the lesson's broader "no `notebook`/DAO/messenger imports" claim unenforced. The fix extends the closed-set forbidden-substring list to also reject `core/pkg/notebook`, `core/pkg/messenger`, and `WatchkeeperSlackAppCredsDAO`. Pattern: when a lesson narrative claims multiple invariants (audit-free + substrate-free + DAO-free + platform-free), the source-grep AC must enumerate ALL of them — partial coverage lets a future maintainer add a forbidden import while still passing the test, and the lesson silently rots.

## 2026-05-09 — M7.1.c.c: BotProfile saga step + register step list in spawn kickoff wiring

**PR**: #116
**Merged**: aece000 on 2026-05-09

### Context

M7.1.c.c added the third concrete saga.Step (`BotProfileStep`) bridging `messenger.Adapter.SetBotProfile` (Slack `users.profile.set`) into the spawn flow, plus extended `spawn.SpawnKickoffer` to register a step list at construction time and seed `saga.SpawnContext` on the per-call ctx. The kickoffer's `Kickoff` signature gained `watchkeeperID uuid.UUID` and `claim saga.SpawnClaim` parameters with fail-fast validation that rejects `uuid.Nil` ids BEFORE any audit/state side effect; the dispatcher mints `uuid.New()` per call and resolves the claim per-call via the new `approval.WithSpawnClaimResolver` option (a `func(ctx, manifestVersionID) (saga.SpawnClaim, error)` so multi-tenant deployments derive the right tenant claim from the row instead of pinning a process-global value). The `approval_wiring` helper exposes `SpawnClaimResolver` + `Steps` deps so the future Slack-bot binary plugs concrete steps without re-plumbing the dispatcher.

### Pattern

1. **Step list registered at constructor; per-saga values flow through SpawnContext** — `SpawnKickoffer` accepts `Steps []saga.Step` in its deps and stamps a defensive copy at construction. On every `Kickoff` it seeds `saga.SpawnContext{ManifestVersionID, AgentID: watchkeeperID, Claim}` onto ctx BEFORE `runner.Run`, so registered steps read per-saga values via `saga.SpawnContextFromContext(ctx)` without re-plumbing each step constructor. Reusable for any saga that wires per-call values to multiple steps without forcing each step's API surface to grow.

2. **Per-call resolver beats process-global static for tenant-scoped values** — Iter-1 codex review caught a multi-tenant correctness bug: a static `WithWatchmasterClaim(saga.SpawnClaim)` option pinned one claim on the dispatcher, so a single bot serving multiple orgs would replay every approval under the same authority tuple. The fix replaces the static option with `WithSpawnClaimResolver(func(ctx, manifestVersionID) (saga.SpawnClaim, error))`. The resolver runs INSIDE the dispatch path, derives the tenant claim from the per-call manifestVersionID, and returns errors that surface as `approval_replay_failed`. Pattern: when a value depends on per-request state (tenant id, request scope), wire it as a resolver func, NOT as a constructor-time static value — even when "every deployment ships one tenant" is true today.

3. **Optional Setter seam decouples step from messenger client construction** — `BotProfileSetter` interface (`SetBotProfile(ctx, watchkeeperID, profile)`) hides the per-watchkeeper bot-token decryption + per-call client clone. The step does not import `messenger/slack` directly; the production wiring layer composes a setter that decrypts the M7.1.c.b.b token, builds a per-call client, and forwards. Same pattern as `SlackAppInstaller` from M7.1.c.b.b — keep the saga step lean, push platform-specific wiring to the production composition layer.

4. **Empty profile is a no-op, not a panic** — `messenger.BotProfile{}` (entirely empty) is a documented adapter-level no-op; the step accepts it without panic at construction. This matches the real-world need where a deployment without a personality profile still completes the saga rather than fail-closing.

5. **Backward-compatible step-list default** — `Steps` is an _optional_ `[]saga.Step` on `SpawnKickoffDeps`; nil keeps the M7.1.b zero-step behaviour (single `manifest_approved_for_spawn` + single `saga_completed`). Pattern: when a long-lived seam grows a new optional field, ship it with a documented zero-value contract so existing callers stay green without a coordinated rollout.

6. **Defensive deep copy on reference-typed step config** — Iter-1 codex review caught that `BotProfileStep` stored `messenger.BotProfile` by shallow copy, leaving `Metadata` (map) and `AvatarPNG` (slice) aliased to caller-owned memory. Under concurrent saga runs OR post-construction caller mutations, this would race AND change what gets sent to Slack without going back through construction. The fix added a `cloneBotProfile` helper called both in `NewBotProfileStep` (snapshot) and in `Execute` (fresh per-call copy to the Setter). Pattern: when a step's "immutable configuration" claim covers a value type with internal map / slice fields, defensively deep-copy in the constructor AND clone again on dispatch — mirrors `NewCreateAppStep`'s `Scopes` defensive copy.

7. **Fail-fast validation precedes audit-emit + state-write** — Iter-1 codex review caught that `SpawnKickoffer.Kickoff` accepted `uuid.Nil` for `sagaID` / `manifestVersionID` / `watchkeeperID` and only failed downstream after emitting `manifest_approved_for_spawn` + inserting the saga row. The fix added an explicit `ErrInvalidKickoffArgs` sentinel and validation guard BEFORE the audit Append. Pattern: a malformed input that's a programmer / wiring bug (not a runtime fault) should leave NO audit row and NO persisted state — the audit row should always reflect "we tried and failed downstream", never "we accepted garbage and persisted it".

## 2026-05-09 — M7.1.c.b.b: OAuthInstall saga step + encrypted bot-token storage using secrets crypto primitive

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

M7.1.c.b.b added the second concrete saga.Step (`OAuthInstallStep`) in the spawn flow. It bridges `messenger.Client.InstallApp` (M4.2.d.2 OAuth `oauth.v2.access` exchange) to encrypted token storage in `watchkeeper.slack_app_creds` via the M7.1.c.b.a `secrets.AESGCMEncrypter`. The step pulls M7.1.c.a's stored `client_id`/`client_secret` via the resolver, receives `InstallTokens` via the sink, encrypts each token, and persists ciphertexts.

### Pattern

1. **Empty-plaintext short-circuit avoids encrypted-empty-string regression** — `encryptIfNonEmpty(ctx, plaintext)` returns `nil, nil` when `len(plaintext) == 0`. Encrypting an empty plaintext via AES-GCM produces a 28-byte ciphertext (12 nonce + 16 tag); storing that as "no refresh_token" would silently disagree with downstream `len() == 0` checks. Reusable for any optional encrypted field where "absent" must round-trip as nil/empty, not as a fixed-size ciphertext. Pinned by AC test asserting stored CT is `nil` when input is empty.

2. **Per-call resolver+sink wrap of a long-lived Client** — `messenger.Client.InstallApp` accepts `WithInstallTokenSink` / `WithInstallParamsResolver` per call (clones config). The saga step builds fresh closures per Execute that capture the SpawnContext-derived `watchkeeperID`, NOT the client-shared install state. Pattern: when an SDK Client takes per-call options, prefer per-Execute closures over construction-time wiring so step concurrency is safe and the closure scope matches the per-saga state.

3. **Highest-risk leak vector for redaction tests is the post-Encrypt-pre-DAO path** — when an Encrypt-then-Persist pipeline fails at Persist, all three plaintext tokens are still in scope on the goroutine; a future maintainer wrapping `tokens.AccessToken` into the DAO error message would slip past the existing redaction harness undetected. Pattern: redaction-grep tests MUST cover EVERY failure path, especially the post-encrypt-pre-store one, not just the obvious "encrypter error" or "missing input" paths. Iter-1 reviewer caught this gap; the iter-1 fix added a 5th case to the redaction table.

4. **Reuse sentinel errors across saga steps** — `OAuthInstallStep` reuses `ErrCredsNotFound` from M7.1.c.a (raised when the M7.1.c.a creds row is missing). Don't duplicate sentinels per step; one sentinel per error class across the M7 saga family. Reusable for M7.1.c.c BotProfileStep + M7.1.d Notebook step + M7.1.e runtime launch step.
