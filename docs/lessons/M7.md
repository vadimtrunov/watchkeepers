## 2026-05-09 — M7.1.a: Saga skeleton: spawn/ package, state machine, saga-state DAO + migration

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

M7.1.a introduced `core/pkg/spawn/saga` with a step-based orchestrator and state-persistence DAO, plus the SQL migration for saga-state storage. The package defines a generic `Step` interface so M7.1.b–.e (Slack interaction, App provisioning, Notebook provision, runtime launch) can plug concrete steps without modifying saga core. Migration `019_spawn_sagas.sql` ships alongside the in-memory DAO so M7.1.b can wire a Postgres adapter without migration churn.

### Pattern

**Step-interface seam for multi-step external flows**: A saga's `Run` loop consumes `[]Step` where each step is `{Name() string, Execute(ctx) error}`. The orchestrator owns state persistence (UpdateStep BEFORE Execute) and audit emit (started BEFORE Execute, completed/failed BEFORE return). Future PRs add concrete steps as struct types — no modification to saga core. Reusable for retire saga (M7.2) and any multi-step flow.

**Audit-emit-before-return ordering generalizes M6.3.e to saga steps**: Every state transition's audit row emits BEFORE the boundary call (Execute / return). On `Step.Execute` error: `MarkFailed` → emit `saga_failed` → return error. Inverting any of the three creates a silent-data-loss defect. Pattern extends M6.3.e's "stream emit before user handler" to the saga domain.

**PII discipline closed-set payload (M2b.7 + M6.3.f wire-shape)**: Saga audit payloads are restricted to three keys (`saga_id`, `step_name`, `last_error_class`) and a regression-test grep-asserts the JSON keys. Step authors cannot leak raw step params or `error.Error()` strings into audit — payload constructors live in saga core, not Step impls.

**Migration-shipped-with-skeleton**: Per M6.3.b "ship in-memory DAO with consumer", M7.1.a ships the migration alongside the in-memory DAO so M7.1.b wires Postgres without migration churn.

## 2026-05-09 — M7.1.b: Slack interaction handler: approval action → Keeper's Log event → saga kickoff

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

M7.1.b wired Slack-approval-action → Keeper's Log audit event → spawn-saga kickoff using the M7.1.a step-interface seam. New seam `SpawnKickoff` interface in approval-package + concrete `SpawnKickoffer` impl in spawn-package + production-wiring helper. The saga runs with empty `[]Step{}` in this PR; concrete steps land in M7.1.c–.e.

### Pattern

**Per-tool branch on PendingApproval.ToolName**: When an approval flow needs to dispatch differently per tool, the cleanest seam is to branch the dispatcher on the closed-set `pending.ToolName` and route `propose_spawn` to a new `SpawnKickoff` seam while leaving the existing `Replayer` path for other tools. Mirrors the M6.3.b two-seam pattern (Replayer + Renderer); keeps the new dispatch path independently testable.

**manifest_version_id flows through ParamsJSON, not the PendingApproval struct**: When the upstream tool already serialised its params snapshot into `pending.ParamsJSON`, downstream branches that need a specific field (here `manifest_version_id`) parse it from the JSON envelope rather than expanding the `PendingApproval` struct. Path B (parse + typed sentinel `ErrMissingManifestVersionID`) keeps the diff inside the M7.1.b file set; path A (extend `PendingApproval` + DAO surface) was rejected because it would force changes to the M6.3.b storage contract for a transient need.

**Audit-emit-before-state-write generalises across the seam**: `SpawnKickoffer.Kickoff` emits `manifest_approved_for_spawn` BEFORE `Insert` BEFORE `Run([]Step{})`. The audit row is canonical even when state-persistence fails (M6.3.e + M7.1.a pattern). Test `TestSpawnKickoff_InsertError_AuditAlreadyEmitted` pins this — Insert error → audit row IS persisted, saga DAO has no row.

**Local seam re-declaration over cross-file unexported interface**: When an impl file needs an interface type already declared in a sibling file in the same package, prefer re-declaring locally (`kickoffLogAppender` in spawnkickoff.go alongside `keepersLogAppender` in slack*app.go) with a compile-time drift-pin assertion `var * kickoffLogAppender = keepersLogAppender(nil)`. Reader of one file sees the contract; build catches signature drift.

**Exact-equality vocabulary pin over HasPrefix tautology**: When guarding event-type vocabulary against future renames, use `if eventType != "manifest_approved_for_spawn"` rather than `!HasPrefix(et, "manifest_approved_for_spawn")` (which is tautological for the literal string). Keep `HasPrefix` checks against COLLIDING prefixes (`llm_turn_cost`, `saga_`) as defense-in-depth.

## 2026-05-09 — M7.1.c.a: CreateApp saga step + watchkeeper secrets column + migration

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

M7.1.c.a wired the first concrete saga.Step impl (CreateApp) bridging the M6.1.b SlackAppRPC privileged RPC to a new watchkeeper.slack_app_creds Postgres table via the existing M4.2.d sink-adapter contract. Introduced saga.SpawnContext value type for cross-step parameter passing and stabilized manifest_version_id + agent_id access across future steps.

### Pattern

**Sink-to-DAO bridge owns watchkeeperID, not appID**: A saga step that bridges an OUT-OF-BAND credential delivery (M4.2.d sink) to a persistence DAO must close over the saga's primary key (watchkeeperID from SpawnContext), NOT the credential's own assigned id (creds.AppID). Re-create scenarios change AppID; watchkeeper id is stable across the saga's lifetime. The DAO column layout reflects this: watchkeeper_id is PK, app_id is a column.

**SpawnContext as saga-level cross-step seam**: Future M7.1.x steps need read access to manifest_version_id, agent_id, and the dispatcher's Claim. Define a typed value (saga.SpawnContext) + accessors (WithSpawnContext / SpawnContextFromContext) in the saga package — NOT in the spawn package — so step impls have a stable contract that does not couple them to step-specific constructors.

**Source-grep AC pinning for negative AC ("does NOT call X")**: When an AC asserts that a unit must NOT call some API, the test that pins it cannot be a runtime mock — even an injected fake passes vacuously if the unit reaches the API via a different path or is later refactored. The robust pin is os.ReadFile on the unit's source + strings.Contains outside //-prefixed lines. Mutation-tested (adding the forbidden call must cause the test to fail).

**Migration ships with deferred encryption-at-rest comment**: Per the M6.3.b "ship in-memory DAO with consumer" rhythm, M7.1.c.a's migration uses text columns for credentials with an explicit -- TODO: encryption-at-rest deferral comment. The Phase-2 secrets pass will rotate to encrypted bytea. Keeps the M7.1.c.a PR scope tight.

### Anti-pattern

Hollow runtime-mock AC pins. The iter-1 reviewer caught a keeperslog.New(keep) that was constructed and immediately discarded (\_ = ...) — the recordingKeepClient could never observe the call because the writer was never injected. The fix replaced it with a source-grep mutation-tested assertion. Lesson: any AC pin that depends on observation through an injected dependency must verify the dependency reaches the unit-under-test; otherwise prefer source-grep.

## 2026-05-09 — M7.1.c.c: BotProfile saga step + register step list in spawn kickoff wiring

**PR**: #116
**Merged**: aece000 on 2026-05-09

### Context

M7.1.c.c added the third concrete saga.Step (`BotProfileStep`) bridging `messenger.Adapter.SetBotProfile` (Slack `users.profile.set`) into the spawn flow, plus extended `spawn.SpawnKickoffer` to register a step list at construction time and seed `saga.SpawnContext` on the per-call ctx. The kickoffer's `Kickoff` signature gained `watchkeeperID uuid.UUID` and `claim saga.SpawnClaim` parameters with fail-fast validation that rejects `uuid.Nil` ids BEFORE any audit/state side effect; the dispatcher mints `uuid.New()` per call and resolves the claim per-call via the new `approval.WithSpawnClaimResolver` option (a `func(ctx, manifestVersionID) (saga.SpawnClaim, error)` so multi-tenant deployments derive the right tenant claim from the row instead of pinning a process-global value). The `approval_wiring` helper exposes `SpawnClaimResolver` + `Steps` deps so the future Slack-bot binary plugs concrete steps without re-plumbing the dispatcher.

### Pattern

1. **Step list registered at constructor; per-saga values flow through SpawnContext** — `SpawnKickoffer` accepts `Steps []saga.Step` in its deps and stamps a defensive copy at construction. On every `Kickoff` it seeds `saga.SpawnContext{ManifestVersionID, AgentID: watchkeeperID, Claim}` onto ctx BEFORE `runner.Run`, so registered steps read per-saga values via `saga.SpawnContextFromContext(ctx)` without re-plumbing each step constructor. Reusable for any saga that wires per-call values to multiple steps without forcing each step's API surface to grow.

2. **Per-call resolver beats process-global static for tenant-scoped values** — Iter-1 codex review caught a multi-tenant correctness bug: a static `WithWatchmasterClaim(saga.SpawnClaim)` option pinned one claim on the dispatcher, so a single bot serving multiple orgs would replay every approval under the same authority tuple. The fix replaces the static option with `WithSpawnClaimResolver(func(ctx, manifestVersionID) (saga.SpawnClaim, error))`. The resolver runs INSIDE the dispatch path, derives the tenant claim from the per-call manifestVersionID, and returns errors that surface as `approval_replay_failed`. Pattern: when a value depends on per-request state (tenant id, request scope), wire it as a resolver func, NOT as a constructor-time static value — even when "every deployment ships one tenant" is true today.

3. **Optional Setter seam decouples step from messenger client construction** — `BotProfileSetter` interface (`SetBotProfile(ctx, watchkeeperID, profile)`) hides the per-watchkeeper bot-token decryption + per-call client clone. The step does not import `messenger/slack` directly; the production wiring layer composes a setter that decrypts the M7.1.c.b.b token, builds a per-call client, and forwards. Same pattern as `SlackAppInstaller` from M7.1.c.b.b — keep the saga step lean, push platform-specific wiring to the production composition layer.

4. **Empty profile is a no-op, not a panic** — `messenger.BotProfile{}` (entirely empty) is a documented adapter-level no-op; the step accepts it without panic at construction. This matches the real-world need where a deployment without a personality profile still completes the saga rather than fail-closing.

5. **Backward-compatible step-list default** — `Steps` is an _optional_ `[]saga.Step` on `SpawnKickoffDeps`; nil keeps the M7.1.b zero-step behaviour (single `manifest_approved_for_spawn` + single `saga_completed`). Pattern: when a long-lived seam grows a new optional field, ship it with a documented zero-value contract so existing callers stay green without a coordinated rollout.

6. **Defensive deep copy on reference-typed step config** — Iter-1 codex review caught that `BotProfileStep` stored `messenger.BotProfile` by shallow copy, leaving `Metadata` (map) and `AvatarPNG` (slice) aliased to caller-owned memory. Under concurrent saga runs OR post-construction caller mutations, this would race AND change what gets sent to Slack without going back through construction. The fix added a `cloneBotProfile` helper called both in `NewBotProfileStep` (snapshot) and in `Execute` (fresh per-call copy to the Setter). Pattern: when a step's "immutable configuration" claim covers a value type with internal map / slice fields, defensively deep-copy in the constructor AND clone again on dispatch — mirrors `NewCreateAppStep`'s `Scopes` defensive copy.

7. **Fail-fast validation precedes audit-emit + state-write** — Iter-1 codex review caught that `SpawnKickoffer.Kickoff` accepted `uuid.Nil` for `sagaID` / `manifestVersionID` / `watchkeeperID` and only failed downstream after emitting `manifest_approved_for_spawn` + inserting the saga row. The fix added an explicit `ErrInvalidKickoffArgs` sentinel and validation guard BEFORE the audit Append. Pattern: a malformed input that's a programmer / wiring bug (not a runtime fault) should leave NO audit row and NO persisted state — the audit row should always reflect "we tried and failed downstream", never "we accepted garbage and persisted it".

## 2026-05-09 — M7.1.c.b.b: OAuthInstall saga step + encrypted bot-token storage using secrets crypto primitive

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Context

M7.1.c.b.b added the second concrete saga.Step (`OAuthInstallStep`) in the spawn flow. It bridges `messenger.Client.InstallApp` (M4.2.d.2 OAuth `oauth.v2.access` exchange) to encrypted token storage in `watchkeeper.slack_app_creds` via the M7.1.c.b.a `secrets.AESGCMEncrypter`. The step pulls M7.1.c.a's stored `client_id`/`client_secret` via the resolver, receives `InstallTokens` via the sink, encrypts each token, and persists ciphertexts.

### Pattern

1. **Empty-plaintext short-circuit avoids encrypted-empty-string regression** — `encryptIfNonEmpty(ctx, plaintext)` returns `nil, nil` when `len(plaintext) == 0`. Encrypting an empty plaintext via AES-GCM produces a 28-byte ciphertext (12 nonce + 16 tag); storing that as "no refresh_token" would silently disagree with downstream `len() == 0` checks. Reusable for any optional encrypted field where "absent" must round-trip as nil/empty, not as a fixed-size ciphertext. Pinned by AC test asserting stored CT is `nil` when input is empty.

2. **Per-call resolver+sink wrap of a long-lived Client** — `messenger.Client.InstallApp` accepts `WithInstallTokenSink` / `WithInstallParamsResolver` per call (clones config). The saga step builds fresh closures per Execute that capture the SpawnContext-derived `watchkeeperID`, NOT the client-shared install state. Pattern: when an SDK Client takes per-call options, prefer per-Execute closures over construction-time wiring so step concurrency is safe and the closure scope matches the per-saga state.

3. **Highest-risk leak vector for redaction tests is the post-Encrypt-pre-DAO path** — when an Encrypt-then-Persist pipeline fails at Persist, all three plaintext tokens are still in scope on the goroutine; a future maintainer wrapping `tokens.AccessToken` into the DAO error message would slip past the existing redaction harness undetected. Pattern: redaction-grep tests MUST cover EVERY failure path, especially the post-encrypt-pre-store one, not just the obvious "encrypter error" or "missing input" paths. Iter-1 reviewer caught this gap; the iter-1 fix added a 5th case to the redaction table.

4. **Reuse sentinel errors across saga steps** — `OAuthInstallStep` reuses `ErrCredsNotFound` from M7.1.c.a (raised when the M7.1.c.a creds row is missing). Don't duplicate sentinels per step; one sentinel per error class across the M7 saga family. Reusable for M7.1.c.c BotProfileStep + M7.1.d Notebook step + M7.1.e runtime launch step.
