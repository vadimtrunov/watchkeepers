## 2026-05-09 тАФ M7.1.a: Saga skeleton: spawn/ package, state machine, saga-state DAO + migration

**PR**: pending тАФ to be opened in Phase 5b
**Merged**: pending тАФ to be merged in Phase 7a

### Context

M7.1.a introduced `core/pkg/spawn/saga` with a step-based orchestrator and state-persistence DAO, plus the SQL migration for saga-state storage. The package defines a generic `Step` interface so M7.1.bтАУ.e (Slack interaction, App provisioning, Notebook provision, runtime launch) can plug concrete steps without modifying saga core. Migration `019_spawn_sagas.sql` ships alongside the in-memory DAO so M7.1.b can wire a Postgres adapter without migration churn.

### Pattern

**Step-interface seam for multi-step external flows**: A saga's `Run` loop consumes `[]Step` where each step is `{Name() string, Execute(ctx) error}`. The orchestrator owns state persistence (UpdateStep BEFORE Execute) and audit emit (started BEFORE Execute, completed/failed BEFORE return). Future PRs add concrete steps as struct types тАФ no modification to saga core. Reusable for retire saga (M7.2) and any multi-step flow.

**Audit-emit-before-return ordering generalizes M6.3.e to saga steps**: Every state transition's audit row emits BEFORE the boundary call (Execute / return). On `Step.Execute` error: `MarkFailed` тЖТ emit `saga_failed` тЖТ return error. Inverting any of the three creates a silent-data-loss defect. Pattern extends M6.3.e's "stream emit before user handler" to the saga domain.

**PII discipline closed-set payload (M2b.7 + M6.3.f wire-shape)**: Saga audit payloads are restricted to three keys (`saga_id`, `step_name`, `last_error_class`) and a regression-test grep-asserts the JSON keys. Step authors cannot leak raw step params or `error.Error()` strings into audit тАФ payload constructors live in saga core, not Step impls.

**Migration-shipped-with-skeleton**: Per M6.3.b "ship in-memory DAO with consumer", M7.1.a ships the migration alongside the in-memory DAO so M7.1.b wires Postgres without migration churn.

## 2026-05-09 тАФ M7.1.b: Slack interaction handler: approval action тЖТ Keeper's Log event тЖТ saga kickoff

**PR**: pending тАФ to be opened in Phase 5b
**Merged**: pending тАФ to be merged in Phase 7a

### Context

M7.1.b wired Slack-approval-action тЖТ Keeper's Log audit event тЖТ spawn-saga kickoff using the M7.1.a step-interface seam. New seam `SpawnKickoff` interface in approval-package + concrete `SpawnKickoffer` impl in spawn-package + production-wiring helper. The saga runs with empty `[]Step{}` in this PR; concrete steps land in M7.1.cтАУ.e.

### Pattern

**Per-tool branch on PendingApproval.ToolName**: When an approval flow needs to dispatch differently per tool, the cleanest seam is to branch the dispatcher on the closed-set `pending.ToolName` and route `propose_spawn` to a new `SpawnKickoff` seam while leaving the existing `Replayer` path for other tools. Mirrors the M6.3.b two-seam pattern (Replayer + Renderer); keeps the new dispatch path independently testable.

**manifest_version_id flows through ParamsJSON, not the PendingApproval struct**: When the upstream tool already serialised its params snapshot into `pending.ParamsJSON`, downstream branches that need a specific field (here `manifest_version_id`) parse it from the JSON envelope rather than expanding the `PendingApproval` struct. Path B (parse + typed sentinel `ErrMissingManifestVersionID`) keeps the diff inside the M7.1.b file set; path A (extend `PendingApproval` + DAO surface) was rejected because it would force changes to the M6.3.b storage contract for a transient need.

**Audit-emit-before-state-write generalises across the seam**: `SpawnKickoffer.Kickoff` emits `manifest_approved_for_spawn` BEFORE `Insert` BEFORE `Run([]Step{})`. The audit row is canonical even when state-persistence fails (M6.3.e + M7.1.a pattern). Test `TestSpawnKickoff_InsertError_AuditAlreadyEmitted` pins this тАФ Insert error тЖТ audit row IS persisted, saga DAO has no row.

**Local seam re-declaration over cross-file unexported interface**: When an impl file needs an interface type already declared in a sibling file in the same package, prefer re-declaring locally (`kickoffLogAppender` in spawnkickoff.go alongside `keepersLogAppender` in slack*app.go) with a compile-time drift-pin assertion `var * kickoffLogAppender = keepersLogAppender(nil)`. Reader of one file sees the contract; build catches signature drift.

**Exact-equality vocabulary pin over HasPrefix tautology**: When guarding event-type vocabulary against future renames, use `if eventType != "manifest_approved_for_spawn"` rather than `!HasPrefix(et, "manifest_approved_for_spawn")` (which is tautological for the literal string). Keep `HasPrefix` checks against COLLIDING prefixes (`llm_turn_cost`, `saga_`) as defense-in-depth.

## 2026-05-09 тАФ M7.1.c.a: CreateApp saga step + watchkeeper secrets column + migration

**PR**: pending тАФ to be opened in Phase 5b
**Merged**: pending тАФ to be merged in Phase 7a

### Context

M7.1.c.a wired the first concrete saga.Step impl (CreateApp) bridging the M6.1.b SlackAppRPC privileged RPC to a new watchkeeper.slack_app_creds Postgres table via the existing M4.2.d sink-adapter contract. Introduced saga.SpawnContext value type for cross-step parameter passing and stabilized manifest_version_id + agent_id access across future steps.

### Pattern

**Sink-to-DAO bridge owns watchkeeperID, not appID**: A saga step that bridges an OUT-OF-BAND credential delivery (M4.2.d sink) to a persistence DAO must close over the saga's primary key (watchkeeperID from SpawnContext), NOT the credential's own assigned id (creds.AppID). Re-create scenarios change AppID; watchkeeper id is stable across the saga's lifetime. The DAO column layout reflects this: watchkeeper_id is PK, app_id is a column.

**SpawnContext as saga-level cross-step seam**: Future M7.1.x steps need read access to manifest_version_id, agent_id, and the dispatcher's Claim. Define a typed value (saga.SpawnContext) + accessors (WithSpawnContext / SpawnContextFromContext) in the saga package тАФ NOT in the spawn package тАФ so step impls have a stable contract that does not couple them to step-specific constructors.

**Source-grep AC pinning for negative AC ("does NOT call X")**: When an AC asserts that a unit must NOT call some API, the test that pins it cannot be a runtime mock тАФ even an injected fake passes vacuously if the unit reaches the API via a different path or is later refactored. The robust pin is os.ReadFile on the unit's source + strings.Contains outside //-prefixed lines. Mutation-tested (adding the forbidden call must cause the test to fail).

**Migration ships with deferred encryption-at-rest comment**: Per the M6.3.b "ship in-memory DAO with consumer" rhythm, M7.1.c.a's migration uses text columns for credentials with an explicit -- TODO: encryption-at-rest deferral comment. The Phase-2 secrets pass will rotate to encrypted bytea. Keeps the M7.1.c.a PR scope tight.

### Anti-pattern

Hollow runtime-mock AC pins. The iter-1 reviewer caught a keeperslog.New(keep) that was constructed and immediately discarded (\_ = ...) тАФ the recordingKeepClient could never observe the call because the writer was never injected. The fix replaced it with a source-grep mutation-tested assertion. Lesson: any AC pin that depends on observation through an injected dependency must verify the dependency reaches the unit-under-test; otherwise prefer source-grep.
