### Iter-1 patterns worth carrying forward

1. **Umbrella + leaf sentinel chaining via Go 1.20+ multi-wrap is
   the canonical shape for "operator triage AND per-row diagnostic"
   error surfaces.** A single-wrap leaf forces the caller to know
   every leaf shape (or write a brittle string-match). A single-
   wrap umbrella loses per-row diagnostic detail. The double-wrap
   shape (`fmt.Errorf("%w: %w: …", umbrella, leaf, detail)`) lets
   operators triage on the umbrella AND specific-error branches
   match on the leaf. `TestIter1_EveryLeafChainsToUmbrellaSentinel`
   is the load-bearing pin — every leaf-producing site MUST chain
   to the umbrella, and the umbrella MUST be matchable on every
   leaf path.

2. **I/O failure is operationally different from content-malformed
   failure; one umbrella sentinel for both is a category error.**
   `LoadFromFile` mid-iter-1 wrapped read errors under
   `ErrInvalidDictionary` — making "file missing" indistinguishable
   from "yaml malformed". An operator catching the structural
   sentinel for triage would mis-route on the wrong cause. Fix:
   distinct `ErrDictionaryRead` sentinel for the I/O failure mode;
   the path that emits it does NOT chain to `ErrInvalidDictionary`.

3. **`utf8.ValidString` upfront is the right shape for
   "validate utf-8 once, then walk bytes for control-byte checks".**
   The pre-iter-1 code used the range-over-string `r ==
unicode.ReplacementChar` check, which conflates "invalid utf-8
   byte sequence" with "legitimately-encoded U+FFFD codepoint".
   Splitting the two checks lets legit U+FFFD flow through (i18n-
   friendly) while still rejecting truly-malformed bytes.

4. **Map-iteration nondeterminism in error-message paths is
   user-facing.** A validator that walks `map[string]V` and
   surfaces the FIRST failure emits run-to-run-different messages
   on identical input. The fix is one `sort.Strings(ids)` line
   before the walk; the test shape is "run the failing payload
   N times and assert the message is byte-for-byte identical
   across runs". `TestIter1_MultiBadEntry_FirstFailureIsDeterministic`
   is the shape.

5. **Distinct length-sentinels for "id too long" vs "description
   too long" beat a shared umbrella when callers want to branch.**
   Pre-iter-1, id-too-long had `ErrInvalidCapabilityID`;
   description-too-long had only the umbrella. Asymmetry creates
   API drift over time (a future reader assumes there's a
   description-length sentinel by analogy with the id-length one).
   `ErrDescriptionTooLong` closes the asymmetry; both length paths
   now chain umbrella + leaf.

6. **Hoist user-visible string literals to package-level consts
   once they appear in two or more sites.** Pre-iter-1, the
   placeholder text appeared in `card.go:388`, `card_test.go:179`,
   and `card_capdict_test.go`. The em-dash (U+2014) is a
   particularly bad shape to leave duplicated — an editor that
   auto-converts ↔ en-dash would silently break either site.
   `FallbackTranslationDictionaryNotLoaded` /
   `FallbackTranslationNotRegistered` consts hoist the shape once.

7. **A "by-design" callout in the lesson trail beats a per-test
   one-line apology.** The per-package `stripGoComments`
   duplication across M9.5 / M9.6 / M9.7 / M9.3.a audit-grep tests
   is deliberate (package boundary). A future contributor reaching
   audit_grep_test.go via the lesson trail sees the duplication
   and either mistakes it for drift OR proposes a "DRY" refactor.
   One paragraph in the lesson entry explaining the boundary
   choice short-circuits both paths.

### Context

M9.3.a closes the dictionary half of the M9.3 parent. The signing
half (cosign / minisign verifier + per-source public keys) is
formally decomposed into M9.3.b and deferred to Phase 2 per the
existing `docs/lessons/M9.md:290` note. The seam for signing
already exists at M9.1.a (`SignatureVerifier` interface +
`NoopSignatureVerifier` default + `ErrSignatureVerification`
sentinel + `source_failed` phase=signature wiring), so Phase 2
lands the real verifier without disturbing the dictionary code.

The dictionary is the lead-facing piece: the M9.4.b approval card
renders one bullet per capability id, and a non-engineer lead
clicks `[Approve]` without seeing raw code. Until M9.3.a, the
renderer carried a documented "translation pending — M9.3"
fallback placeholder — operators saw the raw cap id and knew the
dictionary was not yet wired. M9.3.a wires the real translator,
keeping the fallback as a degradation path (now renamed to
`translation pending — dictionary not loaded` to reflect the post-
M9.3.a operator semantics: the placeholder only fires when the
caller has not yet loaded the dictionary, not because M9.3 is
unshipped).

The Go canonical set in `core/pkg/capdict/canonical.go` is the
authority; the YAML on disk is the lead-facing translation
artefact. A two-way bijection test in `canonical_test.go` pins
both sides — adding a capability requires one Go-slice entry AND
one yaml row. Same shape as M9.7's
`TestEventTypeVocabulary_RoadmapNames` (lesson #6 — a two-way
closed-set pin catches asymmetric drift).

### Pattern

1. **A two-way closed-set bijection is the CI completeness gate.** The forward direction ("every canonical id has a yaml entry") catches a yaml miss when a new capability lands. The reverse direction ("every yaml entry is in canonical") catches a Go-side miss when a yaml entry is added without the matching canonical-slice update. The `len()` equality pin catches the degenerate case where forward + reverse pass but one side carries a duplicate masking a miss. `core/pkg/capdict/canonical_test.go:TestCapabilities_BijectionWithCanonicalSet` mirrors M9.7's `TestEventTypeVocabulary_RoadmapNames`.

2. **A pure-decoder package needs an INVERSE-shape audit-grep.** M9.5 / M9.6 emitter packages assert their production code DOES NOT import `keeperslog` (they emit events; the M9.7 subscriber is the writer). M9.7 inverts: its production code MUST import `keeperslog` (it IS the writer). M9.3.a is a third shape: the decoder NEVER writes audit rows at all — the dictionary-loaded outcome is configuration, not event-shaped. The negative-shape audit-grep (assert NO keeperslog import, NO `.Append(` call) pins the discipline against a future maintainer plumbing audit through the loader.

3. **Per-call resolver-shape closure beats process-global static even for a "global config" decoded once at boot.** A `var Translator func(string) string` package-level singleton would force every tenant to share one dictionary. The `capdict.Translator(*Dictionary) func(string) string` factory returns a fresh closure bound to a specific Dictionary value; a multi-tenant deployment loads per-tenant dictionaries at boot and routes per-call. Mirrors M9.1.a's `AuthSecretResolver` / M9.4.a's `IdentityResolver` shape.

4. **Cross-package func-type assignability removes the need for a shared interface package.** `approval.CapabilityTranslator` is a named `func(string) string` type. `capdict.Translator(*Dictionary)` returns an UNNAMED `func(string) string` value. Go assignability lets the unnamed value satisfy the named parameter at the call site without either package importing the other (capdict must stay a pure decoder; approval must not depend on capdict's loader). Pinned at compile time by a `var _ approvalLikeTranslator = Translator(nil)` declaration in `translator_test.go` against a locally-redeclared duplicate of the approval type — the duplicate is intentional, not drift.

5. **Strict yaml.v3 decoding catches typos at boot, NOT at first use.** `yaml.NewDecoder(...).KnownFields(true)` rejects unknown top-level keys (`capabilites:`) AND unknown per-row fields (`descriiption:`) at decode time. Same discipline as `core/pkg/toolregistry/yaml.go:DecodeSourcesYAML`. The alternative — accepting unknown keys and silently dropping them — produces a dictionary that "loads fine" but renders every cap as the "no translation registered" fallback, a class of operator-confusion bug the strict decode prevents at source.

6. **Versioned schema field unlocks additive evolution without back-compat shims.** The `version: 1` field is enforced via [SupportedSchemaVersion] in Phase 1. Future locale keys (e.g. `descriptions_i18n: {ru: "...", de: "..."}`) land as additive optional fields inside the per-id object — KnownFields stays compatible with new field names by relaxing the decoder OR by introducing a `version: 2` decode path. Operators upgrading carry the same yaml file forward; the loader's version check fails loudly if an operator points an old binary at a v2 dictionary.

7. **A renamed fallback placeholder reflects the post-ship operator semantics.** Pre-M9.3.a, the card renderer carried `translation pending — M9.3`; the placeholder meant "M9.3 is not yet shipped, the dictionary doesn't exist". Post-M9.3.a, the dictionary EXISTS and the fallback should only fire when a caller has not yet LOADED it (boot ordering, test harness without a yaml fixture). Renamed to `translation pending — dictionary not loaded`. The empty-string-from-wired-translator path stays distinct as `no translation registered` so a bijection-violation case is operator-distinguishable from the boot-ordering case. Two distinct placeholders for two distinct failure modes.

## 2026-05-14 — M9.7: Keeper's Log audit surface (`core/pkg/auditsubscriber` package — single eventbus→keeperslog bridge for 11 M9 audit topics)

**PR**: #142
**Merged**: 3b0037f on 2026-05-14

### Review iter-1 fixes

Two parallel reviewers (codex correctness + critic-style) flagged
3 Major + 12 Minor + 7 Nit after dedup; the operator chose
"All findings" and the union was applied on the same branch before
commit:

- **M1 (codex) — `Start` did NOT invoke the unsubscribe returned by
  the FAILING `Bus.Subscribe` call.** The eventbus.Bus contract
  documents the error-path return as a no-op closure, but the
  `Bus` seam is a public interface — a future non-eventbus.Bus
  implementation that PARTIALLY registered a handler before
  returning an error would leak a live subscription past the
  rollback. Fix: capture the unsub from the failing call and
  invoke it before walking the prior-row rollback
  (`subscriber.go:Start`). `fakeBus.failingUnsubCount` test fake
  pins the defensive call.

- **M2 (codex) — Every extractor accepted only the concrete value
  type `T`, not `*T`.** A future emitter refactor from
  `Publish(..., ev)` to `Publish(..., &ev)` would compile and
  silently log+drop every audit row on that topic. Fix: replaced
  the 11 hand-rolled extractor funcs with a single generic
  `makeExtractor[T any]` that accepts BOTH `T` and `*T` (`*T` nil
  is type-mismatched — no fields to surface). `TestExtractor_AcceptsValueAndPointer`
  pins both shapes × all 11 bindings; `TestExtractor_NilPointer_TypeMismatch`
  pins the nil-pointer boundary.

- **M3 (critic) — `ErrUnexpectedPayload` exported, wrapped via
  `%w`, but never observable or testable.** The dispatch closure
  consumed the wrapped err only to log `err_type` (always
  `*fmt.wrapError` — zero diagnostic signal). The sentinel was
  dead API surface. Fix: unexported to `errUnexpectedPayload`;
  added `TestExtractor_WrapsSentinel` asserting `errors.Is`
  matches; replaced the useless `err_type` log field on the type-
  mismatch path with `expected_type` (carried on the new
  `binding.ExpectedType` field) — a useful operator triage field.

- **m1 (codex) — append-failure logf used the raw bus `ctx`, not
  `appendCtx`.** A `Logger` impl that reads correlation id from
  ctx would record the wrong id on the exact path operators need
  to debug. Fix: use `appendCtx` in the failure logf
  (`subscriber.go:dispatch`). `TestDispatch_AppendFailureLogf_UsesAppendCtx`
  pins via a hand-rolled `correlationCapturingLogger`.

- **m2 (codex) — `Start` wrapped + returned the `Bus.Subscribe`
  error verbatim.** With `*eventbus.Bus` this is benign today, but
  a future non-eventbus.Bus impl might embed credentials / URLs /
  payload data in its error strings. Fix: introduced
  `ErrSubscribe` sentinel + private `subscribeError` wrapper whose

### Iter-1 patterns worth carrying forward

1. **An untyped `any` bus surface forces the bridge to accept
   BOTH value and pointer publishes.** A package whose whole job
   is cross-package bridging cannot rely on every emitter author
   to know the value-vs-pointer convention. `makeExtractor[T]`
   accepts both `T` and `*T`; a future `Publish(..., &ev)`
   refactor lands one place safely.

2. **An exported sentinel that the dispatcher does not surface to
   callers is dead API surface.** If the only consumer of an
   error chain is the local logger, unexport the sentinel and
   keep the `errors.Is`-matching contract via an in-package test.
   Otherwise the exported symbol implies an observability contract
   the implementation does not deliver — a future reader writes
   a useless `errors.Is(err, MySentinel)` check that always returns
   false.

3. **Diagnostic surfaces (Logger) need their own redaction
   discipline distinct from the audit-write path.** The
   `Writer.Append` call IS the audit purpose; payloads flow
   verbatim. The Logger is NOT the audit purpose — its entries
   land in operator log aggregators that typically do not redact
   at the same level. Every M9.7 logf carries metadata-shaped
   fields only (topic, event_type, expected_type, got_type
   via `%T`, err_type via `%T`); never the payload body, never
   the error VALUE. Two-axis canary tests (logger redaction +
   writer pass-through) pin both sides per topic.

4. **The `Bus` seam contract is implicit; defensive-unsub closes
   the hole.** `eventbus.Bus.Subscribe` documents the error-path
   unsubscribe as a no-op closure — relying on this for the
   atomic-install rollback is correct today but couples the
   subscriber to the eventbus impl. Calling the failing-Subscribe's
   returned unsub is one extra line + a `fakeBus.failingUnsubCount`
   counter; in exchange the subscriber stays correct against
   any future non-eventbus.Bus implementation.

5. **A redacted-`Error()` wrapper preserves classification AND
   redaction.** When wrapping a downstream-impl error that MIGHT
   embed credentials in its `.Error()` string, a custom error
   type with `Unwrap()` (preserves `errors.Is` / `errors.As`
   chain) but a redacted `Error()` method (carries only Go TYPE
   name + caller-known identifiers) gets both wins. Generalises
   to any future bridge over an interface boundary.

6. **A two-way closed-set vocabulary pin catches asymmetric
   drift.** A one-direction check ("every roadmap name is in
   allBindings") passes when a binding is silently added with a
   typo'd EventType. The reverse direction ("every allBindings
   EventType is in roadmap") fails. Both halves together (plus
   the `len()` symmetric assertion) form a tight closed-set
   contract.

7. **Test-only method on the same-package receiver beats
   production-code test-only surface.** A `func (s *Subscriber)
testUnsubsCount() int` defined in `*_test.go` reaches into
   the private `s.unsubs` field without polluting the exported
   API. Same package = same access. The production type stays
   minimal.

### PII boundary

The audit subscriber is intentionally a thin pass-through: every bus
payload arrives at `keeperslog.Writer.Append` VERBATIM (no field
reshape, no normalisation, no scrubbing). The emitter packages have
already applied per-payload PII allowlists — re-doing the work HERE
would diverge from the on-bus shape and confuse audit consumers.

What the M9.7 subscriber DOES control: the diagnostic `Logger`
surface. Both failure paths (`unexpected payload` and `append
failed`) emit ONLY metadata-shaped fields:

- `topic` — the package-prefixed eventbus topic string.
- `event_type` — the audit-vocabulary bare verb-phrase form.
- `got_type` — `fmt.Sprintf("%T", event)` (Go type name only, never the
  value).
- `err_type` — `fmt.Sprintf("%T", err)` (Go type name only).

Never the payload body. Never the error VALUE. Per-topic PII canary
tests (`TestPIICanary_LoggerOmitsAllPayloadFields_OnAppendFailure`,
`TestPIICanary_LoggerOmitsAllPayloadFields_OnTypeMismatch`) stuff
canary substrings into payload fields AND make the writer fail, then
assert the captured logger entries contain none of the canaries. The
mirror canary (`TestPIICanary_PayloadPassedThroughVerbatim`) asserts
the same canary substrings DO arrive on the writer's `Append.Payload`
— locking the pass-through contract from both sides.

### Concurrency + lifecycle

- `Subscriber` is single-use. `Start` ↔ `Stop` is a one-way
  transition; once `Stop` has run, `Start` returns `ErrStopped`.
- `Start` is atomic: any `Bus.Subscribe` failure unsubscribes every
  prior handler before returning. Same discipline as
  `cron.Scheduler.Start` / `lifecycle.Manager.Start`.
- `Stop` is idempotent (multiple calls return nil without re-running
  the unsubscribe loop). Stop releases the mutex BEFORE invoking the
  per-subscription unsubscribe callbacks so a slow `Bus` impl can't
  deadlock against a concurrent `Start`.
- Handlers run on `*eventbus.Bus` per-topic worker goroutines.
  Sequential within a topic; concurrent across topics. The
  `TestDispatch_ConcurrencyNoRace` test exercises 16 goroutines × 25
  dispatches on a single topic under `-race`.
- A failing `Writer.Append` LOGS metadata + DROPS the event. This is
  the M9.6 "best-effort side surfaces" lesson (Slack-DM) generalised:
  blocking the topic worker on audit failure would propagate
  backpressure into emitters, breaking the M9 contract that
  publishing is non-blocking.
- A type-mismatch on the bus LOGS metadata + DROPS the event. The
  bus is untyped (`event any`); a misconfigured publisher or a test
  pumping the wrong shape is the failure mode. Panicking would kill
  the topic worker (see `eventbus.Handler` godoc).

### Pattern

1. **An inverse-shape audit-grep test pins the bridge contract.**
   M9.5/M9.6 emitter packages assert their production code does
   NOT import keeperslog and does NOT call `.Append(`. M9.7 is the
   one layer down — its production code MUST do both. A
   `TestAuditGrep_KeeperslogImportPresent_AppendCallPresent` test
   (positive: at least one production file imports `keeperslog`;
   at least one calls `.Append(`) is the symmetric pin. Together
   the two flavours of audit-grep enforce the one-way audit flow:
   `emitter → eventbus → auditsubscriber → keeperslog → keepers_log`.
   A future refactor that accidentally moves the writer call to a
   different package surfaces in M9.7's audit-grep (no `.Append(`
   left in this package).

2. **Bare audit-vocabulary constants belong to the subscriber, not
   the emitter.** The M9 emitter packages publish package-prefixed
   topic strings (`toolregistry.source_synced`). The M9.7 audit
   subscriber translates each to the bare verb-phrase form
   (`source_synced`) for the `keepers_log.event_type` column. Owning
   the bare forms in `core/pkg/auditsubscriber/topics.go` (not on
   the emitter packages) means a future event-type rename is a
   one-file change AND the emitter packages stay decoupled from
   the audit-vocabulary. The `TestEventTypeVocabulary_RoadmapNames`
   test pins the bare forms against the roadmap line 512 list.

3. **Per-call seams + thin pass-through beats centralised
   normalisation.** Each topic's extractor type-asserts to the
   declared payload struct and forwards the value VERBATIM to
   `keeperslog.Event.Payload`. The subscriber does NOT reshape /
   normalise / sanitise — each emitter has already applied its own
   PII allowlist. Re-doing the work here would diverge the audit
   row's JSON shape from the on-bus shape, confusing downstream
   consumers (audit dashboards joining on the proposal-id field
   would not know which shape to expect). The
   `TestPIICanary_PayloadPassedThroughVerbatim` test asserts the
   pass-through contract by stuffing canary substrings into bus
   payload fields and checking they arrive on the writer's
   captured Append payload.

4. **Diagnostic Logger surface enforces redaction independent of
   the audit-write path.** The audit `Append` call IS the audit
   purpose: a Reason field with operator accountability text SHOULD
   land in `keepers_log.payload.data`. But the SAME field MUST NOT
   land in a diagnostic Logger entry — operator log aggregators
   typically do not redact at the same level. Both M9.7 failure
   paths log ONLY metadata-shaped fields (topic, event*type,
   got_type as `%T`, err_type as `%T`); the
   `TestPIICanary_LoggerOmitsAllPayloadFields*\*` tests pin this on
   both failure paths × all 11 topics.

5. **Best-effort audit beats topic-worker backpressure.** A failing
   `Writer.Append` (keepclient outage, postgres lock contention,
   etc.) LOGS + DROPS the event. Returning the error from the
   handler would not propagate to the bus (Handlers have no return
   value), so the alternative would be to retry with a backoff
   internally — at which point a slow keep server stalls every
   subsequent audit dispatch on the SAME topic worker. The M9.6
   "Slack DM is best-effort, not the primary outcome" lesson
   generalises: any side surface that can stall MUST NOT block the
   primary outcome.

6. **Atomic install + idempotent teardown for orchestrator
   lifecycles.** `Start` subscribes 11 handlers in a single mutex-
   guarded transaction; on any `Bus.Subscribe` failure every prior
   subscription is rolled back. `Stop` is idempotent (the second
   call is a no-op). The lifecycle state is a simple `(started,
stopped)` pair guarded by `sync.Mutex`; the mutex is released
   BEFORE the unsubscribe callbacks fire so a slow Bus impl cannot
   deadlock against a concurrent `Start`. Same discipline as
   `cron.Scheduler.Start` + `lifecycle.Manager.Start`.

7. **Real-bus end-to-end test pins the seam compatibility.** The
   `*eventbus.Bus` interface satisfaction is asserted at compile
   time (`var _ Bus = (*eventbus.Bus)(nil)`), but a future Bus.
   Subscribe signature change OR a per-topic-worker behaviour
   change would still slip past the type system. A single
   `TestDispatch_WithRealEventbus_EndToEnd` test wires the real
   bus, publishes one event, polls the writer with a 2s deadline,
   and asserts the captured event_type + correlation id. The poll
   shape (`for time.Now().Before(deadline)` + 5ms sleep) tolerates
   the bus's per-topic-worker async dispatch.

### Iter-1 patterns worth carrying forward

1. **A "concurrent-safe" or "OOM-safe" claim requires the bound
   to fire BEFORE the side-effect.** `walkShareFiles` collecting
   all file contents before checking `MaxFilesPerShare` was a
   classic "check after the damage was done" pattern. Move the
   bound inside the walker; bail before the next `ReadFile`.
   Generalises to any future tree walker handling untrusted
   input sizes.

2. **A per-process atomic nonce is durable across calls but NOT
   across restarts.** The branch-name suffix needs the
   nanosecond timestamp prefix to survive process restarts —
   process restarts never reuse a nanosecond, and within-process
   the atomic nonce already guarantees uniqueness. Two-axis
   uniqueness (time + atomic) covers both windows.

3. **Validators on a public API surface must be symmetric across
   read AND write paths.** `validateRefRelative` (GET) refused
   `..`/whitespace/control-chars; `validateRefFull` (POST) did
   not — strictly weaker. A shared helper (`refContentBytesValid`)
   plus a uniform validator-call discipline keeps the surface
   defensible.

4. **CLI flag-validation must accumulate ALL missing-flag /
   bad-value diagnostics in one round-trip** (M9.5 m4 lesson
   restated). An early-return on `--target invalid` ate the
   missing `--reason` complaint AND echoed the bad value
   verbatim. Treating bad-value as a missing-flag entry both
   accumulates AND skips the verbatim echo.

5. **An RPC handler's error-code-to-sentinel doc-block IS the
   wire contract** (M9.4.d Pattern 1 restated). The
   `mapShareCapabilityError` listed three unreachable cases —
   a future reader would assume the wire contract included them.
   `capability.Broker.Validate`'s real surface is the
   one source of truth; the doc-block + the mapper switch + the
   error tests must agree.

### Context

M9.6 closes the hosted ↔ git migration + tool-sharing surfaces of the
M9 self-modification loop in a single PR. Three Go packages and one
RPC handler land:

1. **`core/pkg/hostedexport/`** — operator-driven hosted-tool
   exporter. Reads `<DataDir>/tools/<hosted-source>/<tool>/`,
   copies the tool tree (manifest + src + tests) to an operator-
   supplied destination as a self-contained plain-directory bundle.
   Emits `hostedexport.hosted_tool_exported` with operator id,

   cleanly into a fresh git repo (`cd <dest> && git init && git

2. **`core/pkg/github/` write extensions** — four new methods on

     tree-commit chain would land all files in one commit but adds

3. **`core/pkg/toolshare/`** — orchestrator package that

### Pattern

1. **Per-package re-export of the shared FS interface beats
   sibling-package coupling.** `hostedexport.FS` and
   `toolshare.FS` are both `type X = localpatch.FS` aliases —
   single source of truth for the `[MkdirAll, ReadDir, ReadFile,
Stat, Lstat, WriteFile, RemoveAll]` shape across the M9
   on-disk tree walkers. Same lesson as M9.5's `OSFS`
   re-export from `toolregistry.OSFS`. Production wiring lands
   ONE `localpatch.OSFS{}` value at the CLI; every M9 package
   accepts it. Defining a parallel `FS` interface per package
   would have forced the CLI to wrap `OSFS` three times.

2. **Best-effort side surfaces (Slack DM) MUST NOT fail the
   primary outcome.** `Sharer.Share`'s Slack-DM step is gated by
   `notifyLead` which logs failures but never returns them. The
   PR open IS the durable outcome of `Share` — a Slack outage
   that masked the success would force operators to triage a
   failed-share-that-actually-succeeded scenario. The
   `ShareResult.LeadNotified` bool surfaces the notification
   state to the caller without coupling the contract.
   Generalises to any future best-effort observability hook.

3. **Github write-API chain produces N+3 round-trips per share.**
   The chain is `GetRef → CreateRef → CreateOrUpdateFile × N →
CreatePullRequest`. N is the regular-file count (manifest +
   src + tests, typically 5-30 for a Phase 1 tool). Each file
   becomes ONE commit on the share branch; the PR shows N
   distinct commits. Trade-off vs the Git Data API blob-tree-
   commit chain (which produces 1 atomic commit but adds 3
   round-trips of its own and 4 wire types): for the M9.6
   single-author share flow the per-file commit is operator-
   readable in the PR diff AND simpler to test (4 endpoint shapes
   vs 7). A future refactor lands the Git Data API path behind a
   `Sharer.UseAtomicCommit bool` flag when a real customer demands
   a single-commit history.

4. **MaxFilesPerShare (256) is a pre-flight defence against
   half-uploads.** The github API chain fails atomicity at the
   per-file commit boundary: a network outage between
   `CreateOrUpdateFile` calls leaves a half-populated share
   branch on the target repo. Refusing 256+ files pre-flight
   bounds the operator's manual-cleanup blast radius (`gh
pr close --delete-branch` is one shell command). Tool trees
   regularly under this bound — the M9.4.d CI template's
   `undeclared_fs_net` walker imposes its own structural limits
   on what a "tool" can contain.

5. **Per-call resolver beats process-global static for
   tenant-scoped values.** All five per-call seams on `Sharer`
   (`SourceLookup`, `ProposerIdentityResolver`,
   `TargetRepoResolver`, `LeadResolver`, plus the `TokenSource`
   underlying `GitHubClient`) are per-invocation function-shape
   seams. A tenant rotation (new lead admin, new platform repo
   owner, new PAT) takes effect on the NEXT `Share` call without
   a process restart. Same lesson as M9.1.a's `AuthSecretResolver`,
   M9.5's `OperatorIdentityResolver`. Codex iter-1 has reliably
   flagged process-global static across every M9 step — apply
   pre-emptively.

6. **Closed-set discriminator on payload beats expanding the
   topic vocabulary** (carried forward from M9.5 Pattern 2). The
   `TargetSource` enum (`platform` | `private`) rides on the
   `ToolShareProposed` payload via the typed `TargetSource` field.
   A downstream "filter shares to platform only" subscriber
   gates on `TargetSource` rather than subscribing to a separate
   topic. Avoiding the 2-topic expansion (`tool_share_proposed_platform`
   / `tool_share_proposed_private`) keeps the M9.7 audit subscriber
   wiring single-topic.

7. **Refuse-on-non-empty destination is symmetric with
   refuse-on-undecidable live tree** (M9.5 Pattern 7 generalised).
   `hostedexport.Exporter.Export` refuses with
   `ErrDestinationNotEmpty` when the operator-supplied destination
   exists AND contains entries. Overwriting an arbitrary on-disk
   tree would be an irreversible operator surprise — same shape
   as M9.5's "refuse install when the prior live tree is
   undecidable" discipline. Empty existing directories ARE
   accepted (operators commonly `mkdir -p` ahead of time).

### Review iter-1 fixes

Two parallel reviewers (codex-style correctness + critic-style PII /

Minor + 3 Nit after dedup; the operator chose "All findings" and the

- **C1 — Symlink cycle detection used `filepath.Abs`, not a realpath
  visited set; doc-block claim was empirically wrong.** Codex flagged
  that `walkInto`'s `filepath.Abs(dir)` is purely lexical — it does
  NOT resolve symlinks — so a cycle `a/b → ../a` keyed distinct
  visited entries and recursion fell through to a stack-overflow on
  OSFS. Critic independently flagged the same gap as Minor. Fix:
  replaced the cycle-defence with a non-following [`FS.Lstat`] /

  other non-regular non-directory entries) are skipped silently. A
  cycle is impossible by construction. New `FS.Lstat` method on the
  seam interface + `OSFS.Lstat` production-shim
  (`osfs.go:35-39`, `digest.go:179-217`). The lesson Pattern 5
  documents the refusal-to-follow stance verbatim.

- **M1 — `TestInstall_PublishUsesContextWithoutCancel` was a no-op
  assertion.** Codex flagged that the prior shape cancelled the

  the captured value was always nil regardless of whether the
  implementation used `context.WithoutCancel` or the parent ctx
  verbatim. Fix: new `onWrite` hook on `fakeFS` cancels the parent
  ctx DURING the manifest write (a side-effect that lands BEFORE
  the publish); the publisher then runs under an already-cancelled
  parent ctx and the assertion verifies the publish call site's
  `ctx.Err()` is nil — i.e. the publish IS detached
  (`installer_test.go:TestInstall_PublishUsesContextWithoutCancel`,
  `fakes_test.go:fakeFS.onWrite`).

- **M2 — Rollback over an undecidable live tree silently destroyed
  the prior contents.** Codex flagged the asymmetry with the
  installer's `TestInstall_PriorTreeUndecidableRefused`: install

  target is always preserved); rollback was PROCEEDING. Critic
  independently flagged the same gap. Fix: `readLiveVersion` error
  now propagates verbatim — rollback surfaces `ErrManifestRead`
  BEFORE the destructive `replaceLive`. Symmetric with install
  (`rollback.go:Rollback`,
  `rollback_test.go:TestRollback_UndecidableLiveTreeRefused`).

- **M3 — Buggy `SourceLookup` returning a different `Name` was
  misclassified as `ErrUnknownSource`.** Codex flagged that the
  defensive `if src.Name != req.SourceName` branch wrapped
  `ErrUnknownSource`, but the source IS known — the resolver
  returned the wrong record. An operator's `errors.Is` triage would
  route to the wrong place. Critic flagged the same branch as
  untested. Fix: new `ErrSourceLookupMismatch` sentinel

### Iter-1 patterns worth carrying forward

1. **A "cycle-safe via realpath visited set" claim is only
   defensible if the FS seam exposes an EvalSymlinks-equivalent
   method.** `filepath.Abs` is lexical and provides ZERO cycle
   defence on OSFS; the in-memory fake hid the gap because it has
   no symlinks. The cleaner discipline: REFUSE to follow symlinks
   in the walker — symlinks become silently-skipped entries (like
   sockets / fifos / devices) and a cycle becomes impossible by
   construction. Document the refusal-to-follow stance explicitly
   in both the walker godoc AND the digest's user-facing
   contract.

2. **Per-target mutex MUST be acquired AFTER the resolved key is
   known.** The lock key is `(source, tool)` but the tool name
   comes from the folder manifest, which the installer decodes
   AFTER validation + source-lookup + identity-resolve. Acquiring
   earlier (e.g. on `req.SourceName` alone) would serialise
   different-tool installs against the same source unnecessarily;
   acquiring later (after `ContentDigest`) would let two same-tool
   installs race on the digest read. The right place is "after
   manifest decode, before any side-effect on the live tree".

3. **A CLI exposing a package's operator boundary MUST validate
   inputs against the package's bounded contracts at the
   flag-parse layer, not rely on the package's post-resolver
   diagnostic.** The package error message is for in-process
   triage; a CLI wrapper running `wk-tool: install: %v` lands the
   bad-input on stderr verbatim, which is hygiene-bad for any
   operator wrapper that captures stderr. Mirror the package's
   `Validate` discipline AT the CLI boundary
   (`ValidateOperatorID`, env-allowlist gate).

4. **An "operator-visible" output surface is a PII leak vector
   even when the in-process payload is metadata-only.** The bus
   payload allowlist defends the bus subscriber's view; the JSONL
   stdout publisher defends the operator's view. They are
   distinct surfaces and BOTH need canary coverage — a future
   envelope-wrapper change could surface fields the payload
   struct never declared.

5. **A "safe for concurrent use" doc claim is a load-bearing
   contract.** Code that says it is safe must BE safe even on
   OSFS — not just on the test FS that happens to serialise via a
   single mutex. When the runtime IS designed for one-shot CLI
   dispatch only, the doc should say so explicitly OR a per-target
   mutex should land. Pick one.

### Context

M9.5 closes the operator-driven self-modification loop for `kind: local`
tool sources. Two operator-facing surfaces ship in one atomic pass:

1. **`make tools-local-install`** — copies an operator-supplied folder
   into `<DataDir>/tools/<source>/<tool>/`, snapshotting the prior
   contents under `<DataDir>/_history/<source>/<tool>/<priorVersion>/`
   (sibling to `tools/`, NOT under it — see Pattern 1) so a later
   rollback can restore them. Emits `local_patch_applied` with
   operator identity, content SHA256 (`diff_hash`), and the operator-
   supplied `--reason` text.

2. **`wk-tool rollback`** — restores a previously-snapshotted version
   into the live tools directory AND snapshots the current live tree
   under its own version BEFORE the overwrite (so the operator can
   un-rollback). Emits the SAME `local_patch_applied` topic with
   `Operation=rollback` so a downstream M9.7 audit subscriber sees a
   single chronological feed for the local-patch lifecycle.

Concrete artifacts:

1. **New package `core/pkg/localpatch/`** — `Installer` + `Rollbacker`
   orchestrators with parallel deps shapes (FS / Publisher / Clock /
   SourceLookup / OperatorIdentityResolver / DataDir + optional
   Logger). Per-call seam discipline mirrors M9.1.a (`AuthSecretResolver`),
   M9.4.a (`IdentityResolver`), M9.4.b (`WebhookSecretResolver` /
   `SourceForTarget`), M9.4.c (`ScopeResolver`): every tenant-scoped
   value is a per-invocation function-shape seam, never pinned to the
   constructor.

2. **New binary `core/cmd/wk-tool/`** — `local-install` and `rollback`
   subcommands; flag-driven; JSONL stdout publisher emits
   `localpatch.local_patch_applied` events one per line for downstream
   audit-log capture (production wiring to the durable outbox is M9.7
   work). Exit codes mirror `core/cmd/spawn-dev-bot/` (0 success,
   1 runtime, 2 usage). Operator id resolved via a trust-the-hint
   resolver in the bundled CLI; hosted deployments swap in a
   verified-OIDC resolver via the constructor seam.

3. **Makefile target `tools-local-install`** — wraps the binary's
   `local-install` subcommand with a `--reason`-required guard
   (the env-var literal `?:` syntax fails fast with a stable
   diagnostic when REASON / FOLDER / OPERATOR / WATCHKEEPER_DATA_DIR
   are missing — matches `spawn-dev-bot`'s discipline).

4. **Single-topic audit channel** — `localpatch.local_patch_applied`
   handles BOTH install AND rollback. The `Operation` field on the
   payload (closed-set: `install` | `rollback`) discriminates; M9.7's
   audit subscriber splits the feed without parsing free-form text.
   The roadmap's M9.7 list pins `local_patch_applied` as the only
   local-patch topic, so a `tool_rolled_back` topic would have
   forced a roadmap edit and a second subscriber wiring — single
   topic with a discriminator field is the proportional choice.

The boundary at M9.5 is deliberately upstream of the durable outbox:
the `Publisher` seam consumes `(ctx, topic, event)` and any production
wiring (the M9.7 outbox bridge, an in-process `eventbus.Bus`, a JSONL
writer) plugs in without changing the package surface. The CLI's
JSONL stdout publisher is the M9.5 default; the operator pipes it to
their audit-log capture path until M9.7 lands the durable bridge.

### PII boundary

The `LocalPatchApplied` payload is a 9-field allowlist: `SourceName`,
`ToolName`, `ToolVersion`, `OperatorID`, `Reason`, `DiffHash`,
`Operation`, `AppliedAt`, `CorrelationID`. Two deliberate departures
from the M9.4.a/b/c metadata-only-payload rule:

- **`Reason` is verbatim free-form text.** The operator's
  accountability statement IS the audit purpose — redacting it
  would defeat the entire point of the audit channel. Bounded by
  `MaxReasonLength` (1 KiB) to defend against a buggy CLI invocation
  landing an unbounded payload.

- **`OperatorID` is verbatim.** The resolved operator id is a stable
  public identifier (UUID, agent handle, human handle), explicitly
  NOT a credential — the resolver's contract names this and the
  installer enforces both a length bound (`MaxOperatorIDLength` = 64)
  and a character allowlist (`validOperatorID`) that refuses
  credential-shaped strings.

Tool source code, file paths under `<DataDir>/_history/`, snapshot
trees, and the operator-supplied folder path NEVER land on the
payload — `DiffHash` (lower-hex SHA256, 64 chars) is the
cryptographically-bounded summary the audit subscriber consumes
instead. The reflection-based field-allowlist test
(`TestM95_LocalPatchApplied_FieldAllowlist`) pins both NAME and TYPE
in declaration order so a future field-type swap surfaces here. A
parallel `TestM95_LocalPatchApplied_NoBannedField` asserts no
PII-bearing field name appears (`FolderPath`, `FolderContent`,
`ManifestBody`, `SnapshotPath`, `DataDir`, `LivePath`, `AuthSecret`,
`Token`).

Two canary tests defend the boundary at runtime:

- `TestInstall_PIICanary_EventOmitsCanaryReason` stuffs synthetic
  canary substrings into the operator-supplied folder content +
  filenames; runs `Install`; `%+v %#v`-dumps the published event
  payload AND asserts the canaries DO NOT appear.
- `TestInstall_LoggerOmitsCanaries` exercises the publish-failure
  branch (the densest log-call site) and asserts the same.

### Pattern

1. **Snapshot directory MUST be SIBLING to the source tree the
   M9.1.b scanner walks.** `<DataDir>/_history/<source>/<tool>/<version>/`
   is the snapshot path; the M9.1.b scanner walks
   `<DataDir>/tools/<source>/<tool>/`. A snapshot tree under
   `tools/<source>/_history/...` would be observed by the scanner as
   a "tool" directory and emit decode failures on every recompute
   (`core/pkg/localpatch/snapshot.go:historyDirName`,
   `snapshot.go:snapshotPath`). The same pattern generalises to any
   future "shadow on-disk state next to the live tree" — keep it
   OUTSIDE the live-tree parent.

2. **Single audit topic with an `Operation` discriminator beats
   per-operation topics.** Install + rollback share
   `localpatch.local_patch_applied` (`events.go:TopicLocalPatchApplied`);
   the `Operation` field (`events.go:OperationInstall` /
   `OperationRollback`) discriminates. The downstream M9.7 audit
   subscriber gets a single chronological feed for the local-patch
   lifecycle without parsing free-form text. The roadmap's M9.7 topic
   list (`docs/ROADMAP-phase1.md:512`) only names `local_patch_applied`
   — a per-operation topic would have forced a roadmap edit AND a
   second subscriber wiring. Generalises: a closed-set discriminator
   on the payload beats expanding the topic vocabulary when the
   subscriber is a single audit feed.

3. **Per-call seams beat process-global statics — applied to THREE
   function-shape seams plus the per-call `Clock` read.** The
   `localpatch.SourceLookup`, `OperatorIdentityResolver`, and the
   per-`Publish.Publish` invocations are per-invocation; the
   `Clock.Now()` reads happen per-operation. The `FS` field is
   per-construction (single FS handle binding the process is the
   right scope — it's the deployment's data root, not a tenant-
   scoped value). A tenant rotation (new operator OIDC binding,
   new data dir, new clock skew) takes effect on the NEXT operation
   without a process restart. Same lesson the codex iter-1 reviewer
   has reliably flagged across M9.1.a / M9.4.a / M9.4.b / M9.4.c —
   apply pre-emptively (`installer.go:InstallerDeps`,
   `rollback.go:RollbackerDeps`).

4. **Defensive validation BEFORE any on-disk side effect (M9.4.b
   webhook iter-1 M1 lesson).** The install / rollback resolution
   order is `validate → ctx → SourceLookup → identity → folder
manifest read → folder digest → snapshot prior → replaceLive →
publish`. Validation runs FIRST so a malformed request fails
   before any FS read / write. The publish uses
   `context.WithoutCancel` so a mid-flight caller cancel cannot split
   the on-disk-then-publish pipeline (`installer.go:Install`,
   `rollback.go:Rollback`). Mirror the `Proposer.Submit` /
   `Webhook.handleApproved` discipline.

5. **Content-addressed digest with length-prefix collision defence
   AND exec-bit captured.** `ContentDigest` walks the tree, sorts
   entries by relative path, hashes each `(rel_path, exec_bit,
content)` record with explicit 64-bit length prefixes
   (`digest.go:writeLengthPrefixed`). Without length-prefixing, two
   trees with permuted path/content separators would collide; without
   exec-bit capture, a `chmod +x` would not change the digest even
   though it changes runtime behaviour. The walker is symlink-cycle-
   safe via a realpath visited set so a malicious / misconfigured
   tool tree cannot drive the digest into an infinite loop
   (`digest.go:walkInto`). Mirror M9.4.d Pattern 4 (symlinks demand
   explicit, documented handling).

6. **Forward snapshot on rollback enables un-rollback.** The
   rollback flow snapshots the CURRENT live tree under its own
   version BEFORE replacing it with the target snapshot
   (`rollback.go:Rollback`). A "rollback to v1.2.0 → tested → forward
   to v1.3.0" workflow degenerates to `wk-tool rollback foo --to 1.3.0`
   — no special "un-rollback" command needed. The
   `snapshotIfPresent` collision policy preserves the FIRST snapshot
   so an idempotent re-rollback does not lose the original audit
   trail (`snapshot.go:snapshotIfPresent`).

7. **Refuse install AND refuse rollback when the prior / current
   live tree is undecidable (iter-1 codex M2 symmetric fix).** A
   live tree present but with an undecidable `manifest.json` would
   leave us with no version under which to snapshot the prior
   contents — overwriting it would lose the prior tree without a
   restorable target. BOTH installer AND rollbacker surface
   `ErrManifestRead` BEFORE any destructive write so the operator
   triages (`installer.go:readLiveVersion`,
   `rollback.go:Rollback`,
   `installer_test.go:TestInstall_PriorTreeUndecidableRefused`,
   `rollback_test.go:TestRollback_UndecidableLiveTreeRefused`).
   Mirror the M9.1.a "fail-loud on misconfiguration vs silent
   degradation" discipline.

8. **Snapshot retention is an explicit operator contract, not an
   implicit assumption (iter-1 critic M6 fix).** Every install +
   every rollback adds a directory under
   `<DataDir>/_history/<source>/<tool>/<version>/`. There is no
   automated GC, no max-versions cap, and no scheduled cleanup.
   The operator's contract: snapshots accumulate indefinitely
   until the operator manually prunes via `rm -rf
<DataDir>/_history/<source>/<tool>/<version>/` for the
   versions they no longer want as a rollback target. A future
   roadmap item should land an automated retention sweep; until
   then, the lesson + the `wk-tool` help text + the Makefile
   target's diagnostic line all name the operator-managed
   cleanup contract explicitly so an operator's first encounter
   with `_history/` size is on the docs, not on the disk-full
   alert.

### Iter-1 patterns worth carrying forward

1. **A YAML deliverable needs the same "parity asserted by the
   downstream code" discipline as a wire-format proto.** The
   reviewer.go ↔ tool-ci.yml parity surface is what makes the
   audit-trail join work; without contract tests pinning the
   byte-for-byte mirror (M6), a single edit to either side
   silently produces a slack-native ↔ git-pr asymmetry. The
   M9.4.d iter-1 fixes added 6 parity assertions (fsNetPattern,
   capabilityIDFormat, DryRunMode, action SHAs, permissions,
   input defaults) on top of the 6 the initial commit had.

2. **Strict-decode discipline crosses language boundaries.** The
   Go decoder's `DisallowUnknownFields` + `strings.TrimSpace`
   semantics are part of the manifest contract — not Go
   implementation details. A TS / JS / YAML consumer of the same
   contract MUST mirror them or the language boundary becomes a
   pit of silently-accepted-then-rejected inputs (M2, M8).

3. **Optional outputs require explicit triple-layer wiring in
   reusable workflows.** A step writing to `$GITHUB_OUTPUT` is
   the FIRST of three required layers: the step needs an `id:`,
   the job needs an `outputs:` block mapping `steps.<id>.outputs.X`,
   and the `workflow_call` block needs a top-level `outputs:` block
   mapping `jobs.<job>.outputs.X`. Missing any one silently breaks
   the consumer's `needs.<job>.outputs.X` (M1). GitHub's
   documentation describes this but the failure mode is "output is
   empty string", not a syntax error.

4. **Symlinks in input trees demand explicit, documented
   handling.** `lstat`-shaped walkers (Node's `Dirent` API,
   `find -type f`) silently underscan symlinks; `stat`-shaped
   walkers risk cycles. Either choice is fine but MUST be
   conscious AND documented — both the README's "Consumer
   expectations" and the operator-runbook's "layout expectations"
   need to name the choice so a consumer knows their repo's
   symlink discipline is respected (M9).

### Pattern

1. **Closed-set vocabulary spans the package boundary.** The
   `GateName` constants (`core/pkg/approval/reviewer.go:53-79`) are
   the contract: any new gate landing in the slack-native reviewer
   MUST also land as a `name:`-matching step in the M9.4.d
   template, AND vice versa. The Go contract test
   (`core/pkg/approval/tool_ci_template_test.go:TestM94D_Template_GateNameParity`,
   lines 69-90) reads the YAML from
   `tools-builtin/ci-template/tool-ci.yml` and asserts every
   constant has a matching step. A future GateName rename without
   a template update fails the test loud — the audit-vocabulary
   join breaks at the source rather than silently in production.

2. **Reusable-workflow trigger exclusivity is enforced by test, not
   convention.** A `workflow_call`-only template that accidentally
   gains `on: push` or `on: pull_request` would fire on every
   commit in the watchkeeper-tools repo — disruptive and a CI cost
   surprise. `TestM94D_Template_TriggerIsWorkflowCallOnly`
   (`tool_ci_template_test.go:122-132`) asserts the `on:` map has
   EXACTLY one key (`workflow_call`). A future edit adding a
   second trigger fails closed.

3. **Inlined lint scripts via heredoc beat external dependencies
   for a published-template self-containment.** The
   `undeclared_fs_net` and `capability_declaration` gates inline
   Node scripts via the `cat > "$RUNNER_TEMP/lint-*.mjs"
<<'NODESCRIPT' ... NODESCRIPT` pattern. Consumer-side cost: zero
   — no `npm install @watchkeepers/tool-lint`, no second
   `actions/checkout` step to pull the template's repo, no version-
   skew between the script and the workflow. Trade-off: the
   inlined script is verbose inside the YAML, and a local-mirror
   invocation requires copying the heredoc body to a file. The
   operator-runbook flags this explicitly and tracks the M9.5
   extraction follow-up. The single-quoted heredoc delimiter
   (`<<'NODESCRIPT'`) prevents shell `$` and backtick expansion so
   the embedded JS reads literally. The `$RUNNER_TEMP` (not `/tmp`)
   destination is per-job-isolated so concurrent self-hosted-runner
   jobs cannot collide.

4. **Pattern parity is copied verbatim from production source AND
   pinned byte-for-byte by tests, not just re-described.** The
   `fsNetPattern` needle table in the template is byte-for-byte the
   same as `core/pkg/approval/reviewer.go:150-167`; the
   `capabilityIDFormat` regex (`reviewer.go:143`) and the
   `DryRunMode` enum (`toolregistry/manifest.go:32-61`) are copied
   from the same packages. The iter-1 M6 fix added three contract
   tests pinning the parity at the YAML side, mirroring the
   existing Go-side `Validate()` discipline. Rationale: divergence
   on a needle or an enum value would create a slack-native ↔
   git-pr asymmetry where the in-process reviewer flags a violation
   the real CI does not (or vice versa) — a confidence-destroying
   audit-trail hole.

5. **Optional steps are conditionally guarded on TWO axes.** The
   `sign` step's `if:` clause references BOTH
   `inputs.enable_signing` AND the release ref
   (`refs/heads/main` OR `refs/tags/v*`). The first axis answers
   "did the consumer opt in"; the second answers "is this a
   release-relevant run". A future regression dropping either
   half — running on every PR despite `enable_signing=false`, or
   running on every push to a feature branch — fails the contract
   test (`TestM94D_Template_SigningGatedByInput`,
   `tool_ci_template_test.go:269-294`). Same pattern applies to
   any future optional gate (slsa provenance, npm publish dry-run):
   conditional on user opt-in AND a release-triggering ref.

6. **Reusable-workflow output requires a three-layer wiring; tests
   pin all three.** A step writing to `$GITHUB_OUTPUT` is the
   FIRST of three required layers: the step needs `id:`, the job
   needs an `outputs:` block, and the `workflow_call` block needs
   a top-level `outputs:` block. The iter-1 M1 fix added all three
   layers AND a contract test (`TestM94D_Template_OutputsWired`,
   `tool_ci_template_test.go:200-237`) asserting each. The failure
   mode without this wiring is "consumer reads empty string", not
   a syntax error — a confidence-destroying silent regression.

7. **A YAML deliverable that mirrors a Go validator MUST mirror
   the STRICTNESS, not just the SHAPE.** The Go decoder's
   `DisallowUnknownFields` + `strings.TrimSpace` semantics are
   part of the manifest contract — not implementation details.
   The iter-1 M2 + M8 fixes brought the inline JS up to parity:
   trim-then-empty check on `version`, allow-listed top-level
   keys mirroring the Go decoder's allowlist. The
   `capability_declaration` doc-block names every parity surface
   so a future editor adding a manifest field knows to update
   BOTH sides (`tools-builtin/ci-template/tool-ci.yml:180-208`).

### Iter-1 patterns worth carrying forward

1. **Pass a defensive clone INTO the side-effecting seam, not just
   keep a clone for the trace.** The pattern "clone before Forward;
   store original on Outcome" beats "clone after Forward" — the
   latter is too late because the mutation has already happened on
   the shared map. Generalises to any in-process seam taking a
   reference-typed argument that the caller will keep alive
   post-call.

2. **`sideEffectsFired bool` gates the pre-publish ctx-check.** When
   no real side effect has fired (ghost mode, scoped with nil
   forwarder), the pre-publish ctx-gate refuses phantom events; when
   real side effects have fired, the gate is SKIPPED so the durable
   record lands regardless. The same axis applies to per-iteration
   ctx-checks: gate side-effect firing, but publish whatever fired.

3. **Per-iteration ctx-check inside a side-effecting loop is a
   discrete discipline from "pre-side-effect ctx-check".** The
   M9.4.a/b prior art pinned the pre-side-effect ctx-check at the
   top of an orchestrator method; M9.4.c extends to "per side-effect
   ctx-check inside a fan-out loop". The latter prevents a single
   caller-cancel from amplifying into N real side effects.

4. **Post-rewrite bound enforcement matches pre-rewrite bound
   enforcement.** When a transformation can grow a bounded
   collection (here: scoped rewrite inserting a key), the
   transformation MUST re-validate the bound on the result. The
   authoring-boundary input validator alone is not sufficient.

5. **Closed-set enum hygiene is consistent across every value-type
   in the package.** If one closed-set enum has a `Validate()`,
   every closed-set enum in the same package should — even if the
   current code path does not consult the validator (the future
   audit-decoder will).

### Context

M9.4.c lands the runtime primitive that the M9.4.b
[`DryRunRequester`](https://en.wikipedia.org/wiki/Seam) seam was
designed to plug into. The single artifact is
`core/pkg/approval/dryrun.go`'s `*Executor` — a per-call orchestrator
that branches on a tool manifest's `dry_run_mode` ∈
{`ghost`, `scoped`, `none`} and produces a [`Trace`] of every write-
side broker operation the proposed tool would make.

The boundary at M9.4.c is deliberately upstream of the actual TS
runtime: the executor consumes a typed `Request` carrying a
`ProposalID`, `ToolName`, `Mode`, and a bounded list of
`BrokerInvocation` values (Kind ∈ {`slack`, `jira`}, opaque Op,
bounded Args map). The future TS-runtime adapter (out of scope) reads
a proposal's `code_draft`, statically extracts its broker calls, and
hands them to the executor; the executor never executes the
TypeScript itself.

The three modes behave as the roadmap text specifies:

1. **`ghost`** — every invocation is stubbed
   (`Outcome.Disposition = ghosted`); the `Effective` invocation is a
   fresh defensive clone of `Original` so a downstream consumer
   mutating one cannot corrupt the other. No `BrokerForwarder` is
   consulted regardless of wiring (ghost is "no real write, ever").
2. **`scoped`** — the per-call `ScopeResolver` resolves a `Scope`
   (`LeadDMChannel`, `JiraSandboxProject`); each invocation's
   `channel` (Slack) or `project` (Jira) arg is forced to the scope
   value; the optional `BrokerForwarder.Forward(ctx, effective)` runs
   on the **rewritten** invocation. A forwarder error short-circuits
   the remaining invocations and returns the partial trace built so
   far so the caller surfaces "how far we got".
3. **`none`** — the executor returns `ErrPreApprovalWarning`
   IMMEDIATELY — no resolver call, no clock read, no id mint, no
   publish. The sentinel **is** the warning; the caller (future
   runtime glue) renders it to the lead before the first real call.

The boundary discipline matches every prior M9 atomic ship:

- Per-call resolver (`ScopeResolver`) beats process-global static —
  the lead-DM and sandbox project key are tenant-scoped, so a rotation
  takes effect on the next invocation without a process restart.
  Mirrors M9.1.a's `AuthSecretResolver`, M9.4.a's `IdentityResolver`,
  M9.4.b's `WebhookSecretResolver` / `SourceForTarget`.
- Optional `BrokerForwarder` seam is the "primitive not yet wired"
  degradation path — the Trace still records `DispositionScoped` with
  the rewritten Effective; only the real broker call is skipped.
- Defensive deep-copy of reference-typed Args on `Original` AND
  `Effective` keeps in-process caller mutation from corrupting the
  trace (M9.4.a / M9.4.b discipline, applied here to the per-
  invocation Args map).
- Topic `approval.tool_dry_run_executed` joins the M9.4.b topic
  vocabulary (`approval.` prefix; metadata-only `DryRunExecuted`
  payload with a 7-field reflection allowlist).

### Pattern

1. **Open enum at the parameter boundary; closed enum at the
   discriminator boundary.** `BrokerInvocation.Op` is a free-form
   string (the runtime accepts any broker-specific op name); but
   `BrokerInvocation.Kind` is a closed-set enum with `Validate`. The
   Op string is bounded by `MaxBrokerOpLength` so the open boundary
   does not become an unbounded surface; the Kind closed-set fails
   loud on a typo so a future broker addition forces the executor's
   `rewriteForScope` switch to grow a new case
   (`core/pkg/approval/dryrun.go:34-66`).

2. **Per-call resolver beats process-global static — applied to the
   sandbox config too.** `ScopeResolver` is a function-shape per-call
   seam (`dryrun.go:185-208`), not a `*Executor` field set at
   construction. A tenant rotation (new sandbox project, new lead
   DM channel) takes effect on the NEXT invocation without a process
   restart. Same M9.1.a / M9.4.a / M9.4.b lesson — codex iter-1
   reliably flags a tenant-scoped value pinned as a static.

3. **The "primitive not yet wired" degradation path is documented in
   the seam godoc, not hidden in test fakes.** `BrokerForwarder` is
   optional; a nil forwarder still produces a valid Trace with
   `DispositionScoped` outcomes (the rewrites are surfaced; only the
   real broker call is skipped). The godoc names it the "M9.4.c
   primitive not yet wired to real brokers" path
   (`dryrun.go:218-232`). A future production-wiring task plugs in
   a real Slack + Jira forwarder; in the interim the executor is
   fully testable end-to-end and shippable. Mirrors M9.4.b's
   `DryRunRequester` nil-seam discipline.

4. **`Original` and `Effective` are SIBLING fields, not a "before /
   after rewrite chain".** Both carry a fresh defensive clone of the
   input — `Original` is preserved verbatim; `Effective` carries the
   per-mode rewrite. Under ghost mode they are
   field-equal-but-pointer-distinct (`executeGhost` calls
   `cloneBrokerInvocation` twice). A downstream caller mutating
   `Effective` cannot bleed into `Original`
   (`TestExecute_DefensiveDeepCopy_RewriteDoesNotCorruptOriginal`);
   the caller's input slice mutated post-Execute cannot bleed into
   either (`TestExecute_DefensiveDeepCopy_OriginalArgs`).

5. **The sentinel IS the user-facing warning for `none` mode.**
   `ErrPreApprovalWarning` wraps the tool name in its message so the
   future runtime-glue caller can `errors.As` (or `errors.Is`) and
   render the literal sentinel string as the lead-facing notice. The
   executor refuses to invoke any side effect (no resolver call, no
   id mint, no publish) — the sentinel return preserves the "no
   side-effect before pre-approval warning surfaces" semantic
   (`dryrun.go:418-419`, `errors.go:ErrPreApprovalWarning`). Mirrors
   M9.4.a's "validation precedes side-effect" + M9.4.b's "ctx-check
   precedes publish" disciplines.

6. **Partial-trace return on per-invocation forwarder failure.** When
   the optional `BrokerForwarder.Forward` rejects under scoped mode,
   the executor returns the outcomes built BEFORE the failing
   invocation (zero in the canonical case because the failing call
   short-circuits before its outcome is appended). The caller
   surfaces "we got 0 of 2 invocations through" rather than a
   silently empty trace; the publish does NOT fire (the sequence
   never completed)
   (`dryrun.go:executeScoped:362-381`,
   `TestExecute_ScopedMode_ForwarderErrorReturnsPartialTrace`).
   Asymmetric with `executeGhost` (no forwarder, no partial-failure
   surface) by construction.

7. **PII discipline: in-process Trace surfaces caller bodies; bus
   payload carries metadata-only counts.** The `BrokerInvocation.Args`
   map may carry the would-be Slack message body or the Jira issue
   summary — surface to the lead is the entire point. The Trace
   returned from `Execute` carries these verbatim. The
   `DryRunExecuted` event published on the bus carries
   `BrokerKindCounts` (a per-Kind int) and `InvocationCount` only —
   the field allowlist test pins the boundary and the canary harness
   stuffs synthetic ASCII into every Args entry and asserts the
   verbatim `%+v` dump of the published payload never contains them
   (`events_m94c_test.go`, `dryrun_test.go:TestExecute_EventPayloadOmitsArgsCanary`).
   Symmetric with M9.4.a's `ToolProposed` allowlist + canary
   discipline — extended to "even though the in-process consumer
   gets the body, the bus subscriber never does".

### Iter-1 patterns worth carrying forward

1. **`claim → side-effect → rollback-on-error` is the cheapest
   webhook idempotency primitive that works under concurrency.**
   In-process `sync.RWMutex` + a `decisions[uuid]kind` map gives
   exactly-one-winner semantics without coordinating with the
   eventbus. `UnmarkDecided` on publish failure preserves
   retry-friendliness. The same pattern lands in any future
   "external system retries our http handler" boundary
   (gateway-side ingress, billing webhooks, OAuth callbacks).

2. **Per-button-branch `MarkDecided` placement matters more than
   per-button-branch `ctx.Err` placement.** M9.4.a iter-1 spent
   significant effort on the ctx.Err discipline; M9.4.b iter-1
   adds the decision-claim ordering as a peer concern. The
   pre-claim ctx.Err gate STILL refuses a phantom claim from an
   abandoned caller, but the claim itself runs on a detached ctx
   so a mid-flight cancel cannot strand it.

3. **A PII canary test with no actual canary in the kv path
   passes trivially.** The corrective discipline: every PII-canary
   test MUST include a "sanity gate" (e.g. assert at least one
   log entry carries non-empty kv) so a regression in the
   plumbing-under-test cannot make the assertion trivially pass.
   Add this pattern preemptively to any new redaction test.

4. **`reflect.TypeOf(<struct>{})` walks are the canonical "this
   PII-bearing field must never appear here" test, not type
   aliases.** The prior `type reflectFieldNames = X` pattern
   declares an alias and tests nothing; the real assertion is a
   field-by-field walk against a banned-name allowlist (already
   the M9.4.a discipline for event payloads — now extended to
   `ApprovalCardInput`).

### Pattern

1. **Mirror the canonical HMAC verifier file, header-for-header.**
   `webhook.go` mirrors `core/pkg/messenger/slack/inbound/verify.go`:
   `v0=`-style version prefix (renamed `sha256=` to align with
   GitHub's `X-Hub-Signature-256` convention), 5-minute timestamp
   replay window, `crypto/hmac.Equal` constant-time compare,
   non-integer timestamp bucketed under "stale", exported `Sign`
   helper for test fixtures using the SAME code path the verifier
   consults. A regression-guard test (`TestWebhook_HMACEqualPinned`)
   greps the source for `hmac.Equal(` literal so a future refactor
   to `bytes.Equal` or `==` surfaces at test time. The discipline
   pays for itself: the verifier code reads like the spec, and a
   second-set-of-eyes review can confirm the algorithm by side-by-
   side comparison with the slack-inbound original.

2. **`SchedulerSyncer` 1-method interface beats consuming the
   concrete `*toolregistry.Scheduler`.** `webhook.go:117-128` declares
   `SchedulerSyncer interface { SyncOnce(ctx, sourceName) error }`
   and consumes the interface, not the concrete pointer. Production
   wiring satisfies it via `*toolregistry.Scheduler`; tests
   substitute a `fakeSchedulerSyncer` that drives the
   sync-error-still-publishes branch deterministically without
   spinning up a git server. Same Phase-1 LESSON as M9.4.a's
   `Publisher` / `IdentityResolver` seam discipline — the package
   boundary owns the interface; downstream impls satisfy it
   without coupling.

3. **Per-call resolver for webhook secret + source mapping.**
   `WebhookSecretResolver` and `SourceForTarget` are both function-
   shape per-call seams (`webhook.go:138-152` / `:154-163`), not
   process-global statics. A per-tenant secret rotation takes effect
   on the next delivery without a process restart; a per-deployment
   source-mapping reload does not couple webhook delivery to
   operator-config reload semantics. Mirrors M9.1.a's
   `AuthSecretResolver` and M9.4.a's `IdentityResolver` — codex
   iter-1 reliably catches a tenant-scoped value pinned as a
   static.

4. **Sync failure does NOT abort publish on the approval path.**
   `Webhook.handleApproved` (`webhook.go:265-321`) treats
   `SchedulerSyncer.SyncOnce` as a best-effort side effect: a
   non-nil sync error logs but does NOT short-circuit the
   `Publisher.Publish(TopicToolApproved)` call. The semantic
   contract is "the event is the durable record". HTTP 202
   Accepted communicates the partial-success state to the
   webhook poster — "approval landed but sync needs operator
   attention". The audit subscriber records the failure for
   triage. Same "publish-failure means event lost; sync-failure
   means downstream needs re-drive" discipline as M9.1.a's
   `SourceFailed` / `SourceSynced` split.

5. **Per-button publish helpers with `context.WithoutCancel` and
   pre-publish ctx-check.** Each callback branch in
   `callbacks.go:198-303` runs `ctx.Err()` BEFORE the publish AND
   uses `context.WithoutCancel(ctx)` for the publish itself.
   The pre-check refuses a phantom event on a cancelled ctx; the
   detached child ctx ensures the publish completes deterministically
   regardless of mid-flight caller cancel against the eventbus's
   queue-depth-dependent send fast-path. This is the same race
   the codex iter-1 M1 finding on M9.4.a caught — applied
   preemptively across all 4 callback branches AND across the
   webhook's `handleApproved` / `handleRejected` so a future
   reviewer cannot single out one branch as forgotten.

6. **`CapabilityTranslator` is an optional function-shape seam
   with a documented "M9.3 not yet wired" fallback.** The renderer
   accepts a nil translator and renders `<cap_id>
(translation pending — M9.3)` for every cap (`card.go:332-368`).
   The pattern lets the M9.4.b ship land BEFORE M9.3's dictionary
   loader without leaving an `// TODO M9.3` comment in production
   code; the seam is in place and the fallback string names the
   future milestone so a reader knows the gap is intentional.

7. **Per-package action_id namespace prevents cross-saga
   misrouting.** `card.go:122-130` declares
   `tool_approval:<proposal-uuid>` and an `EncodeApprovalActionID`
   /`DecodeApprovalActionID` codec — deliberately NOT reusing
   `messenger/slack/cards`'s `approval:<tool>:<token>` namespace
   (which is pinned to the M6.3 saga's
   `spawn.PendingApprovalTool*` constants). A unified Slack
   dispatcher branches on the action_id prefix; the per-package
   namespace ensures the M6.3 dispatcher cannot mis-decode an
   M9.4.b click into the spawn approval saga (and vice versa).
   Distinct sentinels `ErrInvalidActionID` in each package make
   `errors.Is` triage unambiguous.

### Pattern

1. **Decompose multi-surface roadmap leaves BEFORE coding.** M9.4
   bundled six orthogonal surfaces in one line — the same
   discipline that decomposed M9.1 (commit `8bf1018`) and M8.2
   landed a `docs(roadmap):` commit splitting M9.4 into four
   lifecycle-stage sub-items (`docs/ROADMAP-phase1.md:505-509`)
   BEFORE any code landed. The decomposition commit lives on
   `main` (no PR — the gated work is the per-sub-item PRs) and
   includes a body paragraph naming the seam choice
   (lifecycle-stage: authoring → approval → runtime → CI). The
   one-PR alternative was rejected because the surface spans four
   packages with different reviewers and verification gates.

2. **Required-enum field with empty-string explicit rejection +
   loud-failure unknown-value.** `Manifest.DryRunMode` is required
   (`Manifest.Validate` checks empty after the other required
   fields), and `DryRunMode.Validate` rejects empty AND unknown
   values via `ErrInvalidDryRunMode`
   (`core/pkg/toolregistry/manifest.go:32-58` for the typed-string
   enum + Validate, `:115-125` for the field, `:148-164` for the
   validation block, `core/pkg/toolregistry/errors.go:201-214` for
   the sentinel). Two distinct sentinels
   (`ErrManifestMissingRequired` for "author forgot the field";
   `ErrInvalidDryRunMode` for "author typoed the value") let
   callers triage authoring mistakes vs schema drift. Same
   pattern as M9.1.a's `SourceKind` / `PullPolicy` (Pattern 7).
   The empty string is explicitly refused by the enum validator
   standalone so a future caller that bypasses
   `Manifest.Validate` (e.g. M9.4.c's runtime executor handed a
   deserialised manifest from a non-strict source) still gets
   the loud failure.

3. **Closed-set enum WITHOUT `local` is the roadmap-text
   contract enforcement.** `TargetSource` has only `platform` and
   `private` constants
   (`core/pkg/approval/target_source.go:27-39`). The roadmap text
   says "local source never offered to the agent"; the enum
   itself enforces the contract — the validator could never
   accept `local` because no constant exists, and
   `TestTargetSource_RejectsLocal` /
   `TestProposalInput_ValidateRejectsLocalTargetExplicitly` pin
   the literal-string rejection so a future enum addition
   (`hosted`, etc.) does not accidentally re-introduce the
   contract gap. Lead-override at approval time (M9.4.b) uses a
   distinct lead-only enum that does not flow through this
   validator.

4. **Cross-field validation lives in a separate function that
   takes BOTH inputs explicitly.** `ValidateModeForSources`
   takes `(Mode, []toolregistry.SourceConfig)`
   (`core/pkg/approval/approval_mode.go:84-103`). The
   single-field `Mode.Validate` / `TargetSource.Validate`
   stays small and composable; cross-field rules live one level
   up so a caller wiring the operator config can choose to run
   the single-field validators first (for early error surfaces)
   and the cross-field validator second. The boundary is
   defensible: a single-field validator that ALSO consumed
   sources would cyclically import the registry into config
   primitives. Same lesson as `toolregistry.SourceConfig.Validate`
   vs `ValidateSources` (M9.1.a).

5. **Bounded-length defensive validation at the agent-facing
   boundary.** Every string field on `ProposalInput` has a max
   byte length enforced by `Validate` and exposed as a public
   constant (`MaxToolNameLength = 64`,
   `MaxCodeDraftLength = 65536`, etc.,
   `core/pkg/approval/proposal.go:13-45`). Bounds exist
   BEFORE the M9.4.b downstream consumers (AI reviewer, Slack
   card renderer) that would otherwise exhibit unbounded
   behaviour under adversarial AI-authored input. The 64 KiB
   `CodeDraft` cap fits any realistic TS source while refusing
   pathological payloads; same defensive-shape discipline as
   M9.1.a's source-name regex allowlist (path-traversal
   defence). The constants are exported so callers can surface
   "max length" hints in their UI without duplicating the
   numbers.

6. **Fail-fast validation BEFORE any side effects: validate →
   ctx → identity → clock → id → ctx-recheck → publish (with
   detached ctx).** `Proposer.Submit` runs the resolution order
   documented at the top of `proposer.go`
   (`core/pkg/approval/proposer.go:1-31`). Each side-effecting
   call is GATED by the prior validation result: `Validate`
   runs first (no identity-resolver call wasted on bad input);
   ctx.Err runs before the resolver (a pre-cancelled ctx aborts
   cheap); identity resolver runs before `Clock.Now` +
   `IDGenerator.NewUUID` so a tenant-rotation failure does not
   waste a process-monotonic correlation id. A SECOND ctx.Err
   check fires AFTER the proposal is built but BEFORE Publish,
   so a caller that cancels mid-flight does not produce a
   phantom proposal record (iter-1 codex M1 fix). The actual
   Publish call uses `context.WithoutCancel(ctx)` so once the
   pre-publish gate passes, the durable record completes
   deterministically — the `eventbus.Bus` send fast-path's
   queue-depth-dependent cancel behaviour cannot race the
   "event is the durable record" contract.
   `TestSubmit_CtxCancelledMidFlight_RefusesPublish` and
   `TestSubmit_PublishCtxDetachedFromCallerCancel` pin both
   halves. Same lesson as M7.1.c.c's "fail-fast precedes audit
   Append" (the M7.1.c.c iter-1 fix that surfaced from codex).

7. \*\*Metadata-only event payload + reflection allowlist test
   - PII canary harness on dynamic emission.\*\* `ToolProposed`
     carries 7 fields (`ProposalID`, `ToolName`, `ProposerID`,
     `TargetSource`, `CapabilityIDs`, `ProposedAt`,
     `CorrelationID`) and DELIBERATELY excludes `CodeDraft`,
     `Purpose`, `PlainLanguageDescription`
     (`core/pkg/approval/events.go:50-84`). The
     `TestToolProposed_FieldAllowlist` reflection test pins the
     payload shape to the documented 7 fields by name + type;
     adding a field requires the author bump the allowlist AND
     document why the new field's PII shape is OK
     (`core/pkg/approval/events_test.go`). The
     `TestSubmit_EventPayloadOmitsBodies` canary test stuffs
     synthetic ASCII substrings into each PII-bearing
     `ProposalInput` field and asserts the verbatim `%+v` dump
     of the event payload never contains them;
     `TestSubmit_EventPayloadOmitsBodiesWithNULByteCanary`
     covers the NUL-byte variant so a future logger that drops
     NULs cannot silently invalidate the substring match.
     Same dual-AC shape as M9.1.b / M9.2: static field allowlist
   - dynamic canary emission. Capability ids land on the
     payload as a defensive deep copy (`newToolProposedEvent`,
     `core/pkg/approval/events.go:93-105`) because they are
     public dictionary entries (M9.3) AND the downstream M9.4.b
     approval card needs them to render human-readable
     translations.

8. **Per-call resolver function shape, not interface, for
   tenant-scoped identity.** `IdentityResolver` is
   `func(ctx context.Context) (string, error)`
   (`core/pkg/approval/proposer.go:86-112`), invoked on every
   `Submit`. Per-tenant rotation takes effect on the next
   `Submit` without restarting the process. The function shape
   (vs an interface) matches M9.1.a's `AuthSecretResolver`
   discipline for the same reason: the surface is one-method
   and stays simpler as a `func` type alias. `("", nil)` is
   refused as `ErrEmptyResolvedIdentity` — a non-nil-but-empty
   return is a fail-loud programmer error, same as
   `toolregistry.ErrEmptyResolvedAuth`. A resolver returning
   a non-empty identifier exceeding `MaxProposerIDLength`
   surfaces `ErrInvalidProposerID` so a buggy resolver cannot
   silently leak a bearer-token-shaped value onto the event
   payload (iter-1 codex finding m2 fix).

9. **`gosec` G101 nolint annotation on synthetic redaction-
   harness canary constants.** The `canaryCodeDraft` /
   `canaryPurpose` / `canaryPlainLanguageDescription`
   constants in `proposer_test.go` carry
   `//nolint:gosec // G101: synthetic redaction-harness
canaries, not real credentials.` annotations because the
   strings contain `_DO_NOT_LEAK` and gosec's secret-detection
   heuristic would otherwise flag them. Same annotation
   pattern as the M7.1.c.c iter-1 fix.

### Context

M9.2 layers the shadow-detection half of the multi-source resolver
onto the M9.1.b `Registry` without rewriting its hot path. M9.1.b's
`BuildEffective` already implemented earlier-source-wins precedence
flattening but SILENTLY dropped the loser; M9.2 turns the loser into
a first-class signal:

- `BuildEffective` now returns `(*EffectiveToolset, []ShadowedTool,
error)`. The new `ShadowedTool` carries the tool name + the
  winner's source/version + the shadowed entry's source/version —
  exactly the fields a downstream subscriber needs to construct the
  lead-facing DM documented on the roadmap entry.
- `Registry.Recompute` consumes the shadow list and emits one
  `TopicToolShadowed` event per shadow on the existing `Publisher`
  seam BEFORE emitting `TopicEffectiveToolsetUpdated`. A subscriber
  observing both topics on the same eventbus thus learns about
  every shadow BEFORE seeing the revision bump.
- `ToolShadowed.Message()` renders the exact roadmap-text DM
  string so the wording stays consistent across deployments and any
  future re-wording lands in one method body.
- A new `ErrPublishToolShadowed` sentinel surfaces partial-delivery
  failures: when a per-shadow publish errors out, Recompute keeps
  publishing the remaining shadows (each is independent), records
  the FIRST failure, and — if the subsequent
  `TopicEffectiveToolsetUpdated` publish succeeds — returns
  `fmt.Errorf("%w: revision %d: %w", ErrPublishToolShadowed, rev,
firstShadowErr)`. If the toolset-updated publish ALSO fails,
  `ErrPublishAfterSwap` takes precedence (the missed revision-bump
  signal is the more critical observability gap).

Slack DM wiring is intentionally NOT in this package — the
`toolregistry` package stays audit/slack-free (same discipline as
M9.1.a/b). A downstream subscriber wires the DM by listening on
`TopicToolShadowed` and calling `ToolShadowed.Message()`; the
production wiring lands alongside the M9.7 audit subscriber.

the roadmap with or without "v" depending on how the manifest was
authored. The PII canary harness pins `Capabilities` and `Schema`
bodies out of the payload — same "event-payload-boundary redaction"
discipline as M9.1.a's `SourceFailed`-carries-only-error-type and
M9.1.b's `EffectiveToolsetUpdated`-carries-only-metadata. The
M9.4-anticipating `dmInjectionScrubber` additionally strips
backticks / newlines / Slack-mention sigils from the dynamic parts
of `Message()` before formatting (iter-1 M3 fix).

### Pattern

1. **Pure-transformation seam returns side-data alongside primary
   result; orchestrator consumes both** — `BuildEffective` is
   pure: takes a context + sources + clock + logger, returns a
   snapshot + shadow list + error
   (`core/pkg/toolregistry/scanner.go:170-225`). It does NOT
   publish events itself even though the shadow list will become
   events; the registry's `Recompute` is the orchestrator that
   reads the side data and emits. Keeping `BuildEffective`
   side-effect-free lets future callers (M9.4 dry-run, M9.5 local
   patch verification) consume shadow information without
   dragging the eventbus into the call graph. Mirrors the
   M9.1.a `Scheduler.SyncOnce` shape — the I/O work is in one
   layer, the event-emission is in another.

2. **`winnerIdx map[string]int` over `winner *EffectiveTool`** —
   the shadow-detection loop tracks winners via
   `winnerIdx map[string]int` keyed by `Manifest.Name`, with the
   value indexing into the growing `merged` slice
   (`scanner.go:202-220`). A pointer-into-slice would be UNSAFE
   because `append` may reallocate the underlying array — a
   pointer captured pre-reallocation would point at orphaned
   memory. The index variant is reallocation-safe AND constant-
   time. Reading the winner's source/version uses `merged[idx]`
   right when the shadow entry is being constructed. Same lesson
   as `core/pkg/spawn/saga`'s step-index tracking pattern.

3. **Event emission order matters for cross-topic causality** —
   `Recompute` emits all `tool_shadowed` events FIRST, then
   `effective_toolset_updated` LAST (`registry.go:316-355`). A
   subscriber observing both topics on the same eventbus thus
   learns about every shadow BEFORE seeing the revision bump.
   Subscribers can rely on the invariant "by the time I see the
   updated event for revision N, I have already seen every
   shadow at revision N" to atomically decide "this revision's
   shadows are X, Y, Z" without polling. The
   `eventbus.Bus`'s per-topic worker is sequential but
   cross-topic ordering depends on emission order at the
   publisher — Recompute owns the order.

4. **Partial-delivery semantics: each event is independent;
   first error wins; remaining errors logged only** — Recompute's
   shadow loop attempts every publish even when an earlier one
   failed (`registry.go:323-345`). The FIRST failure is wrapped
   onto the return via `ErrPublishToolShadowed`; subsequent
   failures land in the optional `Logger` only. The contract:
   subscribers that observe K of N notifications are strictly
   better off than zero, so first-error-aborts is the wrong
   default. Asymmetry vs the toolset-updated publish (which has
   no peers — losing it is a clean miss) — there the single
   error becomes the return outright. Same lesson lives in
   `core/pkg/outbox`'s batch-dispatch pattern.

5. **Multi-error precedence: the "registry-bump invisible" signal
   beats the "one DM missed" signal** — when BOTH the shadow
   publish AND the toolset-updated publish fail, `Recompute`
   returns `ErrPublishAfterSwap` (toolset-updated failure),
   NOT `ErrPublishToolShadowed` (`registry.go:357-365`). The
   reasoning: subscribers that only observe
   `effective_toolset_updated` would otherwise miss the
   revision bump entirely (zero notifications for the snapshot
   change), while a missed `tool_shadowed` only loses one DM
   (the snapshot is still visible via `Snapshot()` /
   `Acquire()`). The more severe observability gap takes the
   error chain. Tested by
   `TestRecompute_EffectiveUpdatedFailureOverridesShadowFailure`.

6. **DM wording lives on the payload type, not the subscriber** —
   `ToolShadowed.Message()` (`events.go:217-225`) returns the
   exact roadmap-text format using `fmt.Sprintf`. A future
   re-wording (e.g. localisation, A/B test, or appending an
   issue-tracker link) lands in this one method body — every
   Slack-DM subscriber, every audit-log entry, every operator
   alert reads from the same source of truth. Same lesson as
   `saga.SpawnContext` carrying canonical IDs vs every caller
   re-deriving them.

7. **Field-allowlist reflection test catches PII drift at
   compile-and-test time** — `TestToolShadowed_FieldAllowlist`
   (`m92_test.go`) pins the payload to 8 fields by name + type;
   adding a field (`Capabilities []string`, `Schema
json.RawMessage`) fails the test until the author bumps the
   `want` map AND consciously documents why the new field's PII
   shape is OK. Mirrors
   `TestEffectiveToolsetUpdated_FieldAllowlist` from M9.1.b's
   iter-1 t5. Combined with the canary-substring test, the AC
   surface is: "the payload's fields are X and Y" (static) AND
   "a runtime emission of the payload does not contain the
   canary" (dynamic).

8. **Intra-source duplicates do NOT surface as shadows** —
   `TestBuildEffective_IntraSourceDuplicateNotSurfacedAsShadow`
   pins the contract: a same-name conflict within ONE source
   lands as an `ErrIntraSourceDuplicateManifestName` log entry
   via `ScanSourceDir` (M9.1.b's M3 iter-1 fix) BEFORE the
   precedence-flattening loop sees it. The shadow list is
   reserved for CROSS-source conflicts because the lead-facing
   DM ("source A's tool conflicts with source B's") only makes
   sense across sources — within a single source, the conflict
   is an authoring bug the upstream owners resolve, not a
   cross-team coordination signal. Two failure surfaces, two
   diagnostic shapes.

9. **Slack DM wiring lives OUTSIDE `toolregistry`** —
   the package stays audit/slack-free (same discipline as
   M9.1.a/b). A future subscriber wraps `slack.OpenIM` +
   `slack.SendMessage` (`core/pkg/messenger/slack/`) against
   `TopicToolShadowed` and renders the body via
   `ToolShadowed.Message()`. Keeping the wiring at the
   subscriber layer means `toolregistry` is safe to depend on
   from any layer (no transitive slack import), AND the
   subscriber can be replaced (e.g. by a Discord or email
   subscriber) without touching the registry. Same
   import-cycle-break precedent as M3.2.b's `LocalPublisher`.

### Pattern

1. **Atomic `Pointer[entry]` + per-entry refcount is the natural
   shape for "in-flight calls complete on the old version while
   new invocations resolve against the new toolset"** — the
   `Registry` holds `current atomic.Pointer[registryEntry]`
   (`registry.go:163-167`); each `registryEntry` bundles a snapshot
   - an `atomic.Int32` refcount + a `retiredAt time.Time` (`registry.go:140-152`).
     `Acquire()` atomically loads the entry, increments its refcount,
     and returns a snapshot + idempotent release closure
     (`registry.go:233-245`). `Recompute()` swaps the pointer in one
     atomic op (`registry.go:286-289`); the old entry's snapshot stays
     reachable on the in-flight caller's stack (Go GC keeps it alive)
     AND the registry tracks its refcount in a retiring list so
     operator dashboards / future M9.3 cleanup can observe "how many
     in-flight calls are still on revision N." No mutex on Acquire
     means the hot path is a single atomic load + a single atomic
     increment.

2. **The grace period is purely a diagnostic-tracking knob — Go
   GC reclaims the snapshot, NOT the registry** — the
   `RegistryDeps.GracePeriod` (`registry.go:73-84`) is the duration
   the registry keeps a retiring entry in its
   `retiring []*registryEntry` slice (`registry.go:178-181`). At each
   `Recompute()` the registry sweeps entries whose
   `retiredAt + GracePeriod` is on or before `now`
   (`registry.go:382-403`), logs an `effective_toolset retired`
   entry, and drops its reference. The on-disk snapshot itself
   stays alive as long as any in-flight caller holds the pointer
   — the grace period doesn't free memory, it only frees the
   registry's bookkeeping slot. This decoupling matters because
   M9.3 will plug heavy resources (loaded code modules, mmaped
   signed tarballs) onto the retiring entry and need a
   "registry-bounded" cleanup signal that doesn't fight Go GC.

3. **The hot-reload event carries metadata only — never manifest
   bodies** — `EffectiveToolsetUpdated` fields are `Revision`,
   `BuiltAt`, `ToolCount`, `SourceCount`, `CorrelationID`
   (`events.go:111-141`). Tool names, capability ids, schema bytes,
   and `Source` fields are deliberately excluded. A subscriber that
   wants the manifest contents calls `Registry.Snapshot()` or
   `Registry.Acquire()`; the bus channel never carries the payload.
   The `TestPIIRedactionCanary_EffectiveToolsetUpdatedPayload` test
   asserts the canary substrings never appear in
   `fmt.Sprintf("%+v", event)`. Same redaction discipline as
   M9.1.a's `SourceFailed`-carries-only-error-type — this time the
   "secret to protect" is the manifest body itself (a customer
   private tool that ends up logged verbatim could reveal proprietary
   logic).

4. **Per-tool failure isolation in the scanner: log + skip never
   poisons the whole source; per-source failure isolation in the
   builder: log + zero contributions never poisons the whole
   recompute** — `ScanSourceDir` runs `LoadManifestFromFile` per
   per-tool subdirectory under a try-catch loop, logs decode
   failures via the optional `Logger`, and continues
   (`scanner.go:60-83`). `BuildEffective` runs `ScanSourceDir` per
   configured source under the same loop, logs per-source
   `ReadDir` failures, and continues with zero contributions from
   that source (`scanner.go:139-167`). A single corrupt manifest or
   a single I/O blip cannot brick the runtime's effective toolset
   — the operator still gets a coherent snapshot from every
   source's good tools. The error-isolation boundaries match
   `core/pkg/cron`'s per-job dispatch isolation (one panicking
   handler does not stop the scheduler) and M9.1.a's per-source
   `SyncOnce` mutex isolation (one bad source's network stall
   does not block siblings).

5. **`fs.ErrNotExist` is the "no work to do" sentinel; anything
   else surfaces wrapped** — `ScanSourceDir` treats
   `errors.Is(err, fs.ErrNotExist)` on `FS.ReadDir` as legitimate
   "source not yet synced — skip" and returns `(nil, nil)`
   (`scanner.go:51-56`). Any other `ReadDir` error wraps a new
   `ErrScanReadDir` sentinel (`errors.go:151-157`). Symmetric to
   the M9.1.a `Scheduler.SyncOnce` `Stat` chain that uses
   `fs.ErrNotExist` as the clone-vs-pull discriminant. The
   stdlib sentinel chain is preferred to a hand-rolled "not exist"
   field because the fake FS already returns
   `*fs.PathError{Err: fs.ErrNotExist}` and the production `OSFS`
   shim forwards `os.ReadDir`'s native chain — no wrapper code,
   no aliasing bugs.

6. **Publisher-error-on-success: ASYMMETRY mirroring M9.1.a's
   M5 contract — success-path publish errors are RETURNED, the
   atomic swap stays committed** — `Recompute` (`registry.go:297-309`)
   atomically swaps the new entry into current BEFORE publishing
   `TopicEffectiveToolsetUpdated`. If the publish fails, the
   registry still holds the new snapshot as current — the next
   `Snapshot()` / `Acquire()` returns it — but the caller observes
   the publisher error so an external retry / monitoring path can
   fire. The test
   `TestRecompute_PublisherErrorSurfaces` asserts both invariants
   (return error + state moved forward). Asymmetric to M9.1.a's
   `emitFailure` (publish errors suppressed because the original
   sync error is already authoritative) — here, the original
   "sync" is the registry's pointer swap, which has no error of
   its own.

7. **Compile-time interface assertion against the concrete
   `*eventbus.Bus`** — `registry_test.go` has
   `var _ Subscriber = (*eventbus.Bus)(nil)`. The `toolregistry.Handler`
   type alias is exactly `func(ctx context.Context, event any)` — the
   same shape as `eventbus.Handler` — so structural satisfaction
   holds. A future `Bus.Subscribe` signature change breaks the
   compile of THIS line at the same time as the production wiring
   would break. Mirrors the M9.1.a `Publisher` discipline + the
   cron / outbox pattern (`cron_test.go` / `consumer_test.go`).

8. **Subscriber seam, not a `*eventbus.Bus` import in production
   code** — `Registry.Start(ctx, sub Subscriber)` takes a
   `Subscriber` interface (`registry.go:28-39`) defined in this
   package. The production wiring will pass a `*eventbus.Bus` (the
   compile-time assertion in test pins the shape); the test path
   uses the real bus (no fake subscriber stub needed) which
   exercises the actual async dispatch + buffered-channel
   semantics. `TestStart_SubscribesAndDispatchesRecompute` uses
   `bus.Publish(SourceSynced{...})` then polls the registry's
   captured-publish list for `TopicEffectiveToolsetUpdated`. Same
   real-bus discipline as `core/pkg/cron`'s integration tests.

9. **`FS.ReadDir` re-introduced with a clarified contract** —
   M9.1.a's m8 iter-1 fix removed `ReadDir` from the FS interface
   because the scheduler did not need it; M9.1.b restores it with
   an explicit godoc contract on the `not-exist sentinel chain`
   (`scheduler.go:35-45`). `OSFS.ReadDir` forwards to
   `os.ReadDir` with the same `//nolint:gosec` annotation as the
   sibling methods (the path is operator-derived, not user
   input — `osfs.go:34-39`). The `fakeFS` adds a
   `dirEntries map[string][]fs.DirEntry` keyed by parent path and
   a `fakeDirEntry` struct implementing `fs.DirEntry`
   (`fakes_test.go`). Adding a method to a closed-set seam is
   mechanical because every layer of the stack uses interfaces.

10. **Initial empty snapshot at revision 0 lets `Acquire` return
    non-nil even before the first `Recompute`** — `NewRegistry`
    stores a synthetic empty `EffectiveToolset` at revision 0 with
    `BuiltAt = Clock.Now()` (`registry.go:208-211`). Callers that
    `Acquire` before any sync event has fired get an empty
    snapshot (zero tools) instead of a nil pointer. The first
    real `Recompute` advances the revision to 1; the synthetic
    revision-0 enters retiring like any other. This eliminates
    a nil-check in every downstream caller (M9.2 resolver, M9.3
    capability enforcer) at the cost of one no-op snapshot per
    registry — a trade-off that mirrors the
    M9.1.a `NoopSignatureVerifier{}` default-zero-value-usable
    discipline.

11. **`recomputeMu` serialises Recompute even though eventbus
    workers are per-topic sequential** — `Registry.recomputeMu`
    (`registry.go:173-181`) is taken at the top of `Recompute`.
    The eventbus's per-topic worker already serialises
    `TopicSourceSynced` handlers, but EXTERNAL callers (tests,
    operator CLI in future milestones) may call `Recompute`
    directly while the subscriber path is firing. Without the
    mutex, two `Recompute` calls would race on the revision
    counter, the swap, and the retiring-list append. The mutex
    is cheap (one atomic CAS per Recompute under no contention)
    and keeps the surface composable.

## 2026-05-11 — M9.1.a: tool_sources config + manifest.json schema + ToolSyncScheduler (new core/pkg/toolregistry)

**PR**: #133
**Merged**: d033536 on 2026-05-11

### Context

M9.1.a opens the M9 milestone family. The roadmap text on M9.1 is too
large to ship atomically (operator config schema + manifest decoding +
clone/pull I/O + scheduler + effective-toolset recompute + runtime
hot-reload signal with grace period — six orthogonal surfaces), so the
operator chose a 2-way decomposition along the data/runtime seam
(commit `8bf1018`):

- **M9.1.a** — the data + sync layer: operator `tool_sources` config,
  per-tool `manifest.json` schema, and `ToolSyncScheduler` that
  clones / pulls each source per its pull policy and emits
  `toolregistry.source_synced` / `toolregistry.source_failed` events.
- **M9.1.b** — the runtime layer: effective-toolset recompute on each
  `source_synced` event + a hot-reload signal to running runtimes with
  configurable grace period (in-flight `InvokeTool` calls complete on
  the old version while new invocations resolve against the new
  toolset).

The new package is `core/pkg/toolregistry/`. It is intentionally
self-contained — no imports from `core/pkg/config`, no audit emission,
no runtime coupling. Downstream subscribers (M9.1.b, M9.2 resolver,
M9.7 audit) join via `eventbus`.

Files:

- `doc.go`, `errors.go` — package documentation + sentinel errors.
- `manifest.go` — `Manifest` struct + `DecodeManifest` (strict JSON
  decode, `DisallowUnknownFields`, required-field validation,
  defensive deep-copy of reference-typed fields) +
  `LoadManifestFromFile` (file read + decode + source-stamping with
  collision detection).
- `config.go` — `SourceConfig` + closed-set `SourceKind`
  (`git` / `local` / `hosted`) and `PullPolicy`
  (`on-boot` / `cron` / `on-demand`) + per-source `Validate()` +
  `ValidateSources()` (covers duplicate names) + `EffectiveBranch()`
  defaulting to `main` + `CloneSources()` defensive copy.
- `yaml.go` — `DecodeSourcesYAML` (strict yaml.v3 `KnownFields(true)`,
  multi-document rejection) + `LoadSourcesYAMLFromFile`.
- `events.go` — `TopicSourceSynced` / `TopicSourceFailed` constants +
  `SourceSynced` (sourceName, syncedAt, localPath, correlationID) and
  `SourceFailed` (sourceName, failedAt, errorType, phase,
  correlationID) payloads.
- `scheduler.go` — `Scheduler` orchestrator with five seams (`FS`,
  `GitClient`, `Clock`, `SignatureVerifier`, `Publisher`) +
  `AuthSecretResolver` per-call resolver + optional `Logger`. `New`
  panics on nil required deps; `SyncOnce(ctx, sourceName)` performs
  one clone-or-pull cycle and emits the appropriate event.
- `osfs.go` — production `OSFS` adapter forwarding to `os.*`.

### Pattern

1. **Atomize a multi-surface roadmap leaf along its natural seam BEFORE
   coding** — M9.1 in the original ROADMAP text bundles six orthogonal
   surfaces. Mirror the M8.2 / M7.3 decomposition pattern: a separate
   `docs(roadmap):` commit on `main` that splits the leaf into named
   sub-items (`M9.1.a` / `M9.1.b`), then ship one sub-item per
   `ship-roadmap-item` invocation. The decomposition commit
   (`docs/ROADMAP-phase1.md:500-502`) is doc-only, lands directly on
   `main` (no PR — the gated work is the per-sub-item PRs), and
   includes a body paragraph explaining the seam choice + why the
   single-PR alternative was rejected. The atomic-ship rule has a
   user-override channel via `AskUserQuestion` when an axis is
   genuinely indivisible.

2. **Seam-driven design: every side-effecting primitive is an
   interface, every default is no-op** — the `Scheduler` consumes five
   interfaces (`FS`, `GitClient`, `Clock`, `SignatureVerifier`,
   `Publisher`) plus a callable `AuthSecretResolver`
   (`core/pkg/toolregistry/scheduler.go:21-118`). Each interface is
   defined in this package (not imported from elsewhere), uses the
   minimal subset of methods the scheduler actually calls, and has a
   hand-rolled fake in `fakes_test.go`. The default
   `SignatureVerifier` is the no-op `NoopSignatureVerifier{}` (zero
   value usable, returns nil for every input) — this lets M9.1.a land
   the verification HOOK without committing to a specific signing
   scheme (cosign / minisign is M9.3's call). The production wiring
   for `FS` is a 4-method shim (`osfs.go`) — adding a sixth seam later
   is mechanical.

3. **Per-call resolver beats process-global static for tenant-scoped
   values** — `AuthSecretResolver` is `func(ctx, sourceName, ref)
(string, error)`, invoked on every `SyncOnce`
   (`scheduler.go` resolveAuth). The test
   `TestSyncOnce_AuthResolverPerCall` asserts the call count matches
   the number of SyncOnce invocations (not cached). Same lesson as
   M8.2.a's `jira.BasicAuthSource` interface (consumed via
   `jira.WithBasicAuth`): a per-tenant rotation takes effect on the
   next sync without restarting the process, and the scheduler never
   holds the plaintext secret across calls. The toolregistry side
   chose a function shape (vs an interface) because the resolver
   surface is one-method and stays simpler as a `func` type alias.

4. **PII discipline at the event-payload boundary, NOT at the logger**
   — `SourceFailed` carries `ErrorType` (the Go type name via
   `fmt.Sprintf("%T", err)`) and `Phase` but NEVER the underlying
   error MESSAGE
   (`core/pkg/toolregistry/events.go:46-76`,
   `scheduler.go:374-395`). A failing git URL with embedded
   credentials would surface its message via `errors.Unwrap` if the
   payload carried `.Error()`; trimming the payload at the type-name
   boundary is what makes a subscriber that logs the event verbatim
   safe. The PII canary harness
   (`scheduler_test.go:TestPIIRedactionCanary_AcrossEveryFailurePath`)
   injects a synthetic `CANARY_PII_AUTH_TOKEN_…` into every failure
   phase (`mkdir` / `stat` / `clone` / `pull` / `signature` / `auth`)
   and asserts neither the event nor any logged kv pair contains the
   substring.

5. **Defensive deep-copy of reference-typed inputs in BOTH constructor
   AND on dispatch** — `New` calls `CloneSources(sources)`
   (`config.go:148-156`) so caller-side mutation post-construction
   cannot bleed into the scheduler's state; `DecodeManifest` copies
   `Capabilities` (string slice) and `Schema` (byte slice) via the
   local `cloneStringSlice` / `cloneBytes` helpers
   (`manifest.go:104-118`) so the caller's `raw` buffer is no longer
   aliased. Mirrors the M7.1.c.c `cloneBotProfile` pattern. The test
   `TestDecodeManifest_DefensiveCopy` mutates the input buffer
   post-decode and asserts the returned manifest is unaffected.

6. **Per-source mutex serialises concurrent SyncOnce on the SAME
   source; different sources run in parallel** — `Scheduler.perSourceMu`
   is a `map[string]*sync.Mutex` populated at construction
   (`scheduler.go:213-219`). `SyncOnce` takes the per-source mutex
   before touching the FS / git
   (`scheduler.go:308-310`). The concurrency tests
   (`TestSyncOnce_ConcurrencyDifferentSources`,
   `TestSyncOnce_ConcurrencySameSourceSerialised`) dispatch 16
   goroutines: the first asserts all 16 calls complete in parallel
   (all 16 sources clone exactly once); the second asserts the
   max-inflight count for the same source never exceeds 1 even
   under a 1ms artificial hold. The map is read-only after `New`, so
   no top-level lock is needed.

7. **Closed-set discriminators with named constants + per-discriminator
   cross-field validation** — `SourceKind` and `PullPolicy` are
   typed string aliases with named constants
   (`config.go:13-58`). `SourceConfig.Validate` switches on the kind
   and policy enums, requiring `URL` for `git` (and forbidding it for
   `local` / `hosted`) and `CronSpec` for `cron` (and ignoring it for
   `on-boot` / `on-demand`)
   (`config.go:96-130`). A `default:` branch on each switch surfaces
   `ErrInvalidSourceKind` / `ErrInvalidPullPolicy` rather than a
   silent acceptance — same loud-failure discipline as
   `runtime.AuthorityFromString`.

8. **Strict YAML decoding at the document root AND inside each entry**
   — `DecodeSourcesYAML` uses yaml.v3's `KnownFields(true)` on the
   `sourcesYAMLDocument` wrapper, which propagates strictness to the
   nested `SourceConfig` entries (`yaml.go:36-44`). A typo in either
   the top-level key (`toolsources:` instead of `tool_sources:`) or
   an entry field (`pulled_policy:` instead of `pull_policy:`)
   surfaces an error rather than silently producing zero values. The
   multi-document guard (`yaml.go:51-58`) catches accidental
   merge-conflict artefacts.

9. **Sentinel-error sprawl is cheap when each sentinel names a
   concrete failure surface; callers `errors.Is` against the union**
   — `errors.go` defines 16 sentinels covering source name / kind /
   policy / cron / URL allowance / data dir / unknown-source /
   manifest parse / manifest unknown field / manifest missing
   required / manifest source collision / signature verification /
   auth resolution / sync clone / sync pull / FS stat / FS mkdir.
   Each is a one-line `errors.New` with a doc-block explaining the
   condition that triggers it. `SyncOnce` wraps via
   `fmt.Errorf("%w: %w", ErrSyncClone, err)` so callers can
   `errors.Is(err, ErrSyncClone)` for kind-of-failure and
   `errors.Is(err, originalErr)` for the underlying cause. Mirrors
   the `core/pkg/eventbus` / `core/pkg/cron` sentinel discipline.

10. **Source-grep AC catches audit-emission drift early** — the test
    `TestSourceGrepAC_NoAuditCallsInProductionSources`
    (`manifest_test.go`) reads every production file in the package
    and grep-asserts that `keeperslog.` and `.Append(` do not appear
    outside comments. Audit emission for the M9 family lives in M9.7;
    the data-layer package must remain audit-free so the scheduler is
    safe to import from M9.1.b's runtime layer without dragging in
    audit dependencies. A `containsOutsideComments` helper
    (`testhelpers_test.go`) skips both line-comment lines and
    in-line `//` tails so doc-blocks discussing the audit boundary
    don't trip the AC.

11. **The Stat-not-exist sentinel chain (`fs.ErrNotExist`) is the
    clone-vs-pull discriminant** —
    `scheduler.go:347-358` calls `FS.Stat(<localPath>/.git)`; an
    `errors.Is(err, fs.ErrNotExist)` return triggers Clone, a nil
    error triggers Pull, and any other error aborts with `ErrFSStat`
    - `SourceFailed.Phase = "stat"`. The fake FS in `fakes_test.go`
      returns a `*fs.PathError{Err: fs.ErrNotExist}` for missing paths
      so the test matrix exercises the stdlib sentinel chain rather
      than a hand-rolled fake.

12. **Stop publish errors from masking sync success** —
    `emitSuccess` / `emitFailure` swallow `Publisher.Publish` errors
    after logging them (`scheduler.go:351-371`). The SyncOnce caller
    already has the original sync outcome in hand; a downstream
    eventbus stall must not turn a successful clone into a reported
    failure. The test
    `TestSyncOnce_PublisherErrorDoesNotMaskSyncSuccess` asserts the
    SyncOnce return is nil even when the publisher returns
    `errSentinel`.
