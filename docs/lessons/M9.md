## 2026-05-12 — M9.4.b: approval execution (git-pr webhook + slack-native AI-reviewer + 4-button approval card + callback dispatcher)

**PR**: pending — to be opened in Phase 5b
**Merged**: pending — to be merged in Phase 7a

### Review iter-1 fixes

Codex + critic (parallel) flagged 3 Major + 6 Minor; the operator
chose "All findings" and the union was applied on the same branch
before commit:

- **M1 — `Webhook.handleApproved` SyncOnce-before-ctx-gate race** —
  the prior flow ran `SourceResolver` → `SyncOnce` → ctx-check →
  Publish; a caller-side cancel arriving between `SyncOnce` and the
  ctx gate would land the source-resync side-effect WITHOUT
  emitting `TopicToolApproved`, producing an orphaned 408 response
  whose registry state diverged from the eventbus record. Same
  class of bug codex flagged on M9.4.a's `Proposer.Submit` (iter-1
  M1). Fix: pre-claim ctx-check fires BEFORE any side-effect; the
  full claim → SourceResolver → SyncOnce → Publish window now runs
  under `context.WithoutCancel(ctx)` so a mid-flight cancel can no
  longer split the pipeline (`core/pkg/approval/webhook.go:378-440`).
  Same pattern applied to `handleRejected`.

- **M2 — duplicate webhook delivery / double-click published as
  fresh approval events** — webhook delivery is retried by GitHub
  on transient 5xx + network glitches; slack-native button clicks
  can fire twice on platform-side BlockActions retry. The prior
  code re-emitted `TopicToolApproved` / `TopicToolRejected` on
  every replay, producing duplicate `tool_approved` events that
  M9.1.b's registry rebuilder would consume as independent
  reloads. Fix: new `DecisionRecorder` seam on
  `InMemoryProposalStore` with `MarkDecided(ctx, id, kind)` →
  `(firstTime bool, err error)` and `UnmarkDecided(ctx, id, kind)`;
  the full critical section runs under a single write-lock so a
  concurrent duplicate either observes the claim (silent 200) or
  wins it. Webhook + callback dispatchers claim → publish →
  unmark-on-publish-error so a retried delivery can re-publish
  (`core/pkg/approval/proposalstore.go:60-200`,
  `webhook.go:407-485`, `callbacks.go:227-289`). Pre-existing
  approve-then-reject scenario raises `ErrDecisionConflict` →
  HTTP 409 / callback error. `DecisionKind` is closed over
  `{approved, rejected}` only — `dry_run_requested` /
  `question_asked` are intermediate interactions that may
  legitimately repeat. A concurrency test
  (`TestInMemoryProposalStore_MarkDecided_ConcurrencyOneWinner`,
  16 goroutines) pins exactly-one-winner under the race detector.

- **M3 — `RenderApprovalCard` did not assert
  `Review.ProposalID == Input.ProposalID`** — an orchestration bug
  upstream of the renderer could ship a card carrying proposal A's
  identity (header + action_id + footer) plus proposal B's
  ReviewResult body (gates + risk badge), so the lead would
  approve the wrong tool. Fix: `validateCardInput` now fails
  closed on the mismatch with a new `ErrCardProposalMismatch`
  sentinel (`core/pkg/approval/card.go:285-307`). Test
  (`TestRenderApprovalCard_ReviewProposalIDMismatch_FailsClosed`)
  pins the boundary.

- **m1 — zero UUID `proposal_id` on the webhook payload was a
  404, not a 400** — `uuid.Parse("00000000-0000-0000-0000-000000000000")`
  succeeds, then `Lookup` returns `ErrProposalNotFound`. The nil
  UUID is reserved as a "no proposal" marker; rejecting at the
  webhook boundary surfaces a malformed-payload 400 (symmetric
  with `DecodeApprovalActionID`'s nil-uuid rejection on the
  slack-native callback path). Fix:
  `core/pkg/approval/webhook.go:332-344` + new
  `TestWebhook_ZeroUUIDProposalID_400`.

- **m2 — `json.Unmarshal` on the webhook body did not refuse
  unknown fields** — asymmetric with `DecodeManifest`'s
  `KnownFields(true)` discipline; a forward-compat extra field
  on the payload would silently flow through. Fix: switched to
  `json.NewDecoder` + `DisallowUnknownFields()`
  (`core/pkg/approval/webhook.go:319-330`) + new
  `TestWebhook_UnknownField_400`.

- **m3 — webhook PII canary test was trivially passing** — the
  prior test scanned `logger.snapshot()` for canary substrings,
  but `Webhook.logErr` discarded the `kv ...any` variadic
  altogether. The assertion ran against an empty kv set on
  every entry and would have passed regardless of whether the
  redaction worked. Fix (two halves): (a) `logErr` now plumbs
  `kv ...any` through `Logger.Log`, with the godoc explicitly
  forbidding `Purpose` / `PlainLanguageDescription` / `CodeDraft`
  in any kv pair (only opaque identifiers — proposal_id,
  request_id — are passed); (b) the canary test now adds a
  sanity guard (`sawKv` → fatal if no entry carries kv) so the
  assertion cannot trivially pass if logErr's plumbing
  regresses. Each failure-surface call site now plumbs
  `"proposal_id", proposal.ID` so the audit subscriber can join
  on the failure (`core/pkg/approval/webhook.go:296-396, 533-549`,
  `webhook_test.go:632-660`).

- **m4 — card PII canary test was a no-op** — the prior test
  declared `type reflectFieldNames = ApprovalCardInput` and
  initialised it zero-valued, asserting nothing. Fix: real
  `reflect.TypeOf(ApprovalCardInput{})` walk that fails when any
  field is named `Purpose` or `CodeDraft`, plus a second test
  that injects a canary into `PlainLanguageDescription` and pins
  (a) the canary surfaces inside the rendered card body
  (visibility AC) and (b) it does NOT leak into the opaque
  `action_id` payload (`core/pkg/approval/card_test.go:235-300`).

- **m5 — HMAC signing-base doc-block falsely claimed "mirrors
  EXACTLY" the slack/inbound algorithm** — slack/inbound uses
  `v0:<ts>:<body>` (version token baked into the signed bytes);
  watchkeeper's webhook uses `<ts>:<body>` (version label lives
  on the header name `X-Watchkeeper-Signature-256`, mirroring
  GitHub's `X-Hub-Signature-256` convention). The doc-block now
  documents the deliberate asymmetry and the GitHub-compatibility
  rationale (`core/pkg/approval/webhook.go:580-619`).

- **m6 — `DryRunRequester.RequestDryRun` ctx semantics
  un-documented** — the dispatcher passes
  `context.WithoutCancel(ctx)` so a caller-side cancel of the
  slack click handler does not abort the dry-run executor.
  Implementers reading the godoc would not know this and might
  rely on caller-cancel propagation. Fix: godoc now spells out
  the detached-ctx contract and requires implementers to apply
  their own bounded timeout (`core/pkg/approval/callbacks.go:36-66`).

### Iter-1 patterns worth carrying forward

1. **`claim → side-effect → rollback-on-error` is the cheapest
   webhook idempotency primitive that works under concurrency.**
   In-process `sync.RWMutex` + a `decisions[uuid]kind` map gives
   exactly-one-winner semantics without coordinating with the
   eventbus. `UnmarkDecided` on publish failure preserves
   retry-friendliness. The same pattern lands in any future
   "external system retries our http handler" boundary
   (gateway-side ingress, billing webhooks, OAuth callbacks).

2. **Per-button-branch `MarkDecided` placement matters more than
   per-button-branch `ctx.Err` placement.** M9.4.a iter-1 spent
   significant effort on the ctx.Err discipline; M9.4.b iter-1
   adds the decision-claim ordering as a peer concern. The
   pre-claim ctx.Err gate STILL refuses a phantom claim from an
   abandoned caller, but the claim itself runs on a detached ctx
   so a mid-flight cancel cannot strand it.

3. **A PII canary test with no actual canary in the kv path
   passes trivially.** The corrective discipline: every PII-canary
   test MUST include a "sanity gate" (e.g. assert at least one
   log entry carries non-empty kv) so a regression in the
   plumbing-under-test cannot make the assertion trivially pass.
   Add this pattern preemptively to any new redaction test.

4. **`reflect.TypeOf(<struct>{})` walks are the canonical "this
   PII-bearing field must never appear here" test, not type
   aliases.** The prior `type reflectFieldNames = X` pattern
   declares an alias and tests nothing; the real assertion is a
   field-by-field walk against a banned-name allowlist (already
   the M9.4.a discipline for event payloads — now extended to
   `ApprovalCardInput`).

### Context

M9.4.b decomposed M9.4.a's "authoring → approval → runtime → CI"
lifecycle by landing the **execution** surface. The single PR ships
five orthogonal but coherent slices in the existing
`core/pkg/approval/` package:

1. **`InMemoryProposalStore` + `ProposalLookup`/`ProposalRecorder`
   seam** (`proposalstore.go`) — the durable-record layer that
   M9.4.a deliberately deferred. Production wiring composes the
   recorder + lookup behind the same `*InMemoryProposalStore`;
   `sync.RWMutex` so concurrent webhook + dispatcher Lookups
   parallelise. Defensive deep-copy on both Store + Lookup
   (`cloneProposal` mirrors `cloneProposalInput`) prevents
   caller-side mutation in either direction.
2. **`*Webhook` http.Handler with HMAC-SHA256 signed-payload
   verification** (`webhook.go`) — mirrors `slack/inbound/verify.go`
   exactly: 5-minute timestamp replay window, `hmac.Equal`
   constant-time compare, `MaxBytesReader` body cap (1 MiB default).
   Per-call `WebhookSecretResolver` seam so per-tenant rotation
   takes effect on the next delivery without a process restart.
   Per-call `SourceForTarget` resolver maps `TargetSource → SourceConfig.Name`
   for the `SchedulerSyncer.SyncOnce` trigger. Sync failure does
   NOT abort publish (the event is the durable record); HTTP 202
   communicates "approval landed but sync needs operator attention".
3. **`*Reviewer` with 4 stub gates + heuristic risk score**
   (`reviewer.go`) — `GateTypecheck` (TS `export` + no `eval(`),
   `GateUndeclaredFSNet` (fs/network needles vs declared
   capabilities), `GateVitest` (`describe`/`test`/`it` blocks —
   missing is `SeverityWarn`), `GateCapabilityDeclaration`
   (`^[a-z][a-z0-9_:.-]*$` grammar). Risk: any `SeverityFail` →
   high; warn or > 5 capabilities → medium; otherwise → low. Real
   CI-grade implementations land in M9.4.d; the stub gates give
   the operator a deterministic preview at approval time.
4. **`RenderApprovalCard` Block-Kit renderer with
   `CapabilityTranslator` seam** (`card.go`) — pure function;
   header → metadata → fenced description → capability bullets →
   gate result lines → risk badge → 4-button actions → context
   footer. Description fenced in a code block so embedded mrkdwn
   (mentions, backticks) renders verbatim. `CapabilityTranslator`
   is the M9.3 dictionary seam with a documented "translation
   pending — M9.3" fallback for the (current) un-wired state.
   `EncodeApprovalActionID` / `DecodeApprovalActionID` use the
   `tool_approval:<proposal-uuid>` namespace — distinct from
   `messenger/slack/cards`'s spawn-saga namespace so the M6.3
   dispatcher cannot accidentally route an M9.4.b click into the
   spawn approval saga.
5. **`*CallbackDispatcher` for the 4 button branches**
   (`callbacks.go`) — `Approve`/`Reject` emit
   `TopicToolApproved`/`TopicToolRejected` with
   `ApprovalRouteSlackNative`; `Test in my DM` emits
   `TopicDryRunRequested` AND optionally invokes the
   `DryRunRequester` seam (M9.4.c-not-wired degradation path:
   nil seam → event still emits); `Ask questions` emits
   `TopicQuestionAsked`. `Click.Validate` enforces lead DM
   channel presence for `ButtonActionTestInDM` so an audit row
   records the gap rather than the executor silently dropping it.

The four new event types (`ToolApproved`, `ToolRejected`,
`DryRunRequested`, `QuestionAsked`) inherit M9.4.a's metadata-only

### Pattern

1. **Mirror the canonical HMAC verifier file, header-for-header.**
   `webhook.go` mirrors `core/pkg/messenger/slack/inbound/verify.go`:
   `v0=`-style version prefix (renamed `sha256=` to align with
   GitHub's `X-Hub-Signature-256` convention), 5-minute timestamp
   replay window, `crypto/hmac.Equal` constant-time compare,
   non-integer timestamp bucketed under "stale", exported `Sign`
   helper for test fixtures using the SAME code path the verifier
   consults. A regression-guard test (`TestWebhook_HMACEqualPinned`)
   greps the source for `hmac.Equal(` literal so a future refactor
   to `bytes.Equal` or `==` surfaces at test time. The discipline
   pays for itself: the verifier code reads like the spec, and a
   second-set-of-eyes review can confirm the algorithm by side-by-
   side comparison with the slack-inbound original.

2. **`SchedulerSyncer` 1-method interface beats consuming the
   concrete `*toolregistry.Scheduler`.** `webhook.go:117-128` declares
   `SchedulerSyncer interface { SyncOnce(ctx, sourceName) error }`
   and consumes the interface, not the concrete pointer. Production
   wiring satisfies it via `*toolregistry.Scheduler`; tests
   substitute a `fakeSchedulerSyncer` that drives the
   sync-error-still-publishes branch deterministically without
   spinning up a git server. Same Phase-1 LESSON as M9.4.a's
   `Publisher` / `IdentityResolver` seam discipline — the package
   boundary owns the interface; downstream impls satisfy it
   without coupling.

3. **Per-call resolver for webhook secret + source mapping.**
   `WebhookSecretResolver` and `SourceForTarget` are both function-
   shape per-call seams (`webhook.go:138-152` / `:154-163`), not
   process-global statics. A per-tenant secret rotation takes effect
   on the next delivery without a process restart; a per-deployment
   source-mapping reload does not couple webhook delivery to
   operator-config reload semantics. Mirrors M9.1.a's
   `AuthSecretResolver` and M9.4.a's `IdentityResolver` — codex
   iter-1 reliably catches a tenant-scoped value pinned as a
   static.

4. **Sync failure does NOT abort publish on the approval path.**
   `Webhook.handleApproved` (`webhook.go:265-321`) treats
   `SchedulerSyncer.SyncOnce` as a best-effort side effect: a
   non-nil sync error logs but does NOT short-circuit the
   `Publisher.Publish(TopicToolApproved)` call. The semantic
   contract is "the event is the durable record". HTTP 202
   Accepted communicates the partial-success state to the
   webhook poster — "approval landed but sync needs operator
   attention". The audit subscriber records the failure for
   triage. Same "publish-failure means event lost; sync-failure
   means downstream needs re-drive" discipline as M9.1.a's
   `SourceFailed` / `SourceSynced` split.

5. **Per-button publish helpers with `context.WithoutCancel` and
   pre-publish ctx-check.** Each callback branch in
   `callbacks.go:198-303` runs `ctx.Err()` BEFORE the publish AND
   uses `context.WithoutCancel(ctx)` for the publish itself.
   The pre-check refuses a phantom event on a cancelled ctx; the
   detached child ctx ensures the publish completes deterministically
   regardless of mid-flight caller cancel against the eventbus's
   queue-depth-dependent send fast-path. This is the same race
   the codex iter-1 M1 finding on M9.4.a caught — applied
   preemptively across all 4 callback branches AND across the
   webhook's `handleApproved` / `handleRejected` so a future
   reviewer cannot single out one branch as forgotten.

6. **`CapabilityTranslator` is an optional function-shape seam
   with a documented "M9.3 not yet wired" fallback.** The renderer
   accepts a nil translator and renders `<cap_id>
(translation pending — M9.3)` for every cap (`card.go:332-368`).
   The pattern lets the M9.4.b ship land BEFORE M9.3's dictionary
   loader without leaving an `// TODO M9.3` comment in production
   code; the seam is in place and the fallback string names the
   future milestone so a reader knows the gap is intentional.

7. **Per-package action_id namespace prevents cross-saga
   misrouting.** `card.go:122-130` declares
   `tool_approval:<proposal-uuid>` and an `EncodeApprovalActionID`
   /`DecodeApprovalActionID` codec — deliberately NOT reusing
   `messenger/slack/cards`'s `approval:<tool>:<token>` namespace
   (which is pinned to the M6.3 saga's
   `spawn.PendingApprovalTool*` constants). A unified Slack
   dispatcher branches on the action_id prefix; the per-package
   namespace ensures the M6.3 dispatcher cannot mis-decode an
   M9.4.b click into the spawn approval saga (and vice versa).
   Distinct sentinels `ErrInvalidActionID` in each package make
   `errors.Is` triage unambiguous.

## 2026-05-12 — M9.4.a: authoring surface (Manifest.DryRunMode + new core/pkg/approval/ package + approval_mode config)

**PR**: #136
**Merged**: 2ae4f62 on 2026-05-12

### Review iter-1 fixes

Codex + critic (parallel) flagged 4 Major + 7 Minor + 3 Nit; the
operator chose "All findings" and the union was applied on the same
branch before commit:

- **M1 — `Submit` ctx-cancel race with eventbus send fast-path** —
  the prior `Submit` checked `ctx.Err()` only before
  `IdentityResolver`, then passed the same `ctx` to
  `Publisher.Publish`. `core/pkg/eventbus/bus.go`'s send fast-path
  ignores a cancelled ctx when the topic buffer has room and only
  aborts on the backpressure path, so the same mid-flight cancel
  could either persist or drop `approval.tool_proposed` purely on
  queue depth — a bad fit for the "event is the durable record"
  contract on `ErrPublishToolProposed`. Fix: a SECOND `ctx.Err()`
  check fires after the proposal is built but BEFORE Publish, and
  the Publish call itself uses `context.WithoutCancel(ctx)`
  (`core/pkg/approval/proposer.go:201-268`). Two new tests
  (`TestSubmit_CtxCancelledMidFlight_RefusesPublish` and
  `TestSubmit_PublishCtxDetachedFromCallerCancel`) pin both halves
  of the contract — a mid-flight cancel before the pre-publish gate
  refuses the publish; a cancel during the publish completes
  deterministically without aborting.

- **M2 — `DecodeModeYAML` false-positive on trailing `---`
  separator** — the prior multi-doc guard treated any successful
  second `Decode` as a multi-doc input. yaml.v3 surfaces a trailing
  `---` separator with no following content as a successful Decode
  into a zero-valued document, so operator configs generated by
  templating tools (Helm, Ansible) that emit a trailing `---` for
  consistency were rejected with the misleading
  `ErrModeYAMLMultiDoc` sentinel. Fix: the guard now
  distinguishes "second doc is zero-valued AND decoder at EOF" from
  "second doc is populated", accepting the former as a single-doc
  input (`core/pkg/approval/approval_mode.go:131-160`).
  `TestDecodeModeYAML_AcceptsTrailingSeparator` pins the
  three operator-config shapes (`---\n`, bare `---`, `---\n   \n`).

- **M3 — `ProposalInput.Validate` did not detect duplicate
  capability ids** — an agent (or AI) producing
  `["github:read", "github:read", "jira:read"]` would land a
  proposal carrying duplicates that flowed downstream as
  duplicated approval-card translations, redundant M9.3 dictionary
  lookups, and inflated audit-log capability counts. Fix: a
  `seen map[string]struct{}` pass alongside the per-entry shape
  check + new `ErrDuplicateProposalCapability` sentinel
  (`core/pkg/approval/proposal.go:174-184`,
  `core/pkg/approval/errors.go:111-118`). Tests cover non-adjacent
  and adjacent duplicates.

- **M4 — `CorrelationID = fmt.Sprintf("%d", now.UnixNano())`
  collided under concurrent submits / fake-clock tests** — codex
  and critic BOTH flagged the same line independently. The doc
  described the field as a per-proposal join key, but UnixNano
  granularity (and the test's fake clock returning the same
  instant for every goroutine) produced collisions that would
  silently collapse unrelated proposals in any downstream subscriber
  that groups by `CorrelationID`. Fix: derive
  `CorrelationID` from `ProposalID.String()` (UUIDv7 → time-ordered
  AND unique by construction), so the field is now both a join key
  AND a uniqueness key (`core/pkg/approval/proposer.go:222-232`).
  `TestSubmit_CorrelationIDEqualsProposalIDString` +
  `TestSubmit_ConcurrencyUniqueCorrelationID` (16×4 goroutines
  under a fixed fake clock) pin the contract.

- **m1 — `IdentityResolver` godoc misled implementers to pre-wrap
  with `ErrIdentityResolution`** — codex flagged that `Submit`
  already wraps any non-nil return with `ErrIdentityResolution`,
  so a resolver following the godoc would produce double-wrap
  chains. Fix: rewrote the godoc to state implementers return any
  error and `Submit` wraps (`core/pkg/approval/proposer.go:86-112`).

- **m2 — `ProposerID` had no length / shape check** — a buggy
  `IdentityResolver` returning a bearer-token-shaped string would
  silently flow onto the event payload AND the publish-failure
  log entry, defeating the otherwise-defensive bounded-input
  story. Fix: new `MaxProposerIDLength = 128` constant

### Pattern

1. **Decompose multi-surface roadmap leaves BEFORE coding.** M9.4
   bundled six orthogonal surfaces in one line — the same
   discipline that decomposed M9.1 (commit `8bf1018`) and M8.2
   landed a `docs(roadmap):` commit splitting M9.4 into four
   lifecycle-stage sub-items (`docs/ROADMAP-phase1.md:505-509`)
   BEFORE any code landed. The decomposition commit lives on
   `main` (no PR — the gated work is the per-sub-item PRs) and
   includes a body paragraph naming the seam choice
   (lifecycle-stage: authoring → approval → runtime → CI). The
   one-PR alternative was rejected because the surface spans four
   packages with different reviewers and verification gates.

2. **Required-enum field with empty-string explicit rejection +
   loud-failure unknown-value.** `Manifest.DryRunMode` is required
   (`Manifest.Validate` checks empty after the other required
   fields), and `DryRunMode.Validate` rejects empty AND unknown
   values via `ErrInvalidDryRunMode`
   (`core/pkg/toolregistry/manifest.go:32-58` for the typed-string
   enum + Validate, `:115-125` for the field, `:148-164` for the
   validation block, `core/pkg/toolregistry/errors.go:201-214` for
   the sentinel). Two distinct sentinels
   (`ErrManifestMissingRequired` for "author forgot the field";
   `ErrInvalidDryRunMode` for "author typoed the value") let
   callers triage authoring mistakes vs schema drift. Same
   pattern as M9.1.a's `SourceKind` / `PullPolicy` (Pattern 7).
   The empty string is explicitly refused by the enum validator
   standalone so a future caller that bypasses
   `Manifest.Validate` (e.g. M9.4.c's runtime executor handed a
   deserialised manifest from a non-strict source) still gets
   the loud failure.

3. **Closed-set enum WITHOUT `local` is the roadmap-text
   contract enforcement.** `TargetSource` has only `platform` and
   `private` constants
   (`core/pkg/approval/target_source.go:27-39`). The roadmap text
   says "local source never offered to the agent"; the enum
   itself enforces the contract — the validator could never
   accept `local` because no constant exists, and
   `TestTargetSource_RejectsLocal` /
   `TestProposalInput_ValidateRejectsLocalTargetExplicitly` pin
   the literal-string rejection so a future enum addition
   (`hosted`, etc.) does not accidentally re-introduce the
   contract gap. Lead-override at approval time (M9.4.b) uses a
   distinct lead-only enum that does not flow through this
   validator.

4. **Cross-field validation lives in a separate function that
   takes BOTH inputs explicitly.** `ValidateModeForSources`
   takes `(Mode, []toolregistry.SourceConfig)`
   (`core/pkg/approval/approval_mode.go:84-103`). The
   single-field `Mode.Validate` / `TargetSource.Validate`
   stays small and composable; cross-field rules live one level
   up so a caller wiring the operator config can choose to run
   the single-field validators first (for early error surfaces)
   and the cross-field validator second. The boundary is
   defensible: a single-field validator that ALSO consumed
   sources would cyclically import the registry into config
   primitives. Same lesson as `toolregistry.SourceConfig.Validate`
   vs `ValidateSources` (M9.1.a).

5. **Bounded-length defensive validation at the agent-facing
   boundary.** Every string field on `ProposalInput` has a max
   byte length enforced by `Validate` and exposed as a public
   constant (`MaxToolNameLength = 64`,
   `MaxCodeDraftLength = 65536`, etc.,
   `core/pkg/approval/proposal.go:13-45`). Bounds exist
   BEFORE the M9.4.b downstream consumers (AI reviewer, Slack
   card renderer) that would otherwise exhibit unbounded
   behaviour under adversarial AI-authored input. The 64 KiB
   `CodeDraft` cap fits any realistic TS source while refusing
   pathological payloads; same defensive-shape discipline as
   M9.1.a's source-name regex allowlist (path-traversal
   defence). The constants are exported so callers can surface
   "max length" hints in their UI without duplicating the
   numbers.

6. **Fail-fast validation BEFORE any side effects: validate →
   ctx → identity → clock → id → ctx-recheck → publish (with
   detached ctx).** `Proposer.Submit` runs the resolution order
   documented at the top of `proposer.go`
   (`core/pkg/approval/proposer.go:1-31`). Each side-effecting
   call is GATED by the prior validation result: `Validate`
   runs first (no identity-resolver call wasted on bad input);
   ctx.Err runs before the resolver (a pre-cancelled ctx aborts
   cheap); identity resolver runs before `Clock.Now` +
   `IDGenerator.NewUUID` so a tenant-rotation failure does not
   waste a process-monotonic correlation id. A SECOND ctx.Err
   check fires AFTER the proposal is built but BEFORE Publish,
   so a caller that cancels mid-flight does not produce a
   phantom proposal record (iter-1 codex M1 fix). The actual
   Publish call uses `context.WithoutCancel(ctx)` so once the
   pre-publish gate passes, the durable record completes
   deterministically — the `eventbus.Bus` send fast-path's
   queue-depth-dependent cancel behaviour cannot race the
   "event is the durable record" contract.
   `TestSubmit_CtxCancelledMidFlight_RefusesPublish` and
   `TestSubmit_PublishCtxDetachedFromCallerCancel` pin both
   halves. Same lesson as M7.1.c.c's "fail-fast precedes audit
   Append" (the M7.1.c.c iter-1 fix that surfaced from codex).

7. \*\*Metadata-only event payload + reflection allowlist test
   - PII canary harness on dynamic emission.\*\* `ToolProposed`
     carries 7 fields (`ProposalID`, `ToolName`, `ProposerID`,
     `TargetSource`, `CapabilityIDs`, `ProposedAt`,
     `CorrelationID`) and DELIBERATELY excludes `CodeDraft`,
     `Purpose`, `PlainLanguageDescription`
     (`core/pkg/approval/events.go:50-84`). The
     `TestToolProposed_FieldAllowlist` reflection test pins the
     payload shape to the documented 7 fields by name + type;
     adding a field requires the author bump the allowlist AND
     document why the new field's PII shape is OK
     (`core/pkg/approval/events_test.go`). The
     `TestSubmit_EventPayloadOmitsBodies` canary test stuffs
     synthetic ASCII substrings into each PII-bearing
     `ProposalInput` field and asserts the verbatim `%+v` dump
     of the event payload never contains them;
     `TestSubmit_EventPayloadOmitsBodiesWithNULByteCanary`
     covers the NUL-byte variant so a future logger that drops
     NULs cannot silently invalidate the substring match.
     Same dual-AC shape as M9.1.b / M9.2: static field allowlist
   - dynamic canary emission. Capability ids land on the
     payload as a defensive deep copy (`newToolProposedEvent`,
     `core/pkg/approval/events.go:93-105`) because they are
     public dictionary entries (M9.3) AND the downstream M9.4.b
     approval card needs them to render human-readable
     translations.

8. **Per-call resolver function shape, not interface, for
   tenant-scoped identity.** `IdentityResolver` is
   `func(ctx context.Context) (string, error)`
   (`core/pkg/approval/proposer.go:86-112`), invoked on every
   `Submit`. Per-tenant rotation takes effect on the next
   `Submit` without restarting the process. The function shape
   (vs an interface) matches M9.1.a's `AuthSecretResolver`
   discipline for the same reason: the surface is one-method
   and stays simpler as a `func` type alias. `("", nil)` is
   refused as `ErrEmptyResolvedIdentity` — a non-nil-but-empty
   return is a fail-loud programmer error, same as
   `toolregistry.ErrEmptyResolvedAuth`. A resolver returning
   a non-empty identifier exceeding `MaxProposerIDLength`
   surfaces `ErrInvalidProposerID` so a buggy resolver cannot
   silently leak a bearer-token-shaped value onto the event
   payload (iter-1 codex finding m2 fix).

9. **`gosec` G101 nolint annotation on synthetic redaction-
   harness canary constants.** The `canaryCodeDraft` /
   `canaryPurpose` / `canaryPlainLanguageDescription`
   constants in `proposer_test.go` carry
   `//nolint:gosec // G101: synthetic redaction-harness
canaries, not real credentials.` annotations because the
   strings contain `_DO_NOT_LEAK` and gosec's secret-detection
   heuristic would otherwise flag them. Same annotation
   pattern as the M7.1.c.c iter-1 fix.

### Pattern

1. **Pure-transformation seam returns side-data alongside primary
   result; orchestrator consumes both** — `BuildEffective` is
   pure: takes a context + sources + clock + logger, returns a
   snapshot + shadow list + error
   (`core/pkg/toolregistry/scanner.go:170-225`). It does NOT
   publish events itself even though the shadow list will become
   events; the registry's `Recompute` is the orchestrator that
   reads the side data and emits. Keeping `BuildEffective`
   side-effect-free lets future callers (M9.4 dry-run, M9.5 local
   patch verification) consume shadow information without
   dragging the eventbus into the call graph. Mirrors the
   M9.1.a `Scheduler.SyncOnce` shape — the I/O work is in one
   layer, the event-emission is in another.

2. **`winnerIdx map[string]int` over `winner *EffectiveTool`** —
   the shadow-detection loop tracks winners via
   `winnerIdx map[string]int` keyed by `Manifest.Name`, with the
   value indexing into the growing `merged` slice
   (`scanner.go:202-220`). A pointer-into-slice would be UNSAFE
   because `append` may reallocate the underlying array — a
   pointer captured pre-reallocation would point at orphaned
   memory. The index variant is reallocation-safe AND constant-
   time. Reading the winner's source/version uses `merged[idx]`
   right when the shadow entry is being constructed. Same lesson
   as `core/pkg/spawn/saga`'s step-index tracking pattern.

3. **Event emission order matters for cross-topic causality** —
   `Recompute` emits all `tool_shadowed` events FIRST, then
   `effective_toolset_updated` LAST (`registry.go:316-355`). A
   subscriber observing both topics on the same eventbus thus
   learns about every shadow BEFORE seeing the revision bump.
   Subscribers can rely on the invariant "by the time I see the
   updated event for revision N, I have already seen every
   shadow at revision N" to atomically decide "this revision's
   shadows are X, Y, Z" without polling. The
   `eventbus.Bus`'s per-topic worker is sequential but
   cross-topic ordering depends on emission order at the
   publisher — Recompute owns the order.

4. **Partial-delivery semantics: each event is independent;
   first error wins; remaining errors logged only** — Recompute's
   shadow loop attempts every publish even when an earlier one
   failed (`registry.go:323-345`). The FIRST failure is wrapped
   onto the return via `ErrPublishToolShadowed`; subsequent
   failures land in the optional `Logger` only. The contract:
   subscribers that observe K of N notifications are strictly
   better off than zero, so first-error-aborts is the wrong
   default. Asymmetry vs the toolset-updated publish (which has
   no peers — losing it is a clean miss) — there the single
   error becomes the return outright. Same lesson lives in
   `core/pkg/outbox`'s batch-dispatch pattern.

5. **Multi-error precedence: the "registry-bump invisible" signal
   beats the "one DM missed" signal** — when BOTH the shadow
   publish AND the toolset-updated publish fail, `Recompute`
   returns `ErrPublishAfterSwap` (toolset-updated failure),
   NOT `ErrPublishToolShadowed` (`registry.go:357-365`). The
   reasoning: subscribers that only observe
   `effective_toolset_updated` would otherwise miss the
   revision bump entirely (zero notifications for the snapshot
   change), while a missed `tool_shadowed` only loses one DM
   (the snapshot is still visible via `Snapshot()` /
   `Acquire()`). The more severe observability gap takes the
   error chain. Tested by
   `TestRecompute_EffectiveUpdatedFailureOverridesShadowFailure`.

6. **DM wording lives on the payload type, not the subscriber** —
   `ToolShadowed.Message()` (`events.go:217-225`) returns the
   exact roadmap-text format using `fmt.Sprintf`. A future
   re-wording (e.g. localisation, A/B test, or appending an
   issue-tracker link) lands in this one method body — every
   Slack-DM subscriber, every audit-log entry, every operator
   alert reads from the same source of truth. Same lesson as
   `saga.SpawnContext` carrying canonical IDs vs every caller
   re-deriving them.

7. **Field-allowlist reflection test catches PII drift at
   compile-and-test time** — `TestToolShadowed_FieldAllowlist`
   (`m92_test.go`) pins the payload to 8 fields by name + type;
   adding a field (`Capabilities []string`, `Schema
json.RawMessage`) fails the test until the author bumps the
   `want` map AND consciously documents why the new field's PII
   shape is OK. Mirrors
   `TestEffectiveToolsetUpdated_FieldAllowlist` from M9.1.b's
   iter-1 t5. Combined with the canary-substring test, the AC
   surface is: "the payload's fields are X and Y" (static) AND
   "a runtime emission of the payload does not contain the
   canary" (dynamic).

8. **Intra-source duplicates do NOT surface as shadows** —
   `TestBuildEffective_IntraSourceDuplicateNotSurfacedAsShadow`
   pins the contract: a same-name conflict within ONE source
   lands as an `ErrIntraSourceDuplicateManifestName` log entry
   via `ScanSourceDir` (M9.1.b's M3 iter-1 fix) BEFORE the
   precedence-flattening loop sees it. The shadow list is
   reserved for CROSS-source conflicts because the lead-facing
   DM ("source A's tool conflicts with source B's") only makes
   sense across sources — within a single source, the conflict
   is an authoring bug the upstream owners resolve, not a
   cross-team coordination signal. Two failure surfaces, two
   diagnostic shapes.

9. **Slack DM wiring lives OUTSIDE `toolregistry`** —
   the package stays audit/slack-free (same discipline as
   M9.1.a/b). A future subscriber wraps `slack.OpenIM` +
   `slack.SendMessage` (`core/pkg/messenger/slack/`) against
   `TopicToolShadowed` and renders the body via
   `ToolShadowed.Message()`. Keeping the wiring at the
   subscriber layer means `toolregistry` is safe to depend on
   from any layer (no transitive slack import), AND the
   subscriber can be replaced (e.g. by a Discord or email
   subscriber) without touching the registry. Same
   import-cycle-break precedent as M3.2.b's `LocalPublisher`.

### Pattern

1. **Atomic `Pointer[entry]` + per-entry refcount is the natural
   shape for "in-flight calls complete on the old version while
   new invocations resolve against the new toolset"** — the
   `Registry` holds `current atomic.Pointer[registryEntry]`
   (`registry.go:163-167`); each `registryEntry` bundles a snapshot
   - an `atomic.Int32` refcount + a `retiredAt time.Time` (`registry.go:140-152`).
     `Acquire()` atomically loads the entry, increments its refcount,
     and returns a snapshot + idempotent release closure
     (`registry.go:233-245`). `Recompute()` swaps the pointer in one
     atomic op (`registry.go:286-289`); the old entry's snapshot stays
     reachable on the in-flight caller's stack (Go GC keeps it alive)
     AND the registry tracks its refcount in a retiring list so
     operator dashboards / future M9.3 cleanup can observe "how many
     in-flight calls are still on revision N." No mutex on Acquire
     means the hot path is a single atomic load + a single atomic
     increment.

2. **The grace period is purely a diagnostic-tracking knob — Go
   GC reclaims the snapshot, NOT the registry** — the
   `RegistryDeps.GracePeriod` (`registry.go:73-84`) is the duration
   the registry keeps a retiring entry in its
   `retiring []*registryEntry` slice (`registry.go:178-181`). At each
   `Recompute()` the registry sweeps entries whose
   `retiredAt + GracePeriod` is on or before `now`
   (`registry.go:382-403`), logs an `effective_toolset retired`
   entry, and drops its reference. The on-disk snapshot itself
   stays alive as long as any in-flight caller holds the pointer
   — the grace period doesn't free memory, it only frees the
   registry's bookkeeping slot. This decoupling matters because
   M9.3 will plug heavy resources (loaded code modules, mmaped
   signed tarballs) onto the retiring entry and need a
   "registry-bounded" cleanup signal that doesn't fight Go GC.

3. **The hot-reload event carries metadata only — never manifest
   bodies** — `EffectiveToolsetUpdated` fields are `Revision`,
   `BuiltAt`, `ToolCount`, `SourceCount`, `CorrelationID`
   (`events.go:111-141`). Tool names, capability ids, schema bytes,
   and `Source` fields are deliberately excluded. A subscriber that
   wants the manifest contents calls `Registry.Snapshot()` or
   `Registry.Acquire()`; the bus channel never carries the payload.
   The `TestPIIRedactionCanary_EffectiveToolsetUpdatedPayload` test
   asserts the canary substrings never appear in
   `fmt.Sprintf("%+v", event)`. Same redaction discipline as
   M9.1.a's `SourceFailed`-carries-only-error-type — this time the
   "secret to protect" is the manifest body itself (a customer
   private tool that ends up logged verbatim could reveal proprietary
   logic).

4. **Per-tool failure isolation in the scanner: log + skip never
   poisons the whole source; per-source failure isolation in the
   builder: log + zero contributions never poisons the whole
   recompute** — `ScanSourceDir` runs `LoadManifestFromFile` per
   per-tool subdirectory under a try-catch loop, logs decode
   failures via the optional `Logger`, and continues
   (`scanner.go:60-83`). `BuildEffective` runs `ScanSourceDir` per
   configured source under the same loop, logs per-source
   `ReadDir` failures, and continues with zero contributions from
   that source (`scanner.go:139-167`). A single corrupt manifest or
   a single I/O blip cannot brick the runtime's effective toolset
   — the operator still gets a coherent snapshot from every
   source's good tools. The error-isolation boundaries match
   `core/pkg/cron`'s per-job dispatch isolation (one panicking
   handler does not stop the scheduler) and M9.1.a's per-source
   `SyncOnce` mutex isolation (one bad source's network stall
   does not block siblings).

5. **`fs.ErrNotExist` is the "no work to do" sentinel; anything
   else surfaces wrapped** — `ScanSourceDir` treats
   `errors.Is(err, fs.ErrNotExist)` on `FS.ReadDir` as legitimate
   "source not yet synced — skip" and returns `(nil, nil)`
   (`scanner.go:51-56`). Any other `ReadDir` error wraps a new
   `ErrScanReadDir` sentinel (`errors.go:151-157`). Symmetric to
   the M9.1.a `Scheduler.SyncOnce` `Stat` chain that uses
   `fs.ErrNotExist` as the clone-vs-pull discriminant. The
   stdlib sentinel chain is preferred to a hand-rolled "not exist"
   field because the fake FS already returns
   `*fs.PathError{Err: fs.ErrNotExist}` and the production `OSFS`
   shim forwards `os.ReadDir`'s native chain — no wrapper code,
   no aliasing bugs.

6. **Publisher-error-on-success: ASYMMETRY mirroring M9.1.a's
   M5 contract — success-path publish errors are RETURNED, the
   atomic swap stays committed** — `Recompute` (`registry.go:297-309`)
   atomically swaps the new entry into current BEFORE publishing
   `TopicEffectiveToolsetUpdated`. If the publish fails, the
   registry still holds the new snapshot as current — the next
   `Snapshot()` / `Acquire()` returns it — but the caller observes
   the publisher error so an external retry / monitoring path can
   fire. The test
   `TestRecompute_PublisherErrorSurfaces` asserts both invariants
   (return error + state moved forward). Asymmetric to M9.1.a's
   `emitFailure` (publish errors suppressed because the original
   sync error is already authoritative) — here, the original
   "sync" is the registry's pointer swap, which has no error of
   its own.

7. **Compile-time interface assertion against the concrete
   `*eventbus.Bus`** — `registry_test.go` has
   `var _ Subscriber = (*eventbus.Bus)(nil)`. The `toolregistry.Handler`
   type alias is exactly `func(ctx context.Context, event any)` — the
   same shape as `eventbus.Handler` — so structural satisfaction
   holds. A future `Bus.Subscribe` signature change breaks the
   compile of THIS line at the same time as the production wiring
   would break. Mirrors the M9.1.a `Publisher` discipline + the
   cron / outbox pattern (`cron_test.go` / `consumer_test.go`).

8. **Subscriber seam, not a `*eventbus.Bus` import in production
   code** — `Registry.Start(ctx, sub Subscriber)` takes a
   `Subscriber` interface (`registry.go:28-39`) defined in this
   package. The production wiring will pass a `*eventbus.Bus` (the
   compile-time assertion in test pins the shape); the test path
   uses the real bus (no fake subscriber stub needed) which
   exercises the actual async dispatch + buffered-channel
   semantics. `TestStart_SubscribesAndDispatchesRecompute` uses
   `bus.Publish(SourceSynced{...})` then polls the registry's
   captured-publish list for `TopicEffectiveToolsetUpdated`. Same
   real-bus discipline as `core/pkg/cron`'s integration tests.

9. **`FS.ReadDir` re-introduced with a clarified contract** —
   M9.1.a's m8 iter-1 fix removed `ReadDir` from the FS interface
   because the scheduler did not need it; M9.1.b restores it with
   an explicit godoc contract on the `not-exist sentinel chain`
   (`scheduler.go:35-45`). `OSFS.ReadDir` forwards to
   `os.ReadDir` with the same `//nolint:gosec` annotation as the
   sibling methods (the path is operator-derived, not user
   input — `osfs.go:34-39`). The `fakeFS` adds a
   `dirEntries map[string][]fs.DirEntry` keyed by parent path and
   a `fakeDirEntry` struct implementing `fs.DirEntry`
   (`fakes_test.go`). Adding a method to a closed-set seam is
   mechanical because every layer of the stack uses interfaces.

10. **Initial empty snapshot at revision 0 lets `Acquire` return
    non-nil even before the first `Recompute`** — `NewRegistry`
    stores a synthetic empty `EffectiveToolset` at revision 0 with
    `BuiltAt = Clock.Now()` (`registry.go:208-211`). Callers that
    `Acquire` before any sync event has fired get an empty
    snapshot (zero tools) instead of a nil pointer. The first
    real `Recompute` advances the revision to 1; the synthetic
    revision-0 enters retiring like any other. This eliminates
    a nil-check in every downstream caller (M9.2 resolver, M9.3
    capability enforcer) at the cost of one no-op snapshot per
    registry — a trade-off that mirrors the
    M9.1.a `NoopSignatureVerifier{}` default-zero-value-usable
    discipline.

11. **`recomputeMu` serialises Recompute even though eventbus
    workers are per-topic sequential** — `Registry.recomputeMu`
    (`registry.go:173-181`) is taken at the top of `Recompute`.
    The eventbus's per-topic worker already serialises
    `TopicSourceSynced` handlers, but EXTERNAL callers (tests,
    operator CLI in future milestones) may call `Recompute`
    directly while the subscriber path is firing. Without the
    mutex, two `Recompute` calls would race on the revision
    counter, the swap, and the retiring-list append. The mutex
    is cheap (one atomic CAS per Recompute under no contention)
    and keeps the surface composable.

## 2026-05-11 — M9.1.a: tool_sources config + manifest.json schema + ToolSyncScheduler (new core/pkg/toolregistry)

**PR**: #133
**Merged**: d033536 on 2026-05-11

### Context

M9.1.a opens the M9 milestone family. The roadmap text on M9.1 is too
large to ship atomically (operator config schema + manifest decoding +
clone/pull I/O + scheduler + effective-toolset recompute + runtime
hot-reload signal with grace period — six orthogonal surfaces), so the
operator chose a 2-way decomposition along the data/runtime seam
(commit `8bf1018`):

- **M9.1.a** — the data + sync layer: operator `tool_sources` config,
  per-tool `manifest.json` schema, and `ToolSyncScheduler` that
  clones / pulls each source per its pull policy and emits
  `toolregistry.source_synced` / `toolregistry.source_failed` events.
- **M9.1.b** — the runtime layer: effective-toolset recompute on each
  `source_synced` event + a hot-reload signal to running runtimes with
  configurable grace period (in-flight `InvokeTool` calls complete on
  the old version while new invocations resolve against the new
  toolset).

The new package is `core/pkg/toolregistry/`. It is intentionally
self-contained — no imports from `core/pkg/config`, no audit emission,
no runtime coupling. Downstream subscribers (M9.1.b, M9.2 resolver,
M9.7 audit) join via `eventbus`.

Files:

- `doc.go`, `errors.go` — package documentation + sentinel errors.
- `manifest.go` — `Manifest` struct + `DecodeManifest` (strict JSON
  decode, `DisallowUnknownFields`, required-field validation,
  defensive deep-copy of reference-typed fields) +
  `LoadManifestFromFile` (file read + decode + source-stamping with
  collision detection).
- `config.go` — `SourceConfig` + closed-set `SourceKind`
  (`git` / `local` / `hosted`) and `PullPolicy`
  (`on-boot` / `cron` / `on-demand`) + per-source `Validate()` +
  `ValidateSources()` (covers duplicate names) + `EffectiveBranch()`
  defaulting to `main` + `CloneSources()` defensive copy.
- `yaml.go` — `DecodeSourcesYAML` (strict yaml.v3 `KnownFields(true)`,
  multi-document rejection) + `LoadSourcesYAMLFromFile`.
- `events.go` — `TopicSourceSynced` / `TopicSourceFailed` constants +
  `SourceSynced` (sourceName, syncedAt, localPath, correlationID) and
  `SourceFailed` (sourceName, failedAt, errorType, phase,
  correlationID) payloads.
- `scheduler.go` — `Scheduler` orchestrator with five seams (`FS`,
  `GitClient`, `Clock`, `SignatureVerifier`, `Publisher`) +
  `AuthSecretResolver` per-call resolver + optional `Logger`. `New`
  panics on nil required deps; `SyncOnce(ctx, sourceName)` performs
  one clone-or-pull cycle and emits the appropriate event.
- `osfs.go` — production `OSFS` adapter forwarding to `os.*`.

### Pattern

1. **Atomize a multi-surface roadmap leaf along its natural seam BEFORE
   coding** — M9.1 in the original ROADMAP text bundles six orthogonal
   surfaces. Mirror the M8.2 / M7.3 decomposition pattern: a separate
   `docs(roadmap):` commit on `main` that splits the leaf into named
   sub-items (`M9.1.a` / `M9.1.b`), then ship one sub-item per
   `ship-roadmap-item` invocation. The decomposition commit
   (`docs/ROADMAP-phase1.md:500-502`) is doc-only, lands directly on
   `main` (no PR — the gated work is the per-sub-item PRs), and
   includes a body paragraph explaining the seam choice + why the
   single-PR alternative was rejected. The atomic-ship rule has a
   user-override channel via `AskUserQuestion` when an axis is
   genuinely indivisible.

2. **Seam-driven design: every side-effecting primitive is an
   interface, every default is no-op** — the `Scheduler` consumes five
   interfaces (`FS`, `GitClient`, `Clock`, `SignatureVerifier`,
   `Publisher`) plus a callable `AuthSecretResolver`
   (`core/pkg/toolregistry/scheduler.go:21-118`). Each interface is
   defined in this package (not imported from elsewhere), uses the
   minimal subset of methods the scheduler actually calls, and has a
   hand-rolled fake in `fakes_test.go`. The default
   `SignatureVerifier` is the no-op `NoopSignatureVerifier{}` (zero
   value usable, returns nil for every input) — this lets M9.1.a land
   the verification HOOK without committing to a specific signing
   scheme (cosign / minisign is M9.3's call). The production wiring
   for `FS` is a 4-method shim (`osfs.go`) — adding a sixth seam later
   is mechanical.

3. **Per-call resolver beats process-global static for tenant-scoped
   values** — `AuthSecretResolver` is `func(ctx, sourceName, ref)
(string, error)`, invoked on every `SyncOnce`
   (`scheduler.go` resolveAuth). The test
   `TestSyncOnce_AuthResolverPerCall` asserts the call count matches
   the number of SyncOnce invocations (not cached). Same lesson as
   M8.2.a's `jira.BasicAuthSource` interface (consumed via
   `jira.WithBasicAuth`): a per-tenant rotation takes effect on the
   next sync without restarting the process, and the scheduler never
   holds the plaintext secret across calls. The toolregistry side
   chose a function shape (vs an interface) because the resolver
   surface is one-method and stays simpler as a `func` type alias.

4. **PII discipline at the event-payload boundary, NOT at the logger**
   — `SourceFailed` carries `ErrorType` (the Go type name via
   `fmt.Sprintf("%T", err)`) and `Phase` but NEVER the underlying
   error MESSAGE
   (`core/pkg/toolregistry/events.go:46-76`,
   `scheduler.go:374-395`). A failing git URL with embedded
   credentials would surface its message via `errors.Unwrap` if the
   payload carried `.Error()`; trimming the payload at the type-name
   boundary is what makes a subscriber that logs the event verbatim
   safe. The PII canary harness
   (`scheduler_test.go:TestPIIRedactionCanary_AcrossEveryFailurePath`)
   injects a synthetic `CANARY_PII_AUTH_TOKEN_…` into every failure
   phase (`mkdir` / `stat` / `clone` / `pull` / `signature` / `auth`)
   and asserts neither the event nor any logged kv pair contains the
   substring.

5. **Defensive deep-copy of reference-typed inputs in BOTH constructor
   AND on dispatch** — `New` calls `CloneSources(sources)`
   (`config.go:148-156`) so caller-side mutation post-construction
   cannot bleed into the scheduler's state; `DecodeManifest` copies
   `Capabilities` (string slice) and `Schema` (byte slice) via the
   local `cloneStringSlice` / `cloneBytes` helpers
   (`manifest.go:104-118`) so the caller's `raw` buffer is no longer
   aliased. Mirrors the M7.1.c.c `cloneBotProfile` pattern. The test
   `TestDecodeManifest_DefensiveCopy` mutates the input buffer
   post-decode and asserts the returned manifest is unaffected.

6. **Per-source mutex serialises concurrent SyncOnce on the SAME
   source; different sources run in parallel** — `Scheduler.perSourceMu`
   is a `map[string]*sync.Mutex` populated at construction
   (`scheduler.go:213-219`). `SyncOnce` takes the per-source mutex
   before touching the FS / git
   (`scheduler.go:308-310`). The concurrency tests
   (`TestSyncOnce_ConcurrencyDifferentSources`,
   `TestSyncOnce_ConcurrencySameSourceSerialised`) dispatch 16
   goroutines: the first asserts all 16 calls complete in parallel
   (all 16 sources clone exactly once); the second asserts the
   max-inflight count for the same source never exceeds 1 even
   under a 1ms artificial hold. The map is read-only after `New`, so
   no top-level lock is needed.

7. **Closed-set discriminators with named constants + per-discriminator
   cross-field validation** — `SourceKind` and `PullPolicy` are
   typed string aliases with named constants
   (`config.go:13-58`). `SourceConfig.Validate` switches on the kind
   and policy enums, requiring `URL` for `git` (and forbidding it for
   `local` / `hosted`) and `CronSpec` for `cron` (and ignoring it for
   `on-boot` / `on-demand`)
   (`config.go:96-130`). A `default:` branch on each switch surfaces
   `ErrInvalidSourceKind` / `ErrInvalidPullPolicy` rather than a
   silent acceptance — same loud-failure discipline as
   `runtime.AuthorityFromString`.

8. **Strict YAML decoding at the document root AND inside each entry**
   — `DecodeSourcesYAML` uses yaml.v3's `KnownFields(true)` on the
   `sourcesYAMLDocument` wrapper, which propagates strictness to the
   nested `SourceConfig` entries (`yaml.go:36-44`). A typo in either
   the top-level key (`toolsources:` instead of `tool_sources:`) or
   an entry field (`pulled_policy:` instead of `pull_policy:`)
   surfaces an error rather than silently producing zero values. The
   multi-document guard (`yaml.go:51-58`) catches accidental
   merge-conflict artefacts.

9. **Sentinel-error sprawl is cheap when each sentinel names a
   concrete failure surface; callers `errors.Is` against the union**
   — `errors.go` defines 16 sentinels covering source name / kind /
   policy / cron / URL allowance / data dir / unknown-source /
   manifest parse / manifest unknown field / manifest missing
   required / manifest source collision / signature verification /
   auth resolution / sync clone / sync pull / FS stat / FS mkdir.
   Each is a one-line `errors.New` with a doc-block explaining the
   condition that triggers it. `SyncOnce` wraps via
   `fmt.Errorf("%w: %w", ErrSyncClone, err)` so callers can
   `errors.Is(err, ErrSyncClone)` for kind-of-failure and
   `errors.Is(err, originalErr)` for the underlying cause. Mirrors
   the `core/pkg/eventbus` / `core/pkg/cron` sentinel discipline.

10. **Source-grep AC catches audit-emission drift early** — the test
    `TestSourceGrepAC_NoAuditCallsInProductionSources`
    (`manifest_test.go`) reads every production file in the package
    and grep-asserts that `keeperslog.` and `.Append(` do not appear
    outside comments. Audit emission for the M9 family lives in M9.7;
    the data-layer package must remain audit-free so the scheduler is
    safe to import from M9.1.b's runtime layer without dragging in
    audit dependencies. A `containsOutsideComments` helper
    (`testhelpers_test.go`) skips both line-comment lines and
    in-line `//` tails so doc-blocks discussing the audit boundary
    don't trip the AC.

11. **The Stat-not-exist sentinel chain (`fs.ErrNotExist`) is the
    clone-vs-pull discriminant** —
    `scheduler.go:347-358` calls `FS.Stat(<localPath>/.git)`; an
    `errors.Is(err, fs.ErrNotExist)` return triggers Clone, a nil
    error triggers Pull, and any other error aborts with `ErrFSStat`
    - `SourceFailed.Phase = "stat"`. The fake FS in `fakes_test.go`
      returns a `*fs.PathError{Err: fs.ErrNotExist}` for missing paths
      so the test matrix exercises the stdlib sentinel chain rather
      than a hand-rolled fake.

12. **Stop publish errors from masking sync success** —
    `emitSuccess` / `emitFailure` swallow `Publisher.Publish` errors
    after logging them (`scheduler.go:351-371`). The SyncOnce caller
    already has the original sync outcome in hand; a downstream
    eventbus stall must not turn a successful clone into a reported
    failure. The test
    `TestSyncOnce_PublisherErrorDoesNotMaskSyncSuccess` asserts the
    SyncOnce return is nil even when the publisher returns
    `errSentinel`.
